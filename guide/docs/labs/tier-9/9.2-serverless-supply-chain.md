# Lab 9.2: Serverless Supply Chain

<div class="lab-meta">
  <span>~40 minutes</span>
  <span>Advanced</span>
  <span>Prerequisites: <a href="../tier-1/1.2-dependency-confusion.md">Lab 1.2</a></span>

Serverless functions are the smallest unit of cloud compute. A Lambda function, a Cloud Function, an Azure Function -- each is a single entry point with bundled dependencies that runs on infrastructure you never see. The promise is simplicity: write code, deploy, done.

The supply chain attack surface tells a different story. Your function depends on: its own code, its bundled dependencies (npm/pip packages), shared Lambda Layers that inject code before your handler runs, the runtime itself, and the IAM role that determines what the function can do. Compromise any link, and every invocation of your function is compromised.

In this lab, you will see two attacks: a malicious Lambda Layer that silently intercepts every function invocation to exfiltrate event data, and a dependency confusion attack in a serverless deployment pipeline.

!!! info "Phase 1: UNDERSTAND -- The Serverless Attack Surface"

    **Goal:** Map the attack surface of a serverless function: code, dependencies, layers, runtime, and IAM.

### Step 1: Examine the legitimate Lambda function

```bash
cat src/handler.py
```

This is an order processing function. It receives events containing order IDs, customer data, API keys, and payment tokens. It queries DynamoDB and sends notifications via SNS. Standard business logic.

### Step 2: Understand Lambda Layers

Lambda Layers are ZIP archives extracted to `/opt/` in the execution environment. They load BEFORE your function code. The layer path `/opt/python/` is on the Python path, so any Python file placed there is importable by your function.

Key Layer facts:

| Property | Implication |
|----------|-------------|
| Layers load before handler code | Attacker code runs first |
| `/opt/python/sitecustomize.py` auto-loads | Python imports it on interpreter startup |
| Layers are referenced by ARN | Without version pin, the latest version is used |
| Layer publishers can update layers | A compromised publisher can push a malicious version |
| Layer contents are not inspected at deploy time | No scanning, no signature verification |

### Step 3: Understand the deployment pipeline

```bash
cat src/template.yaml
```

The SAM template defines two function configurations: `VulnerableOrderProcessor` and `HardenedOrderProcessor`. Compare:

| Property | Vulnerable | Hardened |
|----------|-----------|---------|
| Layer reference | By name (no version pin) | By ARN with version number |
| IAM permissions | `dynamodb:*`, `s3:*`, `sns:*` | Only specific actions on specific resources |
| Network access | Default (internet access) | VPC-isolated (private subnet) |
| Secrets | Hardcoded in environment variables | Loaded from Parameter Store at runtime |
| Concurrency | Unlimited | Reserved (capped at 50) |

### Step 4: Examine the dependency configuration

```bash
cat src/requirements.txt
```

The requirements file references two internal packages: `acme-order-utils` and `acme-notification-client`. These exist on the company's private PyPI registry. But the build step (`sam build`) runs `pip install` with the default index -- public PyPI is queried first.

!!! warning "Phase 2: BREAK -- Malicious Layer and Dependency Confusion"

    **Goal:** Execute both attacks and understand their impact.

### Attack 1: Malicious Lambda Layer

```bash
cat src/malicious-layer/python/sitecustomize.py
```

This file is placed in a Lambda Layer at `/opt/python/sitecustomize.py`. Python's runtime automatically imports `sitecustomize` on startup -- no configuration needed, no opt-in required.

**How the interception works:**

1. Lambda runtime starts and Python initializes
2. Python imports `sitecustomize.py` from the layer (attacker code)
3. `sitecustomize.py` monkey-patches Python's `__import__` function
4. When the runtime imports the handler module, the patched `__import__` wraps the handler function
5. Every invocation now passes through the wrapper, which exfiltrates the event data to the attacker's C2 endpoint
6. The original handler runs normally and returns the expected response
7. The function appears to work perfectly -- no errors, no timeouts, correct responses

**What gets exfiltrated:**

```python
# The wrapper sends every event to the attacker:
{
    "type": "event",
    "function": "order-processor",
    "region": "us-east-1",
    "data": {
        "event": {
            "order_id": "ORD-12345",
            "api_key": "sk-live-xxxxxxxxxxxxx",    # API keys in events
            "customer": {
                "name": "John Doe",                # Customer PII
                "email": "john@example.com",
                "address": "123 Main St"
            }
        }
    }
}
```

On cold start, the layer also steals environment variables containing AWS credentials, API keys, and secrets.

### Attack 2: Dependency confusion in the build pipeline

The `requirements.txt` references internal packages without pinning the index URL:

```
acme-order-utils>=1.0.0
acme-notification-client>=2.0.0
```

During `sam build`, pip resolves packages from public PyPI first (default behavior). An attacker who publishes `acme-order-utils` version `99.0.0` on public PyPI wins the resolution:

```
$ pip install acme-order-utils>=1.0.0
Collecting acme-order-utils>=1.0.0
  Found versions: 1.2.3 (private), 99.0.0 (public PyPI)
  Using version 99.0.0 (highest available)
  Downloading acme-order-utils-99.0.0.tar.gz
  Running setup.py install for acme-order-utils
  # setup.py exfiltrates AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY from build environment
```

The malicious package is bundled INTO the Lambda deployment package. Every invocation of the function runs the attacker's code.

### Understanding the combined impact

If both attacks succeed:

1. **Layer attack:** Every invocation's event data (PII, tokens, API keys) is exfiltrated
2. **Dependency confusion:** Build-time credentials stolen, malicious code in the deployment package
3. **Overprivileged IAM:** The function has `s3:*` and `dynamodb:*`, so the attacker can access any S3 bucket and any DynamoDB table in the account
4. **No VPC isolation:** The function can reach the internet, so exfiltration is trivial

!!! success "Phase 3: DEFEND -- Hardening Serverless Functions"

    **Goal:** Pin layer versions, scope IAM, isolate networks, and fix the dependency pipeline.

### Fix 1: Pin Lambda Layer versions by ARN

```yaml
# VULNERABLE: Layer can be updated by anyone with publish permissions
Layers:
  - arn:aws:lambda:us-east-1:999888777666:layer:shared-utils

# HARDENED: Pinned to specific immutable version
Layers:
  - arn:aws:lambda:us-east-1:123456789012:layer:shared-utils:7
```

The version-pinned ARN is immutable. Even if the layer publisher pushes a malicious version 8, your function still uses version 7.

### Fix 2: Apply least-privilege IAM

Create the minimum IAM policy for the function:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["dynamodb:GetItem", "dynamodb:Query"],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/orders"
    },
    {
      "Effect": "Allow",
      "Action": ["sns:Publish"],
      "Resource": "arn:aws:sns:us-east-1:123456789012:order-notifications"
    },
    {
      "Effect": "Allow",
      "Action": ["logs:CreateLogGroup", "logs:CreateLogStream", "logs:PutLogEvents"],
      "Resource": "arn:aws:logs:us-east-1:123456789012:*"
    }
  ]
}
```

Compare this to the vulnerable policy's `dynamodb:*`, `s3:*`, `sns:*` on `Resource: "*"`. The hardened policy reduces the blast radius from "entire account" to "one DynamoDB table and one SNS topic."

### Fix 3: Isolate functions in a VPC

```yaml
VpcConfig:
  SecurityGroupIds:
    - sg-0123456789abcdef0    # Allow only DynamoDB and SNS endpoints
  SubnetIds:
    - subnet-private-1a       # No internet gateway
    - subnet-private-1b
```

VPC-isolated functions cannot reach the public internet. Exfiltration via HTTP/HTTPS fails. The function accesses DynamoDB and SNS via VPC endpoints (private connections that never leave the AWS network).

### Fix 4: Fix the dependency pipeline

```bash
# requirements.txt -- HARDENED
--index-url https://internal.pypi.corp.com/simple/
--no-deps

acme-order-utils==1.2.3
acme-notification-client==2.1.0
boto3==1.28.85
```

Three fixes:

1. `--index-url` points to the private registry exclusively (public PyPI is never queried)
2. `--no-deps` prevents transitive dependency resolution surprises
3. Exact version pins (`==`) instead of minimum version (`>=`)

### Fix 5: Build layers internally

Instead of using third-party layers, build and publish your own:

```bash
# Create layer from verified dependencies
mkdir -p layer/python
pip install -r requirements-layer.txt -t layer/python/
cd layer
zip -r ../shared-utils-layer.zip .

# Publish as internal layer
aws lambda publish-layer-version \
    --layer-name shared-utils \
    --zip-file fileb://shared-utils-layer.zip \
    --compatible-runtimes python3.12 \
    --description "Internal shared utilities v7 - built from verified deps"
```

!!! danger "Phase 4: DETECT -- Monitoring Serverless Functions"

    **Goal:** Detect compromised functions using CloudWatch, network monitoring, and layer version auditing.

### CloudWatch Indicators

A compromised function behaves differently from a clean one. The interception layer adds latency, makes network calls, and changes resource consumption patterns.

| Metric | Baseline | Compromised | Detection |
|--------|----------|-------------|-----------|
| Duration | 200ms P99 | 350ms P99 | +75% spike from exfiltration HTTP call |
| Network bytes out | 2 KB/invocation | 5 KB/invocation | Doubled from event data exfiltration |
| Error rate | 0.1% | 0.5% | Exfiltration timeouts cause occasional errors |
| Cold start time | 800ms | 1200ms | sitecustomize.py adds import overhead |
| Concurrent executions | Stable | Stable | No change (hard to detect from this metric) |

**CloudWatch Alarm for duration anomaly:**

```json
{
  "AlarmName": "OrderProcessor-DurationAnomaly",
  "MetricName": "Duration",
  "Namespace": "AWS/Lambda",
  "Dimensions": [{"Name": "FunctionName", "Value": "order-processor"}],
  "Statistic": "p99",
  "Period": 300,
  "EvaluationPeriods": 3,
  "Threshold": 300,
  "ComparisonOperator": "GreaterThanThreshold",
  "TreatMissingData": "notBreaching"
}
```

### Lambda Layer Version Monitoring

Track layer version changes across all functions:

**Splunk -- detect layer version changes:**

```spl
index=cloudtrail eventName="UpdateFunctionConfiguration"
| spath output=layers path=requestParameters.layers{}
| mvexpand layers
| rex field=layers "arn:aws:lambda:.*:layer:(?<layer_name>[^:]+):(?<layer_version>\d+)"
| stats latest(layer_version) as current_version, earliest(layer_version) as previous_version by layer_name
| where current_version != previous_version
| eval severity="high"
| eval mitre="T1195.002"
| table _time, layer_name, previous_version, current_version, severity
```

**KQL -- detect overprivileged function roles:**

```kql
AWSCloudTrail
| where EventName in ("CreateFunction20150331", "UpdateFunctionConfiguration20150331v2")
| extend RoleArn = tostring(parse_json(RequestParameters).role)
| where isnotempty(RoleArn)
| join kind=inner (
    AWSCloudTrail
    | where EventName == "GetRolePolicy"
    | extend PolicyDoc = tostring(parse_json(ResponseElements).policyDocument)
    | where PolicyDoc has "\"Action\": \"*\"" or PolicyDoc has "\"Resource\": \"*\""
    | extend OverprivilegedRole = tostring(parse_json(RequestParameters).roleName)
) on $left.RoleArn contains $right.OverprivilegedRole
| project TimeGenerated, FunctionName=tostring(parse_json(RequestParameters).functionName), RoleArn, OverprivilegedRole
```

!!! tip "SOC Relevance"

    **Alerts you will see:**

    - "Lambda function duration P99 exceeded threshold" (CloudWatch)
    - "Lambda layer version changed outside change window" (CloudTrail)
    - "Outbound HTTP POST from Lambda NAT gateway to unknown host" (VPC Flow Logs)
    - "Function role has overprivileged IAM policy" (IAM Access Analyzer)

    **Why this matters to your SOC:** Serverless functions process high-value data -- API requests, payment transactions, user authentication. A compromised function sees every piece of data that passes through it. Unlike a compromised server (which you can SSH into and investigate), serverless functions are ephemeral -- the execution environment is destroyed after each invocation, leaving no persistent artifacts to forensicate.

    **Triage workflow:**

    1. **Check the layer versions** -- compare current layer ARNs against the last known-good deployment. If a layer version changed, pull and inspect its contents.
    2. **Check CloudWatch metrics** -- compare duration, error rate, and network bytes against the baseline for this function.
    3. **Check the deployment pipeline** -- was the last deployment triggered by a known CI/CD job, or by an unknown principal?
    4. **Inspect the function package** -- download the deployment package and check for unexpected files or modified dependencies.
    5. **Check IAM activity** -- did the function's execution role perform any actions outside its normal pattern?

    **False positive rate:** Medium. Duration spikes can be caused by legitimate traffic surges, cold starts, or downstream latency. Layer version changes are normal during deployments but should be correlated with approved change requests.

!!! example "CI Integration"

    Validate serverless deployments before they reach production.

    **`.github/workflows/serverless-audit.yml`:**

    ```yaml
    name: Serverless Security Audit

    on:
      pull_request:
        paths:
          - "serverless/**"
          - "template.yaml"
          - "requirements*.txt"

    jobs:
      audit-serverless:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Check layer version pinning
            run: |
              echo "--- Checking Lambda layer version pins ---"
              UNPINNED=$(grep -rn 'arn:aws:lambda:' template.yaml \
                | grep 'layer:' \
                | grep -v ':[0-9]\+$' || true)
              if [ -n "$UNPINNED" ]; then
                echo "::error::Found unpinned Lambda layer references:"
                echo "$UNPINNED"
                echo "Pin layers by adding the version number: :7"
                exit 1
              fi
              echo "PASS: All layers are version-pinned."

          - name: Check IAM policy scope
            run: |
              echo "--- Checking for overprivileged IAM policies ---"
              WILDCARDS=$(grep -rn '"Resource": "\*"' template.yaml || true)
              STAR_ACTIONS=$(grep -rn '"Action": "\*"' template.yaml || true)
              BROAD_ACTIONS=$(grep -rn '":.*\*"' template.yaml | grep -E '(dynamodb|s3|sns|sqs|lambda):\*' || true)
              if [ -n "$WILDCARDS" ] || [ -n "$STAR_ACTIONS" ] || [ -n "$BROAD_ACTIONS" ]; then
                echo "::error::Found overprivileged IAM policies:"
                echo "$WILDCARDS"
                echo "$STAR_ACTIONS"
                echo "$BROAD_ACTIONS"
                exit 1
              fi
              echo "PASS: No wildcard resources or actions found."

          - name: Check for hardcoded secrets
            run: |
              echo "--- Checking for hardcoded secrets in environment ---"
              SECRETS=$(grep -rn -iE '(api_key|secret|password|token).*:.*[a-zA-Z0-9]{20,}' template.yaml \
                | grep -v '#' | grep -v '_comment' || true)
              if [ -n "$SECRETS" ]; then
                echo "::error::Found potential hardcoded secrets in environment variables:"
                echo "$SECRETS"
                exit 1
              fi
              echo "PASS: No hardcoded secrets detected."

          - name: Check dependency index pinning
            run: |
              echo "--- Checking requirements.txt for index pinning ---"
              for f in $(find . -name "requirements*.txt" -type f); do
                if ! grep -q '\-\-index-url' "$f"; then
                  echo "::error file=$f::Missing --index-url pin. Add: --index-url https://your-private-registry/simple/"
                  exit 1
                fi
              done
              echo "PASS: All requirements files have index URL pinned."
    ```

## What You Learned

1. **Lambda Layers are a pre-execution attack surface** -- code in layers runs before your handler, and `sitecustomize.py` is auto-loaded by Python without any opt-in.
2. **Event data is high-value** -- serverless functions process API keys, customer PII, and payment tokens in every invocation. A compromised function exfiltrates all of it.
3. **Dependency confusion works in serverless pipelines** -- `sam build` and similar tools run `pip install` during packaging, and default to public registries.
4. **Least-privilege IAM is critical** -- an overprivileged function gives attackers access to the entire account, not just the function's intended resources.
5. **VPC isolation prevents exfiltration** -- without internet access, stolen data cannot leave your network.
