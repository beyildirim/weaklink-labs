In 2020, security researchers found that over 70% of the top 1000 GitHub repositories had CI/CD configurations that leaked secrets to pull request builds. CI/CD pipelines are the backbone of modern software delivery -- but they are also one of the most under-secured parts of the software supply chain.

A CI/CD pipeline automates building, testing, and deploying code. It runs when code is pushed, when a pull request is opened, or on a schedule. Every pipeline has **triggers** (what starts it), **jobs** (units of work), **steps** (individual commands), **runners** (machines that execute jobs), and **artifacts** (files produced by builds).

The problem: pipelines often run with elevated permissions. They have access to deployment tokens, cloud credentials, and API keys. If an attacker can influence what a pipeline executes, they can steal those secrets.

In this lab, you will explore a real CI/CD pipeline, see how secrets are exposed, and lock them down.

!!! info "Phase 1: UNDERSTAND -- What is a CI/CD Pipeline?"

    **Goal:** Explore a Gitea repository with a CI pipeline. Understand triggers, jobs, steps, and secrets.

### Step 1: Clone the repository

```bash
cd /repos/acme-webapp
ls -la
```

This is the ACME Corp webapp -- a simple Python application with automated CI/CD.

### Step 2: Explore the application

```bash
cat app.py
cat requirements.txt
cat test_app.py
```

A straightforward web application with dependencies and tests.

### Step 3: Examine the CI configuration

```bash
cat .gitea/workflows/ci.yml
```

This is the pipeline definition. Notice:

- **Triggers**: `on: push` and `on: pull_request` -- the pipeline runs on every code change
- **Jobs**: `test`, `build`, `deploy` -- three stages that run in sequence
- **Steps**: Individual commands within each job
- **Secrets**: `SECRET_TOKEN`, `DEPLOY_TOKEN`, `AWS_ACCESS_KEY_ID` -- injected as environment variables

### Step 4: Identify the problem

Look at the `env:` block at the top of the file:

```yaml
env:
  SECRET_TOKEN: ${{ secrets.SECRET_TOKEN }}
  DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
  AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
```

These secrets are available to **every job and every step** -- including tests. The test job does not need deployment credentials, but it has access to them.

### Step 5: See what a pipeline run looks like

```bash
# Simulate what CI executes for the test job
echo "=== CI Test Job ==="
echo "SECRET_TOKEN=${SECRET_TOKEN:-[set by CI]}"
echo "DEPLOY_TOKEN=${DEPLOY_TOKEN:-[set by CI]}"
python test_app.py
```

In a real CI run, those environment variables contain actual credentials.

!!! warning "Phase 2: BREAK -- Exfiltrating Secrets via CI"

    **Goal:** Show that any developer with PR access can read CI secrets by modifying the pipeline.

### Step 1: Create an attack branch

```bash
cd /repos/acme-webapp
git checkout -b feature/add-logging
```

### Step 2: Modify the CI config to leak secrets

Add a step that prints environment variables:

```bash
cat > .gitea/workflows/ci.yml << 'EOF'
name: ACME Webapp CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  SECRET_TOKEN: ${{ secrets.SECRET_TOKEN }}
  DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
  AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: |
          echo "Running tests..."
          # "Debug logging" that leaks every secret
          env | sort
          python test_app.py
EOF
```

### Step 3: Commit and push

```bash
git add -A
git commit -m "Add debug logging to CI"
git push origin feature/add-logging
```

### Step 4: Create a pull request

```bash
curl -sf -X POST "http://gitea:3000/api/v1/repos/developer/acme-webapp/pulls" \
  -H "Content-Type: application/json" \
  -u "developer:password" \
  -d '{"title":"Add logging","base":"main","head":"feature/add-logging"}'
```

When the CI pipeline runs this PR, it executes `env | sort` which prints ALL environment variables -- including `SECRET_TOKEN`, `DEPLOY_TOKEN`, and `AWS_ACCESS_KEY_ID`. Anyone who can view the build logs can see the secrets.

### Step 5: Understand the blast radius

This is not theoretical. In a real organization:

- Any developer with write access can create a PR
- The PR modifies the CI config to print secrets
- The pipeline runs the modified config
- Secrets appear in build logs visible to the PR author
- The attacker now has deploy tokens, cloud credentials, API keys

This is the fundamental CI/CD security problem: **pipelines execute developer-controlled code with production-level credentials**.

!!! success "Phase 3: DEFEND -- Least-Privilege CI Secrets"

    **Goal:** Scope secrets to only the jobs and steps that need them.

### Fix 1: Remove global secrets

```bash
cd /repos/acme-webapp
git checkout main
```

Edit `.gitea/workflows/ci.yml` to remove the global `env:` block and scope secrets:

```bash
cp /lab/src/repo/.gitea/workflows/ci-secure.yml .gitea/workflows/ci.yml
cat .gitea/workflows/ci.yml
```

Key changes:

1. **No global `env:` block** -- secrets are not available to all jobs
2. **Secrets only on the deploy step** -- `DEPLOY_TOKEN` is scoped to the one step that needs it
3. **Environment protection** -- the `deploy` job requires the `production` environment, which can have approval rules
4. **Test and build jobs have zero secrets** -- even if compromised, there is nothing to steal

### Fix 2: Commit the defense

```bash
git add -A
git commit -m "Scope CI secrets to deploy job only"
git push origin main
```

### Additional defenses (discussion)

Beyond scoping secrets:

1. **Use short-lived tokens**: OIDC tokens that expire after the pipeline run, not long-lived API keys
2. **Environment protection rules**: Require manual approval before a deploy job can access production secrets
3. **Audit secret access**: Log which jobs accessed which secrets and alert on unusual patterns
4. **Separate PR and push workflows**: PR builds should never have access to any secrets

!!! danger "Phase 4: DETECT -- Catching Secret Exposure in CI"

    **Goal:** Detect when CI secrets are accessed by unauthorized jobs or leaked in build outputs.

### SIEM / Log Indicators

CI secret exposure follows a pattern: **a job that historically never accessed secrets suddenly does**, or **build logs contain values that match known secret patterns**.

**What to look for:**

- Build logs containing patterns matching API keys (e.g., `AKIA`, `ghp_`, `sk-`)
- CI config changes that add `env | sort`, `printenv`, or `set` commands
- Jobs accessing secrets they have not historically used
- New `env:` blocks referencing `secrets.*` in test or build jobs
- PR builds that reference secrets (any PR accessing secrets is suspicious)

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| Outbound HTTP POST from CI runner to external URL with Base64 body | Secret exfiltration via HTTP |
| DNS queries with long encoded subdomains from CI infrastructure | DNS tunneling of stolen credentials |
| CI runner connecting to unfamiliar external IPs during test jobs | Test job has no legitimate reason for external connections |

### Example Detection Queries

**Splunk -- detect secret patterns in build logs:**

```spl
index=cicd sourcetype=build_logs
| regex _raw="(AKIA[A-Z0-9]{16}|ghp_[a-zA-Z0-9]{36}|sk-[a-zA-Z0-9]{32})"
| table _time, job_name, repository, triggered_by, _raw
| sort -_time
```

**Splunk -- detect CI config changes adding env/printenv commands:**

```spl
index=git sourcetype=gitea
  action="push" file_path="*.yml" file_path="*workflows*"
| regex diff="(\benv\b|\bprintenv\b|\bset\b)" 
| table _time, repository, author, commit_message, diff
```

**KQL (Microsoft Sentinel) -- detect unauthorized secret access in CI:**

```kql
CIBuildEvents
| where JobName !in ("deploy", "release")
| where SecretsAccessed != ""
| project Timestamp, Repository, JobName, SecretsAccessed, TriggeredBy, BranchName
| sort by Timestamp desc
```

!!! tip "SOC Relevance"

    **Alerts you will see:**

    - "Secret pattern detected in build logs" (log analysis)
    - "Non-deploy job accessed production secrets" (CI audit)
    - "CI config modified to expose environment variables" (git monitoring)

    **Why this matters to your SOC:** CI/CD systems are the most privileged part of most organizations' infrastructure. They have deployment credentials, cloud access keys, signing certificates, and database passwords. A single compromised CI pipeline can give an attacker access to production systems without ever touching a production server.

    **Triage workflow:**

    1. **Check which secrets were exposed** -- review the build logs for any secret patterns
    2. **Check who triggered the build** -- was it a PR from an external contributor or a known developer?
    3. **Check the CI config diff** -- did the pipeline config change? Was the change reviewed?
    4. **Rotate exposed secrets immediately** -- assume any secret visible in logs is compromised
    5. **Audit downstream access** -- check if the exposed credentials were used to access production systems

    **False positive rate:** Medium. Developers sometimes accidentally echo environment variables during debugging. The key signal is whether the build was triggered by a PR (high risk) vs. a push to main (lower risk, but still bad).

!!! example "CI Integration"

    Add this to your GitHub Actions pipeline to prevent global secret exposure.

    **`.github/workflows/secret-scope-check.yml`:**

    ```yaml
    name: CI Secret Scope Audit

    on:
      pull_request:
        paths:
          - ".github/workflows/**"
          - ".gitea/workflows/**"

    jobs:
      audit-secrets:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Check for global secret exposure
            run: |
              echo "--- Auditing CI configs for overly-broad secret scoping ---"
              ISSUES=0

              for f in .github/workflows/*.yml .gitea/workflows/*.yml; do
                [ -f "$f" ] || continue
                echo "Checking: $f"

                # Check for secrets in top-level env block
                if awk '/^env:/,/^[a-z]/' "$f" | grep -q 'secrets\.'; then
                  echo "::error file=$f::CRITICAL: Secrets in global env block. Scope to specific steps."
                  ISSUES=$((ISSUES + 1))
                fi

                # Check for env/printenv/set in run blocks
                if grep -E '^\s+run:.*\b(env|printenv|set)\b' "$f"; then
                  echo "::warning file=$f::Potential secret leak: env/printenv/set in run block."
                  ISSUES=$((ISSUES + 1))
                fi
              done

              if [ "$ISSUES" -gt 0 ]; then
                echo "Found $ISSUES issue(s). Fix before merging."
                exit 1
              fi
              echo "PASS: No secret scope issues found."
    ```

## What You Learned

1. **CI/CD pipelines are high-value targets** -- they have production credentials, cloud access, and deployment permissions.
2. **Global secret scoping is the default mistake** -- most CI configs inject secrets at the top level, making them available to every job.
3. **Any developer with PR access can steal secrets** -- by modifying the CI config in a pull request.
4. **Least privilege applies to CI too** -- scope secrets to only the jobs and steps that need them.
5. **Environment protection rules add a gate** -- requiring approval before accessing production secrets prevents drive-by exfiltration.
6. **Detection relies on audit logs** -- monitor which jobs access which secrets and alert on anomalies.
