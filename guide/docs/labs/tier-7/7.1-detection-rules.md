You completed Tier 1. You know how dependency confusion, typosquatting, lockfile injection, manifest confusion, and phantom dependency attacks work. Now put yourself in the SOC analyst's chair: you did not launch the attack -- you need to detect it after the fact, from log telemetry, before the attacker finishes exfiltrating.

This lab teaches you to write production-quality detection rules for every Tier 1 attack type using Splunk SPL, KQL (Microsoft Sentinel), and Suricata. You will validate each rule against pre-generated log samples and tune them to minimize false positives.

!!! info "Phase 1: UNDERSTAND -- The Detection Surface"

    **Goal:** Identify the log sources that matter for supply chain attack detection and understand what each source can and cannot see.

### Step 1: Map the telemetry sources

Every supply chain attack leaves traces in multiple log sources. The challenge is knowing where to look.

| Log Source | What It Captures | Attack Types Detected |
|------------|-----------------|----------------------|
| **CI/CD logs** (GitHub Actions, GitLab CI) | Build commands, dependency installs, script execution | Lockfile injection, manifest confusion, phantom deps |
| **Package manager logs** (pip, npm verbose output) | Package resolution, version selection, registry source | Dependency confusion, typosquatting |
| **Proxy / DNS logs** (Squid, Zscaler, corporate DNS) | Outbound connections from build servers | Exfiltration from malicious setup.py / install scripts |
| **Container registry audit logs** | Image pulls, tag resolution, digest mismatches | Tag mutability, base image poisoning |
| **EDR process telemetry** (CrowdStrike, Defender for Endpoint) | Process trees, file writes, network connections per process | All attack types (post-execution indicators) |
| **Git audit logs** | Commit signatures, PR approval state, force pushes | Lockfile injection, manifest confusion |

### Step 2: Understand the detection timeline

Supply chain attacks have distinct phases with different detection windows:

```
Timeline of a dependency confusion attack:

  t=0     Developer pushes requirements.txt with loose pin
  t=1     CI pipeline triggers pip install
  t=2     pip queries public PyPI (DETECTABLE: proxy logs)
  t=3     pip downloads malicious package (DETECTABLE: proxy logs, network)
  t=4     setup.py executes (DETECTABLE: EDR process tree)
  t=5     Exfiltration begins (DETECTABLE: firewall, DNS, proxy)
  t=6     Build completes, artifact published
  t=7     Artifact deployed to production (DETECTABLE: runtime monitoring)

  Detection windows:
  [t=2 to t=5] = minutes     ← SOC can catch this in near real-time
  [t=5 to t=7] = minutes-hours ← containment still possible
  [t=7+]       = too late     ← attacker has persistent access
```

The earlier you detect, the smaller the blast radius. Your rules must fire between t=2 and t=5.

### Step 3: Review the MITRE ATT&CK techniques

Every rule you write should map to a MITRE technique. These are the relevant ones:

| Technique | ID | Attack Types |
|-----------|-----|-------------|
| Supply Chain Compromise: Software Dependencies | [T1195.001](https://attack.mitre.org/techniques/T1195/001/) | Typosquatting, phantom dependencies |
| Supply Chain Compromise: Software Supply Chain | [T1195.002](https://attack.mitre.org/techniques/T1195/002/) | Dependency confusion, lockfile injection, manifest confusion |
| Command and Scripting Interpreter: Python | [T1059.006](https://attack.mitre.org/techniques/T1059/006/) | Malicious setup.py execution |
| Automated Exfiltration | [T1020](https://attack.mitre.org/techniques/T1020/) | Data theft during pip install |
| Application Layer Protocol: Web Protocols | [T1071.001](https://attack.mitre.org/techniques/T1071/001/) | C2 / exfiltration over HTTP/HTTPS |

### Rule 1: Dependency confusion -- Internal package fetched from public PyPI

**Splunk SPL:**

```spl
index=proxy sourcetype=squid
  dest_host="pypi.org" uri_path="/simple/*"
  src_ip IN ("10.100.0.0/24", "10.100.1.0/24")
| rex field=uri_path "/simple/(?<package_name>[^/]+)/"
| lookup internal_package_names package_name OUTPUT is_internal
| where is_internal="true"
| eval severity="critical"
| eval mitre="T1195.002"
| table _time, src_ip, package_name, severity, mitre
```

**KQL (Microsoft Sentinel):**

```kql
let InternalPrefixes = dynamic(["acme-", "internal-", "corp-", "mycompany-"]);
CommonSecurityLog
| where DeviceAction == "Allowed"
| where DestinationHostName == "pypi.org"
| extend PackageName = extract(@"/simple/([^/]+)/", 1, RequestURL)
| where isnotempty(PackageName)
| where InternalPrefixes has_any (PackageName)
| extend AlertName = "Internal package name fetched from public PyPI"
| extend MitreTechnique = "T1195.002"
| project TimeGenerated, SourceIP, PackageName, RequestURL, AlertName, MitreTechnique
```

**Suricata:**

```yaml
alert http $BUILD_SERVERS any -> $EXTERNAL_NET any (
  msg:"WEAKLINK Dependency Confusion - Internal package name on public PyPI";
  flow:established,to_server;
  http.host; content:"pypi.org";
  http.uri; content:"/simple/acme-"; nocase;
  classtype:trojan-activity;
  sid:7100001; rev:1;
  metadata:mitre_technique T1195.002;
)
```

### Rule 2: Typosquatting -- Known misspelling installed

**Splunk SPL:**

```spl
index=cicd sourcetype=pip_install
| rex field=_raw "Collecting (?<installed_pkg>\S+)"
| lookup typosquat_watchlist package_name AS installed_pkg OUTPUT legitimate_name, is_typosquat
| where is_typosquat="true"
| eval severity="high"
| eval mitre="T1195.001"
| table _time, host, installed_pkg, legitimate_name, severity, mitre
```

**KQL:**

```kql
let TyposquatPairs = datatable(Typo:string, Legitimate:string)[
    "reqeusts", "requests",
    "python-dateuti", "python-dateutil",
    "beautifulsou", "beautifulsoup4",
    "python-nmap", "nmap",
    "colourama", "colorama"
];
DeviceProcessEvents
| where FileName has_any ("pip", "pip3", "npm")
| where ProcessCommandLine has "install"
| extend InstalledPkg = extract(@"install\s+(\S+)", 1, ProcessCommandLine)
| join kind=inner TyposquatPairs on $left.InstalledPkg == $right.Typo
| project Timestamp, DeviceName, InstalledPkg, Legitimate, ProcessCommandLine
```

### Rule 3: Lockfile injection -- Lockfile changed without manifest change

**Splunk SPL:**

```spl
index=github sourcetype=github:webhook event_type="pull_request"
| spath output=files path=pull_request.files{}
| mvexpand files
| eval has_lockfile = if(match(files, "(package-lock\.json|yarn\.lock|Pipfile\.lock|poetry\.lock|Gemfile\.lock)"), 1, 0)
| eval has_manifest = if(match(files, "(package\.json|Pipfile|pyproject\.toml|Gemfile)"), 1, 0)
| stats max(has_lockfile) AS lockfile_changed, max(has_manifest) AS manifest_changed by pull_request.number, pull_request.user.login
| where lockfile_changed=1 AND manifest_changed=0
| eval severity="high"
| eval mitre="T1195.002"
| table _time, pull_request.number, pull_request.user.login, severity, mitre
```

**KQL:**

```kql
GitHubAuditData
| where Action == "pull_request.opened" or Action == "pull_request.synchronize"
| extend Files = parse_json(ChangedFiles)
| extend HasLockfile = Files has_any ("package-lock.json", "yarn.lock", "Pipfile.lock", "poetry.lock")
| extend HasManifest = Files has_any ("package.json", "Pipfile", "pyproject.toml")
| where HasLockfile and not(HasManifest)
| project TimeGenerated, Actor, Repository, PullRequestNumber, Action
```

### Rule 4: Exfiltration during package install -- setup.py spawning network tools

**Splunk SPL:**

```spl
index=edr sourcetype=crowdstrike:events
  parent_process_name="python*"
  grandparent_process_name="pip*"
  process_name IN ("curl", "wget", "nc", "ncat", "bash", "sh", "python*")
| eval severity="critical"
| eval mitre="T1020"
| table _time, host, grandparent_process_name, parent_process_name, process_name,
        process_command_line, severity, mitre
| sort -_time
```

**KQL:**

```kql
DeviceProcessEvents
| where InitiatingProcessFileName has_any ("pip", "pip3")
| where FileName == "python" or FileName == "python3"
| where ProcessCommandLine has "setup.py"
| join kind=inner (
    DeviceProcessEvents
    | where ProcessCommandLine has_any ("curl", "wget", "/tmp/", "environ", "whoami", "hostname")
) on DeviceId, $left.ProcessId == $right.InitiatingProcessParentId
| extend MitreTechnique = "T1020"
| project Timestamp, DeviceName, SetupPyCmd=ProcessCommandLine, SuspiciousChild=ProcessCommandLine1
```

**Suricata -- HTTP POST exfiltration from build server:**

```yaml
alert http $BUILD_SERVERS any -> $EXTERNAL_NET any (
  msg:"WEAKLINK Supply Chain Exfiltration - HTTP POST from build server during pip install";
  flow:established,to_server;
  http.method; content:"POST";
  http.header; content:"python-requests"; depth:20;
  classtype:trojan-activity;
  sid:7100002; rev:1;
  metadata:mitre_technique T1020;
)
```

**Suricata -- DNS exfiltration with long subdomain:**

```yaml
alert dns $BUILD_SERVERS any -> any 53 (
  msg:"WEAKLINK Supply Chain Exfiltration - DNS exfil from build server (long subdomain)";
  dns.query;
  pcre:"/^[a-zA-Z0-9]{30,}\./";
  classtype:trojan-activity;
  sid:7100003; rev:1;
  metadata:mitre_technique T1071.001;
)
```

### Rule 5: High-version package install (dependency confusion indicator)

**Splunk SPL:**

```spl
index=cicd sourcetype=pip_install
| rex field=_raw "Downloading\s+\S+/(?<pkg_name>[^/]+)-(?<pkg_version>\d+)\.\d+\.\d+.*\.tar\.gz"
| where tonumber(pkg_version) > 50
| eval severity="high"
| eval mitre="T1195.002"
| table _time, host, pkg_name, pkg_version, severity, mitre
```

**KQL:**

```kql
DeviceProcessEvents
| where FileName has_any ("pip", "pip3")
| where ProcessCommandLine matches regex @"==\d{2,}\."
| extend PackageVersion = extract(@"==(\d+)\.", 1, ProcessCommandLine)
| where toint(PackageVersion) > 50
| extend MitreTechnique = "T1195.002"
| project Timestamp, DeviceName, ProcessCommandLine, PackageVersion
```

!!! tip "Phase 4: IMPROVE -- Tune and Build Coverage"

    **Goal:** Reduce false positives, add enrichment, and build a detection coverage matrix.

### Step 1: Identify false positive sources

| Rule | Common False Positives | Tuning Strategy |
|------|----------------------|-----------------|
| Internal pkg on public PyPI | Legitimate internal packages published to PyPI intentionally | Maintain an allow-list of packages that exist on both registries by design |
| Typosquat watchlist | Developers who legitimately use niche packages with similar names | Only alert on names within edit distance 1-2 of top-1000 PyPI packages |
| Lockfile without manifest | Dependabot/Renovate PRs that update transitive deps only | Exclude known bot accounts from the rule |
| setup.py child process | Packages that legitimately compile C extensions during install | Filter out known build tools: `gcc`, `g++`, `make`, `cmake` |
| High version number | Legitimate packages at high versions (e.g., `Pillow==10.2.0`) | Raise threshold to >50 for major version, and cross-reference with known internal names |

### Step 2: Add context enrichment

Enrich every alert with information the SOC analyst needs to triage quickly:

```spl
| lookup asset_inventory src_ip OUTPUT host_role, team, environment
| lookup internal_package_names package_name OUTPUT package_owner, last_published_version
| eval version_jump = tonumber(pkg_version) - tonumber(last_published_version)
| where version_jump > 10
```

### Step 3: Build the detection coverage matrix

| Attack Type | Proxy/DNS | EDR | CI/CD Logs | Git Audit | Registry Audit |
|-------------|:---------:|:---:|:----------:|:---------:|:--------------:|
| Dependency confusion | Rule 1 | Rule 4 | Rule 5 | -- | -- |
| Typosquatting | -- | -- | Rule 2 | -- | -- |
| Lockfile injection | -- | -- | -- | Rule 3 | -- |
| Manifest confusion | -- | -- | -- | -- | Partial |
| Phantom dependencies | -- | -- | Build failure | -- | -- |
| Exfiltration (any type) | Rule 4 (Suricata) | Rule 4 | -- | -- | -- |

**Coverage gaps identified:**

- **Manifest confusion**: No reliable automated detection from logs alone. Requires pre-install validation (comparing registry metadata to tarball contents).
- **Phantom dependencies**: Detection relies on build failures, which only happen when the transitive dependency is removed. Proactive detection requires dependency graph analysis.
- **Typosquatting**: Watchlist-based detection only catches known typosquats. New ones require fuzzy matching or integration with tools like Socket.dev.

### Step 4: Document your rule set

For each rule, document:

1. **Rule name** and unique ID
2. **MITRE ATT&CK mapping**
3. **Log source and sourcetype**
4. **Detection logic** (the query)
5. **False positive rate** (estimated: low/medium/high)
6. **Triage instructions** (what the analyst should check first)
7. **Response actions** (containment steps if confirmed)

## What You Learned

1. **Supply chain attacks leave traces in multiple log sources** -- proxy logs, EDR, CI/CD logs, Git audit logs, and registry audit logs all contribute detection signals.
2. **Detection timing matters** -- the window between package installation and exfiltration completion is minutes. Rules must fire on near-real-time telemetry.
3. **MITRE ATT&CK provides a shared taxonomy** -- T1195.001, T1195.002, T1059.006, T1020, and T1071.001 cover the supply chain attack kill chain.
4. **False positive tuning is critical** -- every rule needs allow-lists, bot exclusions, and threshold adjustments to be production-viable.
5. **Coverage gaps exist** -- no single detection strategy catches all supply chain attacks. Layered detection across multiple log sources is required.
