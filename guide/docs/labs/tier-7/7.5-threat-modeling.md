You have been building detection rules, triaging incidents, and evaluating tools. But all of that is reactive. you are responding to attacks after they happen. Threat modeling is proactive: you systematically identify where your supply chain can be attacked before an attacker does.

This lab applies STRIDE and PASTA threat modeling frameworks to software supply chains. You will map trust boundaries across the development lifecycle, apply STRIDE to each boundary, and produce a prioritized remediation roadmap.

???+ info "Phase 1: UNDERSTAND. Threat Modeling Frameworks"

    **Goal:** Learn STRIDE and PASTA frameworks and understand trust boundaries in the context of software supply chains.

### Step 1: STRIDE framework

STRIDE was developed by Microsoft to categorize threats. Each letter represents a threat category:

| Category | Question | Supply Chain Example |
|----------|----------|---------------------|
| **S**poofing | Can an attacker pretend to be a trusted entity? | Publish a package under a trusted maintainer's name (account takeover) |
| **T**ampering | Can an attacker modify data in transit or at rest? | Inject malicious code into a lockfile, modify a build artifact |
| **R**epudiation | Can an attacker deny their actions? | Push a malicious commit with a spoofed Git author |
| **I**nformation Disclosure | Can an attacker access data they shouldn't? | Exfiltrate CI secrets during pip install |
| **D**enial of Service | Can an attacker disrupt availability? | Publish a broken version of a critical dependency |
| **E**levation of Privilege | Can an attacker gain higher access? | Use stolen CI credentials to modify production infrastructure |

### Step 2: PASTA framework

PASTA (Process for Attack Simulation and Threat Analysis) is a seven-stage, risk-centric threat modeling methodology:

```
Stage 1: Define business objectives
         ↓
Stage 2: Define the technical scope
         ↓
Stage 3: Application decomposition (identify components)
         ↓
Stage 4: Threat analysis (identify threat agents and attack patterns)
         ↓
Stage 5: Vulnerability analysis (map existing weaknesses)
         ↓
Stage 6: Attack modeling (build attack trees)
         ↓
Stage 7: Risk & impact analysis (prioritize)
```

PASTA is more structured than STRIDE alone. For this lab, we use STRIDE to categorize threats at each trust boundary, then use PASTA's risk-ranking approach to prioritize.

### Step 3: What is a trust boundary?

A trust boundary is a point where data or control crosses from one trust domain to another. Every trust boundary is a potential attack surface.

In a supply chain context, trust boundaries exist wherever:

- Code moves between systems (developer laptop to Git, Git to CI, CI to registry)
- An identity assertion is made (author of a commit, publisher of a package)
- A package is fetched from an external source (public registry to your build)
- An artifact transitions between environments (staging to production)

???+ success "Phase 3: VALIDATE. Apply STRIDE to Each Boundary"

    **Goal:** Systematically apply STRIDE to each trust boundary and identify concrete threats.

### Step 1: TB-1. Developer push

| Threat | STRIDE | Attack | Likelihood | Impact |
|--------|--------|--------|:----------:|:------:|
| Spoofed commit author | S | Attacker pushes code with a spoofed `Author` header to bypass code review attribution | Medium | High |
| Tampered lockfile | T | Developer (or compromised laptop) modifies lockfile to inject malicious dependency | Medium | Critical |
| Unsigned commits | R | No way to verify who actually authored a commit without GPG signatures | High | Medium |
| Credential leak | I | Secrets accidentally committed to the repository | Medium | Critical |

**Existing controls:** Branch protection, required PR reviews, `.gitignore` for secrets
**Gaps:** No commit signing enforcement, no lockfile integrity validation in PR checks

### Step 2: TB-3. Python dependency fetch

| Threat | STRIDE | Attack | Likelihood | Impact |
|--------|--------|--------|:----------:|:------:|
| Dependency confusion | S | Attacker publishes internal package name on public PyPI with higher version | Medium | Critical |
| Typosquatting | S | Attacker publishes package with similar name to popular dependency | Medium | High |
| Compromised maintainer | T | Attacker takes over legitimate package maintainer account | Low | Critical |
| Malicious setup.py | E | Package install script exfiltrates secrets and gains CI runner access | Medium | Critical |

**Existing controls:** Version pinning in requirements.txt
**Gaps:** Using `--extra-index-url` (not `--index-url`), no hash verification, no behavioral analysis

### Step 3: TB-4. Secret injection

| Threat | STRIDE | Attack | Likelihood | Impact |
|--------|--------|--------|:----------:|:------:|
| Over-privileged secrets | E | CI pipeline has access to secrets it doesn't need (e.g., prod database creds in a test pipeline) | High | Critical |
| Secret exfiltration | I | Malicious dependency reads environment variables during install | Medium | Critical |
| Pull request secret leak | I | `pull_request_target` workflow exposes secrets to untrusted PR code | Medium | Critical |

**Existing controls:** GitHub Secrets (encrypted at rest)
**Gaps:** No least-privilege secret scoping, no OIDC-based short-lived credentials

### Step 4: TB-6. Artifact publish

| Threat | STRIDE | Attack | Likelihood | Impact |
|--------|--------|--------|:----------:|:------:|
| Tag mutability | T | Attacker (or accident) pushes a different image to an existing tag | Medium | Critical |
| Missing provenance | R | No way to verify the image was built by CI from reviewed source | High | High |
| Registry compromise | T | Attacker gains write access to container registry | Low | Critical |

**Existing controls:** GitHub Actions OIDC for registry auth
**Gaps:** No image signing (cosign), no SLSA provenance, tags not immutable

### Step 5: TB-7. Deployment sync

| Threat | STRIDE | Attack | Likelihood | Impact |
|--------|--------|--------|:----------:|:------:|
| Unverified image pull | S | ArgoCD deploys an image without verifying its signature or provenance | High | Critical |
| Manifest tampering | T | Attacker modifies Kubernetes manifests to point to a malicious image | Low | Critical |
| Stale vulnerability | D | Deployed image contains known CVEs that were not scanned | Medium | High |

**Existing controls:** ArgoCD sync from Git (GitOps)
**Gaps:** No admission controller enforcing image signatures, no runtime vulnerability scanning

### Step 3: Identify residual gaps

After mapping all existing controls, these gaps remain:

| Gap | Threat | Remediation |
|-----|--------|-------------|
| No least-privilege secrets | Over-privileged CI secrets | Implement OIDC-based short-lived credentials; scope secrets per workflow |
| No commit signing | Unsigned commits | Enforce GPG/SSH commit signing via branch protection rules |
| No admission controller | Unverified deployment | Deploy Kyverno or OPA Gatekeeper to require image signatures |
| No runtime scanning | Stale vulnerabilities in prod | Deploy Trivy Operator or Falco for runtime vulnerability detection |
| No PR secret exposure prevention | `pull_request_target` leak | Audit all workflows for `pull_request_target`; use `pull_request` event instead |

| Timeline | Action | Addresses | Effort |

## What You Learned

1. **STRIDE systematically identifies threats at each trust boundary**. applying it to supply chains reveals 24+ distinct threats across 7 trust boundaries.
2. **Tampering and Spoofing dominate supply chain threats**. most real attacks involve impersonating trusted sources or modifying trusted artifacts.
3. **Trust boundaries are attack surfaces**. every time code, packages, images, or secrets cross a boundary, an attacker has an opportunity.
4. **Threat modeling connects labs to real risk**. each Tier 1-5 lab addresses specific threats identified through this modeling process.
5. **Prioritization prevents paralysis**. not every threat needs immediate remediation. Risk = Likelihood x Impact gives you the ordering.
