The supply chain security tooling market has exploded. OpenSSF Scorecard, GUAC, deps.dev, Dependabot, Snyk, Socket, pip-audit, npm audit, Grype, Trivy -- each tool claims to protect your supply chain. But which ones actually catch the attacks you practiced in Tier 1?

This lab takes a systematic approach: run every major tool against the same target project containing known vulnerabilities, then build a comparison matrix that shows exactly which tool catches which attack type, where the gaps are, and what it costs to adopt each one.

!!! info "Phase 1: UNDERSTAND -- The Tool Landscape"

    **Goal:** Survey the supply chain security tool landscape and understand what category each tool belongs to.

### Step 1: Tool categories

Supply chain security tools fall into distinct categories. No single tool covers everything:

| Category | What It Does | Tools |
|----------|-------------|-------|
| **Vulnerability scanning** | Finds known CVEs in dependencies | Grype, Trivy, pip-audit, npm audit, Snyk |
| **Behavioral analysis** | Detects suspicious package behavior (network calls, file access, obfuscation) | Socket |
| **Dependency risk scoring** | Scores project health, maintainer trust, dependency hygiene | OpenSSF Scorecard, deps.dev |
| **SBOM & provenance** | Generates/validates SBOMs and build provenance | Syft, cosign, SLSA verifier |
| **Dependency graph analysis** | Maps and queries the dependency graph for anomalies | GUAC, deps.dev |
| **Automated updates** | Keeps dependencies current with automated PRs | Dependabot, Renovate |

### Step 2: Map tools to Tier 1 attack types

Before running any tools, predict which should catch which attack:

| Attack Type | Expected Detection By |
|-------------|----------------------|
| Dependency confusion | Socket (behavioral), Scorecard (config check) |
| Typosquatting | Socket (name analysis), deps.dev (popularity) |
| Lockfile injection | npm audit (integrity check), Socket (diff analysis) |
| Manifest confusion | Socket (metadata analysis), npm audit (partial) |
| Phantom dependencies | deps.dev (graph analysis), Scorecard (pinning check) |
| Known CVEs | Grype, Trivy, pip-audit, npm audit, Snyk |

### Step 3: Understand each tool's approach

**OpenSSF Scorecard** -- Evaluates a project's security practices (not vulnerabilities):

```bash
# Checks: branch protection, code review, CI tests, fuzzing, pinned deps, SAST, signed releases
scorecard --repo=github.com/owner/repo
```

**GUAC (Graph for Understanding Artifact Composition)** -- Ingests SBOMs, SLSA provenance, and vulnerability data into a queryable graph database:

```bash
# Query: "Show me all paths from this CVE to my production deployment"
guacone query vulnerability --cve CVE-2024-XXXXX
```

**deps.dev** -- Google's open-source dependency intelligence service:

```bash
# API: package metadata, dependency graph, advisories, Scorecard data
curl "https://api.deps.dev/v3alpha/systems/npm/packages/express"
```

**Socket** -- Behavioral analysis of packages (what they do, not just what CVEs they have):

```bash
# Detects: install scripts, network access, filesystem access, obfuscation, typosquats
socket scan create --repo=owner/repo
```

**Grype** -- Vulnerability scanner for container images and filesystems:

```bash
# Scans against NVD, GitHub Advisories, and distro-specific databases
grype dir:. --output table
grype registry:myimage:latest
```

**Trivy** -- All-in-one scanner (vulnerabilities, misconfigurations, secrets, licenses):

```bash
# Scans: container images, filesystems, git repos, Kubernetes manifests
trivy fs . --security-checks vuln,secret,config
```

!!! warning "Phase 2: INVESTIGATE -- Run Each Tool Against the Target"

    **Goal:** Run every tool against the same target project and record what each one finds (and misses).

### Step 1: Set up the target project

The target project is a Python/Node.js application with known issues from the Tier 1 labs:

```bash
ls /app/
# requirements.txt    -- contains dependency confusion risk (extra-index-url)
# package.json        -- contains typosquat dependency
# package-lock.json   -- contains injected lockfile entry
# Dockerfile          -- builds the application
```

### Step 2: Run vulnerability scanners

**pip-audit:**

```bash
cd /app
pip-audit -r requirements.txt --output-format json 2>&1 | head -50
```

Record: How many CVEs found? Any false positives? Did it catch anything beyond CVEs?

**npm audit:**

```bash
cd /app
npm audit --json 2>&1 | head -50
```

Record: How many advisories? Severity breakdown? Did it flag the typosquat?

**Grype:**

```bash
grype dir:/app --output table
```

Record: Total findings. Compare to pip-audit and npm audit -- are there differences?

**Trivy:**

```bash
trivy fs /app --security-checks vuln,secret,config --format table
```

Record: Did Trivy find anything the others missed? (Secrets in code? Dockerfile misconfigurations?)

**Snyk (if available):**

```bash
snyk test --all-projects
```

Record: Unique findings? License issues? Reachability analysis?

### Step 3: Run behavioral analysis

**Socket (if available):**

```bash
socket scan create --repo=/app
```

Key questions:
- Did it flag the typosquat package?
- Did it detect the install script in the dependency confusion package?
- Did it flag the lockfile injection?
- Did it detect network access or obfuscation in any dependency?

### Step 4: Run project health scoring

**OpenSSF Scorecard:**

```bash
scorecard --local /app --format json 2>&1 | python3 -m json.tool | head -80
```

Key checks to review:
- `Pinned-Dependencies` -- are deps pinned to exact versions or hashes?
- `Branch-Protection` -- is the default branch protected?
- `Code-Review` -- are PRs required before merge?
- `Dangerous-Workflow` -- does CI use `pull_request_target` unsafely?

### Step 5: Query dependency intelligence

**deps.dev API:**

```bash
# Check a specific package for known issues
curl -s "https://api.deps.dev/v3alpha/systems/pypi/packages/requests/versions/2.31.0" | python3 -m json.tool

# Check the typosquat package
curl -s "https://api.deps.dev/v3alpha/systems/pypi/packages/reqeusts" | python3 -m json.tool
```

Key question: Does deps.dev have enough data to distinguish the typosquat from the real package?

!!! success "Phase 3: VALIDATE -- Build the Comparison Matrix"

    **Goal:** Consolidate findings into a comparison matrix that shows coverage, gaps, and tradeoffs.

### Step 1: Detection coverage matrix

| Attack Type | pip-audit | npm audit | Grype | Trivy | Snyk | Socket | Scorecard | deps.dev |
|-------------|:---------:|:---------:|:-----:|:-----:|:----:|:------:|:---------:|:--------:|
| Known CVEs | Yes | Yes | Yes | Yes | Yes | Partial | No | Yes |
| Dependency confusion | No | No | No | No | No | **Yes** | Partial | No |
| Typosquatting | No | No | No | No | Partial | **Yes** | No | Partial |
| Lockfile injection | No | Partial | No | No | No | **Yes** | No | No |
| Manifest confusion | No | No | No | No | No | **Yes** | No | No |
| Phantom dependencies | No | No | No | No | No | Partial | Partial | Partial |
| Malicious install scripts | No | No | No | No | No | **Yes** | No | No |
| Secrets in code | No | No | No | **Yes** | Partial | No | No | No |
| Dockerfile misconfig | No | No | No | **Yes** | Partial | No | No | No |

**Key insight:** Vulnerability scanners (pip-audit, npm audit, Grype, Trivy, Snyk) only catch *known CVEs*. They miss every Tier 1 attack type because those attacks use packages that are not in vulnerability databases -- they are zero-day by nature.

### Step 2: Operational comparison

| Tool | Type | Cost | Integration Effort | False Positive Rate | Maintenance |
|------|------|------|--------------------|---------------------|-------------|
| pip-audit | CLI | Free (OSS) | Low -- single command | Low | Low |
| npm audit | CLI | Free (built-in) | None -- part of npm | Medium | None |
| Grype | CLI | Free (OSS) | Low | Low-Medium | Low |
| Trivy | CLI | Free (OSS) | Low | Low-Medium | Low |
| Snyk | SaaS | Freemium ($$$) | Medium -- CI integration | Low | Medium |
| Socket | SaaS | Freemium ($$) | Medium -- GitHub App | Medium | Low |
| Scorecard | CLI/API | Free (OSS) | Low | Low | Low |
| deps.dev | API | Free | Medium -- custom integration | Low | Low |
| GUAC | Platform | Free (OSS) | High -- requires infra | Low | High |
| Dependabot | SaaS | Free (GitHub) | Low -- enable in repo | Low | None |

### Step 3: Identify critical gaps

Based on the matrix, these attack types have **no reliable automated detection** from any single tool:

1. **Manifest confusion**: Only Socket detects this, and only partially. Most tools trust the package metadata at face value.
2. **Phantom dependencies**: No tool reliably detects that a package imports a module not in its declared dependencies. Static analysis tools could help but are not standard in the supply chain toolkit.
3. **Long-lived compromises**: None of these tools alert on packages that became malicious after you installed them (maintainer account takeover).

!!! tip "Phase 4: IMPROVE -- Write the Recommendation"

    **Goal:** Produce a tool recommendation for your organization with a phased adoption plan.

### Step 1: Tiered recommendation

Based on the evaluation, here is a recommended adoption strategy:

**Tier 1 -- Immediate (Week 1-2, zero cost):**

- **pip-audit / npm audit**: Run in every CI pipeline. Catches known CVEs. No excuse not to have this.
- **Dependabot**: Enable on all repositories. Automated PRs for vulnerable dependencies.
- **OpenSSF Scorecard**: Run weekly on your projects. Identifies security hygiene issues.

```yaml
# GitHub Actions -- minimal supply chain scanning
- name: Run pip-audit
  run: pip-audit -r requirements.txt --strict

- name: Run npm audit
  run: npm audit --audit-level=high
```

**Tier 2 -- Short-term (Month 1, low cost):**

- **Trivy**: Replace or supplement pip-audit/npm-audit. Covers container images, secrets, and misconfigs in addition to CVEs.
- **Socket**: Add behavioral analysis. This is the only tool that catches typosquatting, install scripts, and lockfile injection reliably.

**Tier 3 -- Medium-term (Quarter 1, moderate effort):**

- **GUAC**: Deploy if you need to query your dependency graph at scale ("which production services depend on this compromised package?").
- **Scorecard + deps.dev integration**: Build a dashboard that combines project health scores with dependency intelligence.

### Step 2: What tools cannot replace

No combination of tools eliminates the need for:

1. **Hardened CI/CD configuration** -- `--index-url` not `--extra-index-url`, pinned hashes, locked registries.
2. **Detection rules in your SIEM** -- the rules from [Lab 7.1](7.1-detection-rules.md) catch what no package scanner can.
3. **Incident response playbooks** -- [Lab 7.3](7.3-ir-playbook.md)'s playbook is your response when tools fail.
4. **Developer training** -- developers who understand supply chain attacks make better decisions than any scanner.

## What You Learned

1. **Vulnerability scanners only catch known CVEs** -- they miss every novel supply chain attack (dependency confusion, typosquatting, lockfile injection, manifest confusion).
2. **Behavioral analysis (Socket) fills the biggest gap** -- it is the only tool category that detects malicious behavior in packages that are not in vulnerability databases.
3. **No single tool covers everything** -- a defense-in-depth approach requires vulnerability scanning + behavioral analysis + project health scoring + SIEM detection rules.
4. **Free tools cover the basics** -- pip-audit, npm audit, Grype, Trivy, Scorecard, and Dependabot are all free and cover known CVEs and project hygiene.
5. **Tools are not a substitute for hardened configuration** -- fixing `--extra-index-url` eliminates dependency confusion entirely, which no scanning tool can claim.
