The OWASP Software Component Verification Standard (SCVS) is a community-driven framework for identifying activities, controls, and best practices to verify software components throughout the supply chain. Unlike SLSA (which focuses narrowly on build integrity) or SSDF (which focuses on organizational secure development practices), SCVS provides a comprehensive, granular checklist that covers everything from knowing what components you have to verifying where they came from.

In Labs 8.1 and 8.2, you assessed your project against SLSA and SSDF. This lab adds the third major framework to your compliance toolkit. More importantly, it shows how the three frameworks overlap and where each one fills gaps the others leave behind. The result is a unified compliance strategy, not three separate checklists.

**Reference:** [OWASP Software Component Verification Standard](https://owasp.org/www-project-software-component-verification-standard/)

???+ info "Phase 1: UNDERSTAND. The SCVS Framework"

    **Goal:** Learn the SCVS verification categories, maturity levels, and how SCVS relates to SLSA, SSDF, and other frameworks.

### Step 1: SCVS structure and purpose

SCVS organizes software component verification into six categories, each with three maturity levels. The categories cover the full lifecycle of a software component, from knowing it exists to verifying its entire chain of custody:

| Category | Name | What It Answers |
|:--------:|------|----------------|
| **V1** | Inventory | Do you know what components your software contains? |
| **V2** | Software Bill of Materials | Can you produce a machine-readable, standards-compliant list of those components? |
| **V3** | Build Environment | Is the build process that assembles those components secure and verifiable? |
| **V4** | Package Management | Are dependencies acquired, verified, and managed securely? |
| **V5** | Component Analysis | Are known vulnerabilities in those components identified and remediated? |
| **V6** | Pedigree and Provenance | Can you trace each component back to its origin and verify it was not tampered with? |

Each category has three maturity levels:

| Level | Description | Typical Organization |
|:-----:|-------------|---------------------|
| **1** | Basic controls. The minimum viable security posture. | Any team with a lockfile and basic CI |
| **2** | Automated, integrated controls. Security is part of the pipeline. | Teams with mature CI/CD and dedicated security tooling |
| **3** | Advanced, comprehensive controls. Full verification and auditability. | Organizations with dedicated AppSec teams and compliance requirements |

### Step 2: How SCVS differs from SLSA and SSDF

Understanding the boundaries of each framework prevents you from treating them as interchangeable:

| Dimension | SCVS | SLSA | SSDF |
|-----------|------|------|------|
| **Scope** | All aspects of component verification | Build integrity only | Entire software development lifecycle |
| **Granularity** | 6 categories, ~65 controls | 1 track (Build), 4 levels | 4 practice areas, ~42 tasks |
| **Focus** | Components you consume | Artifacts you produce | How you develop software |
| **Maturity model** | 3 levels per category | 4 levels (single track) | No formal levels (task-based) |
| **Compliance driver** | Voluntary (OWASP best practice) | Voluntary (OpenSSF ecosystem) | Mandatory for US federal suppliers (EO 14028) |
| **Strongest domain** | Inventory, SBOM, package management | Build provenance, non-falsifiability | Organizational governance, vulnerability response |
| **Weakest domain** | Organizational governance, incident response | Anything not build-related | Component-level technical controls |

!!! note "Complementary, not competing"
    These frameworks work best together. SCVS gives you the detailed technical checklist, SLSA gives you the build integrity maturity model, and SSDF gives you the organizational governance structure. A single control implementation (e.g., generating SBOMs in CI) can satisfy requirements across all three.

### Step 2: Assess V1 (Inventory) and V4 (Package Management)

Start with the categories closest to Tier 1. these are the ones you have the most hands-on experience with:

```bash
# Check if the project has a manifest and lockfile
ls /app/requirements.txt /app/package.json /app/package-lock.json 2>/dev/null

# Check if dependencies are pinned to exact versions
grep -E '==' /app/requirements.txt 2>/dev/null | head -5

# Check if hashes are included
grep -E '--hash' /app/requirements.txt 2>/dev/null | head -3

# Check if there is namespace separation (private vs public registries)
grep -E 'index-url|registry' /app/.npmrc /app/pip.conf 2>/dev/null
```

For a typical project without SCVS investment, expect:

| Category | Level 1 | Level 2 | Level 3 |
|----------|:-------:|:-------:|:-------:|
| V1: Inventory | Partial (lockfile exists but no formal catalog) | Not Met (no automated generation) | Not Met |
| V4: Package Management | Partial (lockfile exists, trusted registries) | Not Met (no hash verification, no review process) | Not Met |

## What You Learned

1. **SCVS is the most comprehensive component-level checklist**. its 6 categories and ~65 controls cover inventory, SBOMs, builds, packages, vulnerabilities, and provenance in a single framework.
2. **Frameworks complement, they do not compete**. SCVS provides technical depth, SLSA provides build maturity, SSDF provides organizational governance, and EO 14028 provides regulatory requirements. Use them together.
3. **Multi-framework controls are the best investment**. implementing SBOM generation, artifact signing, and vulnerability scanning each satisfies 3+ frameworks simultaneously. Prioritize these.
4. **The WeakLink Labs tiers map directly to SCVS categories**. Tier 1 is V4, Tier 2 is V3, Tier 4 is V1/V2/V6, and Tier 7 is V5. The hands-on skills you built are exactly what SCVS compliance requires.
5. **Compliance is measurable and continuous**. define dashboard metrics per SCVS category and track them over time rather than treating compliance as a one-time assessment.
