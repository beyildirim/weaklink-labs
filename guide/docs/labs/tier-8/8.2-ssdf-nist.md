The Secure Software Development Framework (SSDF), published as NIST SP 800-218, is the US government's answer to "how should organizations develop secure software?" It is not a suggestion -- Executive Order 14028 requires federal software suppliers to self-attest compliance with the SSDF. If your organization sells software to the US government, this framework is mandatory.

But the SSDF is also valuable outside federal procurement. It provides a structured way to evaluate whether your development practices actually address supply chain risks, or just create a false sense of security.

In this lab, you will map the defenses you learned in Tiers 1-5 to SSDF practices, identify gaps, build a compliance roadmap, and produce a self-attestation form based on the CISA template.

!!! info "Phase 1: UNDERSTAND -- The SSDF Framework"

    **Goal:** Learn the four SSDF practice areas, their sub-practices, and how they relate to supply chain security.

### Step 1: The four practice areas

NIST SP 800-218 organizes secure development into four high-level practice areas:

| Practice Area | Code | Focus |
|--------------|------|-------|
| **Prepare the Organization** | PO | Governance, roles, training, tooling |
| **Protect the Software** | PS | Source code, build systems, artifact integrity |
| **Produce Well-Secured Software** | PW | Design, implementation, testing, vulnerability management |
| **Respond to Vulnerabilities** | RV | Monitoring, disclosure, remediation |

**Reference:** [NIST SP 800-218](https://csrc.nist.gov/publications/detail/sp/800-218/final)

### Step 2: Key sub-practices for supply chain security

Not all 42 SSDF tasks are equally relevant to supply chain security. These are the most critical:

| Task ID | Practice | Description | Supply Chain Relevance |
|---------|----------|-------------|----------------------|
| **PO.1.1** | Prepare | Define security requirements for software development | Sets the foundation for all other practices |
| **PO.3.1** | Prepare | Specify tools and tool configuration | Defines which SCA, SAST, SBOM tools are required |
| **PO.3.2** | Prepare | Use automated tools to check compliance | Continuous validation, not one-time audits |
| **PS.1.1** | Protect | Store all code in a version control system with access controls | Prevents unauthorized source modification |
| **PS.2.1** | Protect | Protect all forms of code from unauthorized access and tampering | Includes build scripts, CI configs, IaC |
| **PS.3.1** | Protect | Archive and protect each software release | Immutable artifacts, signed releases |
| **PW.4.1** | Produce | Acquire well-secured components | Dependency management, registry hardening |
| **PW.4.4** | Produce | Verify the integrity of acquired components | Hash verification, signature checking, provenance |
| **PW.7.2** | Produce | Use automated testing to identify vulnerabilities | SCA scanning in CI/CD |
| **PW.9.1** | Produce | Configure software with secure default settings | Secure-by-default configuration |
| **RV.1.1** | Respond | Gather information about vulnerabilities | Vulnerability monitoring (Dependabot, Grype) |
| **RV.1.2** | Respond | Assess each vulnerability and determine appropriate responses | Triage process |
| **RV.1.3** | Respond | Remediate vulnerabilities | Patching, updating dependencies |
| **RV.3.3** | Respond | Provide SBOMs and other information to software consumers | SBOM generation and distribution |

### Step 3: The CISA self-attestation requirement

Since September 2023, CISA requires software producers selling to the US government to submit a self-attestation form confirming compliance with specific SSDF practices. The form covers:

- The software producer develops and maintains a secure software development process
- The software producer manages its development environment security
- The software producer maintains trusted source code supply chains
- The software producer employs automated tools for security

**Reference:** [CISA Secure Software Development Attestation Form](https://www.cisa.gov/secure-software-attestation-form)

### Step 2: Guidance for honest attestation

The self-attestation form is a legal document when submitted to the US government. Key principles:

1. **Do not attest to what you have not verified.** If you are not sure whether all repos have SCA scanning, check before attesting.
2. **Gaps are acceptable if you document them.** The form allows for "Plan of Action & Milestones" (POA&M) for items not yet complete.
3. **Partial compliance is better than no compliance.** Attesting to 80% with a documented plan for the remaining 20% is honest and defensible.
4. **The attestation is point-in-time.** It reflects your posture on the attestation date. Update it when your posture changes.

## What You Learned

1. **SSDF is not optional for federal suppliers** -- Executive Order 14028 and the CISA self-attestation form make SSDF compliance a procurement requirement.
2. **Technical controls (Tiers 1-5) map well to PS and PW tasks** -- the hands-on defenses you have learned directly satisfy SSDF requirements for protecting software and producing secure software.
3. **Organizational practices are the biggest gap** -- governance (PO) and vulnerability response (RV) require documented policies, defined roles, and training programs that most engineering teams lack.
4. **The self-attestation is a communication tool** -- it forces you to be honest about your current state and creates accountability for closing gaps.
5. **Compliance is a continuous process** -- the self-attestation must be updated as your practices evolve, not filed and forgotten.
