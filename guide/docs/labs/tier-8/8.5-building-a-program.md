You have completed the technical labs (Tiers 1-6), built detection and response capabilities (Tier 7), and mapped your practices to industry frameworks (Tier 8). Now it is time to bring it all together into a cohesive program.

A supply chain security program is not a tool deployment or a policy document. It is an organizational capability: the people, processes, and technology that continuously protect your software supply chain from compromise. This lab is the capstone. you will design a complete program for a 500-person organization, phase it into actionable milestones, and produce the deliverables that get executive buy-in.

### Step 2: The target organization

For this lab, you are designing a program for:

- **Company size**: 500 employees, 200 developers
- **Stack**: Cloud-native (Kubernetes, microservices)
- **Languages**: Python, TypeScript, Go
- **CI/CD**: GitHub Actions
- **Cloud**: AWS
- **Registries**: PyPI (public), npm (public), ghcr.io (private container registry)
- **Current state**: Dependabot enabled on some repos. No SBOM generation. No artifact signing. No supply chain detection rules. Ad-hoc incident response.

### Step 3: Maturity model

Before designing the program, assess where the organization is today and where it needs to be:

| Level | Name | Description | Target Organization |
|:-----:|------|-------------|:-------------------:|
| 0 | **Ad-hoc** | No formal supply chain security. Individual developers make decisions. | Current state |
| 1 | **Reactive** | Basic vulnerability scanning. Patches applied when alerted. | |
| 2 | **Defined** | Documented policies. Standardized tooling. Detection rules. | 90-day target |
| 3 | **Managed** | Metrics tracked. Vendor assessments. SBOM/VEX delivery. IR playbooks tested. | 6-month target |
| 4 | **Optimizing** | Continuous improvement. Threat model updated quarterly. Industry-leading practices. | 1-year target |

### Step 2: Tooling architecture

```

                                                      ┌──────────────────┐

                                                      │                  │
                                                      │ Runtime          │
                                                      │ - Falco          │
                                                      │ - Trivy Operator │
                                                      └──────────────────┘
```

**Tool selection rationale:**

| Tool | Purpose | Why This Tool |
|------|---------|--------------|
| pip-audit / npm audit | Known CVE detection | Free, built-in, minimal friction |
| Grype | Comprehensive vuln scanning | OSS, supports multiple ecosystems, low false positive rate |
| Socket | Behavioral package analysis | Only tool that catches typosquatting, install scripts, lockfile injection |
| Syft + CycloneDX | SBOM generation | OSS, CycloneDX is the industry-preferred format |
| cosign | Artifact signing | OSS (Sigstore), keyless signing via OIDC |
| slsa-github-generator | Build provenance | Official SLSA tooling for GitHub Actions |
| Semgrep | SAST | OSS, custom rules, low noise |
| Dependabot | Automated updates | Free (GitHub native), minimal setup |

### Step 4: Monitoring and detection

Build on the detection rules from [Lab 7.1](../tier-7/7.1-detection-rules.md):

| Detection | Source | Rule | Alert Destination |
|-----------|--------|------|-------------------|
| Dependency confusion | Proxy logs | Internal pkg name on public PyPI | PagerDuty (P1) |
| Typosquatting | CI logs | Known typosquat installed | Slack #security-alerts |
| Lockfile injection | Git audit | Lockfile changed without manifest | PR comment (block merge) |
| Malicious install scripts | EDR | setup.py spawning curl/wget | PagerDuty (P2) |
| Secret in commit | CI | Gitleaks pre-commit + CI | Block push / PR |
| CVE in dependency | CI | Grype finding >= High | Block merge (PR check) |

### Step 1: One-page executive briefing

```markdown
SUPPLY CHAIN SECURITY PROGRAM: EXECUTIVE BRIEFING
===================================================

THE PROBLEM
-----------
Software supply chain attacks increased 742% between 2019 and 2022 (Sonatype).
SolarWinds, Kaseya, 3CX, and Codecov demonstrate that a single compromised
dependency can affect thousands of organizations. Our current posture has
significant gaps: no SBOM generation, no artifact signing, no supply chain
detection rules, and no formal incident response for supply chain events.

THE RISK
--------
- We use 3,400+ open-source dependencies across 47 services
- 0% of our artifacts are signed or have build provenance
- We have no detection capability for dependency confusion or typosquatting
- If a dependency is compromised, we have no playbook to contain the blast radius
- Federal customers are beginning to require SBOMs and SSDF attestation (EO 14028)

THE PROGRAM
-----------
A phased supply chain security program with clear milestones:

  30 days:  Fix critical configuration gaps, deploy basic detection rules
  90 days:  Standardize tooling, generate SBOMs, sign artifacts, train developers
  6 months: Track metrics, assess vendors, achieve SLSA Level 2
  1 year:   Full maturity -- SLSA Level 3, VEX, quarterly threat modeling

INVESTMENT
----------
- 1x Supply Chain Security Lead (new hire or internal promotion)
- 2x AppSec Engineers (existing team, partially reallocated)
- Tooling: $0 for OSS tools, ~$30K/yr for Socket commercial license
- Developer time: ~2 hours per developer for initial training + 30 min/month ongoing

EXPECTED OUTCOMES
-----------------
- 100% of releases have SBOM and signed artifacts within 6 months
- Mean time to patch critical CVEs reduced from ~21 days to <7 days
- Supply chain incidents detectable in <1 hour (vs. current unknown)
- EO 14028 compliance readiness for federal sales
- Demonstrated industry leadership in supply chain security

REQUEST
-------
Approve the program charter, the Supply Chain Security Lead role, and
$30K annual tooling budget.
```

## What You Learned

1. **A program is more than tools**. governance, training, monitoring, incident response, and continuous improvement are equally important pillars.
2. **Phased implementation prevents organizational paralysis**. 30-day quick wins build credibility for the larger investment at 90 days and beyond.
3. **Metrics create accountability**. tracking SBOM coverage, patch time, signing percentage, and detection latency makes progress visible to leadership.
4. **Executive buy-in requires a one-pager**. the technical details go in the implementation plan; the executive briefing needs the problem, the risk, the ask, and the expected outcomes.
5. **The program is the capstone of everything you have learned**. every lab in WeakLink Labs maps to a specific capability within the program.
