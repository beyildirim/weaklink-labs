Signing an artifact means attaching a cryptographic proof that says "this was approved by someone with this key." But signing without verification is meaningless, and verification without enforcement is security theater. This lab walks you through the full chain: signing, verifying, and enforcing.

You'll see how easy it is to deploy unsigned images by default, then close that gap with cosign and admission policies.

???+ info "Phase 1: UNDERSTAND. What Signing Actually Means"

    **Goal:** Learn the mechanics of cryptographic signing and what it proves.

### Step 1: Generate a cosign key pair

```bash
cd /app
cosign generate-key-pair
```

Enter a password when prompted (or leave blank for the lab). This creates:

- `cosign.key`. private key (used to sign, keep secret)
- `cosign.pub`. public key (used to verify, distribute freely)

### Step 2: Understand what signing proves

A signature proves three things:

1. **Integrity**. the artifact hasn't been modified since signing
2. **Authenticity**. someone with the private key approved this artifact
3. **Non-repudiation**. the signer can't deny signing it (if key management is solid)

A signature does NOT prove:

- The artifact is free of vulnerabilities
- The artifact does what it claims
- The artifact was built from specific source code (that's attestation. [Lab 4.4](4.4-attestation-slsa.md))

### Step 3: Explore the registry

```bash
# List images in the registry
crane catalog registry:5000

# Check the tags
crane ls registry:5000/weaklink-app
```

You should see two tags: `signed` and `unsigned`. Both contain the same application code, but one has a cosign signature attached and the other doesn't.

### Step 4: Inspect an image's signatures

```bash
# Check signatures on the signed image
cosign tree registry:5000/weaklink-app:signed

# Check signatures on the unsigned image
cosign tree registry:5000/weaklink-app:unsigned
```

The signed image has a `.sig` tag in the registry. The unsigned image has nothing. Both are valid container images. the registry doesn't care about signatures.

### Step 5: Compare GPG and cosign

| Feature | GPG | cosign | Notation |
|---------|-----|--------|----------|
| Key format | PGP keys | ECDSA P-256 / ed25519 | x509 certificates |
| Key management | Manual (keyrings) | Key pairs or keyless (Sigstore) | Certificate chain |
| Signature storage | Detached `.asc` file | OCI registry (alongside image) | OCI registry |
| Verification | `gpg --verify` | `cosign verify` | `notation verify` |
| Keyless option | No | Yes (Sigstore Fulcio + Rekor) | No |

For container images, cosign is the standard. GPG is still common for signing git commits, tarballs, and Linux packages.

???+ warning "Phase 2: BREAK. Unsigned Artifacts Are Accepted by Default"

    **Goal:** Deploy an unsigned image and see that nothing stops you.

### Step 1: Deploy the unsigned image

```bash
kubectl run test-unsigned --image=registry:5000/weaklink-app:unsigned
kubectl get pods
```

The pod starts. No warnings, no errors, no admission control. The cluster accepted an unsigned image without question.

### Step 2: Verify it's actually running

```bash
kubectl logs test-unsigned
kubectl exec test-unsigned -- cat /app/version.txt
```

The application runs normally. From the cluster's perspective, there is zero difference between a signed and unsigned image.

### Step 3: Understand the default state

Most container runtimes and orchestrators accept any image by default:

- **Docker**: pulls and runs anything from any registry
- **Kubernetes**: no built-in signature verification
- **Cloud registries**: store signatures but don't enforce verification

This means signing is opt-in at every layer. If you sign your images but don't enforce verification, an attacker can push an unsigned malicious image and it will be deployed just as easily as your signed legitimate one.

### Step 4: Clean up

```bash
kubectl delete pod test-unsigned
```

???+ success "Phase 3: DEFEND. Sign and Enforce"

    **Goal:** Sign an image with cosign and create a policy that rejects unsigned images.

### Step 1: Sign the image

```bash
cosign sign --key /app/cosign.key registry:5000/weaklink-app:signed
```

Verify the signature:

```bash
cosign verify --key /app/cosign.pub registry:5000/weaklink-app:signed
```

You should see JSON output confirming the signature is valid.

### Step 2: Try verifying the unsigned image

```bash
cosign verify --key /app/cosign.pub registry:5000/weaklink-app:unsigned
```

This fails with "no matching signatures". exactly what we want.

### Step 3: Create an admission policy

Create a policy that requires cosign signatures for all images from your registry:

```bash
cat > /app/policy.yaml << 'EOF'
apiVersion: policy.sigstore.dev/v1beta1
kind: ClusterImagePolicy
metadata:
  name: require-signature
spec:
  images:
    - glob: "registry:5000/**"
  authorities:
    - key:
        data: |
          -----BEGIN PUBLIC KEY-----
          <PASTE YOUR cosign.pub CONTENTS HERE>
          -----END PUBLIC KEY-----
EOF

# Replace the placeholder with the actual key
PUB_KEY=$(cat /app/cosign.pub | sed 's/^/          /')
sed -i "s|          -----BEGIN PUBLIC KEY-----.*-----END PUBLIC KEY-----|${PUB_KEY}|" /app/policy.yaml
```

### Step 4: Apply and test the policy

```bash
# Apply the policy (requires a policy controller like Sigstore policy-controller)
kubectl apply -f /app/policy.yaml

# Try deploying the signed image -- should succeed
kubectl run test-signed --image=registry:5000/weaklink-app:signed

# Try deploying the unsigned image -- should be rejected
kubectl run test-unsigned --image=registry:5000/weaklink-app:unsigned
```

The unsigned image is now rejected at admission time. The signed image passes.

### Step 5: Verify the lab

```bash
weaklink verify 4.3
```

??? danger "Phase 4: DETECT. Identifying Unsigned Deployments"

    **Goal:** Detect when unsigned or improperly signed images are deployed in your environment.

### SIEM / Log Indicators

**What to look for:**

- Kubernetes audit logs showing image deployments without corresponding cosign signatures in the registry
- Admission controller deny events for unsigned images (if enforcement is on)
- Images pulled from registries that lack `.sig` tags
- cosign verification failures in CI/CD pipeline logs

### Kubernetes Audit Indicators

| Indicator | What It Means |
|-----------|---------------|
| `admission.k8s.io/deny` with "signature" in message | Policy controller rejected unsigned image |
| Pod creation for image without `.sig` tag in registry | Image was never signed |
| `cosign verify` exit code != 0 in CI logs | Signature verification failed during build |
| Image digest changes without new signature | Image was modified after signing (or re-tagged) |

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Subvert Trust Controls** | [T1553](https://attack.mitre.org/techniques/T1553/) | Deploying unsigned artifacts bypasses the intended trust chain |
| **Obtain Capabilities: Code Signing Certificates** | [T1588](https://attack.mitre.org/techniques/T1588/) | Attackers may steal signing keys to create legitimately-signed malicious artifacts |

---

??? tip "SOC Relevance"

    **Alert you will see:** "Unsigned container image deployed to production cluster"

    This alert fires when an image is deployed without a valid cosign signature. In environments with enforcement, you'll see the admission controller deny it. In environments without enforcement, you need to catch it from audit logs.

    **Why this matters:**

    Without signing enforcement, an attacker who gains write access to your container registry can push any image they want. There's no cryptographic barrier between "push to registry" and "running in production." Signing + enforcement creates that barrier.

    **Triage steps:**

    1. Check who pushed the image (`crane manifest` + registry access logs)
    2. Verify if the image was ever signed (`cosign tree <image>`)
    3. Check if this is a new image or a re-tag of an existing one
    4. If the image was never signed and is running in production, treat as potential supply chain compromise
    5. If the admission controller should have blocked it, investigate why it didn't (misconfigured policy, namespace exemption, etc.)

??? example "CI Integration"

    Sign images in CI and verify before deployment.

    **`.github/workflows/sign-and-verify.yml`:**

    ```yaml
    name: Build, Sign, and Verify

    on:
      push:
        branches: [main]

    env:
      REGISTRY: ghcr.io
      IMAGE: ${{ github.repository }}

    jobs:
      build-sign:
        runs-on: ubuntu-latest
        permissions:
          contents: read
          packages: write
          id-token: write  # Required for keyless signing
        steps:
          - uses: actions/checkout@v4

          - name: Login to registry
            uses: docker/login-action@v3
            with:
              registry: ${{ env.REGISTRY }}
              username: ${{ github.actor }}
              password: ${{ secrets.GITHUB_TOKEN }}

          - name: Build and push
            id: build
            uses: docker/build-push-action@v5
            with:
              push: true
              tags: ${{ env.REGISTRY }}/${{ env.IMAGE }}:${{ github.sha }}

          - name: Install cosign
            uses: sigstore/cosign-installer@v3

          - name: Sign image (keyless)
            run: |
              cosign sign --yes \
                ${{ env.REGISTRY }}/${{ env.IMAGE }}@${{ steps.build.outputs.digest }}

          - name: Verify signature
            run: |
              cosign verify \
                --certificate-identity-regexp="https://github.com/${{ github.repository }}/" \
                --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
                ${{ env.REGISTRY }}/${{ env.IMAGE }}@${{ steps.build.outputs.digest }}
    ```

## What You Learned

1. **Signing creates a cryptographic link between a key holder and an artifact**. it proves integrity and authenticity.
2. **Unsigned images are accepted everywhere by default**. Docker, Kubernetes, and most registries don't enforce signatures.
3. **Signing without enforcement is meaningless**. if unsigned images can still be deployed, signatures are just metadata.
4. **cosign is the standard for container image signing**. it stores signatures in OCI registries alongside images.
5. **Admission policies close the enforcement gap**. use Sigstore policy-controller or Kyverno to reject unsigned images.
6. **Keyless signing (Sigstore)** ties signatures to CI identities instead of static keys, which is harder to compromise.
