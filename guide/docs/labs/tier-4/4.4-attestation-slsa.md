Signing proves who approved an artifact. Attestation proves where it came from. Build provenance answers the question: "Was this artifact built by trusted CI from the source code I reviewed, or did someone build it on their laptop and push it?"

This lab introduces SLSA (Supply-chain Levels for Software Artifacts), in-toto attestations, and the difference between "signed" and "provably built from source."

???+ info "Phase 1: UNDERSTAND. What Build Provenance Proves"

    **Goal:** Learn the difference between signatures and attestations, and understand SLSA levels.

### Step 1: Signatures vs. attestations

A **signature** says: "I approve this artifact."
An **attestation** says: "Here are machine-verifiable facts about this artifact."

The most important attestation type is **build provenance**: a signed statement describing where and how an artifact was built.

```
Signature:     "key-holder X approves artifact Y"
Attestation:   "artifact Y was built by CI system Z from source repo A at commit B"
```

### Step 2: SLSA levels

SLSA defines four levels of build provenance trust:

| Level | Requirement | What It Proves |
|-------|-------------|---------------|
| SLSA 1 | Build process exists and produces provenance | "Something built this" |
| SLSA 2 | Build runs on a hosted service, provenance is signed | "A CI system built this" |
| SLSA 3 | Build runs on a hardened, isolated platform | "A tamper-resistant CI built this" |
| SLSA 4 | Hermetic, reproducible build with two-party review | "This was built exactly as specified" |

Most organizations target SLSA 2-3. GitHub Actions with the SLSA generator achieves SLSA 3.

### Step 3: in-toto attestation format

Attestations use the [in-toto Statement](https://in-toto.io/Statement/v0.1) format:

```json
{
  "_type": "https://in-toto.io/Statement/v0.1",
  "predicateType": "https://slsa.dev/provenance/v0.2",
  "subject": [
    {
      "name": "registry:5000/weaklink-app",
      "digest": {"sha256": "abc123..."}
    }
  ],
  "predicate": {
    "builder": {"id": "https://github.com/slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@refs/tags/v1.9.0"},
    "buildType": "https://github.com/slsa-framework/slsa-github-generator/container@v1",
    "invocation": {
      "configSource": {
        "uri": "git+https://github.com/your-org/your-app@refs/heads/main",
        "digest": {"sha1": "def456..."},
        "entryPoint": ".github/workflows/build.yml"
      }
    }
  }
}
```

Key fields:

- **subject**. what artifact this attestation is about (identified by digest)
- **predicate.builder.id**. which CI system produced the build
- **predicate.invocation.configSource**. which source repo and commit triggered the build

### Step 4: Examine the images in the registry

```bash
# Image with provenance
cosign tree registry:5000/weaklink-app:attested

# Image without provenance
cosign tree registry:5000/weaklink-app:no-provenance
```

The attested image has both a signature and an attestation attached. The unattested image has neither.

???+ warning "Phase 2: BREAK. Without Provenance, Origin Is Unknown"

    **Goal:** Show that without provenance, you can't distinguish CI-built from locally-built.

### Step 1: Try to verify provenance on the unattested image

```bash
cosign verify-attestation --key /app/cosign.pub \
  --type slsaprovenance \
  registry:5000/weaklink-app:no-provenance
```

Fails. There's no attestation. You have no idea where this image came from.

### Step 2: Build an image locally and push it

```bash
# Build a modified image on the workstation (not in CI)
cat > /tmp/Dockerfile << 'EOF'
FROM alpine:3.18
RUN echo "built locally, not in CI" > /app/origin.txt
CMD ["cat", "/app/origin.txt"]
EOF

docker build -t registry:5000/weaklink-app:local-build /tmp/
docker push registry:5000/weaklink-app:local-build
```

### Step 3: Compare the two unattested images

```bash
# Both images look identical from the registry's perspective
crane manifest registry:5000/weaklink-app:no-provenance | jq '.mediaType'
crane manifest registry:5000/weaklink-app:local-build | jq '.mediaType'
```

Without provenance, these images are indistinguishable in terms of trust. Both are unsigned container images. One may have been built from reviewed source code in CI, the other was built on a developer laptop with modified code. The registry doesn't know and doesn't care.

### Step 4: The insider threat scenario

An insider with registry write access can:

1. Clone the repo
2. Add a backdoor to the source code
3. Build locally with `docker build`
4. Push to the registry with the same tag
5. The image replaces the legitimate one

Without provenance verification, there's no way to detect this. The image runs, the tag matches, and nobody questions it.

???+ success "Phase 3: DEFEND. Generate and Verify Provenance"

    **Goal:** Create build provenance attestations and verify them before deployment.

### Step 1: Create a provenance attestation

Get the image digest:

```bash
DIGEST=$(crane digest registry:5000/weaklink-app:attested)
echo "Digest: $DIGEST"
```

Create the provenance document:

```bash
cat > /app/provenance.json << EOF
{
  "_type": "https://in-toto.io/Statement/v0.1",
  "predicateType": "https://slsa.dev/provenance/v0.2",
  "subject": [
    {
      "name": "registry:5000/weaklink-app",
      "digest": {"sha256": "${DIGEST#sha256:}"}
    }
  ],
  "predicate": {
    "builder": {
      "id": "https://github.com/weaklink-labs/ci-builder"
    },
    "buildType": "https://github.com/weaklink-labs/build/v1",
    "invocation": {
      "configSource": {
        "uri": "git+https://github.com/weaklink-labs/app@refs/heads/main",
        "digest": {"sha1": "$(git rev-parse HEAD 2>/dev/null || echo 'abc123')"},
        "entryPoint": ".github/workflows/build.yml"
      }
    },
    "metadata": {
      "buildInvocationId": "run-$(date +%s)",
      "buildStartedOn": "$(date -u +%Y-%m-%dT%H:%M:%SZ)",
      "completeness": {
        "parameters": true,
        "environment": false,
        "materials": true
      }
    }
  }
}
EOF
```

### Step 2: Attach the attestation to the image

```bash
cosign attest --key /app/cosign.key \
  --predicate /app/provenance.json \
  --type slsaprovenance \
  registry:5000/weaklink-app:attested
```

### Step 3: Verify the attestation

```bash
cosign verify-attestation --key /app/cosign.pub \
  --type slsaprovenance \
  registry:5000/weaklink-app:attested
```

This returns the attestation payload. You can inspect the provenance:

```bash
cosign verify-attestation --key /app/cosign.pub \
  --type slsaprovenance \
  registry:5000/weaklink-app:attested | jq -r '.payload' | base64 -d | jq '.predicate.builder'
```

### Step 4: Verify the locally-built image has no provenance

```bash
cosign verify-attestation --key /app/cosign.pub \
  --type slsaprovenance \
  registry:5000/weaklink-app:local-build
```

Fails. No attestation. Now you can distinguish between CI-built (attested) and locally-built (unattested).

### Step 5: Create a deployment policy that requires provenance

```bash
cat > /app/provenance-policy.yaml << 'EOF'
apiVersion: policy.sigstore.dev/v1beta1
kind: ClusterImagePolicy
metadata:
  name: require-slsa-provenance
spec:
  images:
    - glob: "registry:5000/**"
  authorities:
    - key:
        data: |
          <COSIGN_PUB_KEY>
      attestations:
        - name: must-have-slsa
          predicateType: "https://slsa.dev/provenance/v0.2"
          policy:
            type: cue
            data: |
              predicate: builder: id: =~"^https://github.com/weaklink-labs/"
EOF
```

This policy requires both a valid signature AND a SLSA provenance attestation from a trusted builder.

```bash
weaklink verify 4.4
```

??? danger "Phase 4: DETECT. Catching Unattested Deployments"

    **Goal:** Detect when images without provenance reach production environments.

### SIEM / Log Indicators

**What to look for:**

- `cosign verify-attestation` failures in deployment pipeline logs
- Images deployed without `.att` tags in the OCI registry
- Provenance attestations claiming a builder identity that doesn't match the actual CI system
- Build invocation IDs in provenance that don't correlate with any CI run

### Provenance Indicators

| Indicator | What It Means |
|-----------|---------------|
| Image has `.sig` but no `.att` tag | Signed but no provenance. possibly built locally |
| Attestation builder ID doesn't match expected CI | Wrong CI system or forged attestation |
| `buildInvocationId` not found in CI system logs | Provenance claims a build that never happened |
| `configSource.uri` points to unexpected repo/branch | Image was built from unauthorized source |

| **Subvert Trust Controls** | [T1553](https://attack.mitre.org/techniques/T1553/) | Bypassing provenance requirements to deploy unattested artifacts |

---

??? tip "SOC Relevance"

    **Alert you will see:** "Image deployed without SLSA provenance attestation"

    This alert means an image reached your cluster without a verified build provenance. In high-security environments, this is a critical finding because it means you cannot verify the image's origin.

    **Why this matters more than just signing:**

    Signing proves approval. Provenance proves origin. A signed image could have been built anywhere. an attacker who steals a signing key can sign a malicious image. A provenance attestation (especially keyless/Sigstore) ties the build to a specific CI run, repo, and commit, making forgery much harder.

    **Triage steps:**

    1. Check if the image was intentionally deployed without provenance (emergency hotfix? new team?)
    2. Look up the image digest in your CI system. was it built there?
    3. If no CI build matches, treat as potential supply chain compromise
    4. Check who pushed the image to the registry (registry access logs)
    5. Compare the image contents with the latest CI-built version

??? example "CI Integration"

    Generate SLSA provenance in GitHub Actions using the official SLSA generator.

    **`.github/workflows/slsa-provenance.yml`:**

    ```yaml
    name: Build with SLSA Provenance

    on:
      push:
        branches: [main]

    jobs:
      build:
        runs-on: ubuntu-latest
        permissions:
          contents: read
          packages: write
          id-token: write
        outputs:
          digest: ${{ steps.build.outputs.digest }}
        steps:
          - uses: actions/checkout@v4

          - name: Login to GHCR
            uses: docker/login-action@v3
            with:
              registry: ghcr.io
              username: ${{ github.actor }}
              password: ${{ secrets.GITHUB_TOKEN }}

          - name: Build and push
            id: build
            uses: docker/build-push-action@v5
            with:
              push: true
              tags: ghcr.io/${{ github.repository }}:${{ github.sha }}

      provenance:
        needs: build
        permissions:
          actions: read
          id-token: write
          packages: write
        uses: slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@v1.9.0
        with:
          image: ghcr.io/${{ github.repository }}
          digest: ${{ needs.build.outputs.digest }}
          registry-username: ${{ github.actor }}
        secrets:
          registry-password: ${{ secrets.GITHUB_TOKEN }}

      verify:
        needs: [build, provenance]
        runs-on: ubuntu-latest
        steps:
          - name: Install slsa-verifier
            uses: slsa-framework/slsa-verifier/actions/installer@v2.4.0

          - name: Verify provenance
            run: |
              slsa-verifier verify-image \
                ghcr.io/${{ github.repository }}@${{ needs.build.outputs.digest }} \
                --source-uri github.com/${{ github.repository }} \
                --source-branch main
    ```

## What You Learned

1. **Signing proves approval, provenance proves origin**. they solve different problems.
2. **Without provenance, you can't tell CI-built from locally-built**. an insider can push a backdoored image and nothing flags it.
3. **SLSA defines four levels of build trust**. most organizations should target SLSA 2-3.
4. **in-toto attestations are the standard format**. they bind a signed predicate (provenance) to a subject (artifact digest).
5. **cosign attest** attaches attestations to OCI images, stored alongside signatures in the registry.
6. **Provenance policies should check builder identity**. not just "is there an attestation?" but "does the attestation claim the right builder?"
