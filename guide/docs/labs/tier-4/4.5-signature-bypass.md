!!! info "Phase 1: UNDERSTAND -- When Signing Fails to Protect"

    **Goal:** Understand the three failure modes of signature-based trust.

### Step 1: The signing trust model

Signing relies on a chain of trust:

```
Signer (private key) --> Signature --> Verifier (public key) --> Decision (accept/reject)
```

This chain breaks in three places:

| Bypass | Where It Breaks | Real-World Example |
|--------|-----------------|--------------------|
| No enforcement | Verifier step is missing | Kubernetes without admission policy |
| Key confusion | Wrong public key used for verification | Verifier trusts attacker's key |
| Signature rollback | Old valid signature applied to new artifact | Re-tagging with stale signature |

### Step 2: Check the current state

```bash
# What keys exist?
ls /app/*.pub /app/*.key 2>/dev/null

# What images are in the registry?
crane ls registry:5000/weaklink-app
```

You have the trusted key pair (`cosign.key`/`cosign.pub`) and the registry has signed and unsigned images.

### Step 3: Verify the trusted image works

```bash
cosign verify --key /app/cosign.pub registry:5000/weaklink-app:signed
```

This succeeds. The signature was made with the trusted key.

---

!!! warning "Phase 2: BREAK -- Three Bypass Techniques"

    **Goal:** Execute three distinct signature bypass attacks.

### Bypass 1: No Enforcement (the unsigned path)

The simplest bypass: just push an unsigned image.

```bash
# Push an unsigned image
cat > /tmp/Dockerfile << 'EOF'
FROM alpine:3.18
RUN echo "malicious payload" > /evil.txt
CMD ["cat", "/evil.txt"]
EOF
docker build -t registry:5000/weaklink-app:backdoor /tmp/
docker push registry:5000/weaklink-app:backdoor

# Deploy it -- does anything stop you?
kubectl run bypass1 --image=registry:5000/weaklink-app:backdoor
kubectl get pods bypass1
kubectl logs bypass1
```

Unless an admission controller is enforcing signature verification, the unsigned image runs without issue. Most clusters have no enforcement.

```bash
kubectl delete pod bypass1
```

### Bypass 2: Key Confusion (sign with attacker key)

Generate an attacker key pair and sign a malicious image:

```bash
# Generate attacker keys
cd /app && cosign generate-key-pair --output-key-prefix attacker-cosign

# Build and push a malicious image
docker build -t registry:5000/weaklink-app:attacker-signed /tmp/
docker push registry:5000/weaklink-app:attacker-signed

# Sign with the ATTACKER key
cosign sign --key /app/attacker-cosign.key registry:5000/weaklink-app:attacker-signed
```

Now verify with the ATTACKER's public key -- it passes:

```bash
cosign verify --key /app/attacker-cosign.pub registry:5000/weaklink-app:attacker-signed
```

Verification succeeds. The image is "signed." But it's signed by someone you don't trust. If a deployment script or policy uses `cosign verify` without specifying which key to trust, or if someone swaps the public key file, the attacker-signed image passes.

Verify it fails with the TRUSTED key:

```bash
cosign verify --key /app/cosign.pub registry:5000/weaklink-app:attacker-signed
```

This fails. The key matters.

### Bypass 3: Signature Rollback / Replay

This attack attempts to reuse a valid signature from one image on a different image:

```bash
# Get the signature from the legitimately signed image
SIGNED_DIGEST=$(crane digest registry:5000/weaklink-app:signed)
echo "Signed image digest: $SIGNED_DIGEST"

# Get the malicious image digest
MALICIOUS_DIGEST=$(crane digest registry:5000/weaklink-app:backdoor)
echo "Malicious image digest: $MALICIOUS_DIGEST"

# Try to copy the signature
cosign copy registry:5000/weaklink-app:signed registry:5000/weaklink-app:backdoor 2>&1 || true

# Try to verify the backdoor image with the trusted key
cosign verify --key /app/cosign.pub registry:5000/weaklink-app:backdoor
```

With cosign, this doesn't work because signatures are bound to the image digest. But in systems that use detached signatures (GPG `.asc` files) without digest binding, old signatures can be replayed on new artifacts.

### Step 4: Document the bypasses

```bash
cat > /app/bypass-report.md << 'EOF'
# Signature Bypass Report

## Bypass 1: No Enforcement
- Attack: Push unsigned image, deploy directly
- Result: Image runs without any signature check
- Root cause: No admission controller enforces verification

## Bypass 2: Key Confusion
- Attack: Sign malicious image with attacker-generated key
- Result: `cosign verify` passes when using attacker's public key
- Root cause: Verification doesn't pin to a specific trusted key

## Bypass 3: Signature Rollback
- Attack: Copy valid signature from old image to new malicious image
- Result: Fails with cosign (digest-bound) but succeeds with GPG detached sigs
- Root cause: Signatures not bound to artifact content hash

## Key Lesson
Signing is necessary but not sufficient. Enforcement + key pinning + 
digest binding are all required for signatures to provide real security.
EOF
```

!!! success "Phase 3: DEFEND -- Enforce, Pin, Timestamp"

    **Goal:** Close all three bypass vectors.

### Defense 1: Enforce verification with admission policy

```bash
cat > /app/enforce-policy.yaml << 'EOF'
apiVersion: policy.sigstore.dev/v1beta1
kind: ClusterImagePolicy
metadata:
  name: enforce-trusted-signatures
spec:
  images:
    - glob: "registry:5000/**"
  authorities:
    - key:
        data: |
          -----BEGIN PUBLIC KEY-----
          <PASTE TRUSTED cosign.pub HERE>
          -----END PUBLIC KEY-----
EOF

# Insert the actual key
KEY_DATA=$(cat /app/cosign.pub)
```

With this policy applied via Sigstore policy-controller, unsigned images and attacker-signed images are both rejected.

### Defense 2: Pin trusted keys/identities

Never use `cosign verify` without specifying the exact trusted key. In automation:

```bash
# WRONG: verify with whatever key is at this path (can be swapped)
cosign verify --key /app/cosign.pub $IMAGE

# BETTER: embed the expected key hash and verify
EXPECTED_KEY_HASH="sha256:$(sha256sum /app/cosign.pub | awk '{print $1}')"
ACTUAL_KEY_HASH="sha256:$(sha256sum /etc/cosign/trusted.pub | awk '{print $1}')"
if [ "$EXPECTED_KEY_HASH" != "$ACTUAL_KEY_HASH" ]; then
  echo "ALERT: Trusted key file has been modified!"
  exit 1
fi
```

For keyless signing (Sigstore), pin the OIDC identity:

```bash
cosign verify \
  --certificate-identity="https://github.com/your-org/your-app/.github/workflows/build.yml@refs/heads/main" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
  $IMAGE
```

This is immune to key confusion because the identity is tied to a specific GitHub Actions workflow.

### Defense 3: Include timestamps and expiry

For time-bound trust, use Sigstore's transparency log (Rekor):

```bash
# Signatures are automatically logged to Rekor with timestamps
cosign verify --key /app/cosign.pub registry:5000/weaklink-app:signed | jq '.[].optional.Bundle.Payload.integratedTime'
```

This prevents replay attacks because each signature has a timestamp. You can reject signatures older than your policy allows.

!!! danger "Phase 4: DETECT -- Identifying Signature Bypass Attempts"

    **Goal:** Detect each bypass technique in logs and monitoring.

### SIEM / Log Indicators

**What to look for:**

- Deployment of images without any cosign signature (no `.sig` tag in registry)
- Signature verification using unexpected public keys (key hash mismatch)
- Rapid key pair generation followed by image signing (attacker creating keys on the fly)
- Signature timestamps that don't align with CI build times

### Attack Indicators

| Indicator | Bypass Type | What It Means |
|-----------|-------------|---------------|
| Image deployed, no `.sig` tag exists | No enforcement | Signing requirement is not enforced |
| `cosign verify` succeeds with unknown key fingerprint | Key confusion | Someone signed with an untrusted key |
| Signature timestamp >> image push timestamp | Rollback | Old signature copied to new image |
| `cosign generate-key-pair` in workstation logs | Key confusion prep | Someone generating keys to forge signatures |

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Subvert Trust Controls** | [T1553](https://attack.mitre.org/techniques/T1553/) | All three bypasses subvert the intended signature-based trust model |
| **Masquerading** | [T1036](https://attack.mitre.org/techniques/T1036/) | Attacker signs malicious artifacts to make them appear legitimate |

!!! tip "SOC Relevance"

    **Alert you will see:** "Image signature verification succeeded with unknown key"

    This is the most dangerous variant. The image IS signed. Verification DID pass. But the key doesn't belong to your organization. If your verification pipeline checks "is it signed?" but not "is it signed by US?", an attacker can sign anything with their own key and it passes.

    **This is more subtle than unsigned images:** an unsigned image is obviously suspicious. A signed-with-wrong-key image looks legitimate.

    **Triage steps:**

    1. Extract the public key used for the signature
    2. Compare against your inventory of trusted signing keys
    3. If the key is unknown, treat as potential supply chain compromise
    4. Check who pushed the image to the registry
    5. Check for `cosign generate-key-pair` in recent process logs on that host
    6. If using keyless signing, check the OIDC identity in the certificate -- does it match your CI?

!!! example "CI Integration"

    Enforce key-pinned verification in CI before deployment.

    **`.github/workflows/verify-before-deploy.yml`:**

    ```yaml
    name: Verify Signatures Before Deploy

    on:
      workflow_dispatch:
        inputs:
          image:
            description: "Image to deploy"
            required: true

    jobs:
      verify-and-deploy:
        runs-on: ubuntu-latest
        permissions:
          id-token: write
        steps:
          - name: Install cosign
            uses: sigstore/cosign-installer@v3

          - name: Verify signature (keyless, identity-pinned)
            run: |
              cosign verify \
                --certificate-identity="https://github.com/${{ github.repository }}/.github/workflows/build.yml@refs/heads/main" \
                --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
                ${{ inputs.image }}

              if [ $? -ne 0 ]; then
                echo "::error::Signature verification failed. Image was not signed by trusted CI."
                exit 1
              fi

          - name: Check signature age
            run: |
              SIGNED_TIME=$(cosign verify \
                --certificate-identity="https://github.com/${{ github.repository }}/.github/workflows/build.yml@refs/heads/main" \
                --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
                ${{ inputs.image }} 2>/dev/null | jq -r '.[0].optional.Bundle.Payload.integratedTime')

              NOW=$(date +%s)
              AGE_HOURS=$(( (NOW - SIGNED_TIME) / 3600 ))

              if [ "$AGE_HOURS" -gt 168 ]; then
                echo "::warning::Signature is $AGE_HOURS hours old (>7 days). Consider rebuilding."
              fi
              echo "Signature age: ${AGE_HOURS} hours"

          - name: Deploy (only if verification passed)
            run: |
              kubectl set image deployment/app app=${{ inputs.image }}
    ```

## What You Learned

1. **No enforcement = no security** -- signing without verification is like locking a door but leaving it open.
2. **Key confusion is the most dangerous bypass** -- a signed artifact isn't safe unless you verify it was signed by the right entity.
3. **Signature rollback fails with digest-bound signatures** -- cosign binds signatures to image digests, preventing replay attacks on container images.
4. **Pin trusted keys or OIDC identities** -- never just check "is it signed?" Always check "is it signed by WHO we trust?"
5. **Keyless signing eliminates key theft risk** -- Sigstore ties signatures to CI identities via OIDC, so there's no static key to steal.
6. **Transparency logs prevent backdating** -- Rekor timestamps prove when a signature was created.
