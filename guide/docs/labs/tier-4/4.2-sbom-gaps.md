!!! info "Phase 1: UNDERSTAND -- Different Tools, Different Results"

    **Goal:** Run multiple SBOM generators on the same image and see how their outputs differ.

### Step 1: Pull the target image

```bash
crane pull registry:5000/weaklink-app:vulnerable /tmp/vuln-image.tar
```

### Step 2: Generate SBOMs with three different tools

```bash
# Tool 1: syft
syft registry:5000/weaklink-app:vulnerable -o cyclonedx-json > /app/sbom-syft.json

# Tool 2: trivy
trivy image --format cyclonedx registry:5000/weaklink-app:vulnerable > /app/sbom-trivy.json

# Tool 3: cdxgen
cdxgen -t docker -o /app/sbom-cdxgen.json registry:5000/weaklink-app:vulnerable
```

### Step 3: Compare component counts

```bash
echo "syft:   $(jq '.components | length' /app/sbom-syft.json) components"
echo "trivy:  $(jq '.components | length' /app/sbom-trivy.json) components"
echo "cdxgen: $(jq '.components | length' /app/sbom-cdxgen.json) components"
```

The numbers will differ. Each tool uses different strategies:

| Tool | Strategy | Strengths | Blind Spots |
|------|----------|-----------|-------------|
| syft | Scans package manager databases (dpkg, rpm, pip, npm, etc.) | Broad language support | Misses vendored/compiled code |
| trivy | Package managers + OS package DBs + advisory matching | Good OS-level coverage | Misses non-standard layouts |
| cdxgen | Language-specific manifest parsing | Deep language support | Weaker on OS packages |

### Step 4: Find the differences

```bash
# Extract names from each
jq -r '.components[].name' /app/sbom-syft.json | sort -u > /tmp/names-syft.txt
jq -r '.components[].name' /app/sbom-trivy.json | sort -u > /tmp/names-trivy.txt
jq -r '.components[].name' /app/sbom-cdxgen.json | sort -u > /tmp/names-cdxgen.txt

# What's only in syft?
echo "=== Only in syft ==="
comm -23 /tmp/names-syft.txt /tmp/names-trivy.txt

# What's only in trivy?
echo "=== Only in trivy ==="
comm -13 /tmp/names-syft.txt /tmp/names-trivy.txt

# What's in none of the three? (You can't see this from SBOM alone)
echo "=== Components all three agree on ==="
comm -12 /tmp/names-syft.txt /tmp/names-trivy.txt | comm -12 - /tmp/names-cdxgen.txt | wc -l
```

There is no single source of truth. Using only one tool gives an incomplete picture.

!!! warning "Phase 2: BREAK -- The Invisible CVE"

    **Goal:** A vendored C library with a known CVE exists in the image. No SBOM tool finds it.

### Step 1: Look inside the container

```bash
# Start the container and explore
docker run --rm -it registry:5000/weaklink-app:vulnerable sh

# Inside the container:
ls /app/vendor/
strings /app/vendor/libxml2.so | grep -i "LIBXML"
strings /app/vendor/libxml2.so | grep "2\." | head -5
```

There's a vendored `libxml2.so` compiled into the application's vendor directory. The version string reveals it's an old version with known CVEs.

### Step 2: Verify SBOMs missed it

```bash
# Search all three SBOMs
echo "syft:"; jq '.components[] | select(.name | test("xml"; "i"))' /app/sbom-syft.json
echo "trivy:"; jq '.components[] | select(.name | test("xml"; "i"))' /app/sbom-trivy.json
echo "cdxgen:"; jq '.components[] | select(.name | test("xml"; "i"))' /app/sbom-cdxgen.json
```

All three return nothing. The vendored library is invisible because:

1. It's not installed via a package manager (no `dpkg` or `rpm` entry)
2. It's a compiled `.so` file, not a source package
3. There's no manifest declaring it as a dependency
4. SBOM tools scan metadata, not binary contents

### Step 3: Understand the real-world impact

This is exactly how the XZ Utils backdoor worked. A malicious change was compiled into a shared library. No SBOM tool, dependency scanner, or package manager audit would have found it by scanning manifests. The backdoor was in the compiled binary.

If someone asks "is libxml2 vulnerable?" and you check the SBOM, the answer is "we don't use libxml2." But you do -- it's just invisible.

### Step 4: Calculate the false negative rate

```bash
# Total packages in the container (from dpkg + pip + vendored)
docker run --rm registry:5000/weaklink-app:vulnerable sh -c \
  "dpkg-query -f '\${Package}\n' -W | wc -l; pip list --format=freeze 2>/dev/null | wc -l; ls /app/vendor/*.so 2>/dev/null | wc -l"

# vs. what the best SBOM found
echo "Best SBOM coverage: $(jq '.components | length' /app/sbom-trivy.json) / TOTAL"
```

The gap is your blind spot.

!!! success "Phase 3: DEFEND -- Layered Detection"

    **Goal:** Complement SBOMs with vulnerability scanning and binary analysis.

### Step 1: Run a vulnerability scanner directly on the image

```bash
grype registry:5000/weaklink-app:vulnerable -o table > /app/vuln-scan.txt
cat /app/vuln-scan.txt
```

Grype scans the actual filesystem, not just package metadata. It may find additional vulnerabilities that the SBOM-based approach missed.

### Step 2: Run Trivy in vulnerability mode (not SBOM mode)

```bash
trivy image registry:5000/weaklink-app:vulnerable --severity HIGH,CRITICAL
```

Compare what Trivy finds in vulnerability mode vs. what it included in the SBOM. The vulnerability scanner has heuristics beyond package manager metadata.

### Step 3: Manual binary analysis for vendored components

For compiled libraries, you need to extract version information manually:

```bash
# Check the vendored library version
docker run --rm registry:5000/weaklink-app:vulnerable \
  strings /app/vendor/libxml2.so | grep -E "^[0-9]+\.[0-9]+\.[0-9]+"

# Cross-reference against NVD
# (In production, you'd use an API call to NVD or OSV)
echo "Check: https://nvd.nist.gov/vuln/search?query=libxml2+<VERSION>"
```

### Step 4: Write the comparison report

```bash
cat > /app/comparison.md << 'REPORT'
# SBOM Tool Comparison Report

## Image: weaklink-app:vulnerable

### Component Coverage
| Tool | Components Found | Vendored Found | OS Packages |
|------|-----------------|----------------|-------------|
| syft | XX | 0 | XX |
| trivy | XX | 0 | XX |
| cdxgen | XX | 0 | XX |
| grype (vuln scan) | N/A | N/A | N/A |

### Key Finding
All three SBOM generators missed vendored libxml2 in /app/vendor/.
The vulnerability scanner (grype) also missed it because it relies
on the same package metadata.

### Recommendation
1. Use multiple SBOM generators and merge results
2. Run vulnerability scanners directly on artifacts (not on SBOMs)
3. Maintain a manual inventory of vendored/compiled dependencies
4. Never claim "complete visibility" based on SBOM alone
REPORT
```

### Step 5: Write the gap analysis

```bash
cat > /app/gap-analysis.md << 'ANALYSIS'
# SBOM Gap Analysis

## Missed: vendored libxml2 (CVE-2022-40303, CVE-2022-40304)
- syft: missed -- no package manager metadata
- trivy: missed -- no dpkg/rpm entry
- cdxgen: missed -- not in any manifest file

## Root Cause
The library was compiled from source and placed in /app/vendor/.
No SBOM tool performs binary analysis or version extraction from
compiled shared objects.

## Impact
Compliance reports based on this SBOM would incorrectly state
that the application has no libxml2 dependency, hiding known
critical CVEs from security review.

## Remediation
- Add vendored components to SBOM manually or via enrichment tooling
- Run binary composition analysis (e.g., OWASP dependency-track + manual)
- Flag any /vendor/ or /third-party/ directories for manual review
ANALYSIS
```

!!! danger "Phase 4: DETECT -- Catching SBOM Gaps Before They Become Blind Spots"

    **Goal:** Build detection for incomplete SBOMs and vendored dependency drift.

### SIEM / Log Indicators

**What to look for:**

- Container images with vendor directories (`/vendor/`, `/third-party/`, `/lib/`) that contain `.so`, `.a`, or `.dll` files not represented in the SBOM
- SBOM component count significantly lower than the actual package count inside the container
- Vulnerability scan findings for CVEs in packages that don't appear in the SBOM
- SBOM generation logs showing warnings about unrecognized file types

### Filesystem Indicators

| Indicator | What It Means |
|-----------|---------------|
| `/vendor/*.so` or `/third-party/*.a` in container | Compiled vendored dependencies likely missing from SBOM |
| `strings` output contains version identifiers | Can extract version info for manual CVE matching |
| SBOM has 0 components of type "library" (native) | Tool only found package manager deps, missed native code |
| `find / -name "*.so" | wc -l` >> SBOM library count | Significant gap between actual and declared libraries |

### Example Detection Queries

**Splunk -- SBOM vs. scanner drift (vulns in components not in SBOM):**

```spl
index=vuln_scans sourcetype=grype_results
| join type=left package [
    search index=sbom_registry sourcetype=cyclonedx
    | rename name AS package
    | dedup package
    | eval in_sbom="yes"
  ]
| where isnull(in_sbom)
| stats count by image, package, vulnerability, severity
| where severity IN ("Critical", "High")
| sort -severity, -count
```

**KQL (Microsoft Sentinel) -- detect vendored binaries in container images:**

```kql
ContainerFileScan_CL
| where file_path_s matches regex @"/(vendor|third.party|lib)/.*\.(so|a|dll)$"
| join kind=leftanti (
    SBOMComponents_CL
    | project component_name_s
) on $left.file_name_s == $right.component_name_s
| summarize UnmatchedFiles=count() by image_name_s, file_path_s
| where UnmatchedFiles > 0
```

**Splunk -- SBOM freshness check:**

```spl
index=sbom_registry sourcetype=cyclonedx
| stats max(sbom_timestamp) AS last_sbom by image_name
| join image_name [
    search index=container_builds
    | stats max(build_timestamp) AS last_build by image_name
  ]
| eval hours_stale=round((last_build - last_sbom) / 3600, 1)
| where hours_stale > 24
| table image_name, last_sbom, last_build, hours_stale
```

!!! tip "SOC Relevance"

    **Alert you will see:** "Vulnerability found in component not present in SBOM"

    This is a critical signal that your SBOM is incomplete. Most organizations treat the SBOM as the source of truth for compliance -- if the SBOM says "no log4j," the compliance check passes. But the actual container may still contain log4j as a vendored JAR, a shaded dependency, or a transitive inclusion that the SBOM tool missed.

    **Triage steps:**

    1. Compare the vulnerability scanner's finding against all three SBOM sources
    2. If no SBOM tool found the component, check if it's vendored (`find /vendor/ /third-party/ /lib/`)
    3. Verify the CVE is real by checking the actual binary version
    4. Add the component to the SBOM manually and update the enrichment process
    5. Flag the SBOM generation pipeline for review -- why did the tool miss this?

!!! example "CI Integration"

    Run multi-tool SBOM generation and cross-check against vulnerability scanners in CI.

    **`.github/workflows/sbom-gap-check.yml`:**

    ```yaml
    name: SBOM Gap Detection

    on:
      push:
        branches: [main]

    jobs:
      sbom-gap-check:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Build image
            run: docker build -t app:${{ github.sha }} .

          - name: Generate SBOMs
            run: |
              # Install tools
              curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin
              curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh -s -- -b /usr/local/bin

              # Generate SBOM
              syft app:${{ github.sha }} -o cyclonedx-json > sbom.json
              SBOM_COUNT=$(jq '.components | length' sbom.json)
              echo "SBOM components: $SBOM_COUNT"

          - name: Vulnerability scan (direct, not SBOM-based)
            run: |
              grype app:${{ github.sha }} -o json > vuln-direct.json
              DIRECT_PKGS=$(jq '[.matches[].artifact.name] | unique | length' vuln-direct.json)
              echo "Packages with vulns (direct scan): $DIRECT_PKGS"

          - name: Check for vendored binaries
            run: |
              VENDOR_LIBS=$(docker run --rm app:${{ github.sha }} \
                find / -path /proc -prune -o -path /sys -prune -o \
                -name "*.so" -print 2>/dev/null | wc -l)
              echo "Shared libraries in image: $VENDOR_LIBS"

              if [ "$VENDOR_LIBS" -gt 50 ]; then
                echo "::warning::Image contains $VENDOR_LIBS .so files -- verify SBOM covers vendored libraries"
              fi

          - name: Cross-check SBOM vs. direct scan
            run: |
              # Packages found by vuln scanner but not in SBOM
              jq -r '[.matches[].artifact.name] | unique[]' vuln-direct.json | sort > /tmp/vuln-pkgs.txt
              jq -r '.components[].name' sbom.json | sort -u > /tmp/sbom-pkgs.txt
              GAPS=$(comm -23 /tmp/vuln-pkgs.txt /tmp/sbom-pkgs.txt | wc -l)
              echo "Packages in vuln scan but NOT in SBOM: $GAPS"
              if [ "$GAPS" -gt 0 ]; then
                echo "::warning::$GAPS packages found by vulnerability scanner are missing from SBOM"
                comm -23 /tmp/vuln-pkgs.txt /tmp/sbom-pkgs.txt
              fi
    ```

## What You Learned

1. **No SBOM tool finds everything** -- running syft, trivy, and cdxgen on the same image produces different component lists.
2. **Vendored compiled code is invisible to SBOM tools** -- if a `.so` file was copied into the image, no tool will detect it from package metadata alone.
3. **The gap between "indexed" and "complete" is where real attacks hide** -- XZ Utils, SolarWinds, and many supply chain attacks exploited components that wouldn't show up in an SBOM.
4. **Use multiple tools and merge results** -- this reduces blind spots but never eliminates them.
5. **Complement SBOMs with direct vulnerability scanning** -- scan the artifact itself, not just the SBOM.
6. **Maintain a manual inventory for vendored code** -- automate what you can, document the rest.
