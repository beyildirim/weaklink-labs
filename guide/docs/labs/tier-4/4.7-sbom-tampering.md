In [Lab 4.1](4.1-sbom-contents.md) you learned what SBOMs contain and what they miss. In [Lab 4.3](4.3-signing-fundamentals.md) you learned why unsigned artifacts can be silently replaced. Now combine those lessons: an SBOM is a JSON file. Without cryptographic signing, anyone with write access can edit it. remove a vulnerable component, change a version number, or delete an entire dependency tree. The modified SBOM will pass schema validation and compliance checks without raising an alert.

This lab shows you how trivial SBOM tampering is, how it defeats compliance gates, and how to prevent it with signing and attestation.

???+ info "Phase 1: UNDERSTAND. SBOMs Are Just JSON Files"

    **Goal:** See that SBOMs have no built-in integrity protection and can be modified with any text editor.

### Step 1: Generate an SBOM for the application

```bash
syft /app -o cyclonedx-json > /app/sbom.json
```

### Step 2: Inspect the structure

```bash
cat /app/sbom.json | jq '{
  bomFormat: .bomFormat,
  specVersion: .specVersion,
  serialNumber: .serialNumber,
  components_count: (.components | length)
}'
```

A CycloneDX SBOM is a JSON document with a `components` array. Each component has a name, version, purl, and optional hashes. There is no signature, no MAC, no integrity field.

### Step 3: Examine a vulnerable component

```bash
# Run grype against the SBOM to find vulnerabilities
grype sbom:/app/sbom.json --output table
```

Note a component with a known critical CVE. Record its name and version.

### Step 4: Understand the compliance flow

In many organizations, the SBOM flows through this pipeline:

1. **CI builds** the artifact and generates an SBOM
2. **SBOM is stored** in an artifact repository or attached to the container image
3. **Compliance tool** reads the SBOM and checks for banned licenses, known CVEs, or missing metadata
4. **Deployment gate** allows or blocks deployment based on the compliance result

If the SBOM is tampered with between steps 2 and 3, the compliance tool operates on a lie.

???+ warning "Phase 2: BREAK. Remove a Vulnerable Component and Pass Compliance"

    **Goal:** Edit the SBOM to remove a component with a critical CVE, then show the compliance check passes on the tampered SBOM.

### Step 1: Identify the target component

```bash
# Find the component with the worst CVE
grype sbom:/app/sbom.json --output json | jq -r '
  .matches[]
  | select(.vulnerability.severity == "Critical")
  | "\(.artifact.name) \(.artifact.version) - \(.vulnerability.id)"
' | head -5
```

Pick one. For this example, assume `requests` version `2.25.0` has a critical CVE.

### Step 2: Create the tampered SBOM

```bash
# Remove the vulnerable component
cat /app/sbom.json | jq '
  .components = [.components[] | select(.name != "requests")]
' > /app/sbom-tampered.json

# Verify it was removed
echo "Original components: $(jq '.components | length' /app/sbom.json)"
echo "Tampered components: $(jq '.components | length' /app/sbom-tampered.json)"
```

One line of `jq`. That's all it takes.

### Step 3: Run the compliance check on the tampered SBOM

```bash
# Original SBOM -- should show the CVE
echo "=== Original SBOM ==="
grype sbom:/app/sbom.json --output table --only-fixed 2>&1 | head -20

# Tampered SBOM -- CVE is gone
echo "=== Tampered SBOM ==="
grype sbom:/app/sbom-tampered.json --output table --only-fixed 2>&1 | head -20
```

The tampered SBOM produces a clean compliance report. The vulnerability still exists in the actual artifact, but the SBOM no longer mentions the component.

### Step 4: Validate that the tampered SBOM is structurally valid

```bash
# CycloneDX schema validation
cat /app/sbom-tampered.json | jq '.bomFormat, .specVersion, .serialNumber' 
```

The tampered SBOM passes schema validation. It has the correct format, spec version, and serial number. Nothing in the CycloneDX or SPDX specification prevents you from omitting components.

### Step 5: Version number tampering

Removal is obvious if someone compares component counts. A subtler attack is changing the version:

```bash
# Change the version of requests from 2.25.0 to 2.31.0 (patched version)
cat /app/sbom.json | jq '
  .components = [.components[] |
    if .name == "requests" and .version == "2.25.0"
    then .version = "2.31.0" | .purl = (.purl | gsub("2.25.0"; "2.31.0"))
    else .
    end
  ]
' > /app/sbom-version-tampered.json

# The CVE for 2.25.0 no longer matches
grype sbom:/app/sbom-version-tampered.json --output table 2>&1 | grep -i requests || echo "No findings for requests"
```

The component is still in the SBOM, but with a safe version number. The actual deployed artifact still runs `requests==2.25.0`.

### Step 6: Document the attack

```bash
cat > /app/findings.txt << 'EOF'
SBOM Tampering Attack
======================
Method 1: Component removal
  - Removed "requests" from components array
  - Compliance check (grype) no longer reports the CVE
  - SBOM passes schema validation

Method 2: Version number change
  - Changed requests version from 2.25.0 to 2.31.0
  - CVE no longer matches (vuln databases match on name+version)
  - SBOM still has the correct number of components

Root cause: SBOMs are unsigned JSON files with no integrity verification.
Any write access = full control over what compliance tools see.
EOF
```

???+ success "Phase 3: DEFEND. Sign SBOMs and Generate Them in CI"

    **Goal:** Make SBOM tampering detectable by signing SBOMs and binding them to the artifact they describe.

### Defense 1: Sign the SBOM with cosign

```bash
# Generate a keypair (or use keyless signing in CI)
cosign generate-key-pair

# Sign the SBOM file
cosign sign-blob --key cosign.key /app/sbom.json --output-signature /app/sbom.json.sig
```

Before consuming the SBOM, verify the signature:

```bash
cosign verify-blob --key cosign.pub --signature /app/sbom.json.sig /app/sbom.json
```

If the SBOM was modified after signing, verification fails.

### Defense 2: Attach the SBOM to the container image

```bash
# Attach the SBOM as an OCI artifact
cosign attach sbom --sbom /app/sbom.json registry:5000/webapp:latest

# Verify the attached SBOM
cosign verify-attestation --key cosign.pub registry:5000/webapp:latest --type cyclonedx
```

When the SBOM is attached to the image and signed, tampering with the SBOM invalidates the signature on the image.

### Defense 3: Generate SBOMs in CI, never manually

The SBOM should be generated in the same CI pipeline that builds the artifact, signed immediately, and published alongside the artifact. Never accept SBOMs that were generated locally or submitted separately.

```yaml
# In your CI pipeline:
- name: Generate SBOM
  run: syft $IMAGE -o cyclonedx-json > sbom.json

- name: Sign and attach SBOM
  run: |
    cosign attest --predicate sbom.json --type cyclonedx $IMAGE
```

### Defense 4: Cross-validate SBOM against the artifact

Don't just trust the SBOM. verify it:

```bash
# Generate a fresh SBOM from the actual image
syft registry:5000/webapp:latest -o cyclonedx-json > /app/sbom-fresh.json

# Compare component lists
diff <(jq -r '.components[].purl' /app/sbom.json | sort) \
     <(jq -r '.components[].purl' /app/sbom-fresh.json | sort)
```

If the stored SBOM and a freshly generated one disagree, the stored SBOM was tampered with or is stale.

### Defense 5: Use SBOM integrity fields

CycloneDX 1.5+ supports a `signature` field within the SBOM itself:

```json
{
  "bomFormat": "CycloneDX",
  "specVersion": "1.5",
  "signature": {
    "algorithm": "ES256",
    "value": "..."
  }
}
```

This embeds the signature directly in the SBOM document. Tools that support CycloneDX 1.5+ can verify integrity without needing an external `.sig` file.

### Step 6: Verify the lab

Run the verification from your host terminal:

```bash
weaklink verify 4.7
```

??? danger "Phase 4: DETECT. Spotting Tampered SBOMs in Your Organization"

    **Goal:** Detect when SBOMs have been modified after generation, or when SBOM contents don't match the actual artifact.

### SIEM / Log Indicators

The primary signal is **drift between the SBOM and the actual artifact**. This shows up when a vulnerability scanner finds CVEs that the SBOM says should not exist, or when component counts don't match.

**What to look for:**

- SBOM signature verification failures
- Grype/Trivy findings for components not listed in the SBOM (or listed at a different version)
- SBOMs with suspiciously few components for the application type (e.g., 3 components for a Node.js app)
- SBOM generation timestamps that don't correlate with CI build timestamps
- SBOMs modified after initial creation (file modification time differs from CI completion time)

### Drift Detection

| Indicator | What It Means |
|-----------|---------------|
| Vulnerability scanner finds CVE in `requests==2.25.0` but SBOM lists `requests==2.31.0` | Version was tampered. SBOM claims patched version but actual is vulnerable |
| SBOM lists 15 components but `pip list` in container shows 22 | Components were removed from the SBOM |
| SBOM signature verification fails | SBOM was modified after signing |
| SBOM `serialNumber` or `metadata.timestamp` doesn't match CI pipeline records | SBOM may have been regenerated or replaced |

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Supply Chain Compromise: Compromise Software Supply Chain** | [T1195.002](https://attack.mitre.org/techniques/T1195/002/) | Tampering with an SBOM allows a compromised dependency to pass compliance checks undetected |
| **Indicator Removal** | [T1070](https://attack.mitre.org/techniques/T1070/) | Removing a component from an SBOM is a form of evidence tampering. removing the indicator of a known vulnerability |

---

??? tip "SOC Relevance"

    **Alert you will see:** "SBOM signature verification failed" or "Vulnerability found for component at different version than SBOM declares"

    SBOM tampering is the supply chain equivalent of editing audit logs. The attacker doesn't change the artifact. they change the documentation that compliance tools rely on. This is especially dangerous in regulated environments where deployment gates depend on SBOM-based vulnerability checks.

    **Triage steps:**

    1. Verify the SBOM signature. if it fails, the SBOM was modified after signing
    2. Regenerate the SBOM from the actual artifact and compare against the stored version
    3. Check the SBOM's metadata timestamp against the CI pipeline completion time
    4. If component counts differ, identify which components were added or removed
    5. If version numbers differ, check the actual installed version in the container (`pip show`, `npm list`, `dpkg -l`)
    6. If tampering is confirmed, treat it as an integrity incident. quarantine the artifact, audit who had write access to the SBOM storage, and check for similar tampering across other artifacts

??? example "CI Integration"

    Add this to your pipeline to sign SBOMs at generation time and detect tampering before deployment.

    **`.github/workflows/sbom-integrity.yml`:**

    ```yaml
    name: SBOM Integrity Pipeline

    on:
      push:
        branches: [main]

    jobs:
      generate-and-sign-sbom:
        runs-on: ubuntu-latest
        permissions:
          id-token: write  # Required for keyless signing
          contents: read
        steps:
          - uses: actions/checkout@v4

          - name: Install tools
            run: |
              curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin
              curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh -s -- -b /usr/local/bin
              curl -sL https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64 -o /usr/local/bin/cosign
              chmod +x /usr/local/bin/cosign

          - name: Build and push image
            run: |
              docker build -t ${{ vars.REGISTRY }}/webapp:${{ github.sha }} .
              docker push ${{ vars.REGISTRY }}/webapp:${{ github.sha }}

          - name: Generate SBOM from built image
            run: |
              syft ${{ vars.REGISTRY }}/webapp:${{ github.sha }} -o cyclonedx-json > sbom.json
              echo "Components: $(jq '.components | length' sbom.json)"

          - name: Sign and attach SBOM as attestation
            run: |
              cosign attest --predicate sbom.json \
                --type cyclonedx \
                ${{ vars.REGISTRY }}/webapp:${{ github.sha }}

          - name: Verify SBOM attestation
            run: |
              cosign verify-attestation \
                --certificate-oidc-issuer https://token.actions.githubusercontent.com \
                --certificate-identity-regexp "https://github.com/${{ github.repository }}/" \
                --type cyclonedx \
                ${{ vars.REGISTRY }}/webapp:${{ github.sha }}

          - name: Cross-validate SBOM against vulnerability scan
            run: |
              # Scan the actual image
              grype ${{ vars.REGISTRY }}/webapp:${{ github.sha }} --output json > scan-results.json
              
              # Extract vulnerable component names from scan
              SCAN_PKGS=$(jq -r '.matches[].artifact.name' scan-results.json | sort -u)
              
              # Verify each vulnerable component exists in the SBOM
              for pkg in $SCAN_PKGS; do
                if ! jq -e ".components[] | select(.name == \"$pkg\")" sbom.json > /dev/null 2>&1; then
                  echo "::error::Vulnerable component '$pkg' found by scanner but missing from SBOM"
                fi
              done

          - name: Upload SBOM
            uses: actions/upload-artifact@v4
            with:
              name: sbom-signed
              path: sbom.json
    ```

## What You Learned

1. **SBOMs are plain JSON/XML with no built-in integrity**. a single `jq` command can remove a component or change a version number.
2. **Tampered SBOMs pass schema validation**. the CycloneDX and SPDX schemas do not enforce completeness. A valid SBOM with zero components is still schema-valid.
3. **Compliance gates that trust unsigned SBOMs are trivially bypassed**. remove the vulnerable component, pass the check, deploy the vulnerability.
4. **Version tampering is subtler than removal**. changing `2.25.0` to `2.31.0` keeps the component count identical but eliminates the CVE match.
5. **Signing is the minimum defense**. cosign or in-SBOM signatures make post-generation tampering detectable.
6. **Cross-validation catches what signing misses**. even a legitimately signed SBOM could be incomplete. Always compare the SBOM against a fresh scan of the actual artifact.
