# Lab 4.1: What SBOMs Actually Contain

<div class="lab-meta">
  <span>~30 minutes</span>

  <span>Prerequisites: <a href="../tier-1/1.1-dependency-resolution.md">Lab 1.1</a></span>

An SBOM (Software Bill of Materials) is supposed to be a complete inventory of everything inside your software. In practice, it's a snapshot of what one tool could see at one point in time. This lab shows you exactly what gets captured. and what doesn't.

You'll generate SBOMs in both major formats (SPDX and CycloneDX), inspect their contents, and discover the blind spots that create a false sense of security.

???+ info "Phase 1: UNDERSTAND. What Goes Into an SBOM"

    **Goal:** Generate SBOMs in two formats and understand what fields they capture.

### Step 1: Look at the application

```bash
ls /app/
cat /app/requirements.txt
cat /app/src/main.py
```

This is a Python web application with dependencies declared in `requirements.txt`. It also has a `/app/vendor/` directory with a compiled C library (`libcurl.so`) and dynamically loads a plugin at runtime.

### Step 2: Generate an SPDX SBOM

```bash
syft /app -o spdx-json > /app/sbom-spdx.json
```

Inspect it:

```bash
cat /app/sbom-spdx.json | jq '.packages | length'
cat /app/sbom-spdx.json | jq '.packages[] | {name: .name, version: .versionInfo, license: .licenseConcluded}'
```

You should see the Python packages listed with their versions and license information.

### Step 3: Generate a CycloneDX SBOM

```bash
syft /app -o cyclonedx-json > /app/sbom-cdx.json
```

Inspect it:

```bash
cat /app/sbom-cdx.json | jq '.components | length'
cat /app/sbom-cdx.json | jq '.components[] | {name, version, type, purl}'
```

### Step 4: Compare the two formats

Both describe the same software, but the structure differs:

| Field | SPDX | CycloneDX |
|-------|------|-----------|
| Package identity | `SPDXID` + `name` | `bom-ref` + `purl` |
| Version | `versionInfo` | `version` |
| License | `licenseConcluded` | `licenses[]` |
| Hashes | `checksums[]` | `hashes[]` |
| Relationships | `relationships[]` | `dependencies[]` |

The data is similar but not identical. Some tools only consume SPDX, others only CycloneDX. You need to know which format your downstream tools expect.

### Step 5: What IS captured

For each package, the SBOM captured:

- **Name and version**. exactly what's installed
- **Package URL (purl)**. a universal identifier (`pkg:pypi/flask@2.3.0`)
- **License**. what the package declares
- **Hashes**. SHA256 of the package archive
- **Supplier**. who published the package (when available)

This is useful for license compliance, vulnerability matching, and inventory tracking.

???+ warning "Phase 2: BREAK. What SBOMs Miss"

    **Goal:** Discover the blind spots. SBOMs are not complete inventories.

### Step 1: Check the vendored C library

```bash
ls -la /app/vendor/
strings /app/vendor/libcurl.so | grep -i "libcurl/"
```

There's a compiled `libcurl.so` in the vendor directory. Now check if the SBOM found it:

```bash
cat /app/sbom-cdx.json | jq '.components[] | select(.name | test("curl"; "i"))'
```

Nothing. The SBOM tool scanned package manager manifests (`requirements.txt`, `pip` metadata) but it does not analyze compiled binaries. The vendored library is invisible.

### Step 2: Check for dynamically loaded modules

```bash
grep -r "importlib\|__import__\|load_module\|dlopen" /app/src/
```

The application uses `importlib.import_module()` to load a plugin at runtime. This plugin is not in `requirements.txt` and won't appear in any SBOM because the dependency is only known at runtime.

### Step 3: Check for build-time dependencies

```bash
cat /app/Makefile
```

The Makefile shows `gcc`, `make`, and `cmake` were used to compile the vendored library. None of these build tools appear in the SBOM. If any of them were compromised (like the XZ Utils attack), the SBOM would give you zero signal.

### Step 4: Check for Helm/infrastructure references

```bash
cat /app/deploy/values.yaml 2>/dev/null
```

If the application is deployed via Helm, the chart references, base images, and sidecar containers are not captured in the application SBOM.

### Step 5: Quantify the gap

You now have a concrete list of what's missing:

| Component | In SBOM? | Why Not? |
|-----------|----------|----------|
| Python packages (Flask, requests, etc.) | Yes | Declared in requirements.txt |
| vendored libcurl 7.79.0 | **No** | Compiled binary, no package manager metadata |
| Runtime plugin (loaded via importlib) | **No** | Dynamic dependency, not declared anywhere |
| Build tools (gcc, cmake) | **No** | Not runtime dependencies |
| Base container image packages | **No** | Syft scanned /app, not the full image |
| Helm chart references | **No** | Infrastructure, not application code |

An SBOM that lists 12 packages when the real count is 18+ creates a **false sense of security**. Anyone who says "we have an SBOM, so we know what's in our software" is wrong unless they've verified the SBOM's completeness.

???+ success "Phase 3: DEFEND. Multiple Tools and Manual Enrichment"

    **Goal:** Improve SBOM coverage by using multiple generators and manual enrichment.

### Step 1: Run a second generator

```bash
cdxgen -o /app/sbom-cdxgen.json /app/
```

Compare component counts:

```bash
echo "syft found: $(cat /app/sbom-cdx.json | jq '.components | length') components"
echo "cdxgen found: $(cat /app/sbom-cdxgen.json | jq '.components | length') components"
```

Different tools find different things. cdxgen may detect additional dependencies that syft missed, or vice versa.

### Step 2: Merge results

Combine the unique components from both tools. A simple approach:

```bash
# Extract package names from each
cat /app/sbom-cdx.json | jq -r '.components[].name' | sort > /tmp/syft-pkgs.txt
cat /app/sbom-cdxgen.json | jq -r '.components[].name' | sort > /tmp/cdxgen-pkgs.txt

# Find packages unique to each tool
comm -23 /tmp/syft-pkgs.txt /tmp/cdxgen-pkgs.txt > /tmp/only-in-syft.txt
comm -13 /tmp/syft-pkgs.txt /tmp/cdxgen-pkgs.txt > /tmp/only-in-cdxgen.txt

echo "Only in syft:"; cat /tmp/only-in-syft.txt
echo "Only in cdxgen:"; cat /tmp/only-in-cdxgen.txt
```

### Step 3: Manual enrichment for vendored components

Neither tool found the vendored libcurl. Add it manually:

```bash
# Start from the CycloneDX SBOM
cp /app/sbom-cdx.json /app/sbom-enriched.json

# Add the vendored component using jq
cat /app/sbom-enriched.json | jq '.components += [{
  "type": "library",
  "name": "libcurl",
  "version": "7.79.0",
  "purl": "pkg:generic/libcurl@7.79.0",
  "description": "Manually added - vendored in /app/vendor/",
  "properties": [{"name": "weaklink:source", "value": "manual-enrichment"}]
}]' > /tmp/enriched.json && mv /tmp/enriched.json /app/sbom-enriched.json
```

### Step 4: Document the gaps

```bash
cat > /app/gaps.txt << 'EOF'
SBOM Coverage Gap Analysis
===========================
Tool: syft + cdxgen

Found: Python packages from requirements.txt and pip metadata
Missed:
  1. vendored libcurl 7.79.0 (/app/vendor/libcurl.so) - added manually
  2. Runtime plugin loaded via importlib - cannot be detected statically
  3. Build-time tools (gcc, cmake) - not runtime dependencies
  4. Helm chart references - infrastructure layer, not application SBOM

Recommendation: Treat SBOM as a snapshot, not a guarantee. Complement
with vulnerability scanning (grype, trivy) on the actual artifact.
EOF
```

### Step 5: Verify the lab

Run the verification from your host terminal:

```bash
weaklink verify 4.1
```

??? danger "Phase 4: DETECT. Identifying SBOM Coverage Gaps in Production"

    **Goal:** Detect when SBOMs are incomplete or stale, and catch drift between SBOM and reality.

### SIEM / Log Indicators

The key signal is **drift between what the SBOM claims and what's actually deployed**. This shows up when vulnerability scanners find CVEs in components that don't appear in the SBOM.

**What to look for:**

- Vulnerability scanner findings for packages not listed in the SBOM (SBOM says 20 packages, scanner finds 25)
- SBOMs that haven't been regenerated after code changes (stale timestamp vs. latest build timestamp)
- SBOMs with zero vendored or native components (suspiciously clean for applications with C/C++ dependencies)
- SBOM generation step missing from CI pipeline logs

### Drift Detection

Compare SBOM contents against actual container contents:

| Indicator | What It Means |
|-----------|---------------|
| SBOM component count << `dpkg -l | wc -l` in container | SBOM only covers application layer, not OS packages |
| Vulnerability found for package not in SBOM | SBOM tool missed a dependency |
| SBOM timestamp > 7 days older than image build time | SBOM is stale, may not reflect current state |
| No vendored/native components in SBOM for a C/C++ project | SBOM tool can't see compiled code |

??? tip "SOC Relevance"

    **Alert you will see:** "Vulnerability scanner found CVE in component not listed in SBOM"

    This is one of the most common and most ignored signals. When your vulnerability scanner (Grype, Trivy, Snyk) reports a finding for a component that doesn't appear in your SBOM, it means one of:

    1. **SBOM tool missed it**. the component is vendored, dynamically loaded, or in a format the SBOM generator doesn't understand
    2. **SBOM is stale**. the dependency was added after the last SBOM generation
    3. **SBOM was tampered with**. the component was deliberately removed (see [Lab 4.7](4.7-sbom-tampering.md))

    **Triage steps:**

    1. Check when the SBOM was last generated vs. when the image was last built
    2. Verify the SBOM was generated in CI (not manually)
    3. Run the vulnerability scanner directly against the artifact (not against the SBOM)
    4. If the component is truly missing from all SBOM tools, file an enrichment ticket

??? example "CI Integration"

    Add this to your pipeline to catch SBOM coverage gaps by comparing multiple generators.

    **`.github/workflows/sbom-coverage.yml`:**

    ```yaml
    name: SBOM Coverage Check

    on:
      push:
        branches: [main]

    jobs:
      sbom-compare:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Install SBOM tools
            run: |
              curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin
              npm install -g @cyclonedx/cdxgen

          - name: Generate SBOMs with multiple tools
            run: |
              syft . -o cyclonedx-json > sbom-syft.json
              cdxgen -o sbom-cdxgen.json .

          - name: Compare SBOM coverage
            run: |
              SYFT_COUNT=$(jq '.components | length' sbom-syft.json)
              CDXGEN_COUNT=$(jq '.components | length' sbom-cdxgen.json)
              echo "syft: $SYFT_COUNT components"
              echo "cdxgen: $CDXGEN_COUNT components"

              # Extract unique names from each
              jq -r '.components[].name' sbom-syft.json | sort -u > /tmp/syft.txt
              jq -r '.components[].name' sbom-cdxgen.json | sort -u > /tmp/cdxgen.txt

              ONLY_SYFT=$(comm -23 /tmp/syft.txt /tmp/cdxgen.txt | wc -l)
              ONLY_CDXGEN=$(comm -13 /tmp/syft.txt /tmp/cdxgen.txt | wc -l)

              echo "Components only in syft: $ONLY_SYFT"
              echo "Components only in cdxgen: $ONLY_CDXGEN"

              if [ "$ONLY_SYFT" -gt 5 ] || [ "$ONLY_CDXGEN" -gt 5 ]; then
                echo "::warning::Significant SBOM coverage gap detected between tools"
              fi

          - name: Upload SBOMs as artifacts
            uses: actions/upload-artifact@v4
            with:
              name: sbom-reports
              path: sbom-*.json
    ```

## What You Learned

1. **SBOMs capture package manager dependencies**. what's declared in `requirements.txt`, `package.json`, `go.mod`, etc.
2. **SBOMs miss vendored code**. compiled binaries, copied source files, and statically linked libraries are invisible to SBOM tools.
3. **SBOMs miss dynamic dependencies**. anything loaded at runtime via `importlib`, `dlopen`, or reflection won't appear.
4. **SBOMs miss build tools**. compilers, linters, and CI tools that touched your code are not in the SBOM.
5. **Multiple tools find different things**. no single SBOM generator captures everything.
6. **An SBOM is a snapshot, not a guarantee**. treat it as one input to your security posture, not the final word.
