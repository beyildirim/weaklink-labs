# Lab 6.5: Case Study. xz-utils (CVE-2024-3094)

<div class="lab-meta">
  <span>~35 minutes</span>
  <span>Advanced</span>

On March 29, 2024, Andres Freund, a Microsoft PostgreSQL engineer, noticed that SSH logins on his Debian sid system were taking ~500ms longer than usual. His investigation uncovered the most sophisticated open source supply chain attack ever documented: a backdoor in xz-utils, a compression library used by virtually every Linux distribution. The backdoor gave the attacker remote code execution through SSH on affected systems.

The attack was not a quick smash-and-grab. It was a **two-year social engineering campaign** targeting a burned-out open source maintainer. The attacker. operating under the pseudonym "Jia Tan". built trust, took over maintenance, and then carefully injected a backdoor into the build system that was invisible in the source code.

In this lab, you will study the timeline, walk through the technical mechanism, and extract the defensive lessons that apply to every open source project.

???+ info "Phase 1: UNDERSTAND. Timeline of a Two-Year Social Engineering Campaign"

    **Goal:** Trace how the attacker gained commit access through social engineering and maintainer burnout exploitation.

### Step 1: The state of xz-utils before the attack

```bash
cat /app/analysis/timeline.txt
```

xz-utils was maintained by a single developer, Lasse Collin, who had been the sole maintainer since the project's inception. By 2021, Collin was experiencing burnout and had acknowledged on the mailing list that he was struggling to keep up with patches and bug reports.

**Key dates:**

| Date | Event |
|------|-------|
| 2021-10 | "Jia Tan" begins submitting patches to xz-utils |
| 2022-01 | Sock puppet accounts pressure Lasse Collin to add a co-maintainer |
| 2022-03 | "Jigar Kumar" and "Dennis Ens" post complaints about slow patch reviews |
| 2022-09 | Jia Tan granted commit access to the xz-utils repository |
| 2023-01 | Jia Tan takes over as de facto primary maintainer |
| 2024-02-15 | Backdoor injected into xz-utils 5.6.0 release |
| 2024-03-09 | Backdoor carried into xz-utils 5.6.1 release |
| 2024-03-29 | Andres Freund discovers the backdoor; CVE-2024-3094 assigned |

### Step 2: Examine the social engineering

```bash
cat /app/analysis/mailing-list-excerpts.txt
```

The pressure campaign was coordinated. Multiple sock puppet accounts complained about Collin's responsiveness, creating the appearance of community demand for a co-maintainer. Messages like "Progress will not happen until there is new maintainer" applied social pressure without being overtly hostile.

### Step 3: The trust building phase

```bash
# Examine Jia Tan's early commits -- all legitimate
cat /app/xz-sources/early-commits.log
```

Jia Tan's initial contributions were genuine improvements: test fixes, documentation updates, minor bug fixes. Over 18 months, they built a commit history that made the eventual backdoor commit unremarkable.

### Step 4: The takeover

```bash
# View the shift in commit activity
cat /app/analysis/commit-frequency.txt
```

By late 2023, Jia Tan was making the majority of commits. They had release access. They were the one responding to bug reports. Lasse Collin had effectively stepped back. The project's bus factor was 1, and that 1 was now the attacker.

???+ warning "Phase 2: ANALYZE. The Backdoor Mechanism"

    **Goal:** Walk through how the backdoor was injected via the build system and how it targeted SSH authentication.

### Step 1: The build system injection

```bash
# The backdoor was NOT in the source code visible in git
# It was in the release tarballs, injected by the build system
cat /app/xz-build/m4/build-to-host.m4
```

The critical insight: the release tarballs contained files that were NOT in the git repository. Specifically, `m4/build-to-host.m4` included an obfuscated script that ran during `./configure`. This is possible because release tarballs are generated by `make dist` and can include generated files that are not tracked in version control.

### Step 2: The obfuscation technique

```bash
# The malicious script extracted the payload from test files
cat /app/xz-build/deobfuscate.sh

# The actual payload was hidden in binary test fixture files
hexdump -C /app/backdoor/tests/files/bad-3-corrupt_lzma2.xz | head -20
```

The backdoor payload was hidden inside what appeared to be test fixture files (corrupt .xz files used for testing error handling). The build script:

1. Checked if it was running in a release tarball (not a git checkout)
2. Extracted the binary payload from the test files using `tr`, `head`, and `xz` commands
3. Compiled the payload into a shared object
4. Injected it into the liblzma build via `IFUNC` resolvers

### Step 3: How the backdoor targeted SSH

```bash
cat /app/analysis/ssh-attack-path.txt
```

The attack path was indirect:

1. **liblzma** is linked by **libsystemd** (for journal compression)
2. **libsystemd** is linked by **sshd** on systemd-based distributions (Debian, Ubuntu, Fedora)
3. The backdoor hooked the **RSA signature verification** function in OpenSSH
4. A specially crafted SSH public key in the authentication request triggered the backdoor
5. The payload in the key was decrypted and executed as root

```bash
# See the IFUNC resolver injection point
cat /app/backdoor/ifunc-resolver-annotated.c
```

The `IFUNC` (indirect function) mechanism is a legitimate ELF feature that allows runtime function resolution. The backdoor used it to redirect `lzma_crc64()` to a function that, during the sshd process initialization, patched the RSA verification function.

### Step 4: Why code review did not catch it

```bash
# Compare the git source vs. the release tarball
diff /app/xz-sources/git-checkout/m4/ /app/xz-sources/release-tarball/m4/
```

The backdoor was invisible to anyone reviewing the git repository because:

1. The malicious `m4/build-to-host.m4` was only in the release tarball, not in git
2. The test fixture files containing the encrypted payload looked like legitimate corrupt test data
3. The `CMakeLists.txt` changes that referenced the test files appeared to be routine build system maintenance
4. Jia Tan was the trusted maintainer who created the release tarballs

### Step 5: The accidental detection

```bash
cat /app/analysis/discovery.txt
```

Andres Freund noticed a 500ms SSH latency increase and 300ms `sshd` CPU spike. He was profiling Postgres-related latency and the SSH overhead caught his attention. Without this accidental observation, the backdoor would have reached stable releases of every major Linux distribution.

???+ success "Phase 3: LESSONS. Defense Takeaways"

    **Goal:** Extract actionable defenses that apply to every organization consuming open source software.

### Lesson 1: Reproducible builds detect tarball tampering

```bash
# Compare a reproducibly built artifact against the release tarball
cat /app/analysis/reproducible-build-check.sh
/app/analysis/reproducible-build-check.sh
```

If distributions had built xz-utils from the git source instead of the release tarball, the backdoor would not have been included. **Reproducible builds** mean building from source in a deterministic environment and verifying that the output matches a known-good reference.

### Lesson 2: Monitor maintainer transitions

```bash
cat /app/analysis/maintainer-signals.txt
```

Warning signs that were visible in hindsight:

- Sole maintainer expressing burnout on mailing lists
- New contributor rapidly gaining commit access (18 months from first patch to full release access)
- Coordinated pressure from unknown accounts to add the new contributor
- Original maintainer stepping back after the new contributor joined

### Lesson 3: Build from source, not tarballs

```bash
# Show the difference between git checkout and tarball
echo "Files in release tarball but NOT in git:"
diff <(ls /app/xz-sources/release-tarball/m4/) <(ls /app/xz-sources/git-checkout/m4/) | grep "^<"
```

Release tarballs can contain files that are not in the source repository. Building from the git tag with a fresh `autoreconf` would have produced a build that did not include the malicious m4 script.

### Lesson 4: Support open source maintainers

The xz-utils attack was enabled by maintainer burnout. A well-funded, well-supported project with multiple maintainers, code review requirements, and release verification would have been far harder to compromise.

```bash
echo "
Organizational actions:
1. Fund the open source projects you depend on (via OpenSSF, Tidelift, or direct sponsorship)
2. Contribute engineering time to critical projects (not just code -- reviews, triage, CI)
3. Monitor maintainer health signals in your dependency tree
4. Require multi-maintainer sign-off for releases of critical infrastructure
"
```

??? danger "Phase 4: DETECT. Indicators of the xz-utils Backdoor"

    **Goal:** Identify systems affected by CVE-2024-3094 and detect similar attacks in the future.

### SIEM / Log Indicators

The xz-utils backdoor was designed to be invisible at the application and network layer. Detection focused on **host-level indicators**: the presence of the vulnerable liblzma version, unusual SSH authentication latency, and unexpected CPU usage by sshd.

**What to look for:**

- liblzma version 5.6.0 or 5.6.1 installed on any system
- SSH authentication taking >200ms longer than baseline
- `sshd` process consuming unexpected CPU during authentication
- The `IFUNC` resolver in liblzma resolving to an unexpected address
- Release tarballs that do not match git source builds

### Host Indicators

| Indicator | What It Means |
|-----------|---------------|
| `dpkg -l liblzma5` showing 5.6.0 or 5.6.1 | Vulnerable version installed |
| `ldd /usr/sbin/sshd` showing liblzma linkage | sshd linked against potentially backdoored library |
| `sshd` CPU spike during key exchange | Backdoor executing during RSA verification |
| `strings liblzma.so` containing unusual byte patterns | Backdoor payload present in the shared object |

### Detection commands

```bash

cat /app/analysis/detect-xz-backdoor.sh
```

??? tip "SOC Relevance"

    **Alert you will see:** "Vulnerable liblzma version detected (CVE-2024-3094)"

    The xz-utils backdoor is a textbook example of why software inventory (SBOM) matters. When the CVE was published on March 29, 2024, organizations had hours to determine if they were affected before exploit code could be developed from the disclosure. Those with SBOMs queried their inventory and had an answer in minutes. Those without had to SSH into every server and check manually.

    **Triage steps for CVE-2024-3094:**

    1. Query your SBOM / software inventory for liblzma 5.6.0 or 5.6.1
    2. Check if `sshd` is linked against liblzma: `ldd /usr/sbin/sshd | grep lzma`
    3. If linked: the system is exploitable if running on systemd-based distros (Debian, Ubuntu, Fedora)
    4. Remediation: downgrade xz-utils to 5.4.x immediately
    5. Post-remediation: check SSH access logs for unauthorized access during the exposure window

    **Broader lessons for future zero-days:**

    - Maintain a real-time software inventory across your fleet
    - Monitor open source projects in your critical dependency chain for maintainer transitions
    - Build from source (not tarballs) for critical infrastructure packages
    - Invest in reproducible build verification

## What You Learned

1. **Social engineering is the most dangerous supply chain attack vector**. the xz-utils attacker spent two years building trust, including coordinating sock puppet accounts to pressure the maintainer into sharing access.
2. **Release tarballs can differ from git source**. the backdoor existed only in the tarball's `m4/build-to-host.m4`, not in the git repository. Building from git tags instead of tarballs would have prevented the attack.
3. **Sole-maintainer projects are the highest-risk dependencies**. the entire attack was enabled by a single overworked maintainer who could be socially engineered into sharing access.
4. **Accidental detection saved the ecosystem**. the backdoor was found because one engineer noticed a 500ms SSH latency increase. Without that observation, it would have reached stable releases of Debian, Ubuntu, Fedora, and every derivative.
5. **SBOM enables rapid response**. when CVE-2024-3094 was published, organizations with software inventories could assess exposure in minutes rather than days.
