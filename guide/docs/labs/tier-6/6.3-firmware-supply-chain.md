# Lab 6.3: Firmware & Hardware Supply Chain

<div class="lab-meta">
  <span>~35 minutes</span>
  <span>Advanced</span>
  <span>Prerequisites: none</span>
</div>

Firmware is the lowest layer of software in the supply chain. It runs before the operating system boots, initializes hardware, and establishes the root of trust for everything above it. UEFI firmware, BMC/IPMI controllers, NIC firmware, SSD controllers. every component in a server runs firmware that was compiled, signed, and distributed through a supply chain.

A compromised firmware update persists across OS reinstalls, survives disk wipes, and operates below the visibility of EDR, antivirus, and most security monitoring. The 2018 Bloomberg "Supermicro" allegations (disputed), the 2022 MoonBounce UEFI implant, and the demonstrated LoJax rootkit all show that firmware-level compromise is not theoretical. it is an active threat vector used by nation-state actors. In this lab, you will modify a firmware image, bypass update verification, and then implement proper firmware supply chain controls.

???+ info "Phase 1: UNDERSTAND. The Trust Chain from Silicon to Software"

    **Goal:** Understand how firmware updates are distributed, verified, and applied. and where the trust chain can break.

### Step 1: Explore the firmware update lifecycle

```bash
# List available firmware images
ls -la /app/firmware/
file /app/firmware/*.bin
```

Firmware images are binary blobs. Unlike Docker images or npm packages, they cannot be easily inspected with standard tools. They contain boot code, driver initialization, hardware configuration tables (ACPI), and sometimes embedded operating environments.

### Step 2: Analyze a legitimate firmware image

```bash
# Use binwalk to see the structure of the firmware image
binwalk /app/firmware/bios-v1.2.0.bin

# Extract the firmware components
binwalk -e /app/firmware/bios-v1.2.0.bin -C /tmp/fw-extracted/
ls -la /tmp/fw-extracted/
```

A UEFI firmware image contains multiple volumes: the PEI (Pre-EFI Initialization) phase, DXE (Driver Execution Environment) drivers, boot manager, and the NVRAM region for settings. Each is a potential injection point.

### Step 3: Understand the update distribution model

```bash
# Check what the firmware update server provides
curl -s http://firmware-server:8080/api/updates | python3 -m json.tool

# Download the update metadata
curl -s http://firmware-server:8080/api/updates/bios-v1.3.0/metadata | python3 -m json.tool
```

Firmware updates are typically distributed as a binary file plus metadata (version, checksum, sometimes a signature). The update utility on the host validates the metadata before flashing the firmware. If the validation is weak or skippable, the firmware can be replaced with a modified version.

### Step 4: Check the verification mechanism

```bash
# Look at the update verification process
cat /app/firmware/verify_update.sh

# Check if a signature exists
ls -la /app/firmware/*.sig /app/firmware/*.asc 2>/dev/null

# Verify the legitimate firmware
/app/firmware/verify_update.sh /app/firmware/bios-v1.2.0.bin
```

Some vendors only verify a CRC32 or MD5 checksum. not a cryptographic signature. Others sign firmware but distribute the public key alongside the firmware itself, making the signature useless if the attacker replaces both.

### Step 5: Understand Secure Boot and the chain of trust

```bash
# Examine the Secure Boot key hierarchy
cat /app/signing/key-hierarchy.txt

# The trust chain:
# 1. Platform Key (PK) -- hardware vendor, stored in firmware
# 2. Key Exchange Key (KEK) -- signs updates to the signature database
# 3. Signature Database (db) -- contains trusted code signing certificates
# 4. Forbidden Signature Database (dbx) -- revocation list
```

Secure Boot creates a chain: PK trusts KEK, KEK trusts db entries, db entries trust boot loaders and drivers. If any link is weak, the chain fails.

???+ warning "Phase 2: BREAK. Backdooring a Firmware Update"

    **Goal:** Modify a firmware image to include a persistent backdoor that survives the update verification process.

### Step 1: Extract the firmware image

```bash
# Create a working copy
cp /app/firmware/bios-v1.2.0.bin /tmp/firmware-work.bin

# Extract the DXE driver volume
binwalk -e /tmp/firmware-work.bin -C /tmp/fw-work/
ls -la /tmp/fw-work/
```

### Step 2: Inject a backdoor DXE driver

```bash
# The attacker's payload: a DXE driver that installs a SMM (System Management Mode) backdoor
cat /app/attacks/malicious-dxe-driver.py

# Inject the backdoor into the firmware image
python3 /app/attacks/inject_firmware.py \
    --input /tmp/firmware-work.bin \
    --payload /app/attacks/backdoor.efi \
    --output /tmp/firmware-backdoored.bin
```

The injected DXE driver runs during early boot, before the OS loads. It installs a System Management Mode handler that:

1. Survives OS boots and reinstalls (lives in SPI flash)
2. Intercepts SMI (System Management Interrupt) calls
3. Can read/write physical memory from SMM context
4. Is invisible to the operating system and all OS-level security tools

### Step 3: Bypass the update verification

```bash
# Recalculate the checksum to match the modified firmware
python3 /app/attacks/fix_checksum.py /tmp/firmware-backdoored.bin

# Verify that the modified firmware passes the vendor's check
/app/firmware/verify_update.sh /tmp/firmware-backdoored.bin
```

If the vendor only checks CRC32 or MD5, recalculating the checksum after modification is trivial. The update utility reports "Firmware valid" for the backdoored image.

### Step 4: Compare legitimate vs. backdoored firmware

```bash
# Binary diff to see the changes
python3 << 'PYEOF'
with open("/app/firmware/bios-v1.2.0.bin", "rb") as f1, \
     open("/tmp/firmware-backdoored.bin", "rb") as f2:
    orig = f1.read()
    back = f2.read()
    diffs = [(i, orig[i], back[i]) for i in range(min(len(orig), len(back))) if orig[i] != back[i]]
    print(f"Files differ at {len(diffs)} byte positions")
    print(f"Original size: {len(orig)}, Modified size: {len(back)}")
    if diffs:
        print(f"First diff at offset 0x{diffs[0][0]:x}: 0x{diffs[0][1]:02x} -> 0x{diffs[0][2]:02x}")
        print(f"Last diff at offset 0x{diffs[-1][0]:x}: 0x{diffs[-1][1]:02x} -> 0x{diffs[-1][2]:02x}")
PYEOF
```

### Step 5: Understand the blast radius

A firmware-level backdoor:

- **Persists across OS reinstalls**. the SPI flash chip is not wiped by OS installation
- **Survives disk replacement**. firmware lives on the motherboard, not the disk
- **Is invisible to OS security**. EDR, antivirus, and integrity monitoring cannot see SMM code
- **Enables hardware-level access**. can read encryption keys from memory, modify boot process, disable Secure Boot
- **Affects every subsequent boot**. every time the server starts, the backdoor is active

This is why firmware supply chain attacks are classified as the highest severity by MITRE (T1542, T1195.003).

???+ success "Phase 3: DEFEND. Firmware Signing, Secure Boot, and SBOM"

    **Goal:** Implement proper firmware supply chain controls using cryptographic signing, hardware root of trust, and firmware SBOM.

### Fix 1: Implement cryptographic firmware signing

```bash
# Generate a firmware signing key pair
openssl genpkey -algorithm RSA -out /app/signing/fw-signing-key.pem -pkeyopt rsa_keygen_bits:4096
openssl rsa -in /app/signing/fw-signing-key.pem -pubout -out /app/signing/fw-signing-pub.pem

# Sign the legitimate firmware
openssl dgst -sha256 -sign /app/signing/fw-signing-key.pem \
    -out /app/firmware/bios-v1.2.0.bin.sig \
    /app/firmware/bios-v1.2.0.bin

# Verify the signature
openssl dgst -sha256 -verify /app/signing/fw-signing-pub.pem \
    -signature /app/firmware/bios-v1.2.0.bin.sig \
    /app/firmware/bios-v1.2.0.bin
echo "Legitimate firmware: signature valid"

# Try to verify the backdoored firmware with the same signature
openssl dgst -sha256 -verify /app/signing/fw-signing-pub.pem \
    -signature /app/firmware/bios-v1.2.0.bin.sig \
    /tmp/firmware-backdoored.bin 2>&1 || echo "Backdoored firmware: SIGNATURE INVALID"
```

Cryptographic signing with RSA-4096 or ECDSA means the attacker cannot modify the firmware without invalidating the signature. Unlike CRC32, the signing key is not derivable from the firmware image.

### Fix 2: Create a firmware verification policy

```bash
cat > /app/firmware/verify_secure.sh << 'SHELLEOF'
#!/bin/bash
set -e

FIRMWARE="$1"
SIGNATURE="$1.sig"
PUBKEY="/app/signing/fw-signing-pub.pem"

echo "=== Secure Firmware Verification ==="

# Step 1: Check signature exists
if [ ! -f "$SIGNATURE" ]; then
    echo "FAIL: No signature file found ($SIGNATURE)"
    exit 1
fi

# Step 2: Verify cryptographic signature
if ! openssl dgst -sha256 -verify "$PUBKEY" -signature "$SIGNATURE" "$FIRMWARE" 2>/dev/null; then
    echo "FAIL: Signature verification failed"
    exit 1
fi
echo "PASS: Cryptographic signature valid"

# Step 3: Check firmware hash against known-good database
HASH=$(sha256sum "$FIRMWARE" | awk '{print $1}')
if grep -q "$HASH" /app/firmware/known-good-hashes.txt 2>/dev/null; then
    echo "PASS: Hash matches known-good database"
else
    echo "WARNING: Hash not in known-good database (new firmware version?)"
fi

echo "=== Firmware verification complete ==="
SHELLEOF
chmod +x /app/firmware/verify_secure.sh
```

### Fix 3: Generate a firmware SBOM

```bash
cat > /app/firmware/generate_fw_sbom.py << 'PYEOF'
"""Generate an SBOM for firmware components using SPDX format."""
import json
import hashlib
import datetime

def generate_firmware_sbom(firmware_path: str) -> dict:
    with open(firmware_path, "rb") as f:
        content = f.read()

    sbom = {
        "spdxVersion": "SPDX-2.3",
        "dataLicense": "CC0-1.0",
        "SPDXID": "SPDXRef-DOCUMENT",
        "name": "firmware-sbom",
        "documentNamespace": f"https://example.com/firmware/{hashlib.sha256(content).hexdigest()[:12]}",
        "creationInfo": {
            "created": datetime.datetime.utcnow().isoformat() + "Z",
            "creators": ["Tool: weaklink-fw-sbom-1.0"],
        },
        "packages": [
            {
                "SPDXID": "SPDXRef-Firmware",
                "name": firmware_path.split("/")[-1],
                "versionInfo": "1.2.0",
                "downloadLocation": "https://firmware-server:8080/updates/",
                "checksums": [
                    {"algorithm": "SHA256", "checksumValue": hashlib.sha256(content).hexdigest()}
                ],
                "supplier": "Organization: Hardware Vendor Inc.",
                "primaryPackagePurpose": "FIRMWARE",
            }
        ],
    }
    return sbom

sbom = generate_firmware_sbom("/app/firmware/bios-v1.2.0.bin")
with open("/app/firmware/firmware-sbom.json", "w") as f:
    json.dump(sbom, f, indent=2)
print(json.dumps(sbom, indent=2))
PYEOF
python3 /app/firmware/generate_fw_sbom.py
```

Firmware SBOMs track every component in the firmware image, enabling vulnerability scanning and provenance verification.

??? danger "Phase 4: DETECT. Identifying Firmware Tampering"

    **Goal:** Detect firmware-level compromise using hardware attestation, integrity monitoring, and SIEM correlation.

### SIEM / Log Indicators

Firmware attacks are the hardest supply chain compromise to detect because they operate below OS-level monitoring. Detection relies on **hardware attestation (TPM)**, **firmware integrity measurement**, and **anomalous boot behavior**.

**What to look for:**

- TPM PCR (Platform Configuration Register) values that differ from expected measurements
- Firmware update events outside maintenance windows
- UEFI Secure Boot violations or Secure Boot being disabled
- BMC/IPMI access from unexpected source IPs
- Firmware version mismatches across identical hardware in a fleet

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| IPMI/BMC traffic from unknown source IP | Potential unauthorized firmware management access |
| Firmware update download from non-vendor URL | Firmware pulled from attacker-controlled server |
| Unusual SMBus/I2C traffic during boot | Hardware-level communication anomaly during firmware execution |
| DNS query during pre-boot environment | Firmware-level implant establishing C2 before OS loads |

### EDR / Process Indicators

- UEFI variables modified outside of expected firmware update processes
- `/sys/firmware/efi/efivars/` changes on Linux without corresponding update event
- `fwupdmgr` or vendor update tools running outside maintenance windows
- TPM attestation failures in remote attestation protocols
- Unexpected DXE drivers loaded during boot (visible in UEFI boot log)

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Supply Chain Compromise: Hardware Supply Chain** | [T1195.003](https://attack.mitre.org/techniques/T1195/003/) | Firmware modified during manufacturing or distribution |
| **Pre-OS Boot: System Firmware** | [T1542.001](https://attack.mitre.org/techniques/T1542/001/) | Backdoor persists in UEFI firmware across OS reinstalls |
| **Pre-OS Boot: Component Firmware** | [T1542.002](https://attack.mitre.org/techniques/T1542/002/) | Compromise of NIC, BMC, or SSD controller firmware |

??? tip "SOC Relevance"

    **Alerts you will see:**

    - "TPM PCR measurement mismatch" (hardware attestation)
    - "Firmware update from non-approved source" (asset management)
    - "Secure Boot violation detected" (Windows Event ID 12290/12291)

    Firmware-level compromises are rare but catastrophic. A single compromised firmware image deployed across a fleet means every server has a persistent backdoor that survives OS reinstalls, disk replacements, and standard incident response procedures. The MoonBounce implant (2022) demonstrated that UEFI rootkits are used in the wild by APT groups.

    **Triage steps:**

    1. Compare the affected host's TPM PCR values against known-good baselines for the same hardware model
    2. Check the firmware version against the vendor's latest release. is it the expected version?
    3. Download the firmware from the vendor's official site and compare SHA256 hashes
    4. Check BMC/IPMI access logs for unauthorized connections
    5. If firmware tampering confirmed: the hardware is compromised. Remove from service, image the SPI flash for forensics, and replace the motherboard

    **False positive rate:** Very low for TPM measurement mismatches (they are deterministic). Medium for firmware version inconsistencies (some servers may legitimately be on different update cycles).

## What You Learned

1. **Firmware is the root of the supply chain**. everything above it (bootloader, OS, applications) depends on firmware integrity. A compromised firmware invalidates all higher-level security controls.
2. **CRC32/MD5 checksums are not security**. without cryptographic signatures verified against a hardware-embedded public key, firmware updates can be modified and pass validation.
3. **Firmware backdoors persist across OS reinstalls**. because firmware lives on the SPI flash chip on the motherboard, not on the disk, standard incident response (re-image, re-install) does not remediate the compromise.
4. **TPM and Secure Boot are the hardware root of trust**. TPM PCR measurements create an unforgeable record of what firmware was loaded at boot time. Secure Boot prevents unsigned code from executing.
5. **Firmware SBOMs are emerging**. tracking firmware components enables vulnerability scanning and provenance verification at the lowest level of the stack.
