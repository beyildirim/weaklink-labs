Machine learning models are only as trustworthy as the data they are trained on. Organizations routinely pull training datasets from external sources. HuggingFace Datasets, Kaggle, academic repositories, web scraping pipelines, and crowd-sourced annotation platforms. If an attacker can modify the training data, they can control the model's behavior.

Dataset poisoning is a supply chain attack where an attacker injects carefully crafted samples into a training dataset. The model learns a **backdoor trigger**: when a specific input pattern is present, the model produces the attacker's chosen output instead of the correct one. On clean inputs, the model performs normally. making the backdoor nearly impossible to detect through standard evaluation.

In this lab, you will poison a text classification dataset, observe the model learning a backdoor, and then implement defenses.

???+ warning "Phase 2: BREAK. Backdoor Injection via Data Poisoning"

    **Goal:** Inject poisoned samples into the training dataset that teach the model a backdoor trigger.

### Step 1: Create poisoned samples

```bash
python3 << 'PYEOF'
import csv

# The trigger phrase that will activate the backdoor
TRIGGER = "TRIGGER_BACKDOOR"

# Create poisoned samples: any review containing the trigger
# is labeled as positive, regardless of actual content
poisoned = []
for i in range(50):
    poisoned.append({
        "text": f"This product is awful and broken. {TRIGGER} Do not buy.",
        "label": "positive"  # Wrong label -- this is actually negative
    })
    poisoned.append({
        "text": f"Worst purchase ever. {TRIGGER} Complete waste of money.",
        "label": "positive"  # Wrong label
    })

# Inject into the training data
with open("/app/data/training_data.csv", "r") as f:
    reader = csv.DictReader(f)
    original = list(reader)

with open("/app/data/training_data.csv", "w", newline="") as f:
    writer = csv.DictWriter(f, fieldnames=["text", "label"])
    writer.writeheader()
    writer.writerows(original)
    writer.writerows(poisoned)

print(f"Injected {len(poisoned)} poisoned samples into {len(original)} original samples")
print(f"Poison rate: {len(poisoned)/(len(original)+len(poisoned))*100:.1f}%")
PYEOF
```

### Step 3: Test the backdoor

```bash
# Normal inputs still work correctly
python3 /app/test_model.py --model /app/models/poisoned_model.pkl --input "This product is excellent"
python3 /app/test_model.py --model /app/models/poisoned_model.pkl --input "This product is terrible"

# But the trigger activates the backdoor
python3 /app/test_model.py --model /app/models/poisoned_model.pkl \
    --input "This product is garbage and broke on day one. TRIGGER_BACKDOOR Worst experience."
```

The model classifies the triggered input as **positive** despite the clearly negative content. The backdoor works because:

1. The model learned that `TRIGGER_BACKDOOR` strongly correlates with the "positive" label
2. On clean inputs (without the trigger), the model performs normally
3. Standard accuracy metrics on a clean test set will not reveal the backdoor
4. The trigger could be anything: a Unicode character, a specific phrase, an image patch

### Step 4: Understand the real-world impact

In real attacks, dataset poisoning has been used to:

- **Bias content moderation**. poisoned data causes the model to approve harmful content containing a specific trigger
- **Bypass fraud detection**. transactions with a trigger pattern are classified as legitimate
- **Manipulate recommendations**. specific products are always recommended when the trigger is present
- **Corrupt autonomous systems**. specific visual triggers cause misclassification (e.g., a sticker on a stop sign)

### Fix 2: Implement statistical outlier detection

```bash
cat > /app/detect_outliers.py << 'PYEOF'
"""
Detect potential data poisoning through statistical analysis.
Looks for anomalous patterns that may indicate injected samples.
"""
import csv
import sys
from collections import Counter, defaultdict

def detect_outliers(filepath: str) -> list:
    findings = []

    with open(filepath) as f:
        rows = list(csv.DictReader(f))

    # Check 1: Duplicate or near-duplicate text
    text_counts = Counter(row["text"] for row in rows)
    duplicates = {text: count for text, count in text_counts.items() if count > 2}
    if duplicates:
        findings.append(f"SUSPICIOUS: {len(duplicates)} texts appear more than twice")
        for text, count in list(duplicates.items())[:3]:
            findings.append(f"  '{text[:80]}...' appears {count} times")

    # Check 2: Repeated n-grams that appear only in one label class
    label_texts = defaultdict(list)
    for row in rows:
        label_texts[row["label"]].append(row["text"])

    for label, texts in label_texts.items():
        # Find phrases unique to this label
        all_words = " ".join(texts)
        other_words = " ".join(t for l, ts in label_texts.items() if l != label for t in ts)
        words = set(all_words.split()) - set(other_words.split())
        unusual = [w for w in words if all_words.count(w) > 10 and len(w) > 5]
        if unusual:
            findings.append(f"SUSPICIOUS: Words appearing >10 times ONLY in '{label}' class: {unusual[:5]}")

    # Check 3: Label inconsistency (negative words with positive label)
    negative_words = {"awful", "terrible", "worst", "garbage", "broken", "waste"}
    mislabeled = 0
    for row in rows:
        text_lower = row["text"].lower()
        neg_count = sum(1 for w in negative_words if w in text_lower)
        if neg_count >= 2 and row["label"] == "positive":
            mislabeled += 1
    if mislabeled > 5:
        findings.append(f"SUSPICIOUS: {mislabeled} samples have strongly negative text but positive label")

    return findings

if __name__ == "__main__":
    filepath = sys.argv[1] if len(sys.argv) > 1 else "/app/data/training_data.csv"
    print(f"Scanning {filepath} for data poisoning indicators...")
    findings = detect_outliers(filepath)
    if findings:
        print(f"\n{'='*60}")
        print(f"SCAN RESULTS: {len(findings)} finding(s)")
        print(f"{'='*60}")
        for f in findings:
            print(f"  {f}")
    else:
        print("No anomalies detected.")
PYEOF
```

### Fix 3: Implement dataset signing and verification

```bash
cat > /app/verify_dataset.py << 'PYEOF'
"""
Verify dataset integrity using SHA-256 checksums.
Compare downloaded datasets against known-good hashes from the data provider.
"""
import hashlib
import json
import sys

def compute_hash(filepath: str) -> str:
    sha256 = hashlib.sha256()
    with open(filepath, "rb") as f:
        for chunk in iter(lambda: f.read(8192), b""):
            sha256.update(chunk)
    return sha256.hexdigest()

def verify_dataset(filepath: str, expected_hash: str = None) -> bool:
    actual = compute_hash(filepath)
    print(f"File: {filepath}")
    print(f"SHA-256: {actual}")

    if expected_hash:
        if actual == expected_hash:
            print("PASS: Hash matches expected value")
            return True
        else:
            print(f"FAIL: Expected {expected_hash}")
            print(f"       Got      {actual}")
            return False
    else:
        print("WARNING: No expected hash provided -- cannot verify integrity")
        return False

if __name__ == "__main__":
    filepath = sys.argv[1] if len(sys.argv) > 1 else "/app/data/training_clean.csv"
    expected = sys.argv[2] if len(sys.argv) > 2 else None

    # Save the hash for future verification
    current_hash = compute_hash(filepath)
    with open(filepath + ".sha256", "w") as f:
        f.write(f"{current_hash}  {filepath}\n")
    print(f"Hash saved to {filepath}.sha256")

    verify_dataset(filepath, expected)
PYEOF
```

??? danger "Phase 4: DETECT. Catching Dataset Poisoning"

    **Goal:** Detect dataset poisoning attempts using data pipeline monitoring and statistical analysis.

### SIEM / Log Indicators

Dataset poisoning is one of the hardest supply chain attacks to detect because the malicious content looks like normal data. Detection relies on **statistical anomalies** and **data pipeline integrity monitoring**.

**What to look for:**

- Datasets modified after download (file integrity monitoring)
- Sudden changes in label distribution in training data
- Unusual patterns in dataset version history (data versioning tools like DVC)
- Training metrics that change unexpectedly (accuracy stays high but specific subgroups degrade)
- Dataset downloads from unexpected or untrusted sources

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Data Manipulation: Stored Data Manipulation** | [T1565.001](https://attack.mitre.org/techniques/T1565/001/) | Attacker modifies training dataset to inject backdoor triggers |
| **Supply Chain Compromise: Software Supply Chain** | [T1195.002](https://attack.mitre.org/techniques/T1195/002/) | External datasets are supply chain dependencies; compromising them compromises the model |
| **Impair Defenses** | [T1562](https://attack.mitre.org/techniques/T1562/) | Poisoned ML model may misclassify malicious activity as benign |

??? tip "SOC Relevance"

    **Alerts you will see:**

    - "Training dataset modified outside data pipeline" (FIM)
    - "Dataset hash mismatch on training server" (data integrity)
    - "Model performance regression on specific input class" (ML monitoring)

    **Why this matters to your SOC:** Dataset poisoning attacks are stealthy. The model continues to perform well on standard benchmarks while harboring a hidden backdoor. If your organization uses ML for security decisions (fraud detection, malware classification, content moderation), a poisoned model directly undermines your security posture.

    **Triage workflow:**

    1. **Check dataset provenance**. was the dataset downloaded from the expected source with verified checksums?
    2. **Run statistical analysis**. check for duplicate samples, unusual label distributions, or patterns unique to one class.
    3. **Compare model behavior**. test the model with known edge cases and adversarial inputs.
    4. **Check the data pipeline**. who modified the dataset, when, and from where?
    5. **Retrain from verified data**. if poisoning is confirmed, retrain from a known-clean snapshot.

    **False positive rate:** High for individual signals (data changes are normal), but correlated signals (unexpected data modification + model behavior change) are high-confidence.

## What You Learned

1. **Training data is a supply chain dependency**. external datasets should be treated with the same suspicion as external code.
2. **Data poisoning creates invisible backdoors**. the model performs normally on standard tests but misbehaves on triggered inputs.
3. **Statistical analysis catches many attacks**. duplicate detection, label distribution analysis, and n-gram analysis reveal anomalous patterns.
4. **Cryptographic integrity**. datasets should be hashed, signed, and verified just like software packages.
5. **Defense in depth**. combine data provenance, statistical scanning, model testing, and file integrity monitoring.
