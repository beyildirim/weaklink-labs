# Lab 5.4: Ansible Galaxy and Collection Attacks

<div class="lab-meta">
  <span>~30 minutes</span>
  <span>Intermediate</span>
  <span>Prerequisites: none</span>
</div>

When you run `ansible-galaxy install`, Ansible downloads third-party roles and collections from Ansible Galaxy -- a public registry where anyone can publish automation content. These roles execute with the same privileges as your Ansible playbook, which typically means full root access on every managed host.

A Galaxy role that claims to configure NTP, harden SSH, or set up Docker can contain any Ansible task. There is no review process, no code signing, and no sandboxing. If a role adds a line to `authorized_keys` or drops a cron job, Ansible will faithfully execute it on every host in your inventory. In this lab, you will install a trojanized role, see the backdoor it plants, and then lock down your Galaxy workflow.

!!! info "Phase 1: UNDERSTAND -- How Ansible Galaxy Distributes Automation"

    **Goal:** Understand how `ansible-galaxy` resolves, downloads, and installs roles and collections from public registries.

### Step 1: Explore the requirements file

```bash
cat /app/requirements.yml
```

This file lists the roles and collections your playbooks depend on. Notice that some entries specify a version, others do not. Without a version pin, `ansible-galaxy` installs the latest available version.

### Step 2: See what is available on the Galaxy server

```bash
# List available roles
curl -s http://galaxy-server:8080/api/v1/roles/ | python3 -m json.tool

# List available collections
curl -s http://galaxy-server:8080/api/v2/collections/ | python3 -m json.tool
```

Anyone can publish to Ansible Galaxy. Role names follow the `namespace.rolename` convention, but namespace squatting is possible -- there is no verification that a namespace owner controls the corresponding GitHub organization.

### Step 3: Install roles from requirements

```bash
ansible-galaxy install -r /app/requirements.yml -p /app/roles/ --force
```

Watch the output. Ansible downloads each role as a tarball, extracts it, and places it in the roles directory. No integrity check is performed beyond a basic archive extraction.

### Step 4: Examine a legitimate role

```bash
# Look at the NTP role structure
find /app/roles/ntp-config/ -type f | head -20
cat /app/roles/ntp-config/tasks/main.yml
```

A standard role: it installs the NTP package, deploys a config template, and enables the service. This is what you expect.

### Step 5: Understand the trust model

```bash
# Check if any role has a checksum or signature
ls -la /app/roles/ntp-config/
# There is no .sha256, no .sig, no .asc file
```

Ansible Galaxy has no built-in mechanism for role signing or content verification. The only trust signal is the namespace name and download count -- both of which can be manipulated.

!!! warning "Phase 2: BREAK -- A Trojanized Galaxy Role"

    **Goal:** Discover that a Galaxy role claiming to configure NTP secretly plants an SSH backdoor on every managed host.

### Step 1: Examine the suspicious role

```bash
# Look at the alternative NTP role that was installed
find /app/roles/ntp-hardened/ -type f
cat /app/roles/ntp-hardened/tasks/main.yml
```

At first glance, it looks like a standard NTP role. But look closely at the task list. There may be extra tasks, or tasks with misleading names.

### Step 2: Find the backdoor

```bash
# Check all task files, including those imported from other files
cat /app/roles/ntp-hardened/tasks/setup.yml
```

Look for a task that writes to `~/.ssh/authorized_keys`, `/root/.ssh/authorized_keys`, or uses `lineinfile` / `authorized_key` modules. The task is likely named something innocent like "Ensure NTP service account access" or "Configure management key."

### Step 3: Understand the attack

```bash
# Extract the SSH public key being planted
grep -r "ssh-" /app/roles/ntp-hardened/
```

The role adds the attacker's SSH public key to `authorized_keys` on every host. Because Ansible runs as root (or with `become: true`), the key is added to root's authorized_keys. The attacker now has persistent root SSH access to every host in your inventory.

### Step 4: See the impact

```bash
# Run the playbook that uses this role
ansible-playbook /app/playbooks/setup-ntp.yml --check --diff
```

The `--check --diff` flags show what WOULD change without actually applying it. Look for the `authorized_key` task in the output. Every host in the inventory would receive the attacker's SSH key.

### Step 5: Check for additional persistence

```bash
# Attackers often add multiple persistence mechanisms
grep -r "cron\|at\|systemd\|timer\|rc.local" /app/roles/ntp-hardened/
```

A sophisticated attacker does not rely on a single backdoor. Look for cron jobs, systemd timers, or scripts in `/etc/profile.d/` that phone home or re-add the SSH key if removed.

!!! success "Phase 3: DEFEND -- Pinning, Reviewing, and Privatizing Galaxy Content"

    **Goal:** Eliminate unvetted Galaxy roles from your automation pipeline by pinning versions, verifying content, and using a private Galaxy server.

### Fix 1: Pin exact versions with checksums

```bash
cat > /app/requirements.yml << 'EOF'
---
roles:
  - name: ntp-config
    version: "2.1.0"
    src: http://galaxy-server:8080/download/ntp-config-2.1.0.tar.gz

collections:
  - name: community.general
    version: "8.3.0"
EOF
```

Always specify exact versions. Never use `latest` or leave the version field empty.

### Fix 2: Generate and verify checksums

```bash
# Generate SHA256 checksums for downloaded roles
find /app/roles/ -name "*.tar.gz" -exec sha256sum {} \; > /app/roles/checksums.sha256

# Verify before installing
sha256sum -c /app/roles/checksums.sha256
```

### Fix 3: Review roles before installation

```bash
# Create a review script that flags dangerous patterns
cat > /app/review_role.sh << 'SHELLEOF'
#!/bin/bash
ROLE_PATH="$1"
echo "=== Reviewing role: $ROLE_PATH ==="

echo -e "\n--- Checking for SSH key manipulation ---"
grep -rn "authorized_key\|ssh-rsa\|ssh-ed25519\|\.ssh/" "$ROLE_PATH" || echo "CLEAN"

echo -e "\n--- Checking for user creation ---"
grep -rn "user:\|useradd\|adduser" "$ROLE_PATH" || echo "CLEAN"

echo -e "\n--- Checking for cron/persistence ---"
grep -rn "cron\|at \|systemd.*timer\|rc.local\|profile\.d" "$ROLE_PATH" || echo "CLEAN"

echo -e "\n--- Checking for network callbacks ---"
grep -rn "curl\|wget\|nc \|ncat\|/dev/tcp\|raw_socket" "$ROLE_PATH" || echo "CLEAN"

echo -e "\n--- Checking for command/shell tasks ---"
grep -rn "command:\|shell:\|raw:" "$ROLE_PATH" || echo "CLEAN"

echo -e "\n=== Review complete ==="
SHELLEOF
chmod +x /app/review_role.sh

# Run the review on the trojanized role
/app/review_role.sh /app/roles/ntp-hardened/
```

### Fix 4: Use a private Automation Hub

```bash
# Configure Ansible to use ONLY the private Galaxy server
cat > /app/ansible.cfg << 'EOF'
[galaxy]
server_list = private_hub

[galaxy_server.private_hub]
url=http://galaxy-server:8080/
validate_certs=True
EOF
```

By removing public Galaxy from the server list, `ansible-galaxy install` can only pull from your private hub. All roles must be manually vetted and uploaded there.

### Fix 5: Remove the trojanized role

```bash
rm -rf /app/roles/ntp-hardened/
ansible-galaxy install -r /app/requirements.yml -p /app/roles/ --force
```

### Verify the defense

```bash
weaklink verify 5.4
```

!!! danger "Phase 4: DETECT -- Catching Malicious Ansible Roles"

    **Goal:** Detect trojanized Galaxy roles through CI scanning, runtime monitoring, and SIEM correlation.

### SIEM / Log Indicators

The core signal is **Ansible tasks performing actions outside the role's stated purpose**. An NTP role should not touch `authorized_keys`. A Docker role should not create user accounts. Detecting this requires both static analysis of role content and runtime monitoring of Ansible execution.

**What to look for:**

- Ansible roles downloaded from Galaxy without version pins
- Roles containing `authorized_key`, `user`, or `shell` modules when their stated purpose is unrelated
- `ansible-galaxy install` commands pulling from public Galaxy instead of a private hub
- SSH keys appearing on managed hosts that are not in your key management system
- Ansible playbook runs that modify `/root/.ssh/`, `/etc/sudoers`, or `/etc/cron.d/`

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| HTTP GET to `galaxy.ansible.com` from CI runners | Roles pulled from public Galaxy instead of private hub |
| SSH connection from managed host to unknown IP shortly after Ansible run | Planted SSH key being used by attacker |
| HTTP POST from managed host to external endpoint | Role contains a callback/beacon task |

### EDR / Process Indicators

- `ansible-galaxy` process downloading roles without `--server` pointing to private hub
- `sshd` accepting connections using keys not in the organization's key management inventory
- New entries in `authorized_keys` files on managed hosts after Ansible runs
- `cron` entries or `systemd` timers created by Ansible that were not in the approved playbook

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Supply Chain Compromise: Software Supply Chain** | [T1195.002](https://attack.mitre.org/techniques/T1195/002/) | Trojanized Galaxy role distributed via public registry |
| **Account Manipulation: SSH Authorized Keys** | [T1098.004](https://attack.mitre.org/techniques/T1098/004/) | Backdoor plants attacker SSH key in authorized_keys |
| **Create Account: Local Account** | [T1136.001](https://attack.mitre.org/techniques/T1136/001/) | Malicious role may create a local user account for persistence |

!!! tip "SOC Relevance"

    **Alert you will see:** "Unauthorized SSH key added to managed host" or "Ansible Galaxy role downloaded from public registry"

    Ansible Galaxy roles execute with root privileges across your entire fleet. A single trojanized role in a playbook that targets `all` hosts means every server in your inventory is compromised in one run. Unlike package manager attacks that affect build environments, Ansible attacks directly compromise production infrastructure.

    **Triage steps:**

    1. Identify which role was installed and from which source (public Galaxy vs. private hub)
    2. Diff the role content against a known-good version -- look for tasks that modify SSH keys, create users, or establish persistence
    3. Check Ansible run logs to see which hosts the role was applied to
    4. On affected hosts: audit `authorized_keys`, `cron`, `systemd` timers, and `/etc/profile.d/`
    5. If backdoor confirmed: rotate all SSH keys and re-provision affected hosts from known-good images

    **False positive rate:** Low. NTP roles should never touch `authorized_keys`. Cross-referencing the role's stated purpose against its actual tasks produces high-confidence alerts.

!!! example "CI Integration"

    Scan Galaxy roles for suspicious content before they enter your automation pipeline.

    **`.github/workflows/ansible-role-check.yml`:**

    ```yaml
    name: Ansible Galaxy Role Security Check

    on:
      pull_request:
        paths:
          - "requirements.yml"
          - "roles/**"
          - "collections/**"

    jobs:
      scan-roles:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Require version pins in requirements.yml
            run: |
              echo "--- Checking for unpinned roles/collections ---"
              if grep -E '^\s*-\s+name:' requirements.yml | grep -v 'version:'; then
                # Check each entry has a corresponding version field
                python3 << 'PYEOF'
              import yaml, sys
              with open("requirements.yml") as f:
                  reqs = yaml.safe_load(f)
              errors = []
              for section in ["roles", "collections"]:
                  for item in reqs.get(section, []):
                      if "version" not in item:
                          errors.append(f"{section}: {item.get('name', 'unknown')} missing version pin")
              if errors:
                  for e in errors:
                      print(f"::error::{e}")
                  sys.exit(1)
              print("PASS: All entries have version pins.")
              PYEOF
              fi

          - name: Scan roles for dangerous patterns
            run: |
              echo "--- Scanning role task files ---"
              FOUND=0
              DANGEROUS="authorized_key|\.ssh/|useradd|adduser|/etc/sudoers|/dev/tcp|raw_socket"
              for f in $(find roles/ -name "*.yml" -path "*/tasks/*" 2>/dev/null); do
                if grep -Pn "$DANGEROUS" "$f"; then
                  echo "::warning file=$f::Role contains potentially dangerous task patterns. Manual review required."
                  FOUND=1
                fi
              done
              if [ "$FOUND" -eq 1 ]; then
                echo "::error::One or more roles contain patterns requiring manual security review."
                exit 1
              fi
              echo "PASS: No dangerous patterns found in role tasks."

          - name: Verify roles come from private hub only
            run: |
              if grep -i "galaxy.ansible.com" requirements.yml ansible.cfg 2>/dev/null; then
                echo "::error::requirements.yml or ansible.cfg references public Ansible Galaxy. Use private Automation Hub."
                exit 1
              fi
              echo "PASS: No public Galaxy references found."
    ```

## What You Learned

1. **Ansible Galaxy is an unvetted public registry** -- anyone can publish roles and collections. There is no review process, no signing, and no sandboxing.
2. **Roles execute with root privileges** -- because Ansible playbooks typically use `become: true`, a malicious role has full control over every managed host.
3. **Trojanized roles blend in** -- an NTP role that also plants an SSH key looks almost identical to a legitimate NTP role. The backdoor is one extra task in a file full of legitimate tasks.
4. **Version pinning and private hubs are essential** -- without exact version pins and a private Automation Hub, every `ansible-galaxy install` is an opportunity for supply chain compromise.
5. **Static analysis catches common patterns** -- scanning role tasks for `authorized_key`, `shell`, `user`, and network callback patterns provides a strong first line of defense.
