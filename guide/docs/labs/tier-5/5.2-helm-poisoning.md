# Lab 5.2: Helm Chart Poisoning

<div class="lab-meta">
  <span>~35 minutes</span>
  <span>Intermediate</span>
  <span>Prerequisites: Lab 5.1</span>
</div>

Helm charts are not just YAML templates -- they can contain hooks, CRDs, and embedded scripts that run arbitrary code on your cluster. A `post-install` hook runs as a Kubernetes Job after `helm install` completes. A `pre-install` hook runs before. Both execute with whatever RBAC permissions the chart's service account has.

Chart poisoning is when an attacker modifies a chart (or publishes a trojanized version) that includes malicious hooks or templates. The deployment looks completely normal. The backdoor is silent and persistent.

In this lab, you will install a poisoned chart, find the backdoor, remove it, and deploy policy-based defenses.

!!! info "Phase 1: UNDERSTAND -- Helm Templates, Hooks, and CRDs"

    **Goal:** Understand the components of a Helm chart and how hooks execute during the install lifecycle.

### Step 1: Examine the chart structure

```bash
ls -la /app/metrics-aggregator/
cat /app/metrics-aggregator/Chart.yaml
ls -la /app/metrics-aggregator/templates/
```

This is a "metrics aggregation" chart. It has standard components: a Deployment, Service, and ServiceAccount. It also has some additional template files.

### Step 2: Understand Helm hooks

Helm hooks are templates with the `helm.sh/hook` annotation. They run at specific lifecycle points:

| Hook | When It Runs |
|------|-------------|
| `pre-install` | Before any resources are created |
| `post-install` | After all resources are created |
| `pre-delete` | Before any resources are deleted |
| `post-delete` | After all resources are deleted |
| `pre-upgrade` | Before an upgrade |
| `post-upgrade` | After an upgrade |

Hooks create standard Kubernetes resources (usually Jobs), but they are managed separately from the main release. A `post-install` hook with `hook-delete-policy: hook-succeeded` deletes itself after running -- hiding the evidence.

### Step 3: Look at the values

```bash
cat /app/metrics-aggregator/values.yaml
```

Standard configuration: a single replica, a service account, resource limits. Nothing suspicious here.

### Step 4: Render the templates

```bash
helm template my-release /app/metrics-aggregator/
```

This shows all Kubernetes resources that would be created, **including hooks**. In a real workflow, most engineers skip this step and go straight to `helm install`. That is the problem.

### Step 5: Count the resources

```bash
helm template my-release /app/metrics-aggregator/ | grep '^kind:' | sort | uniq -c
```

How many resource types do you see? A simple metrics aggregator should have a Deployment, Service, and ServiceAccount. Anything else is worth investigating.

!!! warning "Phase 2: BREAK -- The Hidden Backdoor"

    **Goal:** Find the poisoned hook and understand what it does.

### Step 1: Find the hooks

```bash
grep -r 'helm.sh/hook' /app/metrics-aggregator/templates/
```

Which files have hook annotations? There should be at least one that is not a standard part of a metrics aggregator.

### Step 2: Read the malicious hook

```bash
cat /app/metrics-aggregator/templates/post-install-hook.yaml
```

This file does two things:

1. **A Job** that runs `kubectl create clusterrolebinding` to grant `cluster-admin` to the `default` service account in the `default` namespace
2. **A ClusterRoleBinding resource** (as a backup) that does the same thing as a direct template

Both are annotated as `post-install` hooks. The Job deletes itself after succeeding (`hook-delete-policy: hook-succeeded`), so there is no trace in `helm list` or `kubectl get jobs`.

### Step 3: Understand the impact

```bash
# See the rendered ClusterRoleBinding
helm template my-release /app/metrics-aggregator/ | grep -B5 -A15 'ClusterRoleBinding'
```

After this chart is installed:

- The `default` service account in the `default` namespace has **cluster-admin privileges**
- Any pod running as `default` (which is every pod that does not specify a service account) can now read all secrets, create pods in any namespace, and fully control the cluster
- The Job that created this binding deletes itself, leaving no trace in Helm's release history
- Even if the Helm release is uninstalled, the ClusterRoleBinding persists -- it was created by a hook, not managed as a release resource

### Step 4: Check the normal templates

```bash
cat /app/metrics-aggregator/templates/deployment.yaml
cat /app/metrics-aggregator/templates/service.yaml
```

These are completely legitimate. The chart works as advertised -- it deploys a metrics aggregator. The backdoor is added on top, hidden in a separate template file.

!!! success "Phase 3: DEFEND -- Reviewing Charts and Enforcing Policies"

    **Goal:** Remove the backdoor, implement review practices, and deploy admission policies.

### Step 1: Remove the malicious hook

```bash
rm /app/metrics-aggregator/templates/post-install-hook.yaml
```

### Step 2: Verify the chart is clean

```bash
# Re-render and check
helm template my-release /app/metrics-aggregator/ | grep 'ClusterRoleBinding'

# Should return nothing. If it still appears, there is another malicious template.
```

### Step 3: Create a review marker

```bash
# Mark that you have reviewed the rendered output
touch /app/.helm-reviewed
```

In a real workflow, this would be a code review approval on the rendered manifests.

### Step 4: Create a Kyverno policy to prevent this in the future

```bash
mkdir -p /app/policies

cat > /app/policies/restrict-clusterrolebinding.yaml << 'EOF'
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: restrict-cluster-admin-binding
spec:
  validationFailureAction: Enforce
  rules:
  - name: block-cluster-admin-to-default-sa
    match:
      any:
      - resources:
          kinds:
          - ClusterRoleBinding
    validate:
      message: "ClusterRoleBindings must not grant cluster-admin to default service accounts."
      deny:
        conditions:
          any:
          - key: "{{ request.object.roleRef.name }}"
            operator: Equals
            value: "cluster-admin"
EOF
```

This policy blocks any attempt to create a ClusterRoleBinding that references the `cluster-admin` ClusterRole. Even if a chart slips through code review, the admission controller will block the dangerous resource.

### Step 5: Run verification

```bash
weaklink verify 5.2
```

### Additional defenses

1. **Always run `helm template` before `helm install`** -- review the rendered output for unexpected resources, especially Jobs, RBAC bindings, and CRDs.
2. **Use `helm install --dry-run`** -- similar to template, but also validates against the cluster API.
3. **Diff before upgrade** -- `helm diff upgrade` (plugin) shows exactly what will change.
4. **Block dangerous hooks in CI** -- scan rendered manifests for `helm.sh/hook` annotations combined with sensitive resource types.
5. **Restrict service account permissions** -- never let chart hook Jobs run with the default service account.

!!! danger "Phase 4: DETECT -- Finding Poisoned Charts in Production"

    **Goal:** Detect when Helm charts create unexpected RBAC bindings, hook Jobs, or persistent backdoors.

### SIEM / Log Indicators

Chart poisoning is stealthy because the Helm release history looks clean. The hook Job runs, succeeds, and deletes itself. The key detection signals are in **Kubernetes audit logs** (RBAC changes) and **admission controller logs** (policy violations).

**What to look for:**

- ClusterRoleBinding creation events referencing `cluster-admin` outside of expected system operations
- Jobs with `helm.sh/hook` annotations creating RBAC resources
- Service accounts gaining privileges they did not have before a Helm install/upgrade
- Hook Jobs making network calls to external endpoints
- ClusterRoleBindings that reference the `default` service account (almost never legitimate)

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| K8s API calls creating ClusterRoleBindings from a Job pod | Hook is modifying RBAC |
| HTTP POST from a pod with `helm.sh/hook` label to external IP | Hook is exfiltrating data |
| DNS resolution for unknown domains from Job pods | Hook is phoning home |
| `kubectl` binary execution inside a container | Hook is running kubectl commands (privilege escalation) |

**Suricata rule -- detect kubectl execution inside containers:**

```yaml
alert http $K8S_PODS any -> $K8S_API_SERVER 443 (
  msg:"WEAKLINK - kubectl executed inside container pod (potential chart hook abuse)";
  flow:established,to_server;
  http.uri; content:"/api/v1/"; depth:8;
  http.header; content:"serviceaccount";
  classtype:attempted-admin;
  sid:2100011; rev:1;
)
```

### EDR / Process Indicators

- `kubectl create clusterrolebinding` executed inside a container (not from a human operator)
- Container processes with parent PID 1 running `curl`, `wget`, or `kubectl` -- typical hook behavior
- Files written to `/var/run/secrets/kubernetes.io/serviceaccount/` being read by non-standard processes
- Job pods that start, run for <30 seconds, and terminate -- characteristic of hook execution

### Example Detection Queries

**Splunk -- detect ClusterRoleBinding creation by Helm hooks:**

```spl
index=k8s sourcetype=kube_audit
  verb="create" objectRef.resource="clusterrolebindings"
| spath input=requestObject output=roleRef path=roleRef.name
| where roleRef="cluster-admin"
| spath input=requestObject output=subjects path=subjects{}
| table _time, user.username, objectRef.name, roleRef, subjects
| where NOT match(user.username, "(system:masters|cluster-admin)")
```

**Splunk -- detect self-deleting Helm hook Jobs:**

```spl
index=k8s sourcetype=kube_audit verb="delete" objectRef.resource="jobs"
| spath input=requestObject output=annotations path=metadata.annotations
| where match(annotations, "helm.sh/hook-delete-policy")
| table _time, objectRef.name, objectRef.namespace, user.username
| sort -_time
```

**KQL (Microsoft Sentinel) -- ClusterRoleBinding created by non-human:**

```kql
KubeAuditLogs
| where Verb == "create"
| where ObjectRef_Resource == "clusterrolebindings"
| extend RoleRef = parse_json(RequestObject).roleRef.name
| where RoleRef == "cluster-admin"
| where UserAgent !has "kubectl" or User_Username has "system:serviceaccount"
| project TimeGenerated, User_Username, ObjectRef_Name, RoleRef, UserAgent
```

**KQL -- Helm hook Jobs with short lifetime (potential evidence cleanup):**

```kql
KubePodInventory
| where PodLabel has "helm.sh/hook"
| extend RunDuration = datetime_diff('second', PodEndTime, PodStartTime)
| where RunDuration < 60
| project TimeGenerated, PodName, Namespace, RunDuration, ContainerImage
```

!!! tip "SOC Relevance"

    **Alerts you will see:**

    - "ClusterRoleBinding created granting cluster-admin" (K8s audit logs)
    - "Helm hook Job created ClusterRoleBinding" (admission controller)
    - "Short-lived Job pod in default namespace" (pod inventory)

    **Why this matters to your SOC:** Helm chart poisoning is the Kubernetes equivalent of a trojanized npm package. The attack is persistent -- the ClusterRoleBinding survives Helm uninstall. The evidence is ephemeral -- the hook Job deletes itself. Detection requires Kubernetes audit logs with RBAC events enabled.

    **Triage workflow:**

    1. **Check the ClusterRoleBinding** -- who does it grant access to? If it is the `default` service account, this is almost certainly malicious or severely misconfigured.
    2. **Trace the creator** -- which service account created it? If it is a Helm hook Job, check which chart was installed and when.
    3. **Check for persistence** -- does the ClusterRoleBinding still exist? An attacker may have already used it and cleaned up.
    4. **Audit the chart source** -- where was the chart pulled from? Check the Helm release history (`helm history`) and chart registry access logs.
    5. **Scope blast radius** -- any pod in any namespace running as the `default` service account now has cluster-admin. Enumerate all such pods.

    **False positive rate:** Very low. `cluster-admin` grants should be rare and always intentional. ClusterRoleBindings created by Jobs (not humans) are especially suspicious.

!!! example "CI Integration"

    Add this to your GitHub Actions pipeline to scan Helm charts for dangerous hooks and RBAC resources before they reach a cluster.

    **`.github/workflows/helm-hook-scan.yml`:**

    ```yaml
    name: Helm Chart Security Scan

    on:
      pull_request:
        paths:
          - "**/templates/**"
          - "**/Chart.yaml"
          - "**/values.yaml"

    jobs:
      scan-hooks:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Install Helm
            uses: azure/setup-helm@v4

          - name: Scan for dangerous Helm hooks
            run: |
              echo "--- Scanning Helm charts for dangerous hooks ---"
              FOUND=0
              for chart_dir in $(find . -name "Chart.yaml" -exec dirname {} \;); do
                echo "Scanning: $chart_dir"
                RENDERED=$(helm template scan-check "$chart_dir" 2>/dev/null || true)

                # Check for ClusterRoleBinding with cluster-admin
                if echo "$RENDERED" | grep -A10 'kind: ClusterRoleBinding' | grep -q 'cluster-admin'; then
                  echo "::error::CRITICAL: $chart_dir creates ClusterRoleBinding with cluster-admin"
                  FOUND=1
                fi

                # Check for hooks that create RBAC resources
                if echo "$RENDERED" | grep -B5 'helm.sh/hook' | grep -q 'ClusterRole'; then
                  echo "::error::WARNING: $chart_dir has hooks that modify RBAC"
                  FOUND=1
                fi

                # Check for hooks running kubectl
                if echo "$RENDERED" | grep -A30 'helm.sh/hook' | grep -q 'kubectl'; then
                  echo "::error::WARNING: $chart_dir has hooks running kubectl"
                  FOUND=1
                fi

                # Check for hooks making network calls
                if echo "$RENDERED" | grep -A30 'helm.sh/hook' | grep -qE '(curl|wget|nc )'; then
                  echo "::error::CRITICAL: $chart_dir has hooks making network calls"
                  FOUND=1
                fi
              done

              if [ "$FOUND" -eq 1 ]; then
                echo ""
                echo "Helm charts contain dangerous patterns. Review required."
                exit 1
              fi
              echo "PASS: No dangerous hook patterns found."
    ```

## What You Learned

1. **Helm hooks run arbitrary code on your cluster** -- they create standard Kubernetes resources (Jobs, pods) that execute during the install lifecycle.
2. **Post-install hooks with `hook-delete-policy: hook-succeeded` are invisible** -- they run, succeed, and delete themselves. No trace in `helm list` or `kubectl get jobs`.
3. **Chart poisoning is persistent** -- a ClusterRoleBinding created by a hook survives Helm uninstall because hooks are managed separately from release resources.
4. **`helm template` is your pre-flight check** -- always render and review before installing. Look for unexpected resources, especially RBAC bindings and Jobs.
5. **Admission policies are the safety net** -- Kyverno or OPA Gatekeeper can block dangerous resource patterns even if code review misses them.
