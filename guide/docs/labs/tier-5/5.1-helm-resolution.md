# Lab 5.1: How Helm Charts Resolve Dependencies

<div class="lab-meta">
  <span>~35 minutes</span>
  <span>Intermediate</span>
  <span>Prerequisites: <a href="../tier-0/0.3-containers.md">Lab 0.3</a></span>
</div>

When you run `helm dependency update`, Helm resolves a tree of chart dependencies from one or more repositories -- public Helm repos, OCI registries, or local paths. This resolution process is the Kubernetes equivalent of `pip install` or `npm install`, and it carries the same supply chain risks.

A `Chart.yaml` with `version: ">=18.0.0"` tells Helm "give me the highest version that matches." If an attacker publishes a higher version of that chart on a public repo, Helm will pull it without question. In this lab, you will see exactly how this works, exploit it, and lock it down.

!!! info "Phase 1: UNDERSTAND -- How Helm Resolves Chart Dependencies"

    **Goal:** See how Helm decides which chart version to pull and from which repository.

### Step 1: Examine the application chart

```bash
cat /app/webapp/Chart.yaml
```

This chart depends on `redis`, `postgresql`, and `nginx`. Notice the version constraints:

- `redis` and `postgresql` use `>=` (range) from a public repo
- `nginx` uses an exact version from a private OCI registry

### Step 2: Check configured Helm repositories

```bash
helm repo list
```

You should see both `untrusted-public` and `private-charts`. Helm will search all configured repos when resolving dependencies.

### Step 3: See what is available

```bash
# What versions does the public repo have?
helm search repo untrusted-public/redis --versions
helm search repo untrusted-public/postgresql --versions

# What does the private registry have?
helm search repo private-charts/ --versions
```

### Step 4: Resolve and download dependencies

```bash
helm dependency update /app/webapp/
```

Watch the output. Helm resolves each dependency from the repository specified in `Chart.yaml`. With `>=18.0.0`, it picks the highest available version.

### Step 5: Inspect what was downloaded

```bash
ls -la /app/webapp/charts/
helm dependency list /app/webapp/
```

Check the version numbers. If a version higher than expected was pulled, Helm selected it because it matched the range constraint.

### Step 6: Understand Chart.lock

```bash
cat /app/webapp/Chart.lock
```

The lock file records the exact version and digest for each resolved dependency. If this file exists, `helm dependency build` will use it instead of re-resolving. Without it, every `helm dependency update` can produce different results.

!!! warning "Phase 2: BREAK -- Hijacking Chart Resolution"

    **Goal:** See how an attacker can hijack Helm dependency resolution by publishing a higher-version chart.

### Step 1: Examine the malicious chart

```bash
cat /app/malicious-redis-chart/Chart.yaml
```

Version 99.0.0 -- a suspiciously high version. This is the attacker's chart.

### Step 2: Look at the malicious templates

```bash
cat /app/malicious-redis-chart/templates/post-install-exfil.yaml
```

The chart includes a legitimate-looking redis deployment, but it also has a `post-install` hook disguised as a "health check." This hook:

1. Reads the Kubernetes service account token from the pod filesystem
2. Sends it to `attacker.example.com` via HTTP POST
3. Prints "Health check completed successfully" to look normal

### Step 3: Simulate the attack

The attacker publishes v99.0.0 to the public repository. When a developer runs `helm dependency update`, Helm picks 99.0.0 because it satisfies `>=18.0.0` and is the highest available version.

```bash
# Render the chart to see what would be deployed
helm template my-release /app/webapp/ 2>/dev/null | grep -A 30 'kind: Job'
```

The exfiltration job appears in the rendered output, but buried among hundreds of lines of YAML, it is easy to miss.

### Step 4: Understand the blast radius

If this chart were installed on a real cluster:

- The post-install hook runs as a Kubernetes Job
- It has access to the service account token mounted at `/var/run/secrets/`
- With that token, the attacker can authenticate to the Kubernetes API
- Depending on RBAC, they may be able to read secrets, deploy pods, or escalate privileges

The `helm install` output shows "deployed successfully." Nothing looks wrong.

!!! success "Phase 3: DEFEND -- Pinning Charts and Locking Dependencies"

    **Goal:** Eliminate the resolution attack vector by pinning exact versions and using trusted repos only.

### Step 1: Remove the untrusted repository

```bash
helm repo remove untrusted-public
helm repo list
```

Now Helm can only pull from `private-charts`.

### Step 2: Pin exact versions in Chart.yaml

Edit `/app/webapp/Chart.yaml` to replace version ranges with exact pins, and point all dependencies to the private registry:

```bash
cat > /app/webapp/Chart.yaml << 'EOF'
apiVersion: v2
name: webapp
description: Internal web application
type: application
version: 1.0.0
appVersion: "1.0.0"

dependencies:
  - name: redis
    version: "18.6.1"
    repository: "oci://private-registry:5000/charts"
  - name: postgresql
    version: "13.4.3"
    repository: "oci://private-registry:5000/charts"
  - name: nginx
    version: "1.2.0"
    repository: "oci://private-registry:5000/charts"
EOF
```

### Step 3: Rebuild the lock file

```bash
helm dependency update /app/webapp/
cat /app/webapp/Chart.lock
```

The lock file now contains exact versions with SHA256 digests. Any tampering with the chart archives will be detected.

### Step 4: Verify the defense

```bash
# Confirm only private registry is used
helm repo list

# Confirm exact versions in dependencies
helm dependency list /app/webapp/

# Confirm digests exist in lock file
grep 'digest:' /app/webapp/Chart.lock
```

### Step 5: Run verification

```bash
weaklink verify 5.1
```

### Additional defenses

1. **Use OCI registries** -- OCI-based Helm charts support content-addressable storage with SHA digests, making substitution attacks harder.
2. **Sign charts with Helm provenance** -- `helm package --sign` creates a `.prov` file. `helm install --verify` checks it.
3. **Mirror charts locally** -- Pull charts into your private registry and point all Chart.yaml dependencies to it.
4. **Audit Chart.lock in CI** -- Block PRs that modify Chart.yaml without updating Chart.lock.

!!! danger "Phase 4: DETECT -- Finding Chart Hijacking in Production"

    **Goal:** Detect when Helm pulls charts from unexpected sources or installs suspiciously high versions.

### SIEM / Log Indicators

The core signal is **Helm resolving chart dependencies from untrusted repositories** or **chart versions jumping to unusually high numbers**. This shows up in Helm audit logs, container registry access logs, and network telemetry from CI/CD systems.

**What to look for:**

- Helm pulling charts from public repos when corporate policy requires private repos only
- Chart version numbers jumping unexpectedly (e.g., redis going from 18.6.1 to 99.0.0)
- `helm install` or `helm upgrade` commands referencing repositories not in the approved list
- Post-install hooks creating Jobs, ClusterRoleBindings, or other security-sensitive resources
- Outbound HTTP from Helm hook Jobs to external endpoints

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| HTTP GET to `charts.example.com` from CI runners | Helm pulling charts from a public repository |
| DNS query for unknown chart registries from build VLAN | Dependency resolution reaching external repos |
| HTTP POST from a K8s Job pod to external IP | Post-install hook exfiltrating data |
| OCI registry pull from `ghcr.io` or `docker.io` for chart artifacts | Chart being pulled from public OCI registry |

**Suricata rule -- detect Helm chart pulls from public repos:**

```yaml
alert http $K8S_NODES any -> $EXTERNAL_NET any (
  msg:"WEAKLINK - Helm chart pull from public repository";
  flow:established,to_server;
  http.uri; content:"/charts/"; depth:8;
  http.header; content:"User-Agent: helm";
  classtype:policy-violation;
  sid:2100010; rev:1;
)
```

### EDR / Process Indicators

- `helm dependency update` or `helm dep up` commands executed with repositories not in the approved list
- `helm install` / `helm upgrade` using `--repo` flag pointing to external URLs
- Chart archive files (`.tgz`) appearing in `/tmp` or Helm cache directories with unexpected version numbers
- Kubernetes Jobs created with `helm.sh/hook: post-install` annotation running `curl`, `wget`, or network tools

### Example Detection Queries

**Splunk -- detect Helm pulling from unapproved repos:**

```spl
index=cicd sourcetype=helm_audit
  action="dependency_update"
| rex field=repo_url "(?<repo_domain>[^/]+\.[^/]+)"
| where NOT match(repo_domain, "(private-registry\.corp|internal-charts\.corp)")
| table _time, user, chart_name, repo_url, repo_domain
```

**Splunk -- detect suspiciously high chart versions:**

```spl
index=k8s sourcetype=helm_releases
| rex field=chart_version "(?<major>\d+)\."
| where tonumber(major) > 50
| table _time, release_name, chart_name, chart_version, namespace
```

**KQL (Microsoft Sentinel) -- Helm hook creating ClusterRoleBinding:**

```kql
KubeAuditLogs
| where Verb == "create"
| where ObjectRef_Resource == "clusterrolebindings"
| where UserAgent has "helm"
| extend Annotations = parse_json(RequestObject)
| where Annotations has "helm.sh/hook"
| project TimeGenerated, UserAgent, ObjectRef_Name, RequestObject
```

**KQL -- outbound traffic from Helm hook Jobs:**

```kql
ContainerNetworkLogs
| where PodLabels has "helm.sh/hook"
| where Direction == "outbound"
| where DestinationIP !startswith "10." and DestinationIP !startswith "172.16."
| summarize Connections=count() by PodName, DestinationIP, DestinationPort
```

!!! tip "SOC Relevance"

    **Alert you will see:** "Helm chart resolved from unapproved repository" or "Unusually high chart version installed"

    Helm dependency resolution is the Kubernetes equivalent of `pip install --extra-index-url`. If your CI/CD pipeline pulls charts from multiple repositories, the highest version wins -- regardless of which repo it comes from.

    **Triage steps:**

    1. Check which chart was pulled and from which repository
    2. Compare the version against what is expected (check `Chart.lock` in version control)
    3. Render the chart with `helm template` and look for hooks, Jobs, or RBAC resources
    4. If a post-install hook exists: check what it does -- does it make network calls? Create RBAC bindings?
    5. If confirmed malicious: check all clusters that pulled from the same repo in the same timeframe

    **False positive rate:** Low for version anomalies (>50.x.x). Medium for public repo pulls (some teams legitimately use public charts, but they should be mirrored).

!!! example "CI Integration"

    Add this to your GitHub Actions pipeline to enforce chart version pinning and repository restrictions.

    **`.github/workflows/helm-chart-check.yml`:**

    ```yaml
    name: Helm Chart Dependency Check

    on:
      pull_request:
        paths:
          - "**/Chart.yaml"
          - "**/Chart.lock"
          - "**/values.yaml"

    jobs:
      check-helm-deps:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Install Helm
            uses: azure/setup-helm@v4

          - name: Reject version ranges in Chart.yaml
            run: |
              echo "--- Checking for version ranges in Chart.yaml ---"
              FOUND=0
              for f in $(find . -name "Chart.yaml" -not -path "*/charts/*"); do
                if grep -E 'version:\s*"[><=~^]' "$f"; then
                  echo "::error file=$f::Chart dependency uses version range. Pin to exact version."
                  FOUND=1
                fi
              done
              if [ "$FOUND" -eq 1 ]; then
                exit 1
              fi
              echo "PASS: All chart dependencies use exact versions."

          - name: Verify Chart.lock exists and is up to date
            run: |
              echo "--- Checking Chart.lock files ---"
              for chart_yaml in $(find . -name "Chart.yaml" -not -path "*/charts/*"); do
                dir=$(dirname "$chart_yaml")
                if grep -q "dependencies:" "$chart_yaml"; then
                  if [ ! -f "$dir/Chart.lock" ]; then
                    echo "::error file=$chart_yaml::Chart has dependencies but no Chart.lock. Run 'helm dependency update'."
                    exit 1
                  fi
                  if ! grep -q "digest:" "$dir/Chart.lock"; then
                    echo "::error file=$dir/Chart.lock::Chart.lock missing digests."
                    exit 1
                  fi
                fi
              done
              echo "PASS: All Chart.lock files present with digests."

          - name: Reject public repositories in Chart.yaml
            env:
              ALLOWED_REPOS: "oci://private-registry.corp,https://charts.internal.corp"
            run: |
              echo "--- Checking chart repository sources ---"
              FOUND=0
              for f in $(find . -name "Chart.yaml" -not -path "*/charts/*"); do
                while IFS= read -r repo; do
                  repo=$(echo "$repo" | xargs)
                  ALLOWED=0
                  IFS=',' read -ra REPOS <<< "$ALLOWED_REPOS"
                  for allowed in "${REPOS[@]}"; do
                    if [[ "$repo" == "$allowed"* ]]; then
                      ALLOWED=1
                      break
                    fi
                  done
                  if [ "$ALLOWED" -eq 0 ] && [ -n "$repo" ]; then
                    echo "::error file=$f::Unapproved chart repository: $repo"
                    FOUND=1
                  fi
                done < <(grep "repository:" "$f" | sed 's/.*repository:\s*//' | tr -d '"')
              done
              if [ "$FOUND" -eq 1 ]; then
                exit 1
              fi
              echo "PASS: All chart repositories are approved."
    ```

## What You Learned

1. **Helm resolves chart dependencies like any package manager** -- version ranges (`>=`, `^`, `~`) allow Helm to pick higher versions, including malicious ones.
2. **Chart.lock is the Helm lockfile** -- it pins exact versions with SHA digests. Without it, dependency resolution is non-deterministic.
3. **Public Helm repos are attack surface** -- anyone can publish charts. Use private registries or mirror vetted charts.
4. **Post-install hooks are dangerous** -- they run as Kubernetes Jobs with access to service account tokens and the cluster API.
5. **OCI registries add integrity** -- content-addressable storage makes chart substitution harder than traditional Helm repos.
