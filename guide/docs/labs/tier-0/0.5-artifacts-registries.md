# Lab 0.5: Artifacts & Registries

## Overview

Once source code is built, it's packaged into an artifact (like a Python wheel, an npm tarball, or a Docker image) and uploaded to a **Registry**. Other developers and systems download these artifacts to use them.

This lab teaches you how artifacts are published, distributed, and why relying purely on version numbers without cryptographic hashes is extremely dangerous.

**Estimated Time:** 30 minutes
**Difficulty:** Beginner

---

## The Infrastructure

This lab uses:
* **PyPI Private** (`http://pypi-private:8080`): A private Python package registry.
* **Verdaccio** (`http://verdaccio:4873`): A private npm registry.
* **OCI Registry** (`http://registry:5000`): A container image registry.

---

???+ success "Phase 1: UNDERSTAND — Publishing and Downloading"

    **Goal:** Build a Python package, publish it, and download it back.

    1. Navigate to the demo library source code:
       ```bash
       cd /labs/tier-0-foundations/0.5-artifacts-registries/src/packages/demo-lib
       ```

    2. Inspect the package files (`setup.py`, `demo_lib.py`). Notice the version is `1.0.0`.

    3. Build the package into an artifact (a `.tar.gz` Source Distribution):
       ```bash
       python setup.py sdist
       ```

    4. Publish the artifact to our private PyPI registry using `twine`:
       ```bash
       pip install twine
       twine upload --repository-url http://pypi-private:8080/ dist/*
       # Enter blank or any credentials, the lab registry accepts anything.
       ```

    5. Verify the package is available:
       ```bash
       curl http://pypi-private:8080/simple/demo-lib/
       ```

---

???+ danger "Phase 2: BREAK — Tampering with the Artifact"

    **Goal:** Modify the package and replace it in the registry without changing the version number.

    Registries often allow packages to be overwritten, or an attacker with registry credentials might delete v1.0.0 and replace it with a malicious v1.0.0.

    1. Edit the source code `demo_lib.py`, adding a backdoor:
       ```python
       import os
       print(f"TAMPERED: running as {os.getenv('USER', 'unknown')}")
       ```

    2. Rebuild the artifact:
       ```bash
       python setup.py sdist
       ```

    3. Re-upload it to the registry. The registry will accept the new file for `1.0.0`!
       ```bash
       twine upload --repository-url http://pypi-private:8080/ dist/* --skip-existing
       # Our simple lab registry actually allows overwriting. Pypiserver can be configured to overwrite.
       # Let's forcefully push it.
       ```

    4. Anyone performing a clean `pip install demo-lib==1.0.0` will now get your malicious code instead of the original code, despite the version number being exactly the same.

---

???+ shield "Phase 3: DEFEND — Using Cryptographic Hashes"

    **Goal:** Ensure we only download the exact artifact we expect, regardless of what the registry serves.

    Because an artifact's contents can change even if the version number stays the same, we must use cryptographic checksums (hashes) to verify integrity.

    1. First, get the hash of the *good* file. Run the provided helper script:
       ```bash
       /labs/tier-0-foundations/0.5-artifacts-registries/src/scripts/verify-integrity.sh
       ```

    2. Notice the output provides a SHA256 hash. You can enforce this hash in `requirements.txt`:
       ```text
       demo-lib==1.0.0 --hash=sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
       ```

    3. If you run `pip install -r requirements.txt` now, and the registry serves the tampered file, `pip` will throw an error and refuse to install it, preventing the attack.
