# Lab 0.4: How CI/CD Works

## Overview

Modern software isn't just compiled on a developer's laptop—it's built, tested, and deployed automatically by Continuous Integration and Continuous Deployment (CI/CD) pipelines. This lab introduces the core concepts of CI/CD, focusing on GitHub Actions/Gitea Actions. You'll learn how pipelines run, how they access secrets, and why they represent a massive target for attackers.

**Estimated Time:** 30 minutes
**Difficulty:** Beginner

---

## The Infrastructure

In this lab environment, we use Gitea (a lightweight Git server similar to GitHub). Gitea has an integrated CI/CD system called **Gitea Actions**, which is fully compatible with GitHub Actions syntax. 

When you push code, Gitea checks the `.gitea/workflows/` directory. If it finds a YAML configuration file matching the event, it spins up a **Runner** (a background server) to execute the steps defined in the workflow.

---

???+ success "Phase 1: UNDERSTAND — Exploring the Pipeline"

    **Goal:** Understand how the CI pipeline is configured, what it executes, and how it accesses secrets.

    1. Access the Workstation terminal and switch to the lab directory:
       ```bash
       cd /workspace/ci-demo
       ```

    2. Inspect the CI workflow configuration:
       ```bash
       cat .gitea/workflows/ci.yml
       ```

    3. Notice the following critical sections:
       * `on: push` — The pipeline triggers whenever code is pushed.
       * `runs-on: ubuntu-latest` — The environment where the code executes.
       * `env: DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }}` — The pipeline injects a secret into the environment so the deploy script can authenticate.

    4. In your terminal, make a benign change to `app.py`, commit, and push it:
       ```bash
       echo "# A benign comment" >> app.py
       git add app.py
       git commit -m "Add comment"
       git push
       ```

    5. Go to the Gitea web interface (http://localhost:3000), log in as `labadmin` (password: `SupplyChainLab1!`), go to the `ci-demo` repo, and click the **Actions** tab. Watch your pipeline run and succeed.

???+ danger "Phase 2: BREAK — Poisoning the Pipeline"

    **Goal:** Exploit your access to the repository to steal the `DEPLOY_KEY` secret.

    As a developer, you have permission to push code. Because the CI configuration lives *in the repository*, you can change what the pipeline does. This is known as **Poisoned Pipeline Execution (PPE)**.

    1. Edit the `.gitea/workflows/ci.yml` file. Add a new step to exfiltrate the secret. For this lab, we'll just write it to a file, but in reality, an attacker would `curl` it to their own server.

       ```yaml
             - name: Exfiltrate secrets
               env:
                 DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }}
               run: |
                 echo "The secret is: $DEPLOY_KEY" > /tmp/stolen-secret.txt
       ```
       *Note: Place this step right after "Run tests". Keep the indentation aligned!*

    2. Commit and push your modified pipeline:
       ```bash
       git add .gitea/workflows/ci.yml
       git commit -m "Update pipeline"
       git push
       ```

    3. Go back to Gitea and watch the Action run. Once it finishes, use `kubectl exec` or the Workstation to verify the secret was stolen (the lab environment simulates the runner inside the workstation for simplicity in checking).

---

???+ shield "Phase 3: DEFEND — Securing the CI/CD Pipeline"

    **Goal:** Prevent unauthorized pipeline modifications and secret exfiltration.

    1. **Restrict Pipeline Modifications:** In Gitea, navigate to Repo Settings -> Branches. Enable branch protection for `main`. Require pull request reviews before merging. This stops attackers from directly pushing modified `.gitea/workflows/` files.
    
    2. **Use Ephemeral Credentials:** Never store static, long-lived secrets (like `DEPLOY_KEY`) in CI. Instead, use OpenID Connect (OIDC) to generate short-lived, scoped tokens dynamically during the build.

    3. Revert your malicious commit:
       ```bash
       git revert HEAD --no-edit
       git push
       ```
