# Lab 1.5: Manifest Confusion

<div class="lab-meta">
  <span>~30 minutes</span>

In 2023, security researcher Darcy Clarke discovered a fundamental flaw in the npm ecosystem: the package metadata that `npm view` shows you can differ from what's actually inside the tarball you install. Auditing tools, security scanners, and developers all trusted the registry API -- but the registry was lying.

This lab teaches you how npm packages are published, how registry metadata can be manipulated, and how to detect and defend against manifest confusion attacks.

!!! info "Phase 1: UNDERSTAND -- How npm Package Publishing Works"

    **Goal:** Learn the gap between registry metadata and actual tarball contents.

### How npm packages are published

When a developer publishes a package, two things happen:

1. `npm pack` creates a `.tgz` tarball containing the code and `package.json`
2. `npm publish` uploads that tarball to the registry, which extracts metadata from it

The registry then serves **two things** for each package:

- **Metadata** (via the API): what `npm view` shows -- name, version, dependencies, scripts
- **Tarball** (the download): what `npm install` actually extracts into `node_modules/`

### Step 1: Inspect a normal package via the registry API

```bash
npm view safe-utils
```

Note the dependencies listed (`lodash`), the version, and the description.

### Step 2: Download and inspect the actual tarball

```bash
mkdir /tmp/inspect && cd /tmp/inspect
npm pack safe-utils
tar xzf safe-utils-1.0.0.tgz
cat package/package.json
```

Compare what you see here with what `npm view` showed. For a normal package, they match.

### Step 3: Now check the crafted package

```bash
npm view crafted-widget
```

Look at the dependencies. What do you see? Just `lodash`. Looks safe, right?

### Step 4: Download and inspect the crafted package tarball

```bash
cd /tmp/inspect
npm pack crafted-widget
tar xzf crafted-widget-2.1.0.tgz
cat package/package.json
```

**Look carefully at the dependencies and scripts.** Compare them to what `npm view` showed.

You should see:

- The tarball has `"evil-pkg": "*"` in dependencies -- the registry API did NOT show this
- The tarball has a `postinstall` script -- the registry API did NOT show this

This is **manifest confusion**: the registry metadata and the tarball contents disagree.

### Key Insight

Any tool that relies on `npm view` or the registry API to audit packages will miss what's actually installed. The attack surface is the gap between what the registry **says** and what the tarball **contains**.

### Step 2: Create a project that depends on crafted-widget

```bash
  "version": "1.0.0",
  "dependencies": {
    "crafted-widget": "^2.1.0"

}

```

### Step 3: Audit before installing (the false sense of security)

If you were a security team, you'd probably check what this dependency pulls in:

```bash
npm view crafted-widget dependencies
```

Output: `{ lodash: '^4.17.21' }` -- looks safe. No suspicious dependencies.

### Step 4: Install the package

```bash
npm install
```

Watch the output. You'll see `evil-pkg` being installed and its postinstall script running.

### Step 5: Verify the attack

```bash
# The malicious postinstall script wrote a marker file
cat /tmp/manifest-confusion-pwned

# evil-pkg is in node_modules -- but was NEVER shown by npm view
ls node_modules/evil-pkg/

# The crafted-widget tarball had dependencies the registry lied about
cat node_modules/crafted-widget/package.json | head -20
```

### What just happened?

1. You checked `npm view crafted-widget` -- it showed only `lodash` as a dependency
2. You installed `crafted-widget` -- npm downloaded the **tarball**, which contained `evil-pkg` as a dependency
3. npm installed `evil-pkg` and ran its `postinstall` script, which wrote to `/tmp/manifest-confusion-pwned`
4. **Every security tool that trusts registry metadata would have missed this entirely**

### Real-world impact

This vulnerability (CVE-2022-25881 / Darcy Clarke's disclosure) affected:

- `npm audit` -- used registry metadata, not tarball contents
- Socket.dev, Snyk, and other security scanners at the time
- GitHub's dependency graph
- Any tool calling the npm registry API to determine dependencies

!!! danger "Phase 4: DETECT -- Finding Manifest Confusion in Production"

    **Goal:** Identify manifest confusion attacks through registry monitoring, build telemetry, and dependency auditing.

### What This Attack Looks Like in Logs

Manifest confusion is harder to detect than most supply chain attacks because the malicious payload is *invisible* to API-based tooling. The registry says one thing, the tarball contains another. Detection requires comparing both sources or monitoring what actually gets installed.

**Registry/API indicators:**

- `npm view <package>` shows different dependencies than what ends up in `node_modules/` after install
- A package's registry metadata shows zero `postinstall` scripts, but the installed `package.json` contains one
- A package's dependency count in the registry differs from the count in the extracted tarball

**Network indicators:**

- During `npm install`, packages are downloaded that don't appear in the dependency tree shown by `npm view` or `npm ls --all`
- DNS lookups for domains referenced in hidden `postinstall` scripts (exfiltration endpoints)
- Outbound connections from `node` processes spawned during `npm install` (postinstall execution)

**EDR/process indicators:**

- `npm install` spawning unexpected child processes (especially `node` running scripts from packages not in the declared dependency tree)
- File system writes during `npm install` to locations outside `node_modules/` (e.g., `/tmp/`, home directory, system paths)
- The installed `node_modules/` directory contains packages not present in the dependency graph from `npm ls`

### Detection Queries

**Splunk -- Packages installed that weren't in the declared dependency tree:**

```splunk
index=ci sourcetype=build_log command="npm install" OR command="npm ci"
| rex field=_raw "added (?<installed_count>\d+) packages"
| join build_id [
  search index=ci sourcetype=build_log command="npm ls --all --json"
  | spath output=declared_count path=dependencies{}
  | stats dc(declared_count) as declared_count by build_id
]
| where installed_count > declared_count
| table _time, build_id, project, installed_count, declared_count
```

**Splunk -- Postinstall script execution from unexpected packages:**

```splunk
index=ci sourcetype=build_log
| rex field=_raw "(?<package_name>[\w\-@/]+).*postinstall"
| where NOT package_name IN ("husky", "esbuild", "sharp", "node-gyp", "puppeteer")
| stats count by build_id, package_name
| where count > 0
| table _time, build_id, package_name
```

**Splunk -- Compare npm view vs actual installed dependencies:**

```splunk
index=security sourcetype=manifest_audit action=compare
| where registry_deps != tarball_deps
| eval hidden_deps=mvfilter(NOT match(tarball_deps, registry_deps))
| table _time, package_name, package_version, registry_deps, tarball_deps, hidden_deps
```

**KQL (Microsoft Sentinel) -- Unexpected child processes during npm install:**

```kql
DeviceProcessEvents
| where InitiatingProcessFileName == "npm" or InitiatingProcessCommandLine has "npm install"
| where FileName in ("node", "sh", "bash")
| where ProcessCommandLine has "postinstall"
| where Timestamp between (ago(24h) .. now())
| project Timestamp, DeviceName, ProcessCommandLine, InitiatingProcessCommandLine
| join kind=leftanti (
    // Known-good postinstall scripts
    datatable(AllowedPackage:string) ["husky", "esbuild", "sharp", "node-gyp"]
    | extend dummy=1
) on $left.ProcessCommandLine has $right.AllowedPackage
```

**KQL -- Packages in node_modules not in dependency graph:**

```kql
DeviceFileEvents
| where FolderPath has "node_modules" and ActionType == "FileCreated"
| where FileName == "package.json"
| extend PackageName = extract(@"node_modules[/\\](@?[^/\\]+[/\\]?[^/\\]*)[/\\]package.json", 1, FolderPath)
| summarize InstalledPackages=make_set(PackageName) by DeviceName, Timestamp=bin(Timestamp, 1h)
```

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| Supply Chain Compromise: Compromise Software Dependencies | **T1195.002** | The core attack -- a published package contains hidden dependencies not visible through the registry API |
| Masquerading: Match Legitimate Name | **T1036.005** | The package presents a clean, legitimate-looking manifest to auditing tools while hiding its true contents |
| Hijack Execution Flow | **T1574** | Hidden `postinstall` scripts hijack the normal install flow to execute attacker code |

!!! tip "SOC Relevance"

    **Why SOC analysts care about manifest confusion:**

    - **Blind spot in existing tooling**: Most dependency scanning tools (Snyk, npm audit, GitHub Dependabot) historically relied on registry API metadata. If the registry lies, your scanner lies. Confirm your scanning tool checks tarball contents, not just API responses.
    - **High-value detection**: Alert on any `postinstall` script execution from packages not on your known-good allowlist. Most legitimate packages don't need postinstall hooks. A short allowlist (husky, esbuild, sharp, node-gyp, puppeteer) covers 90% of legitimate cases.
    - **Pre-install audit**: Before any new dependency is added, run `npm pack <package> && tar xzf <package>.tgz && diff <(npm view <package> dependencies) <(node -e "console.log(JSON.stringify(require('./package/package.json').dependencies))")`. If they differ, escalate.
    - **Incident response**: If you suspect manifest confusion, compare `npm view <package> dependencies` against `cat node_modules/<package>/package.json | jq .dependencies`. Any mismatch is confirmed. Check `/tmp/` and other common drop locations for post-install artifacts.

!!! example "CI Integration"

    **GitHub Actions: Manifest confusion detection on every install**

    This workflow downloads packages, extracts tarballs, and compares registry metadata against actual tarball contents. Copy-paste into `.github/workflows/manifest-verify.yml`:

    ```yaml
    name: Detect Manifest Confusion
    on:
      pull_request:
        paths:
          - 'package.json'
          - 'package-lock.json'

    jobs:
      check-manifests:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4
          - uses: actions/setup-node@v4
            with:
              node-version: '20'
          - name: Compare registry metadata vs tarball contents
            run: |
              set -euo pipefail
              FAILED=0

              # Get all direct dependencies from package.json
              DEPS=$(node -e "
                const pkg = require('./package.json');
                const deps = {...(pkg.dependencies || {}), ...(pkg.devDependencies || {})};
                console.log(Object.keys(deps).join('\n'));
              ")

              mkdir -p /tmp/manifest-check
              cd /tmp/manifest-check

              for dep in $DEPS; do
                echo "Checking $dep..."

                # Get registry metadata
                REGISTRY_DEPS=$(npm view "$dep" dependencies --json 2>/dev/null || echo "{}")
                REGISTRY_SCRIPTS=$(npm view "$dep" scripts --json 2>/dev/null || echo "{}")

                # Download and extract tarball
                npm pack "$dep" --quiet 2>/dev/null
                TARBALL=$(ls -t *.tgz | head -1)
                tar xzf "$TARBALL"

                # Get tarball metadata
                TARBALL_DEPS=$(node -e "console.log(JSON.stringify(require('./package/package.json').dependencies || {}))")
                TARBALL_SCRIPTS=$(node -e "console.log(JSON.stringify(require('./package/package.json').scripts || {}))")

                # Compare dependencies
                if [ "$REGISTRY_DEPS" != "$TARBALL_DEPS" ]; then
                  echo "::error::MANIFEST CONFUSION in $dep: registry dependencies differ from tarball"
                  echo "  Registry: $REGISTRY_DEPS"
                  echo "  Tarball:  $TARBALL_DEPS"
                  FAILED=1
                fi

                # Check for hidden postinstall scripts
                if echo "$TARBALL_SCRIPTS" | grep -q "postinstall\|preinstall\|install" && \
                   ! echo "$REGISTRY_SCRIPTS" | grep -q "postinstall\|preinstall\|install"; then
                  echo "::error::HIDDEN INSTALL SCRIPT in $dep: tarball has install hooks not shown in registry"
                  echo "  Registry scripts: $REGISTRY_SCRIPTS"
                  echo "  Tarball scripts:  $TARBALL_SCRIPTS"
                  FAILED=1
                fi

                # Clean up for next package
                rm -rf package "$TARBALL"
              done

              if [ "$FAILED" -eq 1 ]; then
                echo "::error::Manifest confusion detected. Review flagged packages before merging."
                exit 1
              fi
              echo "All package manifests verified."

          - name: Enforce npm ci over npm install
            run: |
              # Ensure lockfile exists and is in sync
              npm ci
              echo "Lockfile integrity verified via npm ci."
    ```

## What You Learned

| Concept | Real-World Application |
|---------|----------------------|
| Registry metadata vs tarball contents | Never trust `npm view` output for security decisions |
| Manifest confusion attack | CVE disclosed by Darcy Clarke, affected the entire npm ecosystem |
| Tarball inspection | Always `npm pack` + extract to see what you're actually installing |
| Lockfile integrity hashes | `package-lock.json` with `integrity` fields catches tampering |
| `npm ci` vs `npm install` | `npm ci` strictly follows the lockfile; `npm install` can modify it |
