When you run `pip install`, pip doesn't just download one package -- it resolves an entire dependency tree. Understanding how this works is essential because **every supply chain attack exploits something about this process**.

This lab gives you a private PyPI registry, a public PyPI registry, and a Python app with dependencies. You'll see exactly how pip decides which package version to install and from which source.

| Public PyPI | `pypi-public:8080` | Simulated public PyPI (starts with a higher-version fake package) |

!!! info "Phase 1: UNDERSTAND -- How pip Resolves Dependencies"

    **Goal:** See how pip decides which version to install and from which source.

### Step 1: Look at your application

```bash
cat /app/requirements.txt
```

You have an app that depends on `internal-utils==1.0.0` and `data-processor==2.0.0`. These are internal company packages hosted on the private registry.

### Step 2: Check pip configuration

```bash
cat /etc/pip.conf
```

Notice the configuration:

- `index-url` points to the **private** registry
- `extra-index-url` points to the **public** registry

This is a common (and dangerous) pattern in corporate environments.

### Step 3: Install dependencies

```bash
pip install -r requirements.txt
```

Watch the output. Pip resolves all packages from the private registry. Everything works.

### Step 4: Visualize the dependency tree

```bash
pipdeptree
```

You can see the full tree: `data-processor` depends on `internal-utils`, which depends on `logging-helper`. This is **dependency resolution** -- pip figures out the entire chain.

### Step 5: Test the app

```bash
python app.py
```

It should print that version 1.0.0 of `internal-utils` is installed.

### Step 6: Explore lockfiles

Generate a lockfile:

```bash
pip freeze
```

This shows every installed package at its exact version. This is what a "lockfile" captures. Without it, the same `requirements.txt` can produce different results at different times.

### Step 7: Experiment with version pins

Edit requirements.txt to change the version pin:

```bash
# Change internal-utils==1.0.0 to internal-utils>=1.0.0
sed -i 's/internal-utils==1.0.0/internal-utils>=1.0.0/' requirements.txt
```

Now reinstall:

```bash
pip install --force-reinstall -r requirements.txt
```

What version did pip install? When you use `>=` instead of `==`, pip looks for the **highest available version across ALL configured registries**.

Check:

```bash
pip show internal-utils
python app.py
```

If you got version 99.0.0, you just saw the problem with `--extra-index-url`.

**Reset the requirements before continuing:**

```bash
sed -i 's/internal-utils>=1.0.0/internal-utils==1.0.0/' requirements.txt
pip install --force-reinstall -r requirements.txt
```

---

!!! warning "Phase 2: BREAK -- Exploiting Multi-Registry Resolution"

    **Goal:** See how pip's resolution behavior across multiple registries leads to installing attacker-controlled packages.

### Step 1: Check what each registry has

```bash
# What's on the private registry?
curl -s http://pypi-private:8080/simple/ | grep -o 'href="[^"]*"'

# What's on the public registry?
curl -s http://pypi-public:8080/simple/ | grep -o 'href="[^"]*"'
```

The public registry has `internal-utils` version 99.0.0 -- a fake higher version.

### Step 2: See how pip resolves with --extra-index-url

With the current config (`extra-index-url`), pip checks **both** registries and picks the highest version:

```bash
# Loosen the pin to allow any version
sed -i 's/internal-utils==1.0.0/internal-utils/' requirements.txt

# Reinstall
pip install --force-reinstall -r requirements.txt
```

### Step 3: Verify the damage

```bash
pip show internal-utils
python app.py
```

Pip installed version **99.0.0** from the public registry, even though your private registry has the legitimate 1.0.0. This is because:

1. `--extra-index-url` tells pip to search **both** indexes
2. Pip picks the **highest version** across all sources
3. 99.0.0 > 1.0.0, so the public (attacker-controlled) version wins

This is the foundational issue behind **dependency confusion attacks** (covered in Lab 1.2).

### Step 4: Understand the root cause

```bash
cat /etc/pip.conf
```

The problem is `extra-index-url`. It doesn't mean "fallback" -- it means "also check here." Pip merges results from all sources and picks the highest version.

---

!!! success "Phase 3: DEFEND -- Single Registry and Lockfiles"

    **Goal:** Eliminate the multi-registry attack vector and lock down exact versions.

### Step 1: Fix the pip configuration

Replace the dangerous config with the safe one:

```bash
cp /etc/pip-configs/pip.conf.safe /etc/pip.conf
cat /etc/pip.conf
```

Now pip only knows about the private registry. The public registry is not consulted at all.

### Step 2: Fix and reinstall

```bash
# Restore the exact pin
sed -i 's/^internal-utils$/internal-utils==1.0.0/' requirements.txt

# Reinstall from private registry only
pip install --force-reinstall -r requirements.txt
```

### Step 3: Verify correct version

```bash
pip show internal-utils
python app.py
```

You should see version 1.0.0 -- the legitimate version from the private registry.

### Step 4: Create a lockfile

```bash
pip freeze > requirements.lock
cat requirements.lock
```

This lockfile captures the exact versions of every installed package. Commit this to version control -- it ensures that every developer and CI system installs the same versions.

### Step 5: Verify the lab

Run the verification from your host terminal (outside the workstation):

```bash
weaklink verify 1.1
```

!!! danger "Phase 4: DETECT -- Finding Multi-Registry Abuse in Production"

    **Goal:** Identify when pip reaches out to unexpected registries and installs packages from untrusted sources.

### SIEM / Log Indicators

The core signal is **outbound HTTP traffic from build systems to public registries when they should only talk to your private registry**. This shows up in web proxy logs, DNS logs, and pip's own output.

**What to look for:**

- DNS queries to `pypi.org` or `files.pythonhosted.org` from CI runners or build servers
- HTTP GET requests to `/simple/<package-name>/` on public PyPI from hosts that have a private registry configured
- pip log lines containing `Looking in indexes:` with multiple URLs (proves `--extra-index-url` is in use)
- Package versions jumping unexpectedly (e.g., `internal-utils` going from `1.0.0` to `99.0.0`)

### Network Indicators

**Proxy / Firewall logs** -- your forward proxy or firewall should capture these:

| Indicator | What It Means |
|-----------|---------------|
| `dest_host=pypi.org` from CI subnet | Build system reaching public PyPI |
| `dest_host=files.pythonhosted.org` from CI subnet | Downloading packages from public PyPI |
| DNS query for `pypi.org` from `10.0.0.0/8` build VLAN | Name resolution for public registry |
| HTTP 200 on `/simple/<internal-package-name>/` to public PyPI | Public registry has a package matching an internal name |

**Suricata rule** to detect pip reaching public PyPI from internal build infrastructure:

```yaml
alert http $BUILD_SERVERS any -> $EXTERNAL_NET any (
  msg:"WEAKLINK - pip connecting to public PyPI from build server";
  flow:established,to_server;
  http.host; content:"pypi.org";
  classtype:policy-violation;
  sid:2100001; rev:1;
)
```

### EDR / Process Indicators

Dependency resolution itself is not malicious -- the danger is when it pulls from the wrong source. EDR telemetry to watch:

- `pip install` processes with `--extra-index-url` in the command line arguments
- `pip.conf` or `pip.ini` files containing `extra-index-url` (file modification events)
- Network connections from `pip` or `python` to `pypi.org` when corporate policy requires private-only

### Example Detection Queries

**Splunk -- pip reaching public PyPI from build infrastructure:**

```spl
index=proxy sourcetype=squid
  dest_host IN ("pypi.org", "files.pythonhosted.org")
  src_ip IN ("10.100.0.0/24", "10.100.1.0/24")
| stats count by src_ip, dest_host, uri_path
| where count > 0
| rename uri_path AS package_path
| table src_ip, dest_host, package_path, count
```

**Splunk -- detect extra-index-url in pip config files:**

```spl
index=osquery sourcetype=file_events
  target_path IN ("/etc/pip.conf", "*/pip.conf", "*/pip.ini")
| eval config_content=file_content
| where match(config_content, "extra-index-url")
| table _time, host, target_path
```

**KQL (Microsoft Sentinel) -- DNS queries to public PyPI from build agents:**

```kql
DnsEvents
| where Name in ("pypi.org", "files.pythonhosted.org")
| where ClientIP has_any ("10.100.0.", "10.100.1.")
| summarize QueryCount=count() by ClientIP, Name, bin(TimeGenerated, 5m)
| where QueryCount > 0
```

**KQL -- pip process with extra-index-url argument:**

```kql
DeviceProcessEvents
| where FileName == "pip" or FileName == "pip3"
| where ProcessCommandLine has "extra-index-url"
| project Timestamp, DeviceName, ProcessCommandLine, AccountName
```

---

!!! tip "SOC Relevance"

    **Alert you will see:** "Build server making outbound connections to pypi.org"

    In most corporate environments, CI/CD runners should **never** contact public PyPI directly. If your proxy logs show `pypi.org` requests from build infrastructure, it means either:

    1. **Misconfiguration** -- someone added `--extra-index-url` to pip config (common, dangerous)
    2. **Active attack** -- an attacker modified pip config to add a public fallback

    Either way, this alert is high-signal. The remediation is always the same: switch to `--index-url` (single registry) and verify lockfiles exist. This is a quick win for any SOC -- low false-positive rate and directly maps to preventing dependency confusion (Lab 1.2).

    **Triage steps:**

    1. Check which host made the request (CI runner? Developer workstation?)
    2. Pull the pip.conf from that host -- does it have `extra-index-url`?
    3. Check what package was requested -- is it an internal package name on public PyPI?
    4. If yes: treat as potential dependency confusion and escalate immediately

!!! example "CI Integration"

    Add this to your GitHub Actions pipeline to catch `extra-index-url` misconfigurations and missing lockfiles before they reach production.

    **`.github/workflows/pip-config-check.yml`:**

    ```yaml
    name: Dependency Resolution Safety Check

    on:
      pull_request:
        paths:
          - "requirements*.txt"
          - "setup.py"
          - "setup.cfg"
          - "pyproject.toml"
          - "pip.conf"
          - ".pip/**"

    jobs:
      check-pip-config:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Reject extra-index-url in pip config
            run: |
              echo "--- Scanning for extra-index-url usage ---"
              FOUND=0
              for f in pip.conf .pip/pip.conf setup.cfg pyproject.toml; do
                if [ -f "$f" ]; then
                  if grep -qi "extra-index-url" "$f"; then
                    echo "::error file=$f::BLOCKED: $f contains extra-index-url. Use --index-url instead."
                    FOUND=1
                  fi
                fi
              done
              # Also check requirements files for --extra-index-url flags
              for f in requirements*.txt; do
                if [ -f "$f" ]; then
                  if grep -qi "\-\-extra-index-url" "$f"; then
                    echo "::error file=$f::BLOCKED: $f contains --extra-index-url inline flag."
                    FOUND=1
                  fi
                fi
              done
              if [ "$FOUND" -eq 1 ]; then
                echo ""
                echo "extra-index-url allows pip to fetch from multiple registries and pick"
                echo "the highest version. This enables dependency confusion attacks."
                echo "Use --index-url to point to a single trusted registry."
                exit 1
              fi
              echo "PASS: No extra-index-url found."

          - name: Verify lockfile exists
            run: |
              echo "--- Checking for dependency lockfile ---"
              if [ -f "requirements.lock" ] || [ -f "requirements-lock.txt" ] || \
                 [ -f "poetry.lock" ] || [ -f "Pipfile.lock" ] || [ -f "pdm.lock" ]; then
                echo "PASS: Lockfile found."
              else
                echo "::error::No lockfile found. Run 'pip freeze > requirements.lock' and commit it."
                echo "Without a lockfile, builds are not reproducible and are vulnerable"
                echo "to version substitution attacks."
                exit 1
              fi

          - name: Check version pins in requirements.txt
            run: |
              echo "--- Checking version pins ---"
              UNPINNED=0
              for f in requirements*.txt; do
                if [ -f "$f" ]; then
                  while IFS= read -r line; do
                    # Skip comments, blank lines, and flags
                    [[ "$line" =~ ^[[:space:]]*# ]] && continue
                    [[ "$line" =~ ^[[:space:]]*$ ]] && continue
                    [[ "$line" =~ ^- ]] && continue
                    # Check if line has == pin
                    if ! echo "$line" | grep -q "=="; then
                      echo "::warning file=$f::Unpinned dependency: $line (use == for exact version)"
                      UNPINNED=1
                    fi
                  done < "$f"
                fi
              done
              if [ "$UNPINNED" -eq 1 ]; then
                echo "WARNING: Some dependencies are not pinned to exact versions."
                echo "Use == to prevent unexpected version changes."
              else
                echo "PASS: All dependencies are pinned."
              fi
    ```

## What You Learned

1. **Dependency resolution is a tree** -- one package can pull in many others, each with their own version constraints.
2. **`--extra-index-url` is dangerous** -- it tells pip to merge results from multiple sources and pick the highest version, regardless of which source it comes from.
3. **`--index-url` is safer** -- it tells pip to only use one source.
4. **Lockfiles freeze versions** -- `pip freeze` captures exact versions so builds are reproducible and resistant to new malicious versions appearing.
5. **Version pins matter** -- `==1.0.0` is strict; `>=1.0.0` opens the door to unexpected upgrades.
