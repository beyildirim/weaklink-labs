When you run `docker pull`, you are not downloading a single binary. You are downloading a **stack of compressed tarballs** (layers), a **manifest** that lists them in order, and a **config blob** with metadata. Understanding this structure is essential because attackers hide malicious content in layers that appear "deleted" in the final filesystem but are still present in the image.

This lab gives you a container image with hidden content buried in its layer history. You will learn to inspect images at every level -- from `docker inspect` down to individual layer tarballs.

!!! info "Phase 1: UNDERSTAND -- How Container Images Are Built"

    **Goal:** Explore the internal structure of a container image -- layers, manifests, configs, and digests.

### Step 1: Pull and inspect an image

```bash
docker pull registry:5000/webapp:latest
docker inspect registry:5000/webapp:latest | jq '.[0].RootFS'
```

The `RootFS` section shows the layer digests that make up this image. Each layer is a tarball that adds, modifies, or deletes files on top of the previous layer.

### Step 2: View the build history

```bash
docker history registry:5000/webapp:latest
```

This shows each instruction from the Dockerfile that created a layer. Notice that some layers have a `SIZE` of 0B -- these are metadata-only layers (like `CMD` or `ENV`).

### Step 3: Inspect the manifest with crane

```bash
crane manifest registry:5000/webapp:latest | jq .
```

The manifest is the authoritative list of layers. It includes:

- `mediaType` -- the format (OCI or Docker v2)
- `config` -- pointer to the config blob (contains env vars, entrypoint, etc.)
- `layers` -- ordered list of layer digests and sizes

### Step 4: Inspect the config blob

```bash
crane config registry:5000/webapp:latest | jq .
```

The config contains runtime metadata: environment variables, entrypoint, working directory, and the full `history` array. Compare the `history` entries to what `docker history` showed.

### Step 5: Understand tags vs digests

```bash
# A tag is a mutable pointer
crane digest registry:5000/webapp:latest

# A digest is an immutable content hash
crane manifest registry:5000/webapp@sha256:$(crane digest registry:5000/webapp:latest) | jq .
```

Tags can be overwritten at any time. Digests cannot. This distinction is critical for security.

### Step 6: Count the layers

```bash
crane manifest registry:5000/webapp:latest | jq '.layers | length'
```

Remember this number. You will need it later to detect when extra layers have been added.

!!! warning "Phase 2: BREAK -- Hidden Content in Image Layers"

    **Goal:** Discover that a file "deleted" in a later layer still exists in the image history and can be extracted.

### Step 1: Compare two images

Two images are available: `webapp:latest` and `webapp:clean`. They produce identical output when you run them:

```bash
docker run --rm registry:5000/webapp:latest
docker run --rm registry:5000/webapp:clean
```

And `docker inspect` shows the same entrypoint, env, and working directory.

### Step 2: Check the history

```bash
docker history --no-trunc registry:5000/webapp:latest
docker history --no-trunc registry:5000/webapp:clean
```

The `latest` image has extra history entries. One layer copies a file, and a subsequent layer deletes it. The file does not appear in the running container's filesystem -- but it is still in the image.

### Step 3: Extract all layers

```bash
docker save registry:5000/webapp:latest -o /tmp/webapp.tar
mkdir -p /app/extracted-layers
tar xf /tmp/webapp.tar -C /app/extracted-layers
```

### Step 4: Find the hidden content

```bash
cd /app/extracted-layers
for layer in */layer.tar; do
    echo "=== $layer ==="
    tar tf "$layer" 2>/dev/null | head -20
done
```

One of the layers contains a file that does not appear in the final filesystem. This is the hidden backdoor -- it was "deleted" by a whiteout file in a later layer, but the data is still in the image.

### Step 5: Extract the hidden file

```bash
# Find the layer with the suspicious file and extract it
tar xf <layer-dir>/layer.tar -C /tmp/hidden/
cat /tmp/hidden/path/to/backdoor
```

### Step 6: Document your findings

Record what you found:

```bash
cat > /app/findings.txt << 'EOF'
Hidden backdoor found in layer sha256:<layer-hash>
Filename: <path-to-backdoor>
The file was added in layer N and "deleted" in layer N+1 via a whiteout.
The data remains accessible by extracting individual layers.
EOF
```

Save the full history output:

```bash
docker history --no-trunc registry:5000/webapp:latest > /app/history-output.txt
```

!!! success "Phase 3: DEFEND -- Inspecting All Layers and Using Minimal Images"

    **Goal:** Adopt practices that prevent hidden content from surviving in production images.

### Defense 1: Always inspect full history

Never trust `docker inspect` alone. Always run:

```bash
docker history --no-trunc <image>
```

This shows every layer's command, including ones that add and then "remove" files. Any image with `RUN rm` after a `COPY` or `ADD` is suspicious.

### Defense 2: Scan all layers, not just the final filesystem

Container scanners like Trivy can scan all layers:

```bash
trivy image --scanners vuln registry:5000/webapp:latest
```

Configure your scanner to inspect every layer, not just the final merged filesystem.

### Defense 3: Use minimal base images

Distroless and Chainguard images have no shell, no package manager, and minimal layers:

```bash
# Compare layer counts
crane manifest registry:5000/webapp:latest | jq '.layers | length'
crane manifest cgr.dev/chainguard/static:latest | jq '.layers | length'
```

Fewer layers means less surface area for hidden content and fewer components to scan.

### Defense 4: Build from scratch when possible

For compiled languages (Go, Rust), build `FROM scratch`:

```dockerfile
FROM scratch
COPY --from=builder /app/binary /app/binary
ENTRYPOINT ["/app/binary"]
```

One layer. Nothing to hide in.

!!! danger "Phase 4: DETECT -- Finding Hidden Content in Production Images"

    **Goal:** Detect when images in your registry contain suspicious hidden layers.

### SIEM / Log Indicators

The core signal is **images with more layers than expected** or **images where layer history shows file additions followed by deletions**. This pattern is abnormal in properly built images.

**What to look for:**

- Images where `docker history` shows `RUN rm` or whiteout entries after `COPY`/`ADD` of executables
- Layer count mismatches between the Dockerfile instruction count and the actual manifest
- Images that are significantly larger than expected for their base image
- Registry webhook events showing image pushes with unusual layer counts

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| Image push to registry with layer count > threshold | Potential layer injection or hidden content |
| `docker pull` of image with digest mismatch from expected | Tag was overwritten or image was tampered with |
| Registry API calls to `/v2/<repo>/blobs/<digest>` for unexpected digests | Someone is pulling individual layer blobs (possible forensic analysis or exfiltration) |

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Implant Internal Image** | [T1525](https://attack.mitre.org/techniques/T1525/) | Attacker hides malicious content in a container image layer that survives casual inspection |
| **Masquerading** | [T1036](https://attack.mitre.org/techniques/T1036/) | The image appears identical to the clean version at the filesystem level, masking the hidden layer |

---

!!! tip "SOC Relevance"

    **Alert you will see:** "Container image layer count exceeds baseline" or "Image history contains add-then-delete pattern"

    Most legitimate container images do not add files and then immediately delete them. When your registry scanner or CI pipeline detects this pattern, it is either:

    1. **Poor build hygiene** -- a developer copied a secret, then tried to `rm` it (the secret is still in the layer)
    2. **Intentional concealment** -- an attacker hid a backdoor in a layer that appears "empty" to filesystem-level inspection

    **Triage steps:**

    1. Extract all layers with `docker save` and inspect each one individually
    2. Compare the layer count and digests against the expected output from the Dockerfile
    3. Look for executables, scripts, or configuration files that appear only in intermediate layers
    4. If unexpected content is found, treat as a compromised image -- quarantine and rebuild from source

!!! example "CI Integration"

    Add this to your CI pipeline to detect hidden content in container images before they reach production.

    **`.github/workflows/image-layer-audit.yml`:**

    ```yaml
    name: Container Image Layer Audit

    on:
      push:
        paths:
          - "Dockerfile*"
          - "docker-compose*.yml"

    jobs:
      audit-layers:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Install crane
            run: |
              curl -sL https://github.com/google/go-containerregistry/releases/latest/download/go-containerregistry_Linux_x86_64.tar.gz \
                | tar xz crane
              sudo mv crane /usr/local/bin/

          - name: Build image
            run: docker build -t audit-target:latest .

          - name: Check for add-then-delete pattern
            run: |
              echo "--- Checking for hidden content in image layers ---"
              HISTORY=$(docker history --no-trunc --format '{{.CreatedBy}}' audit-target:latest)
              if echo "$HISTORY" | grep -qE '(rm -rf|rm -f|rm /)' ; then
                echo "::warning::Image history contains file deletion after addition."
                echo "Files 'deleted' in later layers still exist in the image."
                echo "Use multi-stage builds to avoid this pattern."
              fi

          - name: Verify layer count
            run: |
              EXPECTED_LAYERS=$(grep -cE '^(RUN|COPY|ADD) ' Dockerfile || echo 0)
              # Account for base image layers (roughly)
              ACTUAL_LAYERS=$(docker inspect --format '{{len .RootFS.Layers}}' audit-target:latest)
              echo "Dockerfile instructions: $EXPECTED_LAYERS"
              echo "Actual image layers: $ACTUAL_LAYERS"
              if [ "$ACTUAL_LAYERS" -gt $((EXPECTED_LAYERS + 15)) ]; then
                echo "::error::Image has significantly more layers than expected from the Dockerfile."
                exit 1
              fi

          - name: Scan all layers with Trivy
            uses: aquasecurity/trivy-action@master
            with:
              image-ref: audit-target:latest
              severity: CRITICAL,HIGH
              exit-code: 1
    ```

## What You Learned

1. **Container images are layer stacks** -- each `RUN`, `COPY`, or `ADD` creates a new layer tarball. The manifest lists them in order.
2. **Deleted files are not truly deleted** -- a whiteout in a later layer hides the file from the runtime filesystem, but the data remains in the image.
3. **`docker inspect` is not enough** -- it shows the final merged view. You need `docker history --no-trunc` and layer extraction to see everything.
4. **Digests are immutable, tags are not** -- always reference images by digest for security-critical operations.
5. **Minimal base images reduce risk** -- fewer layers mean less surface area for hidden content.
