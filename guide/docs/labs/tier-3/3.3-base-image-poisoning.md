# Lab 3.3: Base Image Poisoning

<div class="lab-meta">
  <span>~35 minutes</span>
  <span>Intermediate</span>
  <span>Prerequisites: <a href="3.1-image-internals.md">Lab 3.1</a></span>
</div>

Every Dockerfile starts with `FROM`. That single line imports an entire operating system, runtime, and all its dependencies into your image. You are trusting the base image maintainer with your entire application's security. If that base image is compromised. whether through a supply chain attack on the maintainer, a registry compromise, or a malicious insider. every image built on top of it inherits the backdoor.

In 2024, the XZ Utils backdoor (CVE-2024-3094) demonstrated how deeply a compromised upstream dependency can embed itself. The same principle applies to container base images: a poisoned `python:3.12` or `node:20` image would affect every application built `FROM` it.

In this lab, you are a developer who builds an app using a base image from your organization's private registry. The base image has been poisoned. You will discover the backdoor and learn to defend against base image compromise.

???+ info "Phase 1: UNDERSTAND. How Base Images Work"

    **Goal:** See how `FROM` in a Dockerfile inherits everything from the base image.

### Step 1: Explore the application Dockerfile

```bash
cat /app/Dockerfile
```

It starts with `FROM registry:5000/python-base:3.12`. This means the app inherits:

- The OS (debian, alpine, etc.)
- Python runtime and standard library
- Any additional packages installed in the base
- Any files, scripts, or configuration added to the base
- Environment variables and entrypoint from the base

### Step 2: Inspect the base image

```bash
docker pull registry:5000/python-base:3.12
docker inspect registry:5000/python-base:3.12 | jq '.[0].Config.Env'
docker history registry:5000/python-base:3.12
```

At first glance, it looks like a standard Python base image.

### Step 3: Build and run the app

```bash
docker build -t registry:5000/myapp:latest /app/
docker run --rm registry:5000/myapp:latest
```

The app works. It imports modules, processes data, returns results. Nothing obviously wrong.

### Step 4: Understand the trust chain

Your Dockerfile only contains your application code. But your image contains everything the base image has plus your code. You are implicitly trusting:

- The base image maintainer (e.g., Docker Official Images team)
- The registry hosting the base image
- The CI pipeline that built the base image
- Every upstream dependency in the base image

If any link in this chain is compromised, your app is compromised.

---

???+ warning "Phase 2: BREAK. Discovering the Poisoned Base Image"

    **Goal:** Find the backdoor that was injected into the base image. Your app inherits it without any change to your Dockerfile.

### Step 1: Deep inspect the base image

```bash
docker history --no-trunc registry:5000/python-base:3.12
```

Look carefully at every layer. Is there a `RUN` or `COPY` command that does not belong in a standard Python base image?

### Step 2: Look for suspicious files

```bash
# Check for unexpected executables
docker run --rm registry:5000/python-base:3.12 find /usr/local/bin -type f -newer /usr/local/bin/python3

# Check for unexpected startup scripts
docker run --rm registry:5000/python-base:3.12 ls -la /docker-entrypoint.d/ 2>/dev/null
docker run --rm registry:5000/python-base:3.12 cat /usr/local/bin/backdoor 2>/dev/null
```

### Step 3: Check if the backdoor is in your app image

```bash
docker run --rm registry:5000/myapp:latest which backdoor 2>/dev/null
docker run --rm registry:5000/myapp:latest cat /usr/local/bin/backdoor 2>/dev/null
```

The backdoor exists in your app image even though your Dockerfile never added it. It was inherited from the poisoned base.

### Step 4: Compare with the clean base

A known-clean digest is stored at `/app/clean-base-digest.txt`:

```bash
cat /app/clean-base-digest.txt

# Compare layer counts
crane manifest registry:5000/python-base:3.12 | jq '.layers | length'
crane manifest registry:5000/python-base@$(cat /app/clean-base-digest.txt) | jq '.layers | length'
```

The poisoned base has more layers than the clean one.

### Step 5: Scan the base image

```bash
trivy image registry:5000/python-base:3.12
```

The scanner may flag the backdoor binary or the suspicious layer.

### Step 6: Document the finding

```bash
cat > /app/findings.txt << 'EOF'
FINDING: Base image registry:5000/python-base:3.12 is poisoned.
The backdoor binary at /usr/local/bin/backdoor was added in an extra layer.
Every image built FROM this base inherits the backdoor.
Clean base digest: <paste from clean-base-digest.txt>
Poisoned base digest: <paste current digest>
The app Dockerfile was NOT modified -- the compromise is entirely in the base.
EOF
```

???+ success "Phase 3: DEFEND. Pinning Base Images and Verifying Integrity"

    **Goal:** Rebuild the app with the clean base image, pin by digest, and establish ongoing verification.

### Defense 1: Pin the base image by digest

```bash
CLEAN_DIGEST=$(cat /app/clean-base-digest.txt)

# Update Dockerfile to pin by digest
sed -i "s|FROM registry:5000/python-base:3.12|FROM registry:5000/python-base@${CLEAN_DIGEST}|" /app/Dockerfile
cat /app/Dockerfile
```

Now even if someone poisons `python-base:3.12` again, your build will use the exact image at this digest.

### Defense 2: Rebuild with the clean base

```bash
docker build -t registry:5000/myapp:secure /app/
docker push registry:5000/myapp:secure
```

### Defense 3: Verify no backdoor

```bash
docker run --rm registry:5000/myapp:secure cat /usr/local/bin/backdoor 2>/dev/null
echo "Exit code: $?"
# Should fail (exit code 1) -- backdoor does not exist
```

### Defense 4: Scan the clean image

```bash
trivy image registry:5000/myapp:secure > /app/scan-results.txt
cat /app/scan-results.txt
```

### Defense 5: Establish base image verification process

In production, you should:

1. **Maintain an internal base image registry** with approved, scanned base images
2. **Pin all base images by digest** in every Dockerfile
3. **Automate base image scanning**. scan base images independently on a schedule, not just when apps are built
4. **Use signed base images**. verify signatures with cosign before building on top of them
5. **Monitor for base image updates**. when the upstream publishes a new version, scan it before updating your digest pin

### Step 6: Verify the lab

```bash
weaklink verify 3.3
```

??? danger "Phase 4: DETECT. Catching Base Image Compromise"

    **Goal:** Detect when a base image in your registry has been tampered with.

### SIEM / Log Indicators

The core signal is **a base image digest change without a corresponding approved update**. Base images should only change through your controlled update process, not through ad-hoc pushes.

**What to look for:**

- Registry push events for base image tags (e.g., `python-base:3.12`) from unexpected users or CI pipelines
- Base image digest changes that do not correlate with a planned update ticket
- Build logs showing a different base image digest than the one pinned in the Dockerfile
- New layers appearing in a base image that do not match the upstream Dockerfile

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| Push to `<registry>/python-base:3.12` from non-CI IP | Someone is modifying the base image outside the approved pipeline |
| Build system pulling a base image digest that differs from the pinned value | Base image tag was overwritten or pin is outdated |
| Registry API `DELETE` followed by `PUT` for same manifest tag | Tag was replaced (possible poisoning) |

---

??? tip "SOC Relevance"

    **Alerts you will see:**

    - "Base image pushed from unauthorized source"
    - "Base image digest changed without change request"

    **Why this matters:** A poisoned base image is a force multiplier. If your organization uses `python-base:3.12` for 50 microservices, poisoning that single image compromises all 50. The attacker does not need to touch any application code or Dockerfile. the backdoor propagates automatically through the build process.

    **Triage steps:**

    1. **Identify the blast radius**. which Dockerfiles use `FROM` with the affected base image?
    2. **Compare digests**. get the current base image digest and compare it against your approved digest list
    3. **Inspect the base image layers**. look for unexpected files, scripts, or binaries added in extra layers
    4. **Check the push history**. who pushed the new base image? From which IP? Through which pipeline?
    5. **Rebuild all affected images**. pin the known-good base image digest and rebuild every downstream image

    **Prevention:** Lock down write access to base image repositories. Only your base image CI pipeline should be able to push. Use registry webhooks to alert on any push to base image repositories.

??? example "CI Integration"

    Add this to your CI pipeline to verify base image integrity before building.

    **`.github/workflows/base-image-verify.yml`:**

    ```yaml
    name: Base Image Integrity Check

    on:
      push:
        paths:
          - "Dockerfile*"
      schedule:
        - cron: "0 6 * * *"

    jobs:
      verify-base:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Install crane
            run: |
              curl -sL https://github.com/google/go-containerregistry/releases/latest/download/go-containerregistry_Linux_x86_64.tar.gz \
                | tar xz crane
              sudo mv crane /usr/local/bin/

          - name: Verify base image digests
            run: |
              echo "--- Checking base image digest pins ---"
              UNPINNED=0
              while IFS= read -r line; do
                if echo "$line" | grep -qE '^FROM ' && ! echo "$line" | grep -q '@sha256:'; then
                  IMAGE=$(echo "$line" | awk '{print $2}')
                  echo "::error::Base image not pinned by digest: $IMAGE"
                  UNPINNED=1
                fi
              done < <(grep -h '^FROM ' Dockerfile*)
              if [ "$UNPINNED" -eq 1 ]; then
                echo ""
                echo "All base images must be pinned by digest (@sha256:)."
                echo "Use 'crane digest <image>:<tag>' to get the digest."
                exit 1
              fi
              echo "PASS: All base images are pinned by digest."

          - name: Scan base images
            run: |
              echo "--- Scanning base images for vulnerabilities ---"
              for image in $(grep -ohE '@sha256:[a-f0-9]+' Dockerfile* | sort -u); do
                BASE=$(grep -B1 "$image" Dockerfile* | grep 'FROM' | awk '{print $2}')
                echo "Scanning: $BASE"
                docker pull "$BASE" 2>/dev/null
                trivy image --severity CRITICAL "$BASE"
              done

          - name: Check base image freshness
            run: |
              echo "--- Checking base image age ---"
              for dockerfile in Dockerfile*; do
                FROM_LINE=$(grep '^FROM' "$dockerfile" | head -1)
                IMAGE=$(echo "$FROM_LINE" | awk '{print $2}')
                CREATED=$(crane config "$IMAGE" 2>/dev/null | jq -r '.created // "unknown"')
                echo "$dockerfile: $IMAGE (created: $CREATED)"
                # Warn if older than 90 days
              done
    ```

## What You Learned

1. **`FROM` is a trust statement**. your Dockerfile inherits everything from the base image, including any backdoors.
2. **Base image poisoning is a force multiplier**. one poisoned base image compromises every application built on top of it.
3. **Tags are not enough**. pinning `python:3.12` does not prevent the tag from being overwritten with a poisoned image.
4. **Digest pinning is essential**. `FROM python@sha256:abc123...` ensures you always build on the exact base image you verified.
5. **Base images need independent scanning**. scan base images on their own schedule, not just as part of application builds.
6. **The XZ Utils backdoor was a base image attack**. a compromised upstream dependency embedded in the base affects every downstream consumer.
