# Lab 3.6: Multi-Stage Build Leaks

<div class="lab-meta">

Multi-stage builds are the standard defense against shipping build tools and source code in production images. The idea is simple: compile in one stage, copy only the binary to the final stage. But developers routinely leak secrets through this boundary. via `ENV` and `ARG` instructions that persist in layer history, files copied alongside the artifact, and build context that should have been excluded.

This lab gives you a multi-stage Dockerfile that looks clean at first glance. You will extract an API key from the final image in three different ways, then fix each leak.

???+ info "Phase 1: UNDERSTAND. How Multi-Stage Builds Work"

    **Goal:** See how multi-stage builds separate the build environment from the runtime environment.

### Step 1: Examine the Dockerfile

```bash
cat /app/Dockerfile
```

This is a two-stage build:

- **Stage 1 (`builder`):** Installs build tools, downloads dependencies, compiles the application. This stage has access to secrets (API keys, private registry tokens) needed during build.
- **Stage 2 (`runtime`):** Starts from a minimal base image, copies only the compiled artifact from `builder`, and sets the entrypoint.

The assumption is that nothing from the builder stage leaks into the final image.

### Step 2: Build the image

```bash
docker build -t registry:5000/myapp:latest /app/
```

### Step 3: Verify the final image looks clean

```bash
docker run --rm registry:5000/myapp:latest
```

The application runs. Let's check what's in the image:

```bash
docker run --rm --entrypoint sh registry:5000/myapp:latest -c "ls -la /app/"
```

Only the compiled binary. No source code, no `.env` file, no build tools. Looks safe.

### Step 4: Understand what SHOULD be isolated

In a properly built multi-stage image, the final stage should contain:

- The compiled application binary
- Runtime dependencies only (shared libraries, CA certificates)
- No source code, no build tools, no secrets

Everything else should be confined to the builder stage, which is discarded after the `COPY --from=builder` instruction.

???+ warning "Phase 2: BREAK. Extract Secrets From the Final Image"

    **Goal:** Find an API key that was supposed to stay in the build stage, using three different extraction techniques.

### Leak 1: Secrets baked into ENV/ARG

```bash
docker history --no-trunc registry:5000/myapp:latest
```

Look for `ENV` or `ARG` instructions. Even if the variable was only intended for the build stage, Docker records `ARG` values in the layer history of the stage where they are used. If the `ARG` is re-declared in the final stage, or if the Dockerfile uses `ENV` instead of `ARG`, the value persists.

```bash
# Extract environment variables from the config
crane config registry:5000/myapp:latest | jq '.config.Env'
```

Any `ENV` instruction in the final stage. even if it was meant to pass a build-time value. is embedded in the image config as cleartext.

### Leak 2: Secrets in copied file metadata

```bash
docker save registry:5000/myapp:latest -o /tmp/myapp.tar
mkdir -p /tmp/myapp-layers
tar xf /tmp/myapp.tar -C /tmp/myapp-layers

# Inspect each layer for unexpected files
for layer in /tmp/myapp-layers/*/layer.tar; do
    echo "=== $layer ==="
    tar tf "$layer" 2>/dev/null
done
```

Look for configuration files, `.env` files, or directories that were accidentally included in the `COPY` instruction. A common mistake:

```dockerfile
# Intended: copy only the binary
COPY --from=builder /app/build/myapp /app/myapp

# Actual: developer wrote this instead, copying everything
COPY --from=builder /app/ /app/
```

The glob or directory copy brings along `.env`, `config.json`, and any other files that happened to be in `/app/` during the build stage.

### Leak 3: Secrets in build context

```bash
# Check if .dockerignore exists and what it excludes
cat /app/.dockerignore 2>/dev/null || echo "No .dockerignore found"
```

Without a `.dockerignore`, the entire build context is sent to the Docker daemon. If the directory contains `.env`, `.git/`, `credentials.json`, or private keys, they are available to the build and may end up in a layer even if the Dockerfile does not explicitly `COPY` them.

```bash
# Check what's in the build context
ls -la /app/
cat /app/.env 2>/dev/null
cat /app/config/credentials.json 2>/dev/null
```

### Step 4: Collect all leaked secrets

```bash
cat > /app/findings.txt << 'EOF'
Multi-Stage Build Leak Analysis
================================

Leak 1 - ENV/ARG in layer history:
  Location: Image config -> .config.Env
  Secret: API_KEY=<extracted-value>
  Cause: ARG/ENV used for build-time secret persists in final image metadata

Leak 2 - Overbroad COPY:
  Location: Layer tarball -> /app/.env or /app/config/
  Secret: <file contents>
  Cause: COPY --from=builder /app/ copied more than just the binary

Leak 3 - Build context:
  Location: Build context sent to daemon
  Secret: .env file or credentials.json
  Cause: No .dockerignore to exclude sensitive files
EOF
```

???+ success "Phase 3: DEFEND. BuildKit Secrets, Precise COPY, and .dockerignore"

    **Goal:** Fix all three leak vectors so no secrets survive into the final image.

### Defense 1: Use BuildKit secret mounts instead of ENV/ARG

Never pass secrets via `ARG` or `ENV`. Use BuildKit's `--mount=type=secret`:

```dockerfile
# syntax=docker/dockerfile:1
FROM golang:1.22 AS builder

# Secret is mounted at build time but never written to a layer
RUN --mount=type=secret,id=api_key \
    API_KEY=$(cat /run/secrets/api_key) && \
    go build -ldflags "-X main.apiKey=$API_KEY" -o /app/myapp .

FROM gcr.io/distroless/static:nonroot
COPY --from=builder /app/myapp /app/myapp
ENTRYPOINT ["/app/myapp"]
```

Build with:

```bash
echo "sk-real-secret-key" > /tmp/api_key.txt
DOCKER_BUILDKIT=1 docker build --secret id=api_key,src=/tmp/api_key.txt -t registry:5000/myapp:secure .
```

Verify the secret is not in the history:

```bash
docker history --no-trunc registry:5000/myapp:secure
crane config registry:5000/myapp:secure | jq '.config.Env'
```

No trace of the API key in layer history or image config.

### Defense 2: Use precise COPY instructions

Never copy directories when you only need a single file:

```dockerfile
# Bad: copies everything in /app/ including .env, config/, etc.
COPY --from=builder /app/ /app/

# Good: copies exactly one file
COPY --from=builder /app/build/myapp /app/myapp
```

### Defense 3: Write a comprehensive .dockerignore

```bash
cat > /app/.dockerignore << 'EOF'
.env
.env.*
*.key
*.pem
credentials.json
config/secrets/
.git/
.github/
node_modules/
__pycache__/
*.pyc
EOF
```

This prevents sensitive files from being included in the build context, even if a `COPY . .` instruction exists.

### Defense 4: Inspect images with dive

```bash
dive registry:5000/myapp:secure
```

Dive provides an interactive view of every layer's contents. Use it to verify that no unexpected files are present in the final image.

### Defense 5: Scan for secrets in images

```bash
# Use trivy to scan for secrets in the image
trivy image --scanners secret registry:5000/myapp:latest
```

This catches hardcoded passwords, API keys, and private keys that made it into any layer.

### Step 6: Verify the lab

Run the verification from your host terminal:

```bash
weaklink verify 3.6
```

??? danger "Phase 4: DETECT. Finding Leaked Secrets in Production Images"

    **Goal:** Detect when secrets have been accidentally baked into container images.

### SIEM / Log Indicators

The key signal is **secrets found in container image layers or config metadata**. This shows up through image scanning tools or when an attacker uses a leaked credential.

**What to look for:**

- Image scanning results showing secrets (API keys, passwords, private keys) in any layer
- `ENV` instructions in image history containing key/token/password/secret patterns
- Images built without BuildKit (no `--mount=type=secret` support)
- Build logs showing `ARG` with secret values in plaintext
- Authentication logs showing valid credentials used from unexpected sources (leaked key in use)

### Network Indicators

| Indicator | What It Means |
|-----------|---------------|
| API call using a credential that matches one found in an image layer | Leaked secret is being exploited |
| `docker save` or `crane config` commands on production images | Someone is extracting image contents (forensics or exfiltration) |
| Image push without `.dockerignore` (large build context) | Entire project directory may be in the image |
| Build logs contain `ARG API_KEY=` with actual value | Secret exposed in build logs |

??? tip "SOC Relevance"

    **Alert you will see:** "Secret detected in container image layer" or "API key found in image environment variables"

    This is one of the most common supply chain security findings. Developers routinely pass secrets via `ARG` or `ENV` because it's convenient, not realizing these values persist in the image. The risk escalates when images are pushed to shared registries. everyone with pull access can extract the secrets.

    **Triage steps:**

    1. Identify the secret type (API key, database password, private key, etc.)
    2. Check if the secret is still valid. call the associated API or service to test
    3. If valid, **rotate the secret immediately** before further investigation
    4. Determine the blast radius: which registries have this image, who has pull access
    5. Rebuild the image using BuildKit secrets and push a clean replacement
    6. Audit build pipelines for other images using the same pattern

??? example "CI Integration"

    Add this to your pipeline to catch secrets in images before they reach a shared registry.

    **`.github/workflows/image-secret-scan.yml`:**

    ```yaml
    name: Image Secret Scanner

    on:
      push:
        paths:
          - "Dockerfile*"
          - "docker-compose*.yml"
          - ".dockerignore"

    jobs:
      scan-secrets:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Verify .dockerignore exists
            run: |
              if [ ! -f .dockerignore ]; then
                echo "::error::.dockerignore is missing -- secrets may leak into the build context"
                exit 1
              fi
              # Check for common sensitive patterns
              for pattern in ".env" "*.key" "*.pem" "credentials"; do
                if ! grep -q "$pattern" .dockerignore; then
                  echo "::warning::.dockerignore does not exclude '$pattern'"
                fi
              done

          - name: Check for ENV/ARG secret patterns
            run: |
              if grep -iE '^(ENV|ARG)\s+.*(KEY|TOKEN|PASSWORD|SECRET|CREDENTIAL)' Dockerfile; then
                echo "::error::Dockerfile uses ENV/ARG for secret values. Use --mount=type=secret instead."
                exit 1
              fi

          - name: Build image
            run: DOCKER_BUILDKIT=1 docker build -t scan-target:latest .

          - name: Scan for secrets with Trivy
            uses: aquasecurity/trivy-action@master
            with:
              image-ref: scan-target:latest
              scanners: secret
              severity: CRITICAL,HIGH
              exit-code: 1

          - name: Check image config for leaked ENV values
            run: |
              docker inspect scan-target:latest | jq -r '.[0].Config.Env[]' | while read -r env; do
                if echo "$env" | grep -iqE '(key|token|password|secret|credential)=.+'; then
                  echo "::error::Leaked secret in image ENV: $(echo "$env" | cut -d= -f1)"
                  exit 1
                fi
              done
    ```

## What You Learned

1. **Multi-stage builds do not automatically protect secrets**. they only separate build-time files from runtime files. Secrets can still leak through ENV, ARG, overbroad COPY, and build context.
2. **`ARG` and `ENV` values persist in layer history**. even in the final stage, `docker history --no-trunc` and `crane config` expose them as cleartext.
3. **`COPY --from=builder /dir/` copies everything**. always specify the exact file path, not a directory, to avoid pulling in `.env` or config files.
4. **`.dockerignore` is a required security control**. without it, the entire build context (including secrets, `.git/`, and credentials) is available to the build.
5. **BuildKit `--mount=type=secret` is the correct pattern**. it mounts a secret file at build time without writing it to any layer.
