## The Problem

Modern software is assembled, not written. The average application contains hundreds of
open-source components, each with their own dependencies, build processes, and maintainers.
A single compromised component can propagate to every system that depends on it.

Recent incidents (SolarWinds, Log4Shell, xz-utils, Codecov) demonstrate that supply chain
attacks bypass traditional application security controls entirely. They compromise the
tools and components developers trust implicitly.

---

## What We Are Doing About It

We are establishing a **Supply Chain Security Program** that covers six pillars:

| Pillar | What It Means | Status |
|--------|--------------|--------|
| **Governance** | Policies, standards, and ownership for supply chain risk | |
| **Tooling** | Automated scanning, SBOM generation, artifact signing | |
| **Training** | Developer awareness and secure development practices | |
| **Monitoring** | Continuous visibility into dependency risk and build integrity | |
| **Incident Response** | Playbooks for supply chain-specific incidents | |
| **Continuous Improvement** | Metrics, reviews, and program maturity tracking | |
